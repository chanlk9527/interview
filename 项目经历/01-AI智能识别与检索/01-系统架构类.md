# 一、系统架构类追问

---

## Q1: 请介绍一下这个自然语言搜图系统的整体架构？核心解决什么问题？

**回答思路：**

这个系统的核心目标是：让用户用自然语言（如"昨天下午穿黄色衣服骑车的人"）直接搜索监控画面，替代传统的拖拽时间轴回放方式。

**业务痛点：**
- 存量IPC摄像机没有AI芯片，无法在设备端做智能识别，所有智能化必须在云端完成
- 传统查录像方式极其低效——用户只能按时间轴拖拽或简单分类（机动车/非机动车/人）
- 面向全球用户，需要支持多语言输入（英语、西语、日语等）

**整体架构分为三层：**

| 层级 | 职责 | 核心组件 |
|---|---|---|
| 接入层 | 设备告警接入 + 用户查询接入 | API Gateway、鉴权、限流 |
| 核心处理层 | 图像打标（VLM）+ 查询理解（LLM） | GPU Worker、QU引擎 |
| 数据存储层 | 结构化存储 + 向量检索 | MySQL、Elasticsearch 8.x |

**最关键的架构决策是"写入流与读取流彻底解耦"：**
- **写入流（异步）**：设备告警图片上传 → MQ削峰 → VLM大模型提取画面特征 → 结构化数据+向量双写MySQL和ES
- **读取流（实时）**：用户自然语言输入 → LLM意图解析（时间/地点/实体提取+翻译） → ES混合检索 → Reranker重排 → 返回结果

写入允许几十秒到几分钟的异步延迟，读取要求1.5~3秒内返回结果。这种异步+实时的混合架构是整个系统的基石。

## Q2: 为什么采用"写入流与读取流解耦"的架构？不能实时处理吗？

**回答思路：**

不能实时处理，原因有三：

1. **VLM大模型推理耗时长**：调用第三方视觉大模型API解析一张图片，单次耗时可能在数百毫秒到数秒级别。如果用户搜索时才去实时分析图片，延迟完全不可接受。
2. **告警流量极度不稳定**：安防场景下，雷雨天、大风天可能导致整个城市的室外摄像头同时触发移动侦测告警，瞬时并发量可能是平时的几十倍。实时处理根本扛不住。
3. **成本考量**：VLM API按调用次数计费，如果每次用户搜索都实时调用，成本会爆炸。预先打标一次，后续无限次搜索零边际成本。

**解耦带来的好处：**
- 写入流通过MQ削峰填谷，GPU Worker按自身能力平滑消费，不会被突发流量打垮
- 读取流只需查ES索引，响应时间稳定在秒级以内
- 两条流可以独立扩缩容：告警高峰期加GPU Worker，搜索高峰期加ES节点

## Q3: 全球化多语言支持是怎么做的？为什么选择Pivot Language策略？

**回答思路：**

我们采用的是**"入口强制翻译，底层绝对纯英文"**的Pivot Language（枢轴语言）策略。

**具体做法：**
- **写入流**：VLM大模型的System Prompt强制要求输出纯英文JSON，无论图片来自哪个国家的摄像头，底层存储的特征描述永远是英文
- **读取流**：用户用任意母语输入搜索词，LLM在做意图解析时同步完成翻译，将搜索实体转为标准英文，并自动扩充同义词
- **底层引擎**：ES的分词器、同义词库、BM25计算全部基于英文，只维护一套

**为什么不维护多语言引擎？**

| 方案 | 维护成本 | 扩展性 |
|---|---|---|
| 每种语言一套分词器+同义词库 | 极高，N种语言×M个领域词汇 | 每新增一个市场都要重建 |
| Pivot Language（纯英文底层） | 只维护一套英文引擎 | 新增市场只需升级前端翻译模块 |

**这个决策对后期扩展性是决定性的**：未来拓展到任何新的国家市场，底层的ES索引、同义词库、向量模型全部不用动，只需要确保LLM能把新语种翻译成英文即可。

## Q4: 写入流的告警接入网关是怎么设计的？为什么要做"极简脱水透传"？

**回答思路：**

网关层的设计原则是**极高吞吐、极低延迟**，所以它不做任何复杂业务逻辑。

**鉴权机制：**
- 基于JWT或Token Auth（`X-Device-Token`），Token中包含`tenant_id`和`camera_id`，防止越权伪造告警
- 单设备限流：令牌桶/漏桶算法，限制同一台设备最多1帧/秒，防止硬件故障导致疯狂上报

**极简脱水透传的含义：**
网关不下载图片、不做识别、不做任何AI处理。它只做一件事——从请求中提取最小元数据，封装成标准JSON丢进MQ：
```json
{
  "event_id": "uuid-v4",
  "camera_id": "cam_893A4F",
  "timestamp": 1715423800,
  "image_url": "https://oss.example.com/alerts/cam_893A4F/1715423800.jpg",
  "trigger_type": "motion_detect"
}
```

**为什么这么设计？**
- 图片已经由设备端预先上传到OSS/S3，网关只需要拿到URL即可
- 网关越轻越好，它是系统的入口瓶颈点，任何重逻辑都会拖垮吞吐量
- 所有重活（图片下载、AI分析）全部交给MQ下游的Worker异步处理

## Q5: 消息队列在这个系统中扮演什么角色？Topic和Partition怎么设计的？

**回答思路：**

MQ（Kafka/RocketMQ）在写入流中承担**异步解耦+削峰填谷**的核心角色。

**为什么必须用MQ：**
- 安防场景流量极度不稳定：雷雨天、大风天树影摇晃，可能导致全城摄像头同时触发告警
- 第三方VLM API有严格的TPM/RPM限制，不可能直接承接突发流量
- MQ让网关层在接收巨额请求时不会因为下游瓶颈而过载，所有告警在MQ中暂存，Worker按自身能力平滑消费

**Partition路由策略——以camera_id做哈希：**
- 生产者发送消息时，以`camera_id`作为Partition Key
- 同一台摄像机的告警会路由到同一个Partition
- **保证时序一致性**：避免出现"12:00的照片比11:59的照片先入库"的倒挂乱序

**追问：为什么时序一致性很重要？**
- 用户搜索结果按时间排序展示，如果入库顺序乱了，可能导致搜索结果时间线混乱
- 同一台摄像机的连续帧可能描述同一个事件（如一个人走过），乱序会破坏事件的连续性

## Q6: MySQL和ES的双写一致性怎么保障？

**回答思路：**

系统需要同时写入MySQL（存储结构化元数据留底）和Elasticsearch（供快速混合搜索），我们评估了两种方案：

**方案A：基于MQ的ACK确认机制（我们采用的）**
- VLM Worker处理完图片后，作为MQ消费者的一部分，先写MySQL，再写ES
- 只有两边都成功写入后，才向MQ提交ACK确认消费完毕
- 任一失败则不ACK，消息会被重新投递重试
- **优点**：实现简单，一致性保障直观
- **缺点**：写入耦合度较高，MySQL和ES的写入在同一个消费逻辑中

**方案B：基于Binlog的最终一致性（备选方案）**
- VLM服务只写MySQL
- 部署Canal/Debezium监听MySQL Binlog增量变化
- 独立同步进程将变更同步到ES
- **优点**：解耦更彻底，适合高并发大集群
- **缺点**：引入额外组件，有同步延迟

**追问：为什么选方案A不选方案B？**
- 当前业务规模下，方案A足够简单可靠
- 写入流本身就是异步的（MQ消费），不需要极致的解耦
- 方案B引入Canal/Debezium增加了运维复杂度，在当前阶段ROI不高
- 但架构上预留了切换到方案B的能力，如果未来设备量暴增可以平滑迁移

## Q7: 读取流的查询理解（QU）子系统是怎么设计的？入参有哪些？

**回答思路：**

QU（Query Understanding）是连接用户和底层搜索引擎的核心桥梁。它通过调用LLM API，将用户的口语化模糊输入结构化为搜索引擎能直接使用的精确条件。

**必选入参设计：**

| 参数 | 类型 | 作用 |
|---|---|---|
| `user_query` | String | 用户原始自然语言输入 |
| `user_timezone` | String | 用户时区（如`Asia/Shanghai`），精准转化相对时间 |
| `current_timestamp` | Long | 提交查询瞬间的Unix时间戳，作为"昨天""上周二"的计算锚点 |
| `device_context` | List | 用户绑定的所有IPC列表（含camera_id和用户自定义别名如"后院""车库"） |
| `subscription_retention_days` | Integer | 用户云存套餐的历史天数上限（7天/30天） |

**为什么需要这么多上下文？**
- LLM本身无状态，不知道"现在几点"、"用户有哪些摄像头"
- 没有时区信息，"昨天下午"无法转换为精确时间戳
- 没有设备列表，"车库的摄像头"无法映射到具体的camera_id
- `subscription_retention_days`用于端云数据生命周期对齐，防止用户搜索超出套餐范围的历史数据

**QU最终输出四个维度：**
1. 时间范围标量（start_time / end_time）
2. 设备列表标量（camera_ids）
3. 标准英文搜索词数组（BM25专用）
4. 搜索意图长句的向量（向量空间打分专用）

## Q8: 系统的高可用和降级策略是怎么设计的？LLM/VLM挂了怎么办？

**回答思路：**

系统对第三方大模型API的依赖是最大的单点风险，必须有完善的降级方案。

**VLM（写入流）降级：**
- VLM挂了，MQ中的消息会堆积但不会丢失（At-least-once语义）
- Worker检测到API超时/限流后，执行指数退避重试（Exponential Backoff）
- 如果持续不可用，消息在MQ中等待，VLM恢复后自动消费积压
- 写入流本身是异步的，短暂不可用对用户搜索体验无直接影响（只是新告警暂时搜不到）

**LLM（读取流）降级——分级策略：**

| 故障级别 | 降级方案 |
|---|---|
| LLM输出格式异常（JSON解析失败） | 舍弃LLM结果，取用户名下所有设备+默认近3天时间窗口作为过滤条件 |
| LLM API完全宕机 | 调用备用的极简翻译API（如云厂商免费翻译服务），拿硬翻译的英文词汇做纯文本匹配检索，放弃语义和向量特征 |

**降级后的体验：**
- 搜索依然可用，不会白屏报错
- 精准度会下降（没有语义理解和同义词扩充），但基本的关键词匹配仍然有效
- 相当于从"AI智能搜索"退化为"传统关键词搜索"

**追问：怎么监控这些降级是否触发了？**
- 核心监控指标：MQ队列积压深度、大模型推理耗时P99、ES查询P99延迟
- 无结果查询（Zero-result Query）日志分析，用于持续优化同义词库和VLM Prompt
- 降级触发时自动告警通知运维团队

## Q9: 端云数据生命周期不对齐的问题怎么解决的？

**回答思路：**

这是一个非常有价值的业务架构问题。用户App本地可能缓存了半年的文本告警消息，但云端的图片特征库受限于存储成本，TTL与用户的云存套餐强绑定（如7天或30天）。如果不处理，会出现"本地明明看得到告警记录，搜却搜不到图片"的系统撕裂感。

**三层防护机制：**

**1. 基于套餐的硬性截断：**
- QU模块拿到LLM解析出的`start_time`/`end_time`后，与用户的`subscription_retention_days`计算出的最长容忍点做交集
- 例如用户套餐是30天，LLM解析出"上个月"，实际搜索范围会被截断到30天前

**2. 检索短路防护（Short-circuit）：**
- 如果LLM解析出的时间范围**完全早于**云端允许的生存周期（如7天套餐的用户搜"半个月前"），直接短路拦截
- 不向ES发起请求，节省检索资源的无效开销

**3. UI体验补偿与增值转化：**
- 后端API返回特定越界警戒码（如`E_OUT_OF_RETENTION_PLAN`）
- 前端捕获后弹出友好提示："您的当前套餐仅支持检索过去30天的录像，点击[升级高级套餐]解锁更长历史"
- **把架构限制转化为增值服务的转化入口**，这是产品和技术结合的典型案例
