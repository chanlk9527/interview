# 二、AI模型与算法类追问

---

## Q1: 系统用了哪些AI模型？各自的角色是什么？

**回答思路：**

系统中涉及四类AI模型，各司其职：

| 模型类型 | 具体选型 | 所在链路 | 核心职责 |
|---|---|---|---|
| VLM（视觉语言模型） | Qwen-VL等 | 写入流 | 分析图片，提取画面特征，输出结构化英文JSON |
| LLM（大语言模型） | GPT-4o / Claude等 | 读取流 | 解析用户自然语言意图，提取时间/地点/实体，翻译为英文 |
| Embedding模型 | all-MiniLM-L6-v2等轻量模型 | 写入流+读取流 | 将英文文本转为高维稠密向量（384/768维） |
| Reranker模型 | BGE-Reranker / Cohere Rerank | 读取流 | 对召回结果做精排，解决属性错位问题 |

**关键设计原则：**
- VLM和LLM调用第三方API（成本高但能力强），Embedding用本地轻量模型（成本低、延迟低）
- **写入流和读取流必须使用同一个Embedding模型**，否则向量空间不一致，余弦相似度计算无意义
- Reranker是整个系统精准度的最后防线，专门解决向量检索的"属性错位幻觉"

## Q2: VLM的Prompt是怎么设计的？怎么抑制大模型幻觉？

**回答思路：**

VLM的Prompt设计是整个写入流质量的关键，核心原则是**"零幻觉"——看不清就不说，绝不猜测**。

**Prompt的核心规则：**
1. **ONLY describe what is CLEARLY visible. Do NOT guess or hallucinate.** — 只描述清晰可见的内容
2. **If an object is blurry, partial, or missing colors, omit its details.** — 模糊/残缺/颜色不清的直接省略
3. **Output strictly in English JSON format** — 强制纯英文JSON输出，无markdown包裹

**提取的维度严格围绕安防要素：**

| 维度 | 提取内容 | 示例 |
|---|---|---|
| environment | 环境特征 | daytime, outdoor, raining |
| persons | 人物特征 | clothing_color, accessories, action, carried_items |
| vehicles | 车辆特征 | type, color, details |
| animals | 动物特征 | species, color |
| packages_and_baggage | 无人看管物品 | type, color, shape |
| physical_security_events | 设施安全状态 | door left open, window broken |
| fire_and_smoke | 火灾烟雾 | type, intensity |
| scene_caption | **全局场景描述长句** | 完整的主谓宾因果关系句 |

**追问：为什么不让VLM自由发挥，要限定这些维度？**
- 自由发挥会输出大量无关信息（如"这是一张高清照片"），浪费token和存储
- 安防场景用户关注的要素是确定的：人、车、物、环境、安全事件
- 限定维度让下游的ES索引结构可以预先设计，不会出现字段混乱

## Q3: 为什么VLM输出中要加scene_caption？它和结构化字段有什么区别？

**回答思路：**

这是整个系统设计中最精妙的一个点。`scene_caption`解决的是**结构化字段丢失复杂语义关系**的致命问题。

**反面例子——只有结构化字段的问题：**
画面中出现一个男人、一辆红车、车窗被砸破。结构化提取后得到：`男人`、`红车`、`破窗`。但这三者有关系吗？
- 是男人在砸车？
- 还是他恰好路过一辆被别人砸坏的红车？
- 结构化字段把画面"切碎"成了孤立的名词积木，**丢失了空间关系、因果互动、动宾结构**

**正面例子——scene_caption的价值：**
```
"scene_caption": "A man is smashing the window of a red car with a tool"
```
这句话完整保留了：谁（man）→ 做了什么（smashing）→ 对象是什么（window of a red car）→ 用什么（tool）

**在检索中的双管齐下：**

| 字段 | 用途 | 解决什么问题 |
|---|---|---|
| entities_text（结构化字段拍平） | BM25词频检索 | 颜色、物种等硬性约束的精确匹配 |
| caption_vector（scene_caption的向量） | k-NN向量检索 | 复杂行为语义的模糊匹配（如"拿着箱子跑入车库"） |

两者缺一不可：BM25保证颜色不会错，向量保证动作关系不会丢。

## Q4: LLM在读取流中具体做了哪些事？Prompt怎么设计的？

**回答思路：**

LLM在读取流中承担**查询重写引擎**的角色，一次调用完成四件事：

1. **相对时间绝对化（Time Anchoring）**：将"昨天下午""上周二"转换为精确的ISO 8601时间范围
2. **空间范围映射**：将"车库""前台"映射到具体的camera_id
3. **多语言强制英译+同义词扩充**：将任意母语翻译为标准英文，并补充同义词（如"红色电动车"→`["red e-bike", "red electric scooter", "red electric bicycle"]`）
4. **生成搜索意图长句（search_intent_caption）**：输出一句完整的英文意图描述，用于向量化

**Prompt动态注入的上下文：**
```text
Current Time: {{current_timestamp_formatted}}
Timezone: {{user_timezone}}
Available Cameras: {{device_context_json}}
```

**真实交互示例：**
- 用户输入：`"车库这两天是不是进野猫了？"`
- LLM输出：
```json
{
  "start_time": "2024-05-13T00:00:00+08:00",
  "end_time": "2024-05-15T23:59:59+08:00",
  "camera_ids": ["cam_2"],
  "search_terms": ["stray cat", "feral cat", "wild cat", "cat"],
  "search_intent_caption": "A stray cat is wandering into the garage or lingering around."
}
```

**追问：为什么要生成search_intent_caption而不只是search_terms？**
- `search_terms`是切碎的名词，给BM25用
- `search_intent_caption`是完整的意图句子，转成向量后能捕捉动作语义（如"wandering into"表示进入动作）
- 这和写入流的`scene_caption`形成对称：写入时存完整场景句，读取时生成完整意图句，两个向量在空间中比对

## Q5: Embedding模型的选型考量是什么？为什么用轻量级模型？

**回答思路：**

Embedding模型负责将英文文本转为高维稠密向量（如384/768维），我们选择了`all-MiniLM-L6-v2`这类轻量级句子向量模型。

**选型考量：**

| 维度 | 轻量级本地模型 | 第三方大模型Embedding API |
|---|---|---|
| 延迟 | 毫秒级，本地推理 | 网络往返+排队，几十到几百毫秒 |
| 成本 | 几乎为零，CPU即可运行 | 按token计费，大规模调用成本高 |
| 可用性 | 不依赖外部服务，100%可控 | 受第三方API稳定性影响 |
| 精度 | 对于短句/中等长度句子足够好 | 略优，但差距不大 |

**关键约束：写入流和读取流必须用同一个Embedding模型。**
- 写入时用模型A把`scene_caption`编码为向量存入ES
- 读取时用模型B把`search_intent_caption`编码为向量去做k-NN搜索
- 如果A和B不是同一个模型，两个向量空间不一致，余弦相似度计算完全无意义

**追问：Embedding模型的维度怎么选？384还是768？**
- 384维：存储空间小，计算快，适合对延迟敏感的场景
- 768维：表达能力更强，召回精度略高
- 我们根据实际A/B测试结果选择，在安防短句场景下384维已经足够，性价比更高

## Q6: 混合检索（Hybrid Search）是怎么做的？BM25和向量检索怎么融合？

**回答思路：**

单纯靠BM25无法理解动作和因果关系，单纯靠向量检索容易忽视具体的颜色等硬性属性。所以必须融合。

**检索流程三步走：**

**第一步：物理前置标量硬过滤（Pre-filtering）**
- 用LLM提取出的`camera_ids`和`start_time`/`end_time`在ES的`filter`子句中做硬截断
- 瞬间排除不属于目标设备、不在时间窗口内的百万级无关记录
- 这一步不参与打分，纯粹缩小搜索范围，保护计算资源

**第二步：双路召回打分**
- **BM25（稀疏检索）**：用`search_terms`（如`"stray cat, feral cat"`）匹配`entities_text`字段。保证实体单词的精确命中，尤其是颜色这种硬性约束
- **k-NN（稠密检索）**：用`search_intent_caption`的向量匹配`caption_vector`。捕捉复杂行为语义（如"拿着箱子跑入车库"的动作关系）

**第三步：分数融合**
- 使用RRF（Reciprocal Rank Fusion）或Alpha线性加权
- 安防场景下BM25略占上风：`Alpha = 0.6`给文本匹配，`0.4`给向量
- **原因**：用户对颜色和物种极其敏感，颜色都对不上的话，语义再相似也不行

**追问：Alpha = 0.6是怎么定出来的？**
- 不是拍脑袋，是通过标注数据集做A/B测试调出来的
- 测试发现在安防场景下，用户投诉最多的是"颜色错了"而不是"动作没理解"
- 所以给BM25（精确匹配颜色）更高权重

## Q7: Reranker重排模型是怎么工作的？为什么需要二阶段检索？

**回答思路：**

Reranker是整个系统精准度的**最后防线**，专门解决混合检索仍然无法消除的"属性错位幻觉"。

**什么是属性错位幻觉？**
用户搜"黄衣服男人和黑车"，BM25和向量检索都可能召回"黑衣服男人和黄车"——因为两者包含完全相同的关键词（黄、黑、衣服、车、男人），BM25词频得分几乎一样；向量空间中两句话的语义也高度相似。

**二阶段级联架构：**

| 阶段 | 模型 | 输入规模 | 输出规模 | 耗时 |
|---|---|---|---|---|
| 一阶段召回 | ES Hybrid Search | 百万级图片池 | Top 100 | 毫秒级 |
| 二阶段重排 | Cross-Encoder Reranker | 100条候选 | Top 10 | 百毫秒级 |

**Reranker的原理——和普通Embedding的本质区别：**
- **普通Embedding**：把两句话分别编码为两个独立向量，再算余弦相似度。两句话之间没有交互，无法感知"黄色绑定的是衣服还是车"
- **Cross-Encoder**：直接把两句话拼在一起扔进模型，做逐字的注意力交互比对。它能一眼看穿：用户要找的"黄色"绑定的是衬衫，而底库里的"黄色"绑定的是车，立刻打低分

**严格阈值截断（Hard Threshold）：**
- 重排后设置硬性拦截阈值（如`min_relevance_score = 0.85`）
- 超过阈值的结果才返回，哪怕只有一张
- 没有任何结果超过阈值，直接返回空列表，提示"未发现高度相似目标"
- **宁缺毋滥**：坚决拒绝给用户凑数推荐无关画面，这是建立AI信任度的底线

## Q8: LLM的异常兜底策略具体怎么实现的？有哪些降级层级？

**回答思路：**

首先要区分"正常业务逻辑"和"异常降级"：
- 用户没输入时间、没指定地点——这是**正常业务逻辑**，QU模块有默认策略处理（默认近3天、默认全部设备）
- LLM API超时、限流、输出格式错乱——这才是**异常降级**场景

**降级层级：**

**Level 1：LLM输出格式失控**
- LLM返回的内容无法被JSON反序列化
- 降级策略：舍弃LLM结果，取用户名下所有设备列表 + 默认近3天时间窗口作为基础过滤项
- 用户原始输入直接做简单分词后走纯BM25检索

**Level 2：LLM API完全宕机**
- 网络超时、Rate Limit、服务不可用
- 降级策略：调用备用的极简翻译API（如云厂商预置的免费翻译服务），拿硬翻译的英文词汇做纯文本全词匹配
- 放弃语义理解、同义词扩充、向量特征匹配
- 相当于退化为"传统关键词搜索"

**核心原则：搜图功能永远可用，不能白屏报错。**
- 精准度可以降级，但可用性不能降级
- 用户宁可看到"结果不太准"，也不能接受"功能挂了"

**追问：单设备用户有什么优化？**
- 如果用户账号只绑定了1台设备，直接跳过LLM的空间意图推断
- 不浪费token去调用LLM判断"用户说的是哪个摄像头"
- 直接将唯一的`camera_id`设为强制过滤项

## Q9: 写入流和读取流的Embedding为什么必须用同一个模型？如果要换模型怎么办？

**回答思路：**

**为什么必须同一个模型：**
- 不同的Embedding模型训练数据和目标函数不同，生成的向量空间完全不同
- 模型A把"red car"编码到向量空间的某个位置，模型B可能把同样的"red car"编码到完全不同的位置
- 用模型A的向量去和模型B的向量做余弦相似度，结果毫无意义，等于随机数

**如果要换模型（比如升级到更好的Embedding模型）：**

这是一个**全量数据迁移**问题，步骤如下：
1. 用新模型对ES中所有历史`scene_caption`重新编码，生成新的`caption_vector`
2. 可以采用双索引策略：新索引用新模型，旧索引保留，灰度切换
3. 切换完成后下线旧索引

**追问：这个迁移成本大吗？**
- 取决于数据量。如果有千万级图片记录，全量重新Embedding需要相当的计算资源和时间
- 但Embedding模型是轻量级的（CPU即可运行），不像VLM那样需要GPU
- 可以用离线批处理任务在低峰期完成，不影响线上服务
