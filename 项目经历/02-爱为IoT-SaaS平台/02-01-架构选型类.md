# 一、架构选型类追问

---

## Q1: 为什么选择 WebFlux + Netty，而不是传统的 Spring MVC？

**标准回答话术：**

> 我们选择 WebFlux + Netty，核心原因是 IoT 场景的业务特点决定的。
>
> IoT 平台需要支撑海量设备的长连接接入，特点是**高并发、IO密集、单次计算量小**。传统的 Spring MVC 基于 Servlet 模型，一个请求占用一个线程，当设备连接数上来之后，线程池很快就会被耗尽，即使大部分线程其实都在等待IO，并没有真正做计算。
>
> 而 WebFlux 基于 Reactor 响应式模型，底层用的是 Netty 的 EventLoop 线程模型。它的核心优势是**用少量线程就能处理大量并发连接**——线程不会因为等待IO而阻塞，而是通过事件回调的方式驱动，天然适合这种IO密集型场景。
>
> 具体来说，Netty 在我们项目中承担了设备接入层的角色，处理设备的 TCP/MQTT 长连接。BossGroup 负责接受连接，WorkerGroup 负责IO读写，通过自定义的 ChannelHandler Pipeline 完成协议解析、鉴权、心跳检测和业务转发。
>
> 从实际效果来看，同等硬件条件下，相比传统的 BIO 模型，我们的连接承载能力提升了一个数量级。单个 Netty 节点可以稳定维持数万级别的设备长连接，而且 CPU 和内存的利用率都比较健康。
>
> 当然，响应式编程也不是没有代价。调试比同步代码难，堆栈信息不直观，团队的学习曲线也比较陡。但综合来看，对于 IoT 这种场景，这个选型是值得的。

**回答结构拆解：**

1. **先说场景特点**（IoT = 高并发、IO密集、长连接）→ 建立选型的前提
2. **再说传统方案的痛点**（Spring MVC 线程模型的瓶颈）→ 说明为什么不行
3. **然后说新方案的优势**（WebFlux + Netty 的 EventLoop 模型）→ 说明为什么行
4. **补充实际落地细节**（Netty 在项目中的具体角色）→ 证明真的用了
5. **给出效果数据**（连接承载能力提升）→ 量化价值
6. **最后主动提代价**（调试难、学习曲线）→ 展示客观和深度

> ⚠️ **注意**：回答中"数万级别连接"和"提升一个数量级"需根据实际压测数据调整，有真实数字更有说服力。


### Q1-追问①: 为什么不用 EMQX 或云服务商提供的设备接入服务？

**标准回答话术：**

> EMQX 和云厂商的设备接入服务我们前期都调研过，确实是很成熟的方案。最终选择基于 Netty 自研接入层，主要是几个方面的考量：
>
> **第一，协议定制化需求强。** 我们平台要对接大量异构设备，除了标准的 MQTT，还有很多厂商的私有协议走的是 TCP 自定义报文。EMQX 虽然 MQTT 能力很强，但对私有协议的支持需要写插件扩展，灵活度不如自己基于 Netty 做 Pipeline 来得直接。我们在 ChannelHandler 里可以很方便地插拔不同的协议解码器，适配新协议的成本很低。
>
> **第二，业务耦合度高。** 设备接入不只是收发消息，我们在接入层就要做鉴权、租户识别、流量管控、消息路由这些事情，这些逻辑和业务强相关。如果用 EMQX，这些逻辑要么通过 Webhook 回调到业务服务（多一跳网络开销），要么写 Erlang 插件（团队没有 Erlang 能力）。自研的话，这些逻辑直接在 Pipeline 里处理，链路更短，延迟更低。
>
> **第三，成本和数据自主性。** 云服务按连接数和消息量计费，我们设备量大、消息频繁，算下来长期成本不低。而且作为 SaaS 平台，客户对数据安全比较敏感，自研方案数据全程在自己的基础设施内流转，不经过第三方，这一点在商务谈判中也是加分项。
>
> 当然，自研也意味着要自己处理集群管理、连接迁移、高可用这些问题，运维成本确实更高。如果是设备量不大、协议标准化的场景，直接用 EMQX 或云服务肯定是更高效的选择。我们是因为业务复杂度到了那个程度，自研才更划算。

**回答要点拆解：**

| 层次 | 内容 | 目的 |
|------|------|------|
| 先肯定对方 | "都调研过，确实成熟" | 避免显得无知或偏执 |
| 理由一：协议定制 | 私有协议多，Netty Pipeline 更灵活 | 说明技术上的刚需 |
| 理由二：业务耦合 | 接入层要做鉴权/租户/路由，自研链路更短 | 说明架构上的合理性 |
| 理由三：成本与数据 | 长期成本 + 数据自主性 | 说明商业上的考量 |
| 最后补代价 | 运维成本高，场景简单时推荐用现成方案 | 展示客观判断力 |

> 💡 **核心逻辑**：不是 EMQX 不好，而是我们的场景复杂到需要自研。既展示技术视野，又体现决策能力。关于 Vert.x MQTT Broker 的具体实现细节，见 [02-02 Q7](02-02-设备接入与协议类.md)。


### Q1-追问②: 具体压测数据是多少？QPS、连接数、延迟分别是什么水平？

**标准回答话术：**

> 我们做过专门的压测对比。测试环境是 4核8G 的云服务器，用自研的压测客户端模拟设备连接和消息上报。
>
> **连接数方面**，单个 Netty 节点可以稳定维持 **3~5 万**的 TCP 长连接，CPU 占用在 30% 左右，内存稳定在 4G 以内。作为对比，同样的硬件跑传统的 Spring MVC + BIO 方案，Tomcat 默认线程池 200 个线程，即使调大到 500，也就能撑几千个并发连接，再往上线程切换的开销就很明显了。
>
> **消息吞吐量方面**，在 3 万连接的基础上，每个设备每 5 秒上报一次数据，相当于每秒大约 **6000 条消息**的持续吞吐，系统处理得很稳定，没有出现消息积压。
>
> **延迟方面**，设备消息从到达 Netty 接入层到完成协议解析、鉴权、转发到消息队列，P99 延迟控制在 **5ms 以内**。端到端的延迟（设备上报到用户 APP 收到推送）大概在 **200~500ms**，这个主要取决于下游消费链路的处理速度。
>
> 压测工具我们是基于 Netty 客户端自己写的，可以模拟不同协议的设备批量连接和消息上报，比用 JMeter 更贴近真实场景。

**回答要点拆解：**

| 维度 | 数据 | 说明 |
|------|------|------|
| 硬件环境 | 4核8G 云服务器 | 先交代环境，数据才有意义 |
| 连接数 | 单节点 3~5 万 | 和 BIO 的几千形成对比 |
| 消息吞吐 | ~6000 条/秒（持续） | 说明不是峰值而是稳态 |
| 接入层延迟 | P99 < 5ms | 只算接入层，边界清晰 |
| 端到端延迟 | 200~500ms | 说明瓶颈在下游，不在接入层 |
| 压测工具 | 自研 Netty 客户端 | 展示工程能力 |

> ⚠️ **注意**：以上数字是合理的参考值，你需要根据实际情况调整。面试官不会纠结精确数字，但量级必须对。如果你说单节点撑百万连接，那就穿帮了。

### Q1-追问③: EventLoop 线程模型具体是怎么工作的？

**标准回答话术：**

> Netty 的线程模型核心就是 **EventLoop**，可以把它理解为一个"单线程的事件循环"。
>
> 整体分两组线程：**BossGroup** 和 **WorkerGroup**。BossGroup 通常就 1 个线程，专门负责监听端口、接受新连接（Accept）。每当有新设备连上来，BossGroup 就把这个连接注册到 WorkerGroup 中的某一个 EventLoop 上。WorkerGroup 一般配置为 **CPU 核数 × 2** 个线程，每个线程就是一个 EventLoop。
>
> 关键设计是：**一个 Channel（连接）在整个生命周期内只绑定一个 EventLoop 线程，不会切换**。这意味着同一个连接上的所有 IO 事件和 Handler 回调都在同一个线程里执行，天然避免了多线程竞争问题，不需要加锁。
>
> 每个 EventLoop 线程内部做的事情其实就是一个无限循环，三步走：
> 1. **轮询 IO 事件**：通过 Selector 检查绑定在自己身上的所有 Channel 有没有可读、可写的事件
> 2. **处理 IO 事件**：有数据来了就沿着 Pipeline 依次调用 ChannelHandler 处理
> 3. **执行任务队列**：处理通过 `channel.eventLoop().execute()` 提交的异步任务，比如定时心跳检测
>
> 这三步不断循环，一个 EventLoop 线程就能同时服务成百上千个 Channel。
>
> 但这里有一个**非常重要的约束**：Handler 里面绝对不能做阻塞操作。因为一个 EventLoop 线程管着几百上千个连接，你在处理某个连接的时候阻塞了，比如调了一个同步的数据库查询，那这个线程上所有其他连接的事件都得等着，整体吞吐量会断崖式下降。
>
> 我们项目中如果确实需要做耗时操作，比如查数据库或者调外部接口，会通过 `EventExecutorGroup` 把任务切到独立的业务线程池去执行，不占用 EventLoop 线程。

**回答要点拆解：**

| 层次 | 内容 | 目的 |
|------|------|------|
| 两组线程 | BossGroup（Accept）+ WorkerGroup（IO） | 先画出整体结构 |
| Channel 绑定 | 一个连接绑定一个 EventLoop，不切换 | 说明为什么不需要加锁 |
| 三步循环 | 轮询IO → 处理IO → 执行任务队列 | 说清楚 EventLoop 到底在干什么 |
| 核心约束 | 不能阻塞 EventLoop | 说明你理解这个模型的关键陷阱 |
| 实际做法 | 耗时操作切到业务线程池 | 证明你在项目中真正处理过这个问题 |

> 💡 **面试加分点**：如果面试官继续追问"Reactor 模式和 Proactor 模式的区别"，可以补充：Netty 用的是 **Reactor 模式**（应用程序主动轮询 IO 就绪事件，然后自己读写数据），而 Proactor 模式是操作系统完成 IO 操作后通知应用程序（如 Windows 的 IOCP）。Java NIO 的 Selector 本质上就是 Reactor 模式的实现。


### Q1-追问⑤: 你们是全站 WebFlux 还是部分模块用？怎么划分的？

**标准回答话术：**

> 不是全站 WebFlux，我们是**按模块的业务特点来选择的**。
>
> 设备接入层，也就是处理设备长连接、消息收发的部分，用的是 WebFlux + Netty。这一层的特点是高并发、IO密集，响应式模型的优势能充分发挥出来。
>
> 但管理后台的 API，比如设备管理、租户管理、权限配置这些，本质上就是 CRUD，并发量也不高，我们用的还是传统的 Spring MVC。原因很简单：**MVC 写起来快、调试方便、团队每个人都熟**。用 WebFlux 写 CRUD 反而是给自己找麻烦，代码可读性差，排查问题也慢。
>
> 规则引擎和数据处理这些内部服务，我们根据具体情况决定。如果是需要大量调外部接口或者做消息转发的，倾向用响应式；如果是计算密集型的逻辑处理，用传统同步模型就够了。
>
> 这些服务之间通过消息队列和 HTTP 互相调用，技术栈不同不影响协作。Spring Boot 本身也支持在同一个项目里同时跑 MVC 和 WebFlux，只是我们选择按微服务拆开，职责更清晰。
>
> 总结一句话就是：**IO密集、高并发的用 WebFlux，逻辑复杂、并发不高的用 MVC，不搞一刀切**。

**回答要点拆解：**

| 层次 | 内容 | 目的 |
|------|------|------|
| 明确回答"不是全站" | 按模块特点选择 | 展示务实，不盲目追新 |
| 接入层用 WebFlux | 高并发IO密集，响应式有优势 | 说明用在刀刃上 |
| 后台用 MVC | CRUD 场景，MVC 更高效 | 说明不是不会用，是没必要 |
| 内部服务按需选 | 根据IO密集程度决定 | 展示灵活判断力 |
| 服务间协作 | MQ + HTTP，技术栈不影响 | 说明架构上兼容 |

### Q1-追问⑥: 响应式代码出了线上问题怎么排查？

**标准回答话术：**

> 这确实是响应式编程最头疼的地方。传统同步代码出了问题，看堆栈信息基本就能定位到哪一行。但响应式链路的堆栈是断裂的，你看到的异常堆栈全是 Reactor 内部的调度代码，根本看不出是业务哪里出了问题。
>
> 我们在项目中用了几层手段来解决这个问题：
>
> **第一层，日志链路追踪。** 响应式环境下 ThreadLocal 会失效，因为一个请求可能跨多个线程执行。所以我们用 **Reactor Context** 来传递 traceId，在每个操作符的日志里都带上这个 traceId，这样即使线程切换了，日志链路也不会断。具体实现是写一个全局的 `contextWrite` 把 traceId 注入进去，然后在日志框架里通过 `Mono.deferContextual` 取出来塞到 MDC 里。
>
> **第二层，开发阶段的阻塞检测。** 我们在 CI 流水线和本地开发环境引入了 **BlockHound**，它能检测到响应式链路中任何阻塞调用，比如有人不小心在 Mono 链里调了 `Thread.sleep()` 或者同步的 JDBC 操作，BlockHound 会直接抛异常。这样很多问题在开发阶段就能发现，不用等到线上。
>
> **第三层，调试信息增强。** 开发环境我们会开启 **Reactor Debug Agent**（通过 `ReactorDebugAgent.init()` ），它能在异常堆栈里补充响应式操作符的组装点信息，让你看到是哪一行代码构建的 Mono/Flux 链出了问题。生产环境不开，因为有性能开销。
>
> **第四层，运行时监控。** 通过 **Micrometer + Grafana** 监控 EventLoop 线程的繁忙度和任务队列长度。如果某个 EventLoop 线程的繁忙度突然飙高，大概率是有阻塞操作混进来了。我们还会监控 Reactor 的 Scheduler 线程池利用率，提前发现瓶颈。
>
> 总结下来就是：**开发阶段靠 BlockHound 和 Debug Agent 提前拦截，线上靠 traceId 链路追踪和监控指标定位问题**。

**回答要点拆解：**

| 层次 | 工具/手段 | 阶段 | 解决什么问题 |
|------|-----------|------|-------------|
| 日志链路 | Reactor Context + MDC | 全阶段 | 跨线程日志追踪 |
| 阻塞检测 | BlockHound | 开发/CI | 提前发现阻塞调用 |
| 调试增强 | Reactor Debug Agent | 开发 | 异常堆栈可读性 |
| 运行监控 | Micrometer + Grafana | 生产 | 发现性能异常征兆 |

> 💡 **面试加分点**：如果面试官问"Reactor Context 和 ThreadLocal 有什么区别"，可以补充：ThreadLocal 绑定的是线程，响应式链路会跨线程所以失效；Reactor Context 绑定的是订阅链路（Subscription），不管在哪个线程执行都能拿到，是响应式世界里 ThreadLocal 的替代品。

## Q2: WebFlux 在实际使用中遇到过什么坑？

准备要点：
- 调试困难：响应式链路的堆栈信息不直观，排查问题比同步代码难
- 阻塞操作污染：团队成员不小心在响应式链路中写了阻塞代码（如JDBC调用），导致EventLoop线程被阻塞
- 解决方案：引入BlockHound检测阻塞调用、制定编码规范、Code Review重点关注
- R2DBC生态不成熟：部分复杂SQL不好写，需要退回到传统JDBC（用subscribeOn切到弹性线程池）
- 学习曲线：团队从命令式编程转向响应式需要时间适应

## Q3: Netty 在这个项目中具体怎么用的？

**标准回答话术：**

> Netty 在我们项目中是设备接入层的底层引擎，但我们并不是所有协议都直接用原生 Netty 来做的，而是**根据协议类型选择不同的接入方式**。
>
> 我们平台要对接的设备协议比较杂，主要有这么几类：**MQTT、私有 TCP 协议、HTTP、CoAP** 等。不同协议的接入方式不一样：
>
> **MQTT 协议**，我们用的是 **Vert.x MQTT** 库来搭建自己的 Broker，而不是直接用 Netty 从零写 MQTT 协议栈。原因很简单：MQTT 协议本身有完整的规范——连接、订阅、发布、QoS、遗嘱消息、保活机制这些，自己用 Netty 从头实现一遍工作量很大，而且容易出 bug。Vert.x MQTT 底层也是基于 Netty 的，它帮我们处理了 MQTT 协议的编解码和会话管理，我们只需要关注业务逻辑：设备鉴权、消息路由、Topic 权限控制这些。而且 Vert.x 本身也是事件驱动、非阻塞的模型，和我们整体的响应式架构理念一致。
>
> **私有 TCP 协议**，这部分就是直接基于 Netty 来做的。因为很多硬件厂商有自己的私有报文格式，没有现成的库可以用，只能自己写。我们在 Netty 的 Pipeline 里设计了一套**可插拔的协议适配架构**：
> - **IdleStateHandler**：做心跳检测，设备超时未发心跳就主动断开，释放资源
> - **自定义协议解码器**：根据不同厂商的报文格式做拆包和解码，处理粘包/半包问题
> - **鉴权 Handler**：连接建立后第一个报文必须是鉴权包，验证设备身份和租户归属
> - **业务 Handler**：鉴权通过后，把解析好的业务数据统一转换成内部的标准消息格式，投递到消息队列
>
> BossGroup 负责接受连接，WorkerGroup 负责 IO 读写，这是标准的 Reactor 主从模型。
>
> **HTTP 和 CoAP**，HTTP 走的是 WebFlux 那一套，设备通过 HTTP 接口上报数据，比较简单。CoAP 是轻量级的 IoT 协议，适合资源受限的设备，我们用了 **Californium**（Eclipse 的 CoAP 框架）来处理。
>
> **连接管理方面**，不管设备走哪种协议接入，鉴权通过后我们都会在本地用 **ConcurrentHashMap** 维护 `deviceId → Connection` 的映射关系（MQTT 的是 MqttEndpoint，TCP 的是 Channel），同时在 Redis 里注册路由信息。这样上层业务下发指令时，不需要关心设备用的是什么协议，统一通过路由表找到对应的连接写出去。
>
> 总结一下就是：**MQTT 用 Vert.x MQTT 做 Broker，私有协议用 Netty 原生 Pipeline，HTTP 走 WebFlux，CoAP 用 Californium**。核心思路是能用成熟库的就不重复造轮子，必须自己写的才用 Netty 从底层做。
>
> 关于多协议适配的完整设计（统一消息模型、适配器接口、代码实现），在 [02-02 Q6](02-02-设备接入与协议类.md) 里有详细展开。

**回答结构拆解：**

| 层次 | 内容 | 目的 |
|------|------|------|
| 先说整体定位 | Netty 是底层引擎，但不是所有协议都直接用 | 避免给人"什么都自己写"的印象 |
| MQTT 接入 | Vert.x MQTT 做 Broker，说明为什么不自己写 | 展示技术选型的务实判断 |
| 私有 TCP 接入 | Netty Pipeline 可插拔架构，逐层说明 Handler | 展示 Netty 的实际使用深度 |
| HTTP/CoAP | WebFlux + Californium | 说明多协议的完整覆盖 |
| 连接管理 | 统一抽象，屏蔽协议差异 | 展示架构设计能力 |
| 总结 | 能用库就用库，必须自己写才用 Netty | 体现工程判断力 |

> 💡 **面试加分点**：如果面试官追问"为什么不直接用 EMQX 做 MQTT Broker"，可以回答：EMQX 是独立部署的 Broker，我们需要在接入层就做租户隔离、鉴权、消息路由这些和业务强耦合的逻辑，用 Vert.x MQTT 嵌入到我们的服务里，可以在同一个进程内完成这些操作，链路更短、更灵活。而且我们还需要统一管理 MQTT 和 TCP 设备的连接状态，嵌入式 Broker 更容易和我们的连接管理模块集成。

### Q3-追问①: 为什么 MQTT 选 Vert.x MQTT 而不是直接用 Netty 实现？

**标准回答话术：**

> 核心原因是**投入产出比**。
>
> MQTT 协议虽然看起来不复杂，但要完整实现一个 Broker，要处理的东西很多：CONNECT/CONNACK 握手流程、SUBSCRIBE/UNSUBSCRIBE 的 Topic 匹配（包括通配符 `+` 和 `#`）、PUBLISH 的三种 QoS 级别（QoS 0 最多一次、QoS 1 至少一次、QoS 2 恰好一次）、Session 管理、遗嘱消息、Keep Alive 超时检测、Retained 消息……如果用 Netty 从零写，光协议编解码和状态机就要花很长时间，而且很容易在边界情况上出 bug。
>
> Vert.x MQTT 底层就是基于 Netty 的，它把 MQTT 协议的编解码、连接生命周期管理这些都封装好了，暴露出来的是事件回调的 API——`endpointHandler` 处理新连接、`publishHandler` 处理消息发布、`subscribeHandler` 处理订阅。我们只需要在这些回调里写业务逻辑就行。
>
> 而且 Vert.x 本身的编程模型和 Netty 的 EventLoop 是一脉相承的，都是事件驱动、非阻塞。它的 Verticle 部署模型还天然支持多核利用——部署多个 Verticle 实例，每个跑在自己的 EventLoop 上，不需要我们自己去管线程。
>
> 简单说就是：**协议层的脏活累活让 Vert.x MQTT 干，我们专注在业务层的鉴权、路由、租户隔离这些真正有价值的事情上**。

### Q3-追问②: 多协议接入的统一抽象是怎么做的？

> 这个展开讲比较长，简单说就是我们定义了统一的 **DeviceMessage 消息模型**和 **ProtocolAdapter 适配器接口**，不管什么协议进来，经过适配器转换后输出到下游的格式是统一的。上层业务只面向这个统一模型编程，不感知底层协议差异。具体的设计细节可以看 [02-02 Q6: 多协议适配](02-02-设备接入与协议类.md)，那边有完整的接口定义和代码示例。

### Q3-追问③: 私有 TCP 协议的粘包/半包问题怎么处理的？

**标准回答话术：**

> 粘包和半包是 TCP 编程的经典问题，因为 TCP 是流式协议，不保证消息边界。
>
> 我们对接的私有协议大多采用的是**固定头部 + 长度字段**的报文格式，比如前 2 个字节是魔数（用来快速识别是不是合法报文），接下来 2 个字节是报文总长度，后面是具体的数据内容。
>
> 针对这种格式，我们在 Netty Pipeline 的最前面放了一个 **LengthFieldBasedFrameDecoder**，配置好长度字段的偏移量和长度，Netty 就会自动帮我们按报文边界拆帧。只有收到完整的一帧数据，才会往后面的 Handler 传递，粘包和半包的问题在这一层就解决了。
>
> 也有少数厂商的协议用的是**特殊分隔符**来标记报文边界，比如以 `\r\n` 结尾，这种我们就用 **DelimiterBasedFrameDecoder**。
>
> 还有一种情况是报文格式比较奇葩的，既没有长度字段也没有固定分隔符，这种我们就自己继承 **ByteToMessageDecoder**，在 `decode` 方法里手动解析报文头，判断数据是否收齐，没收齐就等下一次数据到达再继续解析。
>
> 另外我们还加了一个**最大帧长度限制**，防止异常报文导致内存溢出。如果单帧超过限制（比如 64KB），直接丢弃并断开连接。

**回答要点拆解：**

| 场景 | 方案 | Netty 组件 |
|------|------|-----------|
| 固定头部 + 长度字段 | 按长度拆帧 | LengthFieldBasedFrameDecoder |
| 特殊分隔符 | 按分隔符拆帧 | DelimiterBasedFrameDecoder |
| 非标格式 | 手动解析 | 自定义 ByteToMessageDecoder |
| 防护 | 最大帧长度限制 | maxFrameLength 参数 |

## Q4: 集群部署方案是怎样的？如何保证高可用？

**标准回答话术：**

> 我们的接入层是多节点部署的，前面挂一个四层负载均衡（LVS 或云厂商的 NLB），设备通过域名连接，由负载均衡把 TCP 连接分发到不同的 Netty 节点上。
>
> 集群部署最核心的问题是**连接路由**：设备 A 连在节点 1 上，这时候后台要给设备 A 下发一条指令，怎么知道它在哪个节点？
>
> 我们用的方案是 **Redis 维护路由表 + MQ 内部转发**。具体来说：
>
> - 设备连接成功后，Netty 节点会在 Redis 里写一条映射：`device:{deviceId} → nodeId`，同时在本地用 ConcurrentHashMap 维护 `deviceId → Channel` 的映射。
> - 当业务服务需要给某个设备下发消息时，先查 Redis 拿到目标节点 ID，然后把消息投递到 MQ 的对应队列（每个节点监听自己的队列），目标节点收到后从本地 Map 里找到 Channel 写出去。
> - 设备断开连接时，清除 Redis 中的路由记录。
>
> **高可用方面**，主要考虑节点宕机的场景：
>
> - Netty 节点宕机后，该节点上所有设备的 TCP 连接会断开。设备端内置了**指数退避重连策略**（避免"惊群效应"），重连时负载均衡会把设备分配到其他存活节点，新节点完成鉴权后更新 Redis 路由表，整个过程对上层业务透明。重连的具体策略和状态恢复流程见 [02-02 Q8: 设备掉线重连](02-02-设备接入与协议类.md)。
> - Redis 路由表里的记录设置了 TTL（比如 5 分钟），即使节点宕机来不及清理，过期后也会自动失效，不会出现脏数据。
> - 我们还有一个**心跳巡检任务**，定期扫描 Redis 路由表，检查对应节点是否存活，主动清理已失效的路由记录。
>
> 另外我们也评估过**一致性哈希**方案，就是按设备 ID 哈希到固定节点。好处是不需要路由表，但问题是节点扩缩容时会导致大量连接迁移，对 IoT 场景来说抖动太大，所以没有采用。

**回答要点拆解：**

| 层次 | 内容 | 目的 |
|------|------|------|
| 整体架构 | 多节点 + 四层负载均衡 | 先画出全貌 |
| 核心问题 | 连接路由：怎么找到设备在哪个节点 | 点出集群部署的关键难点 |
| 路由方案 | Redis 路由表 + MQ 内部转发 | 说清楚具体怎么做的 |
| 宕机处理 | 指数退避重连 + TTL 兜底 + 巡检清理 | 展示高可用的完整考虑 |
| 方案对比 | 一致性哈希的优劣 | 说明你不是只知道一种方案 |

> ⚠️ **注意**：面试官可能追问"你们部署了几个节点？"——根据实际情况回答，比如 3~5 个接入节点，每个节点 3~5 万连接，总共支撑十几万设备在线。

## Q5: 整体的部署架构是什么样的？能从设备到用户端完整描述一下吗？

**标准回答话术：**

> 我们整个 IoT 平台的部署架构可以分为**五层**，从下往上依次是：设备接入层、消息中间件层、业务服务层、数据存储层、用户交互层。
>
> **第一层，设备接入层。** 这是离设备最近的一层，负责处理设备的连接和协议解析。我们部署了多个接入节点，前面挂四层负载均衡（NLB），设备通过域名解析连上来。MQTT 设备连到我们基于 Vert.x MQTT 搭建的 Broker 节点，私有 TCP 协议的设备连到 Netty 接入节点，HTTP 设备走 WebFlux 的 API 网关。接入层完成协议解析、设备鉴权、租户识别后，把消息统一转换成内部标准格式，投递到消息队列。这一层是**无状态的**（连接状态通过 Redis 路由表维护），可以水平扩展。
>
> **第二层，消息通信层。** 核心是平台自研的 **EventBus + Broker 机制**，而不是 Kafka 这类外部消息中间件。EventBus 是一个基于层级 Topic 树的发布-订阅引擎，支持 `/device/{productId}/{deviceId}/message/{type}` 这种层级路由和通配符匹配（`*` 单层、`**` 多层）。单节点内，EventBus 负责本地的 topic 匹配和投递；跨节点时，上层的 Broker 机制通过**双向通道**实现精确的跨节点转发——节点之间会同步各自的订阅信息，只有真正被订阅的消息才会跨网络传输，而不是广播所有消息让对端过滤。这套设计来自 JetLinks 的架构思路。之所以不直接用 Kafka/RocketMQ 的 topic 做业务路由，是因为它们的 topic 模型是扁平的，不支持层级路由和通配符订阅——IoT 场景下设备消息的路由维度很多（产品、设备ID、消息类型），如果用 MQ 的 topic 要么 topic 数量爆炸，要么在消费端自己做过滤，都不理想。EventBus 的 topic 树天然解决了这个问题。Broker 层的跨节点消息转发底层用的是 **RocketMQ**，利用它的持久化和可靠投递能力做节点间的消息传输。同时我们还用了 **Redis** 做设备在线状态管理和连接路由表。
>
> **第三层，业务服务层。** 这是核心业务逻辑所在，按功能拆成了多个微服务，用 Spring Cloud 做服务治理：
> - **设备管理服务**：设备注册、生命周期管理、固件升级（OTA）
> - **规则引擎服务**：处理用户配置的自动化规则，比如"温度超过 40°C 就发告警"
> - **数据处理服务**：消费 Kafka 消息，做数据清洗、聚合、持久化
> - **消息推送服务**：把告警、事件通过 WebSocket / 长轮询推送给用户 APP
> - **租户管理服务**：租户开通、套餐管理、资源配额控制
> - **API 网关**：统一鉴权、限流、路由，对外暴露 RESTful API
>
> 这些服务之间通过 **EventBus 异步通信**为主、**HTTP/gRPC 同步调用**为辅。注册中心用的 Nacos，同时也承担配置中心的角色。
>
> **第四层，数据存储层。** 不同类型的数据用不同的存储：
> - **MySQL**：存设备元数据、租户信息、规则配置这些结构化的业务数据，按租户做了分库
> - **TDengine / InfluxDB**：存设备上报的时序数据（传感器读数、状态变化），这类数据量大、写入频繁，关系型数据库扛不住
> - **Redis**：设备在线状态、路由表、热点数据缓存、分布式锁
> - **Elasticsearch**：设备日志和操作审计日志的检索
>
> **第五层，用户交互层。** 包括管理后台的 Web 前端（Vue）和用户侧的移动 APP。Web 前端通过 API 网关调后端服务，APP 除了 REST API 之外，还通过 WebSocket 接收实时推送（设备状态变化、告警通知）。
>
> **整体数据流走一遍**：设备上报数据 → 接入层协议解析 → 接入层通过 EventBus publish 到对应的设备 topic（如 `/device/product-01/dev-001/message/property`）→ 本地 EventBus 匹配订阅者投递 + Broker 转发到其他节点 → 数据处理服务持久化到时序库 + 规则引擎判断是否触发规则 → 触发了就 publish 告警事件到 EventBus → 推送服务订阅告警 topic，通知用户 APP。反过来，用户在 APP 上下发指令 → API 网关 → 业务服务 → 查 Redis 路由表找到设备所在的接入节点 → 通过 EventBus Broker（经 RocketMQ）转发到目标节点 → 接入节点写出到设备连接。
>
> 部署方面，我们用的是 **Docker + Kubernetes**，每个微服务独立镜像、独立部署。接入层因为是长连接服务，用的是 **DaemonSet + 固定节点**的方式部署，避免 Pod 漂移导致大量设备断连。其他业务服务用 Deployment，根据负载做 HPA 自动扩缩容。

**回答结构拆解（五层架构）：**

```
┌─────────────────────────────────────────────────────┐
│              用户交互层（Web / APP）                    │
│         Vue 管理后台 · 移动 APP · WebSocket            │
├─────────────────────────────────────────────────────┤
│              业务服务层（微服务集群）                     │
│  设备管理 · 规则引擎 · 数据处理 · 推送 · 租户管理 · 网关   │
│         Nacos 注册/配置中心 · Spring Cloud              │
├─────────────────────────────────────────────────────┤
│              消息通信层                                 │
│     EventBus（层级Topic路由）+ Broker（跨节点转发）       │
│       RocketMQ（Broker传输）· Redis（路由表 / 状态）     │
├─────────────────────────────────────────────────────┤
│              设备接入层（多协议网关）                     │
│  Vert.x MQTT Broker · Netty TCP · WebFlux HTTP       │
│         NLB 四层负载均衡 · Redis 路由表                  │
├─────────────────────────────────────────────────────┤
│              数据存储层                                 │
│  MySQL（业务数据）· TDengine（时序数据）                  │
│  Redis（缓存/状态）· ES（日志检索）                      │
└─────────────────────────────────────────────────────┘
```

| 层次 | 核心组件 | 职责 |
|------|---------|------|
| 设备接入层 | Vert.x MQTT / Netty / WebFlux + NLB | 多协议接入、鉴权、协议转换 |
| 消息通信层 | EventBus + Broker（RocketMQ）+ Redis | 层级Topic路由、跨节点转发、异步解耦 |
| 业务服务层 | Spring Cloud 微服务 + Nacos | 核心业务逻辑处理 |
| 数据存储层 | MySQL + TDengine + Redis + ES | 多模型存储，各取所长 |
| 用户交互层 | Vue + APP + WebSocket | 管理后台 + 用户端实时交互 |

> 💡 **回答技巧**：面试官问整体架构时，先用一句话概括"五层架构"，然后从下往上（或从上往下）逐层讲，最后走一遍完整的数据流。这样既有全局视角，又有细节支撑。

### Q5-追问①: 为什么接入层和业务层要分开部署，不能合在一起？

**标准回答话术：**

> 分开部署的核心原因是**关注点不同、资源特征不同、扩缩容策略不同**。
>
> **关注点不同。** 接入层的核心职责是维持海量设备的长连接，做协议解析和消息转发，逻辑相对简单但对连接数和 IO 吞吐要求极高。业务层做的是复杂的业务逻辑——规则匹配、数据聚合、权限校验，对 CPU 和内存的需求更大，但并发量没那么夸张。如果合在一起，一个业务逻辑的 bug 导致 OOM 或者 CPU 打满，会直接把设备连接全部断掉，影响面太大。
>
> **扩缩容策略不同。** 接入层的瓶颈是连接数，扩容是为了承载更多设备；业务层的瓶颈是计算资源，扩容是为了处理更多业务请求。两者的扩容触发条件和节奏完全不一样。分开部署后，接入层可以按连接数来扩，业务层按 CPU/内存来扩，互不干扰。
>
> **稳定性隔离。** 接入层是长连接服务，最怕频繁重启——每次重启都意味着上面几万个设备要断连重连。业务层发版迭代比较频繁，如果合在一起，每次业务代码更新都要重启，设备就要跟着断一次连，这在生产环境是不可接受的。分开之后，业务层随便发版，接入层只有在协议变更或者接入逻辑调整时才需要动。
>
> **技术栈差异。** 接入层是 WebFlux + Netty / Vert.x，纯响应式；业务层大部分是 Spring MVC，同步模型。混在一个进程里容易出问题，比如同步代码不小心阻塞了 EventLoop 线程。

**回答要点拆解：**

| 维度 | 合在一起的问题 | 分开的好处 |
|------|--------------|-----------|
| 故障隔离 | 业务 bug 拖垮设备连接 | 互不影响 |
| 扩缩容 | 扩容粒度粗，浪费资源 | 按需独立扩容 |
| 发版频率 | 业务发版导致设备断连 | 接入层稳定不动 |
| 技术栈 | 响应式和同步代码混用易出错 | 各自用最合适的模型 |

### Q5-追问②: 服务之间的通信方式是怎么选择的？什么时候用 EventBus，什么时候用 HTTP？

**标准回答话术：**

> 我们的原则很简单：**能异步的就异步，必须同步的才同步**。
>
> **用 EventBus 异步通信的场景：**
> - 设备消息的处理链路。设备上报的数据从接入层 publish 到 EventBus，下游的数据处理模块、规则引擎模块各自通过 topic 订阅，互不阻塞。这是典型的**发布-订阅**模式，一条消息多个订阅者各取所需，而且 EventBus 的层级 topic + 通配符能做到非常精细的路由——比如规则引擎只订阅 `/device/**/message/event`，数据处理订阅 `/device/**/message/**`，各自只收到自己关心的消息。
> - 规则引擎触发的动作。比如规则判断要发告警，它 publish 一条告警事件到 `/alarm/**` topic，推送服务订阅了这个 topic 就会收到。规则引擎不需要等推送完成，处理速度更快。
> - 跨节点的消息转发。EventBus 上层的 Broker 机制负责把消息透明地转发到其他节点，业务代码不需要关心消息是本地消费还是远程消费，统一用 subscribe/publish 就行。
> - 任何**不需要立即拿到结果**的操作，比如日志写入、审计记录、统计数据更新。
>
> **用 HTTP 同步调用的场景：**
> - 用户在管理后台操作，需要**立即看到结果**的。比如用户点"查询设备列表"，API 网关调设备管理服务，必须同步等结果返回给前端。
> - 服务间需要**强依赖对方结果**才能继续的。比如接入层鉴权时需要调租户服务验证设备身份，必须拿到鉴权结果才能决定是否允许连接。
> - 配置类的读取，比如规则引擎启动时从配置服务拉取规则列表。
>
> **还有一种折中方案**：对于需要同步语义但又想解耦的场景，我们用的是 **RPC + 超时降级**。比如数据处理服务在持久化之前需要查设备的产品模型做数据校验，这个调用走的是 HTTP，但设置了比较短的超时（比如 500ms），如果超时就用缓存的模型兜底，不阻塞主流程。
>
> 总结就是：**数据流转用 EventBus（层级路由 + 跨节点透明转发），用户交互用 HTTP，核心链路上的同步调用要有超时和降级**。

**回答要点拆解：**

| 通信方式 | 适用场景 | 典型例子 |
|---------|---------|---------|
| EventBus 异步 | 不需要立即结果、一对多消费、细粒度路由 | 设备消息处理、告警通知、日志写入 |
| HTTP 同步 | 需要立即结果、强依赖 | 用户查询、设备鉴权、配置读取 |
| HTTP + 降级 | 需要结果但可容忍降级 | 数据校验（超时用缓存兜底） |

### Q5-追问③: K8s 部署长连接服务有什么特殊考虑？

**标准回答话术：**

> 长连接服务在 K8s 上部署确实有几个坑，和普通的无状态 HTTP 服务不一样。
>
> **第一，Pod 漂移问题。** 普通服务的 Pod 被调度到哪个 Node 都无所谓，但接入层每个 Pod 上面挂着几万个设备连接，如果 Pod 被 K8s 调度器随意迁移（比如节点资源不足触发驱逐），所有连接都会断。我们的做法是给接入层的 Pod 设置了**较高的 QoS 等级**（Guaranteed 级别，requests = limits），降低被驱逐的优先级。同时用 **PodDisruptionBudget** 限制同时不可用的 Pod 数量，确保滚动更新时不会一次性断太多连接。
>
> **第二，优雅关闭。** 接入层 Pod 关闭时不能直接 kill，需要一个优雅关闭的过程：先从负载均衡摘掉（不再接受新连接），然后给已有连接发一个"服务端主动断开"的信号，等设备重连到其他节点后再关闭。我们通过 **preStop Hook** 实现这个逻辑，配合比较长的 `terminationGracePeriodSeconds`（比如 120 秒），给设备足够的时间重连。
>
> **第三，负载均衡的选择。** K8s 默认的 Service 是四层负载均衡（基于 iptables/IPVS），对 TCP 长连接来说有个问题：连接一旦建立就固定在某个 Pod 上，新扩容的 Pod 分不到已有连接的流量。所以我们在 K8s Service 之上还用了云厂商的 **NLB**（Network Load Balancer），配合**最少连接数**的调度策略，让新连接优先分配到连接数少的 Pod 上。
>
> **第四，健康检查。** 接入层的健康检查不能只检查 HTTP 端口是否存活，还要检查 EventLoop 线程是否正常工作。我们自定义了 **Readiness Probe**，除了检查进程存活，还会检查当前连接数是否超过阈值、EventLoop 线程是否有阻塞。如果连接数已经打满，就让 Readiness 探针返回失败，K8s 会自动把这个 Pod 从 Service 的 Endpoints 里摘掉，新连接就不会再分过来。

**回答要点拆解：**

| 问题 | 解决方案 | 关键配置 |
|------|---------|---------|
| Pod 漂移/驱逐 | Guaranteed QoS + PDB | requests=limits, PDB minAvailable |
| 优雅关闭 | preStop Hook + 长 grace period | terminationGracePeriodSeconds: 120 |
| 负载均衡不均 | NLB + 最少连接数策略 | 云厂商 NLB 配置 |
| 健康检查 | 自定义 Readiness Probe | 检查连接数 + EventLoop 状态 |

> 💡 **面试加分点**：如果面试官问"你们有没有考虑过 Service Mesh（如 Istio）"，可以回答：评估过，但 Istio 的 Sidecar（Envoy）对长连接的支持有一些限制，比如连接级别的负载均衡效果不好，而且 Sidecar 会增加一跳延迟和资源开销。对于接入层这种对延迟敏感、连接数极高的服务，我们选择不走 Sidecar，直接用 NLB。业务层的 HTTP 服务倒是可以考虑接入 Service Mesh。

### Q5-追问④: 如果让你重新设计这个架构，有什么会改进的地方？

**标准回答话术：**

> 回过头来看，有几个地方如果重新做会有不同的选择：
>
> **第一，接入层的协议网关可以做得更通用。** 我们当时是按协议类型分了不同的接入服务（MQTT 一个、TCP 一个），虽然后来做了统一抽象，但早期的代码还是有一些耦合。如果重新设计，我会一开始就设计一个**统一的协议网关框架**，协议解码器作为插件动态加载，新增协议不需要部署新服务，只需要加一个插件 JAR 包。
>
> **第二，时序数据库的选型可以更早确定。** 我们早期用 MySQL 存设备时序数据，后来数据量上来扛不住才迁到时序库。这个迁移过程比较痛苦，涉及数据迁移、查询接口改造、历史数据兼容。如果一开始就用时序库，能省很多事。
>
> **第三，可观测性应该从第一天就建设。** 我们的监控体系是后面逐步补的，早期出了问题基本靠看日志。如果重新来，我会从项目初期就把 **OpenTelemetry** 集成进去，统一 Metrics、Tracing、Logging 三个维度，尤其是响应式链路的追踪，越早做越好。
>
> **第四，考虑引入边缘计算层。** 现在所有设备数据都直接上云处理，对于一些延迟敏感或者带宽受限的场景（比如工厂内网的设备），如果在边缘侧部署一个轻量级的网关做预处理和本地决策，能减少云端压力，也能降低端到端延迟。
>
> 当然，这些都是事后诸葛亮。当时的选择在当时的资源和认知条件下是合理的，架构本来就是演进的，不可能一步到位。

**回答要点拆解：**

| 改进点 | 当时的做法 | 改进方向 | 展示什么 |
|--------|-----------|---------|---------|
| 协议网关 | 按协议分服务 | 统一网关 + 插件化 | 架构抽象能力 |
| 时序库 | 先 MySQL 后迁移 | 一开始就用时序库 | 技术预判能力 |
| 可观测性 | 后期补建 | Day 1 就集成 OpenTelemetry | 工程成熟度 |
| 边缘计算 | 全部上云 | 边缘预处理 + 云端协同 | 技术视野 |

> 💡 **核心逻辑**：这类"如果重新设计"的问题，面试官考察的不是你当时做错了什么，而是你有没有**反思和成长**。回答时既要承认不足，也要说明当时的选择在当时是合理的，展示你的工程判断力在不断进化。
