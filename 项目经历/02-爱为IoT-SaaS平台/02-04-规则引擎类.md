# 四、规则引擎类追问

## Q12: 规则引擎的核心设计思路是什么？为什么不用Drools而选择自研？

**回答思路：**

我们的规则引擎核心思想是：**把一个规则图（节点+连线）编译成路由表，然后让数据从入口节点进来，沿着连线流转到下游节点，每条连线上可以带条件判断。**

整条链路可以浓缩为：
```
RuleModel（规则图）→ RuleCompiler（编译）→ LocalRuleRuntime（运行时）→ 数据在节点间流转
```

**为什么不用Drools：**
- Drools是"事实+规则+推理"模式，适合大量业务规则的匹配场景
- IoT场景的核心需求是"数据进来→转换→判断→动作"，是数据流驱动的
- 我们用有向图建模，天然适合IoT：每个节点是一种处理能力，每条边是数据流向
- 图模型的优势：可视化友好（拖线连节点）、可组合（节点任意连接）、可扩展（新增节点不影响引擎主体）

**分层架构：**
- 模型层：RuleModel / RuleNodeModel / RuleLink / Condition — 定义规则图的静态结构
- 数据层：RuleData — 统一数据信封，在节点间流转
- 条件层：ConditionEvaluator + Registry — 边上的条件判断
- 执行器层：TaskExecutor + TaskExecutorProvider — 节点的处理能力
- 编译层：RuleCompiler → CompiledRule — 预处理路由表
- 运行时层：LocalRuleEngine / LocalRuleRuntime / ExecutionContext — 驱动数据流转


**追问：为什么用图而不是if-else？**

- 节点只管"做事"（处理数据），边负责"路由"（决定数据去哪）
- 同一个节点可以有多条输出边，每条边带不同条件，自然实现分支逻辑
- 修改路由逻辑不需要改节点代码，只需要改边的条件配置
- 新增一种节点类型完全不需要改引擎核心代码

## Q13: 规则引擎的模型层怎么设计的？节点和连线怎么表达？

**回答思路：**

模型层是所有其他层的基础，由四个纯数据类组成：

| 类 | 职责 |
|---|---|
| `RuleModel` | 整张规则图（节点列表 + 连线列表 + 规则级配置） |
| `RuleNodeModel` | 单个节点（id + executor标识 + 配置 + outputs/events列表） |
| `RuleLink` | 一条连线（source → target，type区分普通边/事件边，可带Condition） |
| `Condition` | 条件定义（type + configuration，非常轻量） |

**关键设计决策：**

1. **executor字段是字符串标识，不是执行器本身。** `RuleNodeModel`里的executor（如`"log"`、`"function"`）只是查找key，真正的执行器在运行时通过`TaskExecutorProvider`创建。这是策略模式的核心——模型只说"我要什么类型的节点"，不管"怎么实现"。

2. **RuleLink用type字段区分两种边。** `type`为`"output"`是普通数据边，其他值（如`"error"`、`"complete"`）是事件边。同一个类表达两种路由语义，不需要搞两套继承体系。

3. **Condition只有type+configuration。** 判断逻辑交给条件层去做，新增条件类型时模型层完全不用改。

4. **addLink会自动同步到source节点的outputs/events列表。** 模型既能从"全图"视角看，也能从"节点局部"视角看。

## Q14: RuleData统一数据信封是怎么设计的？为什么不直接传业务对象？

**回答思路：**

RuleData是整个引擎的"血液"，结构如下：
```
RuleData
├── id           — 每份数据包的唯一标识（UUID）
├── contextId    — 同一次规则执行的关联id（UUID）
├── data         — 真正的业务数据（Object）
├── headers      — 元信息（错误信息、标记位等）
└── sourceNodeId — 这份数据从哪个节点出来的
```

**为什么不直接传业务对象，要包一层信封？三个原因：**

1. **条件判断需要读headers。** 没有信封就没地方挂元信息。比如错误传播时需要在headers里塞`_error`和`_errorType`字段。
2. **链路追踪。** `contextId`让同一次执行链路上的所有数据包可以关联起来，方便调试。
3. **数据隔离。** `copyFor()`方法让下游节点拿到的是一份新的数据包（新id），但保留上游的contextId和headers。既不会污染上游数据，又能保持链路上下文。

**追问：copyFor()的精确行为？**
- 新id（UUID），保留原contextId
- 使用新的业务数据，复制headers（浅拷贝）
- 更新sourceNodeId为当前节点
- headers浅拷贝意味着新数据包往里加东西不会影响上游，这在错误传播时非常重要

## Q15: 条件判断为什么放在边上而不是节点内？条件层怎么设计的？

**回答思路：**

**条件放在边上的好处：**
- 节点只管"处理数据"
- 边负责"决定数据去哪"
- 修改路由逻辑不需要改节点代码，只需要改边的条件配置
- 同一个节点可以有多条输出边，每条边带不同条件 → 自然实现分支

**条件层三个类：**

| 类 | 职责 |
|---|---|
| `ConditionEvaluator` | 接口：给一个Condition和一份RuleData，返回条件是否成立 |
| `SimpleConditionEvaluator` | 用`BiPredicate<Condition, RuleData>`快速注册条件策略 |
| `ConditionEvaluatorRegistry` | 注册表：按type查找evaluator，统一对外入口 |

**内置7种条件类型：**

| type | 说明 |
|---|---|
| `always_true` | 永远通过，用于默认通路 |
| `header_equals` | 比较headers中某个key的值 |
| `field_equals` | 比较data中某个字段的值 |
| `field_gt` / `field_gte` | 字段值 > 或 >= 配置值 |
| `field_lt` / `field_lte` | 字段值 < 或 <= 配置值 |

**关键设计：** 注册表内部处理condition为null的情况——直接返回true。运行时路由代码不需要自己判空，直接调`conditionRegistry.evaluate(link.getCondition(), data)`就行。"无条件即放行"的语义封装在注册表内部。

**追问：数值比较怎么处理类型差异？**
- 数值比较类条件内部统一转成double再比较
- Integer、Long、Double等类型都能正常处理
- 如果任意一侧无法转换为数字，直接抛异常让调用方感知配置错误

## Q16: 执行器层怎么做到可插拔？TaskExecutor和Provider的关系是什么？

**回答思路：**

引擎最体现功力的一点是把**引擎骨架**和**节点能力**彻底拆开：
- 引擎不知道"设备告警节点"怎么实现
- 引擎也不知道"SQL节点"怎么处理数据
- 引擎只知道：每个节点会由一个TaskExecutor执行

**三个核心抽象：**

| 类 | 职责 |
|---|---|
| `TaskExecutor` | 接口：节点执行器，负责处理输入数据 |
| `TaskExecutorProvider` | 接口：工厂，根据节点配置创建执行器实例 |
| `AbstractTaskExecutor` | 基类：统一保存context + 统一兜底异常处理 |

**为什么每个执行器都是Provider+内部Executor的结构？**
- 同一种类型的节点可能在规则图里出现多次，每次配置不同
- Provider是单例的（注册一次），但每次`createExecutor()`都创建新的Executor实例，带着各自的配置
- 这是标准的**策略模式+工厂模式**组合

**AbstractTaskExecutor的模板方法——最精妙的设计：**
```java
public void execute(RuleData input, ExecutionContext context) {
    try {
        doExecute(input);      // 子类只管写正常逻辑
    } catch (Exception error) {
        context.onError(error, input);  // 异常自动走error事件边
    }
}
```
子类只需实现`doExecute()`，完全不用操心异常处理。异常自动交给`context.onError()`，转成error事件走事件边路由到错误处理节点。

**6个内置执行器：**

| Provider | executor标识 | 功能 |
|---|---|---|
| `StartTaskExecutorProvider` | `"start"` | 入口节点，透传数据 |
| `LogTaskExecutorProvider` | `"log"` | 打印数据，继续透传 |
| `FunctionTaskExecutorProvider` | `"function"` | 对Map数据做remove/rename/set变换 |
| `FilterTaskExecutorProvider` | `"filter"` | 节点内部条件过滤，满足才输出 |
| `DelayTaskExecutorProvider` | `"delay"` | Thread.sleep延迟后继续传递 |
| `NotifyTaskExecutorProvider` | `"notify"` | 通知下发，通过NotifySender接口解耦 |

## Q17: 为什么要有"编译"阶段？RuleCompiler做了什么？

**回答思路：**

**为什么要编译？**
- 如果运行时每次路由都遍历所有links查找"从当前节点出发的边"，效率太低
- 编译阶段把这些信息预处理成Map，运行时直接O(1)查找

**RuleCompiler.compile()做了这些事：**

1. **校验合法性：** 规则id不能为空、至少一个节点、节点id不能重复、executor不能为空
2. **构建nodeMap：** `Map<nodeId, RuleNodeModel>`，方便快速查找
3. **构建outputLinks（普通边路由表）：** `Map<sourceNodeId, List<OutputLink>>`，按源节点分组。运行时只需`outputLinks.get(sourceNodeId)`就能拿到所有下游
4. **构建eventLinks（事件边路由表）：** `Map<sourceNodeId, Map<eventName, List<OutputLink>>>`，按源节点+事件名二级分组。两次Map.get就能定位目标
5. **识别起始节点：** 优先找executor为`"start"`的节点，找不到就退化为"所有无入边的节点"（通过编译阶段统计的入度inboundCount判断）

**编译阶段的好处：**
- 把"模型"和"执行"隔开：模型不含执行逻辑，编译结果不含模型冗余数据
- 预处理路由表：运行时不需要每次遍历links
- 校验前置：编译阶段就发现图的问题（孤立节点、缺失执行器等）
- 一次编译多次执行：同一个规则可以创建多个运行时实例

**追问：CompiledRule的不可变性？**
- 构造时复制传入的集合（`new LinkedHashMap<>(nodeMap)`等），避免外部修改影响运行时
- `startNodeIds`用`List.copyOf()`生成不可变列表

## Q18: 运行时数据流转的完整路径是怎样的？

**回答思路：**

这是整个引擎的灵魂，完整路径如下：

```
用户调用 engine.fireRule("rule-1", rawData)
  ↓
engine把rawData包装成RuleData，为每个start节点copyFor一份副本
  ↓
runtime.routeToNode("start", ruleData)
  ↓
LocalTask.execute(ruleData) → executor.execute(ruleData, context)
  ↓
executor内部调用 context.output(data)  ← 节点不直接调用下游！
  ↓
context.output() → runtime.handleOutput(nodeId, data)
  ↓
runtime查编译好的outputLinks，逐条调用conditionRegistry.evaluate()
  ↓
条件通过 → routeToNode(targetNodeId, data) → 下一个节点继续处理
```

**核心要点：**
1. **节点不直接调用下游节点**，而是通过`ExecutionContext.output()`写出数据
2. **运行时根据编译后的路由表**决定数据去哪
3. **条件判断发生在路由时**，由`conditionRegistry.evaluate()`统一处理
4. **错误也是一种事件**，通过事件边流向错误处理节点

**ExecutionContext——最关键的桥梁：**

节点执行器永远不知道自己的下游是谁。它只知道"我处理完了，调`context.output(data)`把数据扔出去"。至于数据去哪、走哪条边、条件怎么判断，全部由运行时负责。

| 方法 | 作用 |
|---|---|
| `output(data)` | 把数据写到当前节点的普通输出边 |
| `fireEvent(event, data)` | 触发事件，走对应的事件边 |
| `onError(error, data)` | 统一错误处理入口 |
| `newRuleData(data)` | 创建以当前节点为来源的新数据 |
| `newRuleData(data, parent)` | 基于父数据创建下游数据，保留contextId和headers |
| `testCondition(condition, data)` | 让执行器复用条件注册表（filter节点用到） |

## Q19: 错误是怎么传播的？error事件边是怎么工作的？

**回答思路：**

错误传播没有任何特殊机制，它和正常数据流用的是**同一套路由机制**。

**onError()的精确实现：**
```java
public void onError(Throwable error, RuleData data) {
    RuleData errorData = data == null
        ? newRuleData(null)
        : data.copyFor(data.getData(), nodeId);   // 先copyFor创建副本！
    errorData.getHeaders().put("_error", error.getMessage());
    errorData.getHeaders().put("_errorType", error.getClass().getName());
    fireEvent("error", errorData);
}
```

**两个容易忽略的细节：**
1. **不会污染上游数据。** 先通过`copyFor()`创建副本，再往副本headers里塞`_error`和`_errorType`。原始数据不受影响。
2. **除了`_error`还有`_errorType`。** 存的是异常类的全限定名（如`java.lang.IllegalStateException`），方便下游节点根据异常类型做不同处理。

**完整链路：**
```
executor.doExecute()抛异常
  ↓
AbstractTaskExecutor.execute()捕获异常 → context.onError(error, input)
  ↓
onError()创建errorData副本，塞入_error和_errorType headers
  ↓
fireEvent("error", errorData)
  ↓
runtime.handleEvent(nodeId, "error", errorData)
  ↓
查eventLinks路由表，找到error事件边的目标节点
  ↓
routeToNode(errorHandlerNodeId, errorData)
```

**设计精髓：** 错误就是事件，事件就是路由。没有特殊的错误处理机制，一切都是边上的条件路由。

## Q20: LocalRuleEngine的启动和停止流程？多规则怎么管理？

**回答思路：**

**启动流程 `startRule(model)`：**
1. `compiler.compile(model)` — 编译规则图为路由表
2. 创建`LocalRuleRuntime`实例
3. `runtime.start()` — 为每个节点创建独立的执行器和上下文
4. 存入`runtimes` Map（key为ruleId）
5. 如果有`RuleTrigger`，启动所有触发器

**runtime.start()的细节：**
```java
for (RuleNodeModel node : compiledRule.getNodeMap().values()) {
    TaskExecutorProvider provider = providers.get(node.getExecutor());
    TaskExecutor executor = provider.createExecutor(node.getConfiguration());
    ExecutionContext context = new ExecutionContext(node.getId(), this, conditionRegistry, node.getConfiguration());
    LocalTask task = new LocalTask(node.getId(), executor, context);
    tasks.put(node.getId(), task);
    task.start();
}
```

**重复启动的处理：** 用`compiled.getRuleId()`作为map key，重复启动时先put新的再stop旧的（`runtimes.put`在`previous.stop()`之前），确保不会出现短暂的"规则不存在"窗口。

**fireRule()的细节：** 会为每个起始节点`copyFor`一份输入副本，多入口规则也能正常工作，各入口拿到独立的数据包。

**RuleTrigger——外部事件入口：**
```java
public interface RuleTrigger {
    void start(String ruleId, LocalRuleEngine engine);
    void stop();
}
```
触发器负责监听外部事件源（设备上报、消息总线、定时任务等），事件到来时自动调用`engine.fireRule()`注入数据。规则引擎不需要知道"数据从哪来"，触发器也不需要知道"规则内部怎么跑"。

## Q21: 这个规则引擎用了哪些设计模式？能具体说说吗？

**回答思路：**

| 设计模式 | 在哪里用 | 具体体现 |
|---|---|---|
| **策略模式** | 执行器层 | 不同TaskExecutor是不同策略，引擎不关心具体实现 |
| **工厂模式** | TaskExecutorProvider | 根据配置创建不同的执行器实例 |
| **注册表模式** | Engine维护`Map<String, TaskExecutorProvider>` | 按executor标识查找对应工厂 |
| **模板方法** | AbstractTaskExecutor | `execute()`定义骨架，子类只实现`doExecute()` |
| **中介者模式** | ExecutionContext | 节点不直接通信，通过context和runtime中转 |
| **信封模式** | RuleData | 统一数据包装，携带元信息和业务数据 |
| **编译器模式** | RuleCompiler | 将声明式模型转换为可执行的运行时结构 |

**整个引擎的设计精髓，浓缩成三句话：**
1. **模型只描述"是什么"，不管"怎么执行"** — 策略模式让节点类型可插拔
2. **节点不知道下游是谁，只管往context里扔数据** — 运行时负责路由，节点负责处理
3. **错误就是事件，事件就是路由** — 没有特殊的错误处理机制，一切都是边上的条件路由

## Q22: 规则引擎怎么保证可扩展性？如果要新增一种节点类型怎么做？

**回答思路：**

新增一种节点类型只需要两步：

1. **实现TaskExecutorProvider接口：**
```java
public class MyCustomProvider implements TaskExecutorProvider {
    public String getExecutor() { return "my_custom"; }
    public TaskExecutor createExecutor(Map<String, Object> config) {
        return new MyCustomExecutor(config);
    }
}
```

2. **注册到引擎：**
```java
engine.registerProvider(new MyCustomProvider());
```

然后在规则图的节点配置里把executor设为`"my_custom"`就能用了。

**为什么这么简单？**
- 引擎骨架和节点能力彻底拆开
- 引擎只知道"每个节点会由一个TaskExecutor执行"
- 新节点通过Provider注册进来，引擎按executor字段查找对应工厂
- 节点内部通过ExecutionContext与运行时交互，不需要知道引擎内部实现

**实际例子：测试里的自定义Provider**
- `CollectingProvider`（executor=`"collect"`）：只把收到的RuleData按bucket分桶存起来，方便断言
- `FailingProvider`（executor=`"boom"`）：`doExecute()`直接抛异常，验证错误传播

这两个Provider的存在本身就证明了执行器的可插拔性。

## Q23: 规则引擎的性能怎么保证？如果要扩展到分布式怎么做？

**回答思路：**

**当前版本的性能设计：**
- 编译阶段预处理路由表，运行时O(1)查找下游节点，不需要每次遍历links
- 条件注册表按type直接Map.get查找evaluator，不做全量遍历
- RuleData的copyFor是浅拷贝headers，避免深拷贝开销
- CompiledRule不可变，可以安全地被多个运行时实例共享

**从本地版扩展到分布式的路线：**

当前实现中，节点间通信通过直接方法调用（`runtime.routeToNode()`）。架构预留了集成点：

| 当前实现 | 未来可替换为 |
|---|---|
| `context.output()` → 直接调用`runtime.handleOutput()` | 通过EventBus发布到output topic |
| `context.fireEvent()` → 直接调用`runtime.handleEvent()` | 通过EventBus发布到event topic |

JetLinks正是走的后一条路（`EventBusOutput`/`EventBusInput`），把直接方法调用替换为消息总线路由，就能实现节点跨进程执行。

**扩展路线图：**
1. 引入`ExecutorService`，让节点执行和数据路由异步化
2. Timer源节点：用`ScheduledExecutorService`实现定时触发
3. EventBus集成：把直接方法调用替换为topic路由
4. 规则Builder：提供fluent API更优雅地构建规则图
5. Scope状态：按global/flow/node三级scope设计状态管理

## Q24: 能举一个完整的规则执行场景吗？比如温度告警？

**回答思路：**

**场景：温度超过30度走高温告警分支，否则走正常分支**

规则图结构：
```
[start] → [function: set status=alarm] → [high: 高温告警通知]  (条件: field_gt temperature 30)
                                        → [normal: 正常记录]    (条件: field_lte temperature 30)
```

执行过程（注入temperature=35）：
1. `engine.fireRule("rule-1", Map.of("temperature", 35))`
2. engine包装成RuleData，copyFor到start节点
3. start节点透传，调用`context.output(data)`
4. runtime查outputLinks，找到start→function的边（无条件），路由到function节点
5. function节点对Map做set操作，加上`status: "alarm"`字段，调用`context.output(newData)`
6. runtime查outputLinks，找到function的两条输出边：
   - 边1条件：`field_gt, key=temperature, value=30` → 35>30 ✓ → 路由到high节点
   - 边2条件：`field_lte, key=temperature, value=30` → 35<=30 ✗ → 不路由
7. high节点收到数据，执行告警通知

如果注入temperature=20，则边1条件不满足，边2条件满足，数据走到normal节点。

**错误场景：**
```
[start] → [function: 可能抛异常] --output--> [log: 成功处理]
                                 --error-->  [notify: 错误通知]
```
function节点抛异常 → AbstractTaskExecutor捕获 → context.onError() → fireEvent("error") → 走error事件边到notify节点。
