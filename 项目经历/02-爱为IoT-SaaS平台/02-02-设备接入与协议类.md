## Q5: 物模型抽象层怎么设计的？

准备要点：
- 物模型（Thing Model）的概念：用统一的数据结构描述设备的属性、服务、事件
- 属性（Property）：设备状态，如温度、开关状态，支持读写
- 服务（Service）：设备能力，如重启、云台转动，支持调用
- 事件（Event）：设备主动上报，如告警、故障
- 数据格式：JSON Schema定义每个字段的类型、取值范围、单位
- 好处：上层业务代码只面向物模型编程，不关心底层是MQTT还是HTTP

### 参考回答

#### 1. 为什么需要物模型

我们平台要接入大量异构设备——摄像头、传感器、智能门锁、网关等等，每种设备的能力完全不同。如果没有一层统一的抽象，上层业务代码就要针对每种设备单独写逻辑，设备类型一多就不可维护了。

物模型就是解决这个问题的。它的核心思想是：**用一套标准化的数据结构来描述任意设备"是什么、能做什么、会报什么"**，让上层业务代码只面向物模型编程，不关心底层是什么设备、走的什么协议。

类比一下的话，物模型就像数据库的 Schema——你定义好表结构，应用层按 Schema 读写就行，不需要关心底层是 MySQL 还是 PostgreSQL。

#### 2. 物模型的三要素

我们的物模型把设备能力抽象为三类：**属性、服务、事件**。

**属性（Property）—— 设备"是什么状态"**
- 描述设备的持续性状态，比如温度、湿度、开关状态、电量
- 支持读写：读取当前值（GET）、设置目标值（SET）
- 每个属性定义了数据类型、取值范围、单位、读写权限
- 属性值的变化会更新到设备影子的 reported 中

**服务（Service）—— 设备"能做什么"**
- 描述设备可被调用的能力，比如重启、云台转动、固件升级
- 本质上是一个 RPC 调用：有输入参数、有输出结果、有超时时间
- 和属性设置的区别：属性设置是改状态（幂等的），服务调用是触发动作（可能不幂等）

**事件（Event）—— 设备"发生了什么"**
- 描述设备主动上报的离散事件，比如告警、故障、阈值触发
- 和属性上报的区别：属性是持续性的状态值，事件是一次性的通知
- 事件有级别之分：信息（info）、告警（warn）、故障（error）

#### 3. 物模型的数据结构设计

每个产品（Product）关联一个物模型定义，用 JSON 描述。以一个温湿度传感器为例：

```json
{
  "productKey": "sensor_th_01",
  "productName": "温湿度传感器",
  "properties": [
    {
      "identifier": "temperature",
      "name": "温度",
      "dataType": {
        "type": "float",
        "min": -40,
        "max": 125,
        "step": 0.1,
        "unit": "℃"
      },
      "accessMode": "r"
    },
    {
      "identifier": "reportInterval",
      "name": "上报间隔",
      "dataType": {
        "type": "int",
        "min": 1,
        "max": 3600,
        "unit": "s"
      },
      "accessMode": "rw"
    }
  ],
  "services": [
    {
      "identifier": "reboot",
      "name": "重启设备",
      "inputParams": [],
      "outputParams": [
        {
          "identifier": "result",
          "dataType": { "type": "bool" }
        }
      ],
      "timeout": 10000
    }
  ],
  "events": [
    {
      "identifier": "highTempAlarm",
      "name": "高温告警",
      "level": "warn",
      "outputParams": [
        {
          "identifier": "currentTemp",
          "dataType": { "type": "float" }
        }
      ]
    }
  ]
}
```

这个 JSON 定义存在 MySQL 中（`product_thing_model` 表，用 JSON 字段存储），同时缓存到 Redis 里，Broker 收到设备消息时直接从缓存读取物模型做校验，不走数据库。

#### 4. 物模型在数据链路中的作用

物模型不只是一个静态的定义文件，它在整个数据链路中都在发挥作用：

**设备上报时（上行）：**
```
设备 PUBLISH 消息 → Broker 收到 → 根据 Topic 中的 productKey 查物模型定义
→ 校验 payload 中的字段是否符合物模型（类型对不对、值在不在范围内）
→ 校验通过：更新设备影子、写入时序数据库、推送到规则引擎
→ 校验失败：丢弃消息，记录异常日志
```

**指令下发时（下行）：**
```
APP 调用"设置属性"API → 平台根据物模型校验参数合法性
→ 校验通过：组装 MQTT 消息（Topic + JSON payload），通过 MqttEndpoint 下发
→ 校验失败：直接返回错误，不下发
```

**规则引擎中：**
- 规则条件可以直接引用物模型的属性标识符，比如 `temperature > 50`
- 规则引擎根据物模型知道 `temperature` 是 float 类型，做类型安全的比较

#### 5. 物模型的管理

**产品级定义，设备级继承：**
- 物模型是定义在产品（Product）级别的，同一个产品下的所有设备共享同一套物模型
- 比如"温湿度传感器"这个产品定义了 temperature、humidity 两个属性，那这个产品下的每台设备都有这两个属性
- 这样做的好处是：新增一台设备不需要单独定义物模型，只要指定它属于哪个产品就行

**版本管理：**
- 物模型支持版本化，修改物模型定义时生成新版本
- 已发布的版本不可修改（防止影响线上设备），只能创建新版本
- 设备固件升级后可以切换到新版本的物模型

**可视化编辑：**
- 管理后台提供了可视化的物模型编辑器，租户可以通过拖拽的方式定义属性、服务、事件
- 也支持直接导入 JSON 定义（方便开发者批量配置）

#### 6. 物模型的核心价值

总结一下物模型在我们平台中的核心价值：

| 层面 | 没有物模型 | 有物模型 |
|------|-----------|---------|
| 设备接入 | 每种设备写一套解析逻辑 | 统一按物模型定义解析，新设备只需定义物模型 |
| 数据校验 | 应用层各自校验，容易遗漏 | Broker 层统一校验，不合规数据直接拦截 |
| 业务开发 | 面向具体设备编程，耦合严重 | 面向物模型编程，业务代码和设备解耦 |
| 规则引擎 | 规则条件要硬编码设备字段 | 规则条件引用物模型标识符，通用且类型安全 |
| 设备影子 | 影子结构不统一，难以维护 | 影子结构由物模型驱动，自动保持一致 |

一句话概括：**物模型是整个 IoT 平台的"数据契约"，上层所有模块都基于这个契约来工作，它是平台能支撑异构设备统一管理的基础。**

## Q6: 多协议适配怎么实现的？屏蔽协议差异的核心设计？

> 💡 **与 02-01 Q3 的关系**：02-01 Q3 从架构选型角度概述了"Netty 在项目中怎么用的"，本题是多协议适配层的**完整设计细节**，包含接口定义、代码实现和扩展机制。面试中如果从 Q3 被追问到协议适配细节，直接切到这里的内容。

准备要点：
- 设计模式：适配器模式 + 策略模式
- 定义统一的 DeviceConnector 接口：connect()、disconnect()、send()、onMessage()
- 每种协议实现一个适配器：MqttConnector、HttpConnector、CoapConnector
- 协议识别：设备注册时声明协议类型，或通过端口区分（MQTT用1883，HTTP用8080，CoAP用5683）
- 消息标准化：不同协议的原始消息统一转换为内部标准消息格式（包含设备ID、消息类型、payload）
- SDK设计：设备厂商引入SDK后，只需实现几个回调方法，SDK内部处理协议细节

### 参考回答

#### 1. 问题背景

我们平台要对接的设备种类很多，不同厂商、不同品类的设备使用的通信协议不一样：

- 大部分 IoT 设备走 **MQTT**（轻量、长连接、适合弱网）
- 一些资源受限的传感器走 **CoAP**（基于 UDP，更省电）
- 部分老旧设备或第三方平台对接走 **HTTP**（简单通用，但不支持服务端主动推送）
- 还有一些厂商的私有协议走 **TCP 自定义报文**

如果每种协议都在业务层单独处理，代码会非常混乱。所以我们在接入层做了一个**协议适配层**，把协议差异屏蔽掉，上层业务只面向统一的消息格式编程。

#### 2. 整体架构分层

```
┌─────────────────────────────────────────────────┐
│                   业务层                          │
│   （规则引擎、设备影子、物模型校验、消息路由）       │
│         只面向 DeviceMessage 编程                  │
├─────────────────────────────────────────────────┤
│                 协议适配层                         │
│   ┌──────────┬──────────┬──────────┬──────────┐ │
│   │  MQTT    │  CoAP    │  HTTP    │  TCP     │ │
│   │ Adapter  │ Adapter  │ Adapter  │ Adapter  │ │
│   └──────────┴──────────┴──────────┴──────────┘ │
├─────────────────────────────────────────────────┤
│                 网络传输层                         │
│   ┌──────────┬──────────┬──────────┬──────────┐ │
│   │ Vert.x   │Californium│ WebFlux │  Netty   │ │
│   │ MQTT     │ CoAP     │ HTTP    │  TCP     │ │
│   │ Server   │ Server   │ Server  │  Server  │ │
│   └──────────┴──────────┴──────────┴──────────┘ │
│        :1883      :5683      :8080      :9000    │
└─────────────────────────────────────────────────┘
```

每种协议监听不同的端口，设备连接到对应端口后，由对应的 Adapter 把协议报文转换为统一的内部消息格式，再交给业务层处理。

#### 3. 核心设计：统一消息模型

不管设备走什么协议，最终都要转换成同一个 `DeviceMessage` 对象：

```java
public class DeviceMessage {
    private String deviceId;        // 设备唯一标识
    private String productKey;      // 产品标识（关联物模型）
    private MessageType type;       // 消息类型：PROPERTY_REPORT / SERVICE_REPLY / EVENT_REPORT
    private String identifier;      // 属性/服务/事件的标识符（对应物模型定义）
    private JsonObject payload;     // 消息体（统一用 JSON）
    private long timestamp;         // 消息时间戳
    private Map<String, String> headers;  // 扩展头（协议类型、QoS等元信息）
}
```

这个 `DeviceMessage` 就是协议适配层和业务层之间的"契约"。业务层拿到的永远是这个对象，不需要知道它原来是 MQTT 报文还是 HTTP 请求。

#### 4. 协议适配器的设计

用的是**策略模式 + 适配器模式**的组合。

**定义统一接口：**

```java
public interface ProtocolAdapter {
    
    /** 协议类型标识 */
    String getProtocol();
    
    /** 启动协议服务端（监听端口） */
    void start();
    
    /** 上行：将协议原始报文解码为统一的 DeviceMessage */
    DeviceMessage decode(Object rawMessage);
    
    /** 下行：将统一的 DeviceMessage 编码为协议报文，下发给设备 */
    void sendToDevice(String deviceId, DeviceMessage message);
    
    /** 停止协议服务端 */
    void stop();
}
```

**MQTT 适配器实现（核心协议）：**

```java
public class MqttProtocolAdapter implements ProtocolAdapter {
    
    private final MqttServer mqttServer;
    private final ConcurrentMap<String, MqttEndpoint> endpointMap;
    
    @Override
    public String getProtocol() { return "MQTT"; }
    
    @Override
    public void start() {
        mqttServer.endpointHandler(endpoint -> {
            // 鉴权、维护连接映射...
            endpoint.publishHandler(mqttMessage -> {
                // 将 MQTT 报文转换为统一的 DeviceMessage
                DeviceMessage msg = decode(mqttMessage);
                // 交给业务层处理
                messageDispatcher.dispatch(msg);
            });
        }).listen(1883);
    }
    
    @Override
    public DeviceMessage decode(Object rawMessage) {
        MqttPublishMessage mqtt = (MqttPublishMessage) rawMessage;
        // 解析 Topic: device/{productKey}/{deviceId}/property/report
        String[] topicParts = mqtt.topicName().split("/");
        return DeviceMessage.builder()
            .productKey(topicParts[1])
            .deviceId(topicParts[2])
            .type(resolveType(topicParts[3], topicParts[4]))  // property/report → PROPERTY_REPORT
            .payload(mqtt.payload().toJsonObject())
            .timestamp(System.currentTimeMillis())
            .header("protocol", "MQTT")
            .header("qos", String.valueOf(mqtt.qosLevel()))
            .build();
    }
    
    @Override
    public void sendToDevice(String deviceId, DeviceMessage message) {
        MqttEndpoint endpoint = endpointMap.get(deviceId);
        if (endpoint != null && endpoint.isConnected()) {
            String topic = buildTopic(message);  // 根据消息类型组装 Topic
            endpoint.publish(topic, message.getPayload().toBuffer(), 
                           MqttQoS.AT_LEAST_ONCE, false, false);
        }
    }
}
```

**HTTP 适配器实现（对接第三方平台或老旧设备）：**

```java
public class HttpProtocolAdapter implements ProtocolAdapter {
    
    @Override
    public String getProtocol() { return "HTTP"; }
    
    @Override
    public void start() {
        Router router = Router.router(vertx);
        // POST /api/device/{productKey}/{deviceId}/property/report
        router.post("/api/device/:productKey/:deviceId/:category/:action")
            .handler(ctx -> {
                DeviceMessage msg = decode(ctx);
                messageDispatcher.dispatch(msg);
                ctx.response().setStatusCode(200).end();
            });
        vertx.createHttpServer().requestHandler(router).listen(8080);
    }
    
    @Override
    public DeviceMessage decode(Object rawMessage) {
        RoutingContext ctx = (RoutingContext) rawMessage;
        return DeviceMessage.builder()
            .productKey(ctx.pathParam("productKey"))
            .deviceId(ctx.pathParam("deviceId"))
            .type(resolveType(ctx.pathParam("category"), ctx.pathParam("action")))
            .payload(ctx.body().asJsonObject())
            .timestamp(System.currentTimeMillis())
            .header("protocol", "HTTP")
            .build();
    }
    
    @Override
    public void sendToDevice(String deviceId, DeviceMessage message) {
        // HTTP 是短连接，无法主动推送
        // 方案：将消息写入待拉取队列，设备下次轮询时获取
        offlineMessageQueue.enqueue(deviceId, message);
    }
}
```

#### 5. 协议路由与注册

所有适配器通过一个注册中心统一管理：

```java
public class ProtocolAdapterRegistry {
    
    private final Map<String, ProtocolAdapter> adapters = new HashMap<>();
    
    /** 注册适配器（启动时自动扫描） */
    public void register(ProtocolAdapter adapter) {
        adapters.put(adapter.getProtocol(), adapter);
    }
    
    /** 根据协议类型获取适配器（下发消息时用） */
    public ProtocolAdapter getAdapter(String protocol) {
        return adapters.get(protocol);
    }
    
    /** 启动所有协议服务端 */
    public void startAll() {
        adapters.values().forEach(ProtocolAdapter::start);
    }
}
```

设备注册到平台时会声明自己使用的协议类型，这个信息存在设备元数据中。当需要给设备下发消息时，先查设备的协议类型，再通过注册中心拿到对应的适配器来发送：

```java
public void sendCommand(String deviceId, DeviceMessage message) {
    DeviceInfo device = deviceRegistry.get(deviceId);
    ProtocolAdapter adapter = adapterRegistry.getAdapter(device.getProtocol());
    adapter.sendToDevice(deviceId, message);
}
```

#### 6. 新增协议的扩展流程

这套设计的好处是扩展性好。如果要接入一种新协议（比如 Modbus），只需要：

1. 实现 `ProtocolAdapter` 接口，写一个 `ModbusProtocolAdapter`
2. 在 `decode()` 中把 Modbus 报文转换为 `DeviceMessage`
3. 在 `sendToDevice()` 中把 `DeviceMessage` 转换为 Modbus 指令
4. 注册到 `ProtocolAdapterRegistry`

**不需要改任何业务层代码**——规则引擎、设备影子、物模型校验这些模块完全不感知新协议的存在，因为它们只和 `DeviceMessage` 打交道。

#### 7. 实际中的协议占比

在我们平台上，各协议的使用占比大致是：

| 协议 | 占比 | 典型设备 |
|------|------|---------|
| MQTT | ~85% | 摄像头、智能门锁、网关、大部分传感器 |
| HTTP | ~10% | 第三方平台对接、老旧设备 |
| CoAP | ~3% | 低功耗传感器（电池供电） |
| TCP 私有协议 | ~2% | 部分工业设备 |

MQTT 是绝对主力，所以我们在 MQTT 适配器上投入最多，其他协议的适配器相对简单。

## Q7: MQTT在IoT中的使用细节？

> 💡 **与 02-01 的关系**：02-01 Q1-追问① 从选型角度解释了"为什么不用 EMQX"，02-01 Q3-追问① 解释了"为什么选 Vert.x MQTT 而不是 Netty 裸写"。本题聚焦 MQTT 协议本身的**使用细节**——QoS 选择、Topic 设计、遗嘱消息、安全机制等。

准备要点：
- 为什么选MQTT：轻量级、低带宽、支持QoS、适合弱网环境
- QoS级别选择：QoS 0（最多一次）、QoS 1（至少一次）、QoS 2（恰好一次），你们用的哪个？为什么？
- Topic设计：如 `device/{deviceId}/property/report`、`device/{deviceId}/service/invoke`
- 遗嘱消息（Will Message）：设备异常断开时通知服务端更新在线状态
- 保留消息（Retained Message）：新订阅者能立即获取设备最新状态
- MQTT Broker选型：EMQ X？Mosquitto？自研？

### 参考回答

#### 1. 为什么选 MQTT

我们平台面向的是大量 IoT 设备，很多设备部署在弱网环境下（比如 2G/3G 网络、Wi-Fi 信号不稳定的场景），所以选择 MQTT 作为核心通信协议，主要基于几点考虑：

- **协议轻量**：MQTT 固定头部最小只有 2 字节，相比 HTTP 的头部动辄几百字节，在低带宽场景下优势明显
- **长连接 + 推送模型**：设备和 Broker 之间维持长连接，服务端可以主动下发指令，不需要设备轮询
- **内置 QoS 机制**：协议层面就支持消息可靠性保障，不需要应用层自己实现重传
- **遗嘱消息机制**：设备异常断开时 Broker 能自动发布遗嘱消息，方便我们做在线状态管理

#### 2. 基于 Vert.x MQTT 自研 Broker

> 选型原因（为什么不用 EMQX、为什么选 Vert.x MQTT 而不是 Netty 裸写）在 [02-01 Q1-追问①](02-01-架构选型类.md) 和 [02-01 Q3-追问①](02-01-架构选型类.md) 中已详细说明，这里聚焦实现层面。

**简化消息下发链路：**

自研 Broker 的一个核心优势是——**设备端不需要主动订阅 Topic**。因为 Broker 本身就是我们的应用服务，它直接维护了设备连接（`MqttEndpoint`）的映射关系。当需要下发指令时，应用层根据设备 ID 直接找到对应的 `MqttEndpoint`，调用 `endpoint.publish()` 写入消息即可，不需要走"发布-订阅"的间接路径。

而如果用 EMQ X，下发消息必须依赖设备先订阅对应的 Topic，EMQ X 再根据订阅关系转发，多了一层间接性。

**核心实现：**

```java
// 启动 MQTT Server
MqttServer mqttServer = MqttServer.create(vertx, options);
mqttServer.endpointHandler(endpoint -> {
    // 1. 设备鉴权：校验 clientId、username、password，关联租户信息
    if (!authService.authenticate(endpoint.auth())) {
        endpoint.reject(MqttConnectReturnCode.CONNECTION_REFUSED_BAD_USER_NAME_OR_PASSWORD);
        return;
    }
    
    // 2. 接受连接，维护 deviceId → MqttEndpoint 映射
    endpoint.accept(false); // false 表示不使用 cleanSession
    String deviceId = endpoint.clientIdentifier();
    endpointMap.put(deviceId, endpoint);
    
    // 3. 处理设备上报消息
    endpoint.publishHandler(message -> {
        String topic = message.topicName();
        Buffer payload = message.payload();
        // 解析 Topic，路由到物模型处理逻辑
        messageRouter.route(deviceId, topic, payload);
    });
    
    // 4. 处理设备断开
    endpoint.closeHandler(v -> {
        endpointMap.remove(deviceId);
        deviceStatusService.offline(deviceId);
        // 触发离线事件，推送到规则引擎
        eventBus.publish("device.offline", deviceId);
    });
    
}).listen(1883);

// 下发消息：直接通过 endpoint 推送，不需要设备订阅
public void sendToDevice(String deviceId, String topic, Buffer payload) {
    MqttEndpoint endpoint = endpointMap.get(deviceId);
    if (endpoint != null && endpoint.isConnected()) {
        endpoint.publish(topic, payload, MqttQoS.AT_LEAST_ONCE, false, false);
    }
}
```

- 每个 MQTT 连接对应一个 `MqttEndpoint` 对象，Vert.x 自动管理其生命周期
- 维护一个 **deviceId → MqttEndpoint 的映射表**（ConcurrentHashMap），用于消息下发时直接定位设备连接
- 利用 Vert.x 的 **EventBus** 做内部事件分发（设备上下线、消息路由等），解耦 Broker 层和业务层
- 集群部署时，通过 Kafka 在节点间转发需要跨节点下发的消息；也可以利用 Vert.x 的集群模式通过 EventBus 跨节点通信

#### 3. QoS 级别选择

我们根据不同的业务场景选择不同的 QoS 级别：

| 场景 | QoS | 原因 |
|------|-----|------|
| 设备属性上报（如温度、湿度） | QoS 0 | 数据是周期性上报的，丢一两条影响不大，下一次上报会覆盖 |
| 服务调用下发（如重启、云台转动） | QoS 1 | 指令必须送达，但重复执行一次也可以接受（幂等设计） |
| 告警事件上报 | QoS 1 | 告警不能丢，服务端做去重处理 |

我们没有使用 QoS 2，因为 QoS 2 需要四次握手（PUBLISH → PUBREC → PUBREL → PUBCOMP），在大量设备并发的场景下对 Broker 的性能压力太大，而且通过应用层的幂等设计完全可以替代 QoS 2 的"恰好一次"语义。

#### 4. Topic 设计

我们的 Topic 设计遵循分层命名规范，结构清晰且便于权限控制：

```
# 属性上报（设备 → 平台）
device/{productKey}/{deviceId}/property/report

# 属性设置（平台 → 设备）
device/{productKey}/{deviceId}/property/set

# 服务调用（平台 → 设备）
device/{productKey}/{deviceId}/service/invoke/{serviceId}

# 服务响应（设备 → 平台）
device/{productKey}/{deviceId}/service/reply/{serviceId}

# 事件上报（设备 → 平台）
device/{productKey}/{deviceId}/event/report/{eventId}
```

Topic 中包含 `productKey` 是为了在 Broker 层面快速定位产品对应的物模型定义，方便做数据校验和解析。

#### 5. 遗嘱消息与连接管理

设备在 CONNECT 报文中可以携带遗嘱消息，Vert.x MQTT 通过 `endpoint.will()` 可以直接获取遗嘱配置。但由于我们是自研 Broker，对连接断开的处理比标准遗嘱机制更灵活：

- 通过 `endpoint.closeHandler()` 监听连接断开事件，不需要真的"发布"一条遗嘱消息到某个 Topic
- 连接断开后直接执行业务逻辑：更新设备在线状态为离线、记录离线时间
- 通过 EventBus 发布离线事件，触发规则引擎的告警联动
- 心跳检测：利用 MQTT 协议自带的 Keep Alive 机制，`MqttEndpoint` 会自动处理 PINGREQ/PINGRESP，超时未收到心跳则触发 `closeHandler`

#### 6. 安全机制

- **连接鉴权**：设备连接时通过 `endpoint.auth()` 获取 username/password，校验设备 Token 或证书
- **TLS 加密**：通过 `MqttServerOptions` 配置 SSL/TLS，生产环境使用 TLS 1.2 加密传输，端口 8883
- **Topic 权限控制**：在 `publishHandler` 中校验设备只能发布属于自己的 Topic（根据 clientId 和 Topic 中的 deviceId 匹配），防止越权
- **租户隔离**：Topic 中的 productKey 关联到租户，Broker 在处理消息时校验租户归属

## Q8: 设备掉线重连怎么处理？

> 💡 **与 02-01 Q4 的关系**：02-01 Q4 从集群高可用角度提到了"节点宕机后设备指数退避重连"，本题是掉线重连的**完整处理流程**——检测机制、服务端处理、客户端重连策略、状态恢复。

准备要点：
- 心跳机制：MQTT 用协议自带的 Keep Alive（PINGREQ/PINGRESP），TCP 私有协议用 Netty 的 IdleStateHandler + 自定义心跳包
- 应用层巡检兜底：定时扫描连接映射表，清理僵尸连接（MQTT 和 TCP 统一处理）
- 服务端处理：更新设备在线状态、触发离线事件、通知相关业务模块（协议无关）
- 客户端重连策略：指数退避（Exponential Backoff）+ 随机抖动，避免"惊群效应"
- 重连后的状态恢复：重新鉴权（MQTT 走 CONNECT，TCP 走鉴权包）、上报最新状态、离线消息补发
- 异常场景：连接冲突（新连接踢旧连接）、频繁上下线（告警风暴抑制）

### 参考回答

设备掉线重连这个问题，我从三个阶段来说：**怎么发现掉线、掉线后服务端怎么处理、设备怎么重连恢复**。

#### 1. 掉线检测机制

我们的设备有走 MQTT 的，也有走私有 TCP 协议的，两种协议的心跳机制不一样，但检测思路是一致的：**协议层心跳 + 应用层巡检兜底**。

**MQTT 设备：Keep Alive 机制**

MQTT 协议本身就有心跳机制。设备在 CONNECT 报文中会声明一个 Keep Alive 时间间隔（我们默认要求设备设置为 **60 秒**）。设备需要在这个间隔内至少发送一次 PINGREQ 或其他报文，Broker 收到后回复 PINGRESP。

在 Vert.x MQTT 中，`MqttEndpoint` 会自动处理 PINGREQ/PINGRESP 的交互。如果超过 **1.5 倍 Keep Alive 时间**（即 90 秒）没有收到设备的任何报文，Vert.x 底层会判定连接超时，自动关闭连接并触发 `closeHandler`。这个 1.5 倍是 MQTT 协议规范建议的容忍系数，给弱网环境留一定的缓冲。

**TCP 私有协议设备：IdleStateHandler + 自定义心跳包**

私有 TCP 协议没有像 MQTT 那样内置心跳规范，需要我们自己在 Netty Pipeline 中实现。做法是在 Pipeline 最前面加一个 **IdleStateHandler**，配置读空闲超时时间（比如 90 秒）：

```java
pipeline.addLast(new IdleStateHandler(90, 0, 0, TimeUnit.SECONDS));
pipeline.addLast(new HeartbeatHandler());
```

`IdleStateHandler` 会在指定时间内没有收到任何数据时触发一个 `IdleStateEvent`，我们在自定义的 `HeartbeatHandler` 中捕获这个事件：

```java
public class HeartbeatHandler extends ChannelDuplexHandler {
    private int missedHeartbeats = 0;
    
    @Override
    public void userEventTriggered(ChannelHandlerContext ctx, Object evt) {
        if (evt instanceof IdleStateEvent) {
            missedHeartbeats++;
            if (missedHeartbeats >= 2) {
                // 连续2次读空闲（约3分钟无数据），判定离线，关闭连接
                ctx.close();
            } else {
                // 第1次空闲，可以主动下发一个心跳探测包（如果协议支持的话）
                ctx.writeAndFlush(HeartbeatPacket.PING);
            }
        }
    }
    
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        missedHeartbeats = 0; // 收到任何数据都重置计数
        ctx.fireChannelRead(msg);
    }
}
```

和 MQTT 的区别在于：MQTT 的心跳是协议规范定义好的（PINGREQ/PINGRESP），Vert.x 自动处理；TCP 私有协议的心跳包格式是我们和设备厂商约定的，有的厂商有自己的心跳包，有的没有，没有的就只能靠读空闲超时来判断。

**应用层巡检（通用兜底）**

不管是 MQTT 还是 TCP，都有可能出现 TCP 连接"半开"的情况（设备突然断电，FIN 包没发出来），协议层心跳可能要等一个完整的超时周期才能发现。所以我们额外做了一个**定时巡检任务**，每隔 2 分钟扫描一次连接映射表（MQTT 的 `endpointMap` 和 TCP 的 `channelMap`），检查每个连接的最后活跃时间。超过 3 分钟无活动的，主动关闭并清理资源。

```java
// 定时巡检任务（统一处理 MQTT 和 TCP 连接）
vertx.setPeriodic(2 * 60 * 1000, id -> {
    long now = System.currentTimeMillis();
    connectionManager.getAllConnections().forEach((deviceId, conn) -> {
        if (now - conn.getLastActiveTime() > 3 * 60 * 1000) {
            conn.close();
            connectionManager.remove(deviceId);
            deviceStatusService.offline(deviceId);
        }
    });
});
```

这里的 `connectionManager.getAllConnections()` 返回的是统一的 `DeviceConnection` 接口（见 Q6 的协议适配设计），不区分底层是 MQTT 还是 TCP。

#### 2. 掉线后服务端处理

设备掉线后（无论是心跳超时还是主动断开），服务端的处理流程是**协议无关的**——不管设备走的是 MQTT 还是 TCP，触发的都是统一的离线处理逻辑：

**① 更新设备状态**
- 从连接映射表中移除该设备（MQTT 移除 `MqttEndpoint`，TCP 移除 `Channel`，都通过 `DeviceConnection` 接口统一操作）
- 更新数据库中设备的在线状态为"离线"，记录离线时间戳
- 清除 Redis 中的路由记录（集群部署时，其他节点不再往这个节点转发该设备的消息）

**② 触发离线事件**
- 通过 EventBus 发布 `device.offline` 事件
- 规则引擎监听到离线事件后，根据用户配置的规则执行联动操作，比如：
  - 发送告警通知（短信、邮件、APP 推送）
  - 如果是关键设备（如安防摄像头），触发更高级别的告警
  - 记录离线日志，用于后续的设备稳定性分析

**③ 离线消息缓存**
- 设备离线期间，如果有需要下发的指令（比如用户在 APP 上操作了设备），不会直接丢弃
- 将待下发的消息写入 **Redis 的离线消息队列**（按 deviceId 分 key，设置 TTL 过期时间，比如 24 小时）
- 设备重新上线后，从队列中取出并依次下发

#### 3. 设备端重连策略

重连策略主要是嵌入式团队在设备固件中实现的，我们在 SDK 中提供了推荐方案：

**指数退避（Exponential Backoff）+ 随机抖动（Jitter）**

```
第1次重连：等待 1秒 + random(0~500ms)
第2次重连：等待 2秒 + random(0~500ms)
第3次重连：等待 4秒 + random(0~500ms)
第4次重连：等待 8秒 + random(0~500ms)
...
上限：最长等待 60秒
```

为什么要加随机抖动？假设某个机房网络闪断，几万台设备同时掉线。如果所有设备都在完全相同的时间点重连，会造成**"惊群效应"**——Broker 瞬间收到几万个 CONNECT 请求，可能直接被打垮。加了随机抖动后，重连请求会在一个时间窗口内分散开来，Broker 的压力就平滑多了。

#### 4. 重连后的状态恢复

设备重连成功后，需要完成以下恢复流程：

**① 重新鉴权**

MQTT 设备和 TCP 设备的鉴权方式不同：
- **MQTT 设备**：发送 CONNECT 报文，Broker 校验 clientId、username、password，鉴权通过后 `endpoint.accept()`
- **TCP 设备**：TCP 连接建立后，设备发送的第一个报文必须是鉴权包（包含 deviceId、token 或签名），鉴权 Handler 校验通过后才把连接注册到映射表中；如果第一个包不是鉴权包或者鉴权失败，直接关闭连接

两种协议鉴权通过后，后续操作是统一的：更新连接映射表、更新 Redis 路由表、更新设备在线状态为"在线"。

**② 状态同步**
- 设备重连后主动上报一次当前的完整属性状态（而不是增量），确保平台侧的设备影子数据是最新的
- 这一步很重要，因为设备离线期间可能发生了状态变化（比如温度变了），平台侧的缓存数据已经过时了

**③ 离线消息补发**
- 服务端检测到设备重新上线后，从 Redis 离线消息队列中取出该设备的待下发消息
- 按时间顺序依次下发：MQTT 设备使用 QoS 1 确保送达；TCP 设备通过 Channel 写出后等待应用层 ACK（私有协议一般会约定一个响应包来确认收到）
- 下发完成后清空该设备的离线消息队列

**④ 发布上线事件**
- 通过 EventBus 发布 `device.online` 事件
- 规则引擎可以根据配置执行联动，比如设备上线后自动下发最新的配置参数

#### 5. 异常场景处理

**旧连接未断、新连接进来（连接冲突）：**

同一个 deviceId 的设备，如果旧连接还没被检测到断开，新连接就来了（比如设备快速重启），两种协议的处理方式：

**MQTT 设备**：MQTT 协议规范本身就定义了这个行为——同一个 clientId 的新连接应该替换旧连接。我们在 `endpointHandler` 中实现：

```java
endpoint.accept(false);
String deviceId = endpoint.clientIdentifier();

// 踢掉旧连接
MqttEndpoint oldEndpoint = endpointMap.get(deviceId);
if (oldEndpoint != null && oldEndpoint.isConnected()) {
    oldEndpoint.close();
}
endpointMap.put(deviceId, endpoint);
```

**TCP 设备**：私有协议没有类似 MQTT 的 clientId 机制，设备身份是在鉴权包中携带的。鉴权 Handler 校验通过后，检查连接映射表中是否已有该 deviceId 的旧连接，有的话先关闭旧 Channel 再注册新的：

```java
public class AuthHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        AuthPacket authPacket = (AuthPacket) msg;
        if (authService.verify(authPacket)) {
            String deviceId = authPacket.getDeviceId();
            // 踢掉旧连接
            DeviceConnection oldConn = connectionManager.get(deviceId);
            if (oldConn != null && oldConn.isActive()) {
                oldConn.close();
            }
            // 注册新连接
            connectionManager.put(deviceId, new NettyDeviceConnection(ctx.channel()));
            // 移除鉴权Handler，后续报文直接走业务Handler
            ctx.pipeline().remove(this);
        } else {
            ctx.close(); // 鉴权失败，直接断开
        }
    }
}
```

逻辑是一样的——**新连接踢旧连接**，只是触发时机不同：MQTT 在 CONNECT 阶段，TCP 在鉴权包阶段。

**网络抖动导致频繁上下线：**

如果设备在短时间内反复上下线（比如 1 分钟内掉线 5 次以上），规则引擎会识别为"网络不稳定"，触发专门的告警，而不是每次上下线都发通知，避免告警风暴。

## Q8-追问: 设备影子是什么？你们怎么实现的？

### 参考回答

#### 1. 什么是设备影子

设备影子（Device Shadow）本质上是**设备在云端的一份状态缓存**，它解决的核心问题是：**设备不一定随时在线，但上层业务随时可能需要查询或控制设备**。

举个具体的例子：用户在 APP 上想看家里摄像头的当前状态（是否开启、分辨率设置等），如果没有设备影子，就必须实时去问设备——但设备可能离线了、可能在弱网环境下响应很慢。有了设备影子，APP 直接读云端缓存的状态就行，毫秒级返回。

再比如：用户想把空调温度设成 26 度，但空调当前离线。没有设备影子的话，这个指令就丢了。有了设备影子，期望状态会先写入影子，等设备上线后自动同步过去。

#### 2. 设备影子的数据结构

我们的设备影子用 JSON 文档表示，核心包含两部分：

```json
{
  "deviceId": "device_001",
  "state": {
    "reported": {
      "temperature": 25.6,
      "humidity": 60,
      "switch": true,
      "firmwareVersion": "1.2.3"
    },
    "desired": {
      "switch": false,
      "temperature": 26
    }
  },
  "metadata": {
    "reported": {
      "temperature": { "timestamp": 1713340800000 },
      "humidity": { "timestamp": 1713340800000 }
    },
    "desired": {
      "switch": { "timestamp": 1713341000000 }
    }
  },
  "version": 42,
  "timestamp": 1713341000000
}
```

- **reported**：设备上报的实际状态，由设备端更新。"我现在是什么状态"
- **desired**：应用端期望的目标状态，由业务侧更新。"我希望你变成什么状态"
- **metadata**：每个属性的最后更新时间，用于冲突判断
- **version**：乐观锁版本号，每次更新自增，防止并发写覆盖

#### 3. 设备影子的工作流程

**场景一：设备在线，APP 下发指令**

```
APP → 平台：设置 desired.switch = false
平台：更新影子的 desired 部分
平台 → 设备：通过 MqttEndpoint 直接下发 desired 变更（delta）
设备：执行关闭操作
设备 → 平台：上报 reported.switch = false
平台：更新影子的 reported 部分，清除 desired.switch（因为已经一致了）
```

**场景二：设备离线，APP 下发指令**

```
APP → 平台：设置 desired.temperature = 26
平台：更新影子的 desired 部分（此时 reported.temperature 还是 25.6）
设备上线后：
  平台：计算 delta（desired 和 reported 的差异），下发给设备
  设备：执行调温操作
  设备 → 平台：上报 reported.temperature = 26
  平台：更新 reported，清除 desired.temperature
```

**场景三：APP 查询设备状态**

```
APP → 平台：查询设备状态
平台：直接返回影子中的 reported 数据，不需要实时问设备
```

#### 4. 存储方案

设备影子的存储我们用的是 **Redis Hash + MySQL 持久化**，两层配合：

**Redis（热数据层）：**
- 每个设备一个 Hash Key：`shadow:{deviceId}`
- 字段包括 reported、desired、version 等
- 所有读写操作优先走 Redis，保证毫秒级响应
- 设备每次上报属性，更新 Redis 中的 reported 字段

**MySQL（持久化层）：**
- 影子数据异步写入 MySQL，作为持久化备份
- 表结构大致是：`device_shadow(device_id, reported_state JSON, desired_state JSON, version, updated_at)`
- Redis 挂了或者冷启动时，从 MySQL 加载恢复

为什么不只用 Redis？因为 Redis 是内存存储，设备量大了之后内存成本高，而且需要持久化兜底。为什么不只用 MySQL？因为设备属性上报频率高（可能每秒几千条），直接写 MySQL 扛不住，而且查询延迟也不满足 APP 端的体验要求。

#### 5. 版本控制与冲突处理

设备影子的并发写是一个需要处理的问题。比如 APP 在设置温度的同时，设备也在上报温度，两边同时写影子就可能冲突。

我们用**乐观锁 + 版本号**来解决：

- 影子文档有一个 `version` 字段，每次更新自增
- 写入时带上当前版本号，Redis 端用 Lua 脚本做 CAS（Compare And Swap）操作：

```lua
-- Redis Lua 脚本：原子性更新影子
local currentVersion = tonumber(redis.call('HGET', KEYS[1], 'version'))
local expectedVersion = tonumber(ARGV[1])
if currentVersion == expectedVersion then
    redis.call('HSET', KEYS[1], 'reported', ARGV[2])
    redis.call('HSET', KEYS[1], 'version', currentVersion + 1)
    return currentVersion + 1
else
    return -1  -- 版本冲突，返回失败
end
```

- 如果版本冲突，重新读取最新版本再重试（通常一两次就能成功）

#### 6. 和物模型的关系

设备影子不是独立存在的，它和物模型紧密关联：

- 影子中 reported/desired 的字段必须符合物模型定义（属性名、数据类型、取值范围）
- 设备上报数据时，Broker 先根据物模型做数据校验，校验通过才更新影子
- 物模型变更（比如新增了一个属性），影子结构自动扩展，不需要手动迁移

#### 7. 和 Q8 掉线重连的关系

设备影子在掉线重连场景中起到关键作用：

- **设备离线期间**：用户的操作指令写入影子的 desired 部分，不会丢失
- **设备重连后**：平台计算 desired 和 reported 的 delta，将差异部分下发给设备，实现状态同步
- **设备上报全量状态**：重连后设备上报完整的 reported 状态，平台更新影子，确保云端数据和设备实际状态一致

这比 Q8 中提到的"离线消息队列"更优雅——离线消息队列是按指令顺序回放，而设备影子是**只关心最终状态**。比如用户离线期间先设了 25 度又改成 26 度，影子只会保留 desired = 26，不需要把两条指令都下发。
