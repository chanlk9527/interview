## Q5: 物模型抽象层怎么设计的？

准备要点：
- 物模型（Thing Model）的概念：用统一的数据结构描述设备的属性、服务、事件
- 属性（Property）：设备状态，如温度、开关状态，支持读写
- 服务（Service）：设备能力，如重启、云台转动，支持调用
- 事件（Event）：设备主动上报，如告警、故障
- 数据格式：JSON Schema定义每个字段的类型、取值范围、单位
- 好处：上层业务代码只面向物模型编程，不关心底层是MQTT还是HTTP

## Q6: 多协议适配SDK怎么实现的？屏蔽协议差异的核心设计？

准备要点：
- 设计模式：适配器模式 + 策略模式
- 定义统一的 DeviceConnector 接口：connect()、disconnect()、send()、onMessage()
- 每种协议实现一个适配器：MqttConnector、HttpConnector、CoapConnector
- 协议识别：设备注册时声明协议类型，或通过端口区分（MQTT用1883，HTTP用8080，CoAP用5683）
- 消息标准化：不同协议的原始消息统一转换为内部标准消息格式（包含设备ID、消息类型、payload）
- SDK设计：设备厂商引入SDK后，只需实现几个回调方法，SDK内部处理协议细节

## Q7: MQTT在IoT中的使用细节？

准备要点：
- 为什么选MQTT：轻量级、低带宽、支持QoS、适合弱网环境
- QoS级别选择：QoS 0（最多一次）、QoS 1（至少一次）、QoS 2（恰好一次），你们用的哪个？为什么？
- Topic设计：如 `device/{deviceId}/property/report`、`device/{deviceId}/service/invoke`
- 遗嘱消息（Will Message）：设备异常断开时通知服务端更新在线状态
- 保留消息（Retained Message）：新订阅者能立即获取设备最新状态
- MQTT Broker选型：EMQ X？Mosquitto？自研？

### 参考回答

#### 1. 为什么选 MQTT

我们平台面向的是大量 IoT 设备，很多设备部署在弱网环境下（比如 2G/3G 网络、Wi-Fi 信号不稳定的场景），所以选择 MQTT 作为核心通信协议，主要基于几点考虑：

- **协议轻量**：MQTT 固定头部最小只有 2 字节，相比 HTTP 的头部动辄几百字节，在低带宽场景下优势明显
- **长连接 + 推送模型**：设备和 Broker 之间维持长连接，服务端可以主动下发指令，不需要设备轮询
- **内置 QoS 机制**：协议层面就支持消息可靠性保障，不需要应用层自己实现重传
- **遗嘱消息机制**：设备异常断开时 Broker 能自动发布遗嘱消息，方便我们做在线状态管理

#### 2. 基于 Vert.x MQTT 自研 Broker 的原因和实现

我们没有使用 EMQ X 或 Mosquitto 这类开源 Broker，而是基于 **Vert.x MQTT** 库自研了 MQTT Broker。

**为什么选 Vert.x MQTT：**

首先说为什么要自研——我们的平台需要在 Broker 层面直接处理物模型解析、设备鉴权、多租户隔离等业务逻辑。如果用 EMQ X，这些逻辑只能通过插件或 Webhook 回调来做，多了一层网络开销，而且 EMQ X 的插件要用 Erlang 写，团队没有这个技术栈。

在自研方案的选型上，我们选择 Vert.x MQTT 而不是直接用 Netty 裸写，原因是：

- **开发效率高**：Vert.x MQTT 已经封装好了 MQTT 协议的编解码和会话管理，提供了 `MqttServer` 和 `MqttEndpoint` 这样的高层 API，不需要自己去处理 MQTT 报文的拆包解包。直接通过 `endpoint.publishHandler()`、`endpoint.subscribeHandler()` 等回调就能处理各种 MQTT 报文，比用 Netty 的 `MqttDecoder`/`MqttEncoder` + 自定义 `ChannelHandler` 要简洁很多
- **和平台技术栈统一**：我们整个平台就是基于 Vert.x 生态构建的（WebFlux 底层也是 Netty，而 Vert.x 同样基于 Netty），用 Vert.x MQTT 可以直接复用 Vert.x 的 EventBus、Verticle 部署模型、集群管理等基础设施，不需要额外引入一套框架
- **天然非阻塞**：Vert.x 的编程模型本身就是事件驱动、非阻塞的，和 MQTT 的长连接场景天然契合。所有 IO 操作都在 EventLoop 线程上执行，不会出现线程阻塞的问题

**简化消息下发链路：**

自研 Broker 的一个核心优势是——**设备端不需要主动订阅 Topic**。因为 Broker 本身就是我们的应用服务，它直接维护了设备连接（`MqttEndpoint`）的映射关系。当需要下发指令时，应用层根据设备 ID 直接找到对应的 `MqttEndpoint`，调用 `endpoint.publish()` 写入消息即可，不需要走"发布-订阅"的间接路径。

而如果用 EMQ X，下发消息必须依赖设备先订阅对应的 Topic，EMQ X 再根据订阅关系转发，多了一层间接性。

**核心实现：**

```java
// 启动 MQTT Server
MqttServer mqttServer = MqttServer.create(vertx, options);
mqttServer.endpointHandler(endpoint -> {
    // 1. 设备鉴权：校验 clientId、username、password，关联租户信息
    if (!authService.authenticate(endpoint.auth())) {
        endpoint.reject(MqttConnectReturnCode.CONNECTION_REFUSED_BAD_USER_NAME_OR_PASSWORD);
        return;
    }
    
    // 2. 接受连接，维护 deviceId → MqttEndpoint 映射
    endpoint.accept(false); // false 表示不使用 cleanSession
    String deviceId = endpoint.clientIdentifier();
    endpointMap.put(deviceId, endpoint);
    
    // 3. 处理设备上报消息
    endpoint.publishHandler(message -> {
        String topic = message.topicName();
        Buffer payload = message.payload();
        // 解析 Topic，路由到物模型处理逻辑
        messageRouter.route(deviceId, topic, payload);
    });
    
    // 4. 处理设备断开
    endpoint.closeHandler(v -> {
        endpointMap.remove(deviceId);
        deviceStatusService.offline(deviceId);
        // 触发离线事件，推送到规则引擎
        eventBus.publish("device.offline", deviceId);
    });
    
}).listen(1883);

// 下发消息：直接通过 endpoint 推送，不需要设备订阅
public void sendToDevice(String deviceId, String topic, Buffer payload) {
    MqttEndpoint endpoint = endpointMap.get(deviceId);
    if (endpoint != null && endpoint.isConnected()) {
        endpoint.publish(topic, payload, MqttQoS.AT_LEAST_ONCE, false, false);
    }
}
```

- 每个 MQTT 连接对应一个 `MqttEndpoint` 对象，Vert.x 自动管理其生命周期
- 维护一个 **deviceId → MqttEndpoint 的映射表**（ConcurrentHashMap），用于消息下发时直接定位设备连接
- 利用 Vert.x 的 **EventBus** 做内部事件分发（设备上下线、消息路由等），解耦 Broker 层和业务层
- 集群部署时，通过 Kafka 在节点间转发需要跨节点下发的消息；也可以利用 Vert.x 的集群模式通过 EventBus 跨节点通信

#### 3. QoS 级别选择

我们根据不同的业务场景选择不同的 QoS 级别：

| 场景 | QoS | 原因 |
|------|-----|------|
| 设备属性上报（如温度、湿度） | QoS 0 | 数据是周期性上报的，丢一两条影响不大，下一次上报会覆盖 |
| 服务调用下发（如重启、云台转动） | QoS 1 | 指令必须送达，但重复执行一次也可以接受（幂等设计） |
| 告警事件上报 | QoS 1 | 告警不能丢，服务端做去重处理 |

我们没有使用 QoS 2，因为 QoS 2 需要四次握手（PUBLISH → PUBREC → PUBREL → PUBCOMP），在大量设备并发的场景下对 Broker 的性能压力太大，而且通过应用层的幂等设计完全可以替代 QoS 2 的"恰好一次"语义。

#### 4. Topic 设计

我们的 Topic 设计遵循分层命名规范，结构清晰且便于权限控制：

```
# 属性上报（设备 → 平台）
device/{productKey}/{deviceId}/property/report

# 属性设置（平台 → 设备）
device/{productKey}/{deviceId}/property/set

# 服务调用（平台 → 设备）
device/{productKey}/{deviceId}/service/invoke/{serviceId}

# 服务响应（设备 → 平台）
device/{productKey}/{deviceId}/service/reply/{serviceId}

# 事件上报（设备 → 平台）
device/{productKey}/{deviceId}/event/report/{eventId}
```

Topic 中包含 `productKey` 是为了在 Broker 层面快速定位产品对应的物模型定义，方便做数据校验和解析。

#### 5. 遗嘱消息与连接管理

设备在 CONNECT 报文中可以携带遗嘱消息，Vert.x MQTT 通过 `endpoint.will()` 可以直接获取遗嘱配置。但由于我们是自研 Broker，对连接断开的处理比标准遗嘱机制更灵活：

- 通过 `endpoint.closeHandler()` 监听连接断开事件，不需要真的"发布"一条遗嘱消息到某个 Topic
- 连接断开后直接执行业务逻辑：更新设备在线状态为离线、记录离线时间
- 通过 EventBus 发布离线事件，触发规则引擎的告警联动
- 心跳检测：利用 MQTT 协议自带的 Keep Alive 机制，`MqttEndpoint` 会自动处理 PINGREQ/PINGRESP，超时未收到心跳则触发 `closeHandler`

#### 6. 安全机制

- **连接鉴权**：设备连接时通过 `endpoint.auth()` 获取 username/password，校验设备 Token 或证书
- **TLS 加密**：通过 `MqttServerOptions` 配置 SSL/TLS，生产环境使用 TLS 1.2 加密传输，端口 8883
- **Topic 权限控制**：在 `publishHandler` 中校验设备只能发布属于自己的 Topic（根据 clientId 和 Topic 中的 deviceId 匹配），防止越权
- **租户隔离**：Topic 中的 productKey 关联到租户，Broker 在处理消息时校验租户归属

## Q8: 设备掉线重连怎么处理？

准备要点：
- 心跳机制：Netty的IdleStateHandler检测读空闲，超过阈值（如60秒无数据）判定设备离线
- 服务端处理：更新设备在线状态、触发离线事件、通知相关业务模块
- 客户端重连策略：指数退避（Exponential Backoff），避免大量设备同时重连造成"惊群效应"
- 重连后的状态恢复：设备重连后需要重新鉴权、上报最新状态、恢复订阅关系
- 需要补充：你们的心跳间隔是多少？掉线到重连的平均耗时？

### 参考回答

设备掉线重连这个问题，我从三个阶段来说：**怎么发现掉线、掉线后服务端怎么处理、设备怎么重连恢复**。

#### 1. 掉线检测机制

我们有两层检测机制，互为补充：

**第一层：MQTT Keep Alive 机制**

MQTT 协议本身就有心跳机制。设备在 CONNECT 报文中会声明一个 Keep Alive 时间间隔（我们默认要求设备设置为 **60 秒**）。设备需要在这个间隔内至少发送一次 PINGREQ 或其他报文，Broker 收到后回复 PINGRESP。

在 Vert.x MQTT 中，`MqttEndpoint` 会自动处理 PINGREQ/PINGRESP 的交互。如果超过 **1.5 倍 Keep Alive 时间**（即 90 秒）没有收到设备的任何报文，Vert.x 底层会判定连接超时，自动关闭连接并触发 `closeHandler`。这个 1.5 倍是 MQTT 协议规范建议的容忍系数，给弱网环境留一定的缓冲。

**第二层：应用层心跳巡检**

光靠 MQTT Keep Alive 有一个问题——如果 TCP 连接处于"半开"状态（比如设备突然断电，TCP FIN 包没发出来），服务端可能感知不到连接已经断了，要等到 Keep Alive 超时才能发现。

所以我们额外做了一个**定时巡检任务**，每隔 2 分钟扫描一次 `endpointMap`，检查每个连接的最后活跃时间。如果某个设备超过 3 分钟没有任何数据交互（包括心跳），主动关闭连接并清理资源。这是一个兜底机制，确保不会有"僵尸连接"长期占用资源。

```java
// 定时巡检任务
vertx.setPeriodic(2 * 60 * 1000, id -> {
    long now = System.currentTimeMillis();
    endpointMap.forEach((deviceId, endpoint) -> {
        long lastActive = lastActiveTimeMap.getOrDefault(deviceId, 0L);
        if (now - lastActive > 3 * 60 * 1000) {
            // 超过3分钟无活动，主动关闭
            endpoint.close();
            endpointMap.remove(deviceId);
            deviceStatusService.offline(deviceId);
        }
    });
});
```

#### 2. 掉线后服务端处理

设备掉线后（无论是 Keep Alive 超时还是主动断开），`closeHandler` 被触发，服务端执行以下操作：

**① 更新设备状态**
- 从 `endpointMap` 中移除该设备的连接映射
- 更新数据库中设备的在线状态为"离线"，记录离线时间戳
- 清除 Redis 中的路由记录（集群部署时，其他节点不再往这个节点转发该设备的消息）

**② 触发离线事件**
- 通过 EventBus 发布 `device.offline` 事件
- 规则引擎监听到离线事件后，根据用户配置的规则执行联动操作，比如：
  - 发送告警通知（短信、邮件、APP 推送）
  - 如果是关键设备（如安防摄像头），触发更高级别的告警
  - 记录离线日志，用于后续的设备稳定性分析

**③ 离线消息缓存**
- 设备离线期间，如果有需要下发的指令（比如用户在 APP 上操作了设备），不会直接丢弃
- 将待下发的消息写入 **Redis 的离线消息队列**（按 deviceId 分 key，设置 TTL 过期时间，比如 24 小时）
- 设备重新上线后，从队列中取出并依次下发

#### 3. 设备端重连策略

重连策略主要是嵌入式团队在设备固件中实现的，我们在 SDK 中提供了推荐方案：

**指数退避（Exponential Backoff）+ 随机抖动（Jitter）**

```
第1次重连：等待 1秒 + random(0~500ms)
第2次重连：等待 2秒 + random(0~500ms)
第3次重连：等待 4秒 + random(0~500ms)
第4次重连：等待 8秒 + random(0~500ms)
...
上限：最长等待 60秒
```

为什么要加随机抖动？假设某个机房网络闪断，几万台设备同时掉线。如果所有设备都在完全相同的时间点重连，会造成**"惊群效应"**——Broker 瞬间收到几万个 CONNECT 请求，可能直接被打垮。加了随机抖动后，重连请求会在一个时间窗口内分散开来，Broker 的压力就平滑多了。

#### 4. 重连后的状态恢复

设备重连成功后，需要完成以下恢复流程：

**① 重新鉴权**
- 设备发送 CONNECT 报文，Broker 重新校验 clientId、username、password
- 鉴权通过后，更新 `endpointMap` 映射，更新 Redis 路由表
- 更新设备在线状态为"在线"，记录上线时间

**② 状态同步**
- 设备重连后主动上报一次当前的完整属性状态（而不是增量），确保平台侧的设备影子数据是最新的
- 这一步很重要，因为设备离线期间可能发生了状态变化（比如温度变了），平台侧的缓存数据已经过时了

**③ 离线消息补发**
- Broker 检测到设备重新上线后，从 Redis 离线消息队列中取出该设备的待下发消息
- 按时间顺序依次下发，每条消息使用 QoS 1 确保送达
- 下发完成后清空该设备的离线消息队列

**④ 发布上线事件**
- 通过 EventBus 发布 `device.online` 事件
- 规则引擎可以根据配置执行联动，比如设备上线后自动下发最新的配置参数

#### 5. 异常场景处理

**旧连接未断、新连接进来（clientId 冲突）：**

同一个 deviceId 的设备，如果旧连接还没被检测到断开，新的 CONNECT 请求就来了（比如设备快速重启），我们的处理逻辑是：
- 先关闭旧的 `MqttEndpoint`（踢掉旧连接）
- 再接受新连接，更新 `endpointMap`
- 这符合 MQTT 协议规范：同一个 clientId 的新连接应该替换旧连接

```java
endpoint.accept(false);
String deviceId = endpoint.clientIdentifier();

// 踢掉旧连接
MqttEndpoint oldEndpoint = endpointMap.get(deviceId);
if (oldEndpoint != null && oldEndpoint.isConnected()) {
    oldEndpoint.close();
}
endpointMap.put(deviceId, endpoint);
```

**网络抖动导致频繁上下线：**

如果设备在短时间内反复上下线（比如 1 分钟内掉线 5 次以上），规则引擎会识别为"网络不稳定"，触发专门的告警，而不是每次上下线都发通知，避免告警风暴。

## Q8-追问: 设备影子是什么？你们怎么实现的？

### 参考回答

#### 1. 什么是设备影子

设备影子（Device Shadow）本质上是**设备在云端的一份状态缓存**，它解决的核心问题是：**设备不一定随时在线，但上层业务随时可能需要查询或控制设备**。

举个具体的例子：用户在 APP 上想看家里摄像头的当前状态（是否开启、分辨率设置等），如果没有设备影子，就必须实时去问设备——但设备可能离线了、可能在弱网环境下响应很慢。有了设备影子，APP 直接读云端缓存的状态就行，毫秒级返回。

再比如：用户想把空调温度设成 26 度，但空调当前离线。没有设备影子的话，这个指令就丢了。有了设备影子，期望状态会先写入影子，等设备上线后自动同步过去。

#### 2. 设备影子的数据结构

我们的设备影子用 JSON 文档表示，核心包含两部分：

```json
{
  "deviceId": "device_001",
  "state": {
    "reported": {
      "temperature": 25.6,
      "humidity": 60,
      "switch": true,
      "firmwareVersion": "1.2.3"
    },
    "desired": {
      "switch": false,
      "temperature": 26
    }
  },
  "metadata": {
    "reported": {
      "temperature": { "timestamp": 1713340800000 },
      "humidity": { "timestamp": 1713340800000 }
    },
    "desired": {
      "switch": { "timestamp": 1713341000000 }
    }
  },
  "version": 42,
  "timestamp": 1713341000000
}
```

- **reported**：设备上报的实际状态，由设备端更新。"我现在是什么状态"
- **desired**：应用端期望的目标状态，由业务侧更新。"我希望你变成什么状态"
- **metadata**：每个属性的最后更新时间，用于冲突判断
- **version**：乐观锁版本号，每次更新自增，防止并发写覆盖

#### 3. 设备影子的工作流程

**场景一：设备在线，APP 下发指令**

```
APP → 平台：设置 desired.switch = false
平台：更新影子的 desired 部分
平台 → 设备：通过 MqttEndpoint 直接下发 desired 变更（delta）
设备：执行关闭操作
设备 → 平台：上报 reported.switch = false
平台：更新影子的 reported 部分，清除 desired.switch（因为已经一致了）
```

**场景二：设备离线，APP 下发指令**

```
APP → 平台：设置 desired.temperature = 26
平台：更新影子的 desired 部分（此时 reported.temperature 还是 25.6）
设备上线后：
  平台：计算 delta（desired 和 reported 的差异），下发给设备
  设备：执行调温操作
  设备 → 平台：上报 reported.temperature = 26
  平台：更新 reported，清除 desired.temperature
```

**场景三：APP 查询设备状态**

```
APP → 平台：查询设备状态
平台：直接返回影子中的 reported 数据，不需要实时问设备
```

#### 4. 存储方案

设备影子的存储我们用的是 **Redis Hash + MySQL 持久化**，两层配合：

**Redis（热数据层）：**
- 每个设备一个 Hash Key：`shadow:{deviceId}`
- 字段包括 reported、desired、version 等
- 所有读写操作优先走 Redis，保证毫秒级响应
- 设备每次上报属性，更新 Redis 中的 reported 字段

**MySQL（持久化层）：**
- 影子数据异步写入 MySQL，作为持久化备份
- 表结构大致是：`device_shadow(device_id, reported_state JSON, desired_state JSON, version, updated_at)`
- Redis 挂了或者冷启动时，从 MySQL 加载恢复

为什么不只用 Redis？因为 Redis 是内存存储，设备量大了之后内存成本高，而且需要持久化兜底。为什么不只用 MySQL？因为设备属性上报频率高（可能每秒几千条），直接写 MySQL 扛不住，而且查询延迟也不满足 APP 端的体验要求。

#### 5. 版本控制与冲突处理

设备影子的并发写是一个需要处理的问题。比如 APP 在设置温度的同时，设备也在上报温度，两边同时写影子就可能冲突。

我们用**乐观锁 + 版本号**来解决：

- 影子文档有一个 `version` 字段，每次更新自增
- 写入时带上当前版本号，Redis 端用 Lua 脚本做 CAS（Compare And Swap）操作：

```lua
-- Redis Lua 脚本：原子性更新影子
local currentVersion = tonumber(redis.call('HGET', KEYS[1], 'version'))
local expectedVersion = tonumber(ARGV[1])
if currentVersion == expectedVersion then
    redis.call('HSET', KEYS[1], 'reported', ARGV[2])
    redis.call('HSET', KEYS[1], 'version', currentVersion + 1)
    return currentVersion + 1
else
    return -1  -- 版本冲突，返回失败
end
```

- 如果版本冲突，重新读取最新版本再重试（通常一两次就能成功）

#### 6. 和物模型的关系

设备影子不是独立存在的，它和物模型紧密关联：

- 影子中 reported/desired 的字段必须符合物模型定义（属性名、数据类型、取值范围）
- 设备上报数据时，Broker 先根据物模型做数据校验，校验通过才更新影子
- 物模型变更（比如新增了一个属性），影子结构自动扩展，不需要手动迁移

#### 7. 和 Q8 掉线重连的关系

设备影子在掉线重连场景中起到关键作用：

- **设备离线期间**：用户的操作指令写入影子的 desired 部分，不会丢失
- **设备重连后**：平台计算 desired 和 reported 的 delta，将差异部分下发给设备，实现状态同步
- **设备上报全量状态**：重连后设备上报完整的 reported 状态，平台更新影子，确保云端数据和设备实际状态一致

这比 Q8 中提到的"离线消息队列"更优雅——离线消息队列是按指令顺序回放，而设备影子是**只关心最终状态**。比如用户离线期间先设了 25 度又改成 26 度，影子只会保留 desired = 26，不需要把两条指令都下发。
