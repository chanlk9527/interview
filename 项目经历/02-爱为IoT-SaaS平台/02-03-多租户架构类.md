# 三、多租户与数据权限架构类追问

---

## 设计背景

平台的权限体系分为两层：
1. **功能权限（RBAC）**：控制用户能做什么操作（菜单、按钮）
2. **数据权限（资源归属模型）**：控制用户能看到哪些数据

核心设计思想：
- 所有业务数据抽象为**资源**（设备、订单、告警等都是资源）
- 资源可以划拨给**用户**或**商户**
- 用户只能归属于**一个商户**
- 同一资源可以划拨给**不同的用户和商户**（支持资源共享）
- 如果用户是商户的**管理员**，则可以看到该商户下所有资源

## Q9: 你们的多租户数据隔离方案是怎么设计的？

**详细回答：**

我们平台的数据隔离没有采用传统的"每张表加tenant_id"的方案，而是设计了一套**资源归属模型**来实现。

### 1. 整体思路

我们把所有业务数据抽象为"资源"这个概念。设备是资源、订单是资源、告警记录也是资源。然后通过一张独立的**归属关系表**来描述"这个资源属于谁"。

系统中有三个核心角色：
- **平台管理员**：我们运营方，能看到所有数据
- **商户（Merchant）**：我们的B端客户，每个商户是一个独立的业务主体
- **用户（User）**：商户下的具体操作人员，每个用户必须且只能归属于一个商户

### 2. 核心模型设计

```
商户（Merchant）
  └── 用户（User）：归属于商户，角色为 ADMIN 或 MEMBER
  
资源（Resource）：业务数据的抽象
  └── 归属关系（Resource Bindling）：描述资源划拨给了谁
       - 可以划拨给商户
       - 可以划拨给用户
       - 同一资源可以有多条归属记录（支持共享）
```

数据可见性规则：

| 角色 | 可见范围 | 说明 |
|------|----------|------|
| 平台管理员 | 全部资源 | 运营侧全量可见 |
| 商户管理员 | 商户下所有资源 | 包括划拨给商户的 + 划拨给商户下所有用户的 |
| 普通用户 | 仅自己名下的资源 | 只能看到直接划拨给自己的 |

举个具体例子：商户A有3个用户（管理员张三、普通用户李四、普通用户王五）。平台把10台设备划拨给商户A，管理员张三再把其中3台划拨给李四、2台划拨给王五。那么：
- 张三（管理员）能看到全部10台
- 李四只能看到自己名下的3台
- 王五只能看到自己名下的2台

### 3. 为什么不用传统的tenant_id方案？

传统方案是每张业务表加一个`tenant_id`字段，通过拦截器自动注入`WHERE tenant_id = ?`来隔离。我们评估后没有采用，主要原因：

**（1）资源共享需求**

我们有共享设备的场景——同一台设备可能同时服务于多个商户。比如一台公共区域的摄像头，商户A和商户B都需要查看。tenant_id是一对一的，一条记录只能属于一个租户，无法表达"一个资源属于多个主体"。

**（2）归属关系动态变化**

设备的归属不是固定的。运营过程中经常发生：
- 设备从商户A转移到商户B（客户退租、设备调拨）
- 商户内部重新分配设备给不同用户
- 临时共享设备给合作商户

如果用tenant_id，每次转移都要改业务表的数据，侵入性太强。用独立的归属关系表，转移就是删一条记录加一条记录，业务表完全不动。

**（3）权限粒度不够**

tenant_id只能做到商户级别的隔离，但我们需要做到**用户级别**。同一个商户下，不同用户看到的设备是不一样的。如果用tenant_id，还得再加一套用户级别的过滤逻辑，不如统一用归属关系表一次性解决。

### 4. 归属关系表的核心设计

```sql
CREATE TABLE resource_bindling (
    id BIGINT PRIMARY KEY,
    resource_type VARCHAR(32),    -- 资源类型：DEVICE / ORDER / ALERT ...
    resource_biz_id BIGINT,       -- 业务资源ID（如设备ID）
    bindling_type VARCHAR(16),    -- 归属主体类型：USER / MERCHANT
    bindling_id BIGINT,           -- 归属主体ID
    bindling_time DATETIME,       -- 划拨时间
    bindling_source VARCHAR(32),  -- 划拨来源：PLATFORM_ASSIGN / PURCHASE / TRANSFER
    status TINYINT DEFAULT 1      -- 状态：1有效 0失效
);

-- 核心索引
CREATE INDEX idx_bindling_owner ON resource_bindling(bindling_type, bindling_id, resource_type);
CREATE INDEX idx_bindling_resource ON resource_bindling(resource_type, resource_biz_id);
CREATE UNIQUE INDEX uk_bindling ON resource_bindling(resource_type, resource_biz_id, bindling_type, bindling_id);
```

设计要点：
- `bindling_type` 用多态关联，一张表同时描述"划拨给用户"和"划拨给商户"
- 唯一索引防止重复划拨（同一资源不会重复划拨给同一主体）
- `bindling_source` 记录来源，方便审计追溯
- `status` 字段支持软删除，转移时标记失效而非物理删除，保留历史记录

### 5. 数据查询时怎么过滤？

通过MyBatis拦截器自动改写SQL，根据当前用户角色注入不同的过滤条件：

```java
@Intercepts({@Signature(type = Executor.class, method = "query", ...)})
public class DataScopeInterceptor implements Interceptor {
    
    @Override
    public Object intercept(Invocation invocation) {
        UserContext ctx = UserContextHolder.get();
        
        if (ctx.isPlatformAdmin()) {
            // 平台管理员：不加过滤，直接放行
            return invocation.proceed();
        }
        
        // 获取原始SQL并改写
        BoundSql boundSql = ...;
        String originalSql = boundSql.getSql();
        
        String scopeSql;
        if (ctx.isMerchantAdmin()) {
            // 商户管理员：查商户下所有资源
            scopeSql = wrapWithMerchantScope(originalSql, ctx.getMerchantId());
        } else {
            // 普通用户：只查自己名下资源
            scopeSql = wrapWithUserScope(originalSql, ctx.getUserId());
        }
        
        // 替换SQL执行
        ...
    }
}
```

这样业务代码完全不需要关心数据权限，拦截器统一处理，既降低了开发心智负担，也避免了遗漏过滤条件导致数据泄露。

### 6. 这个方案的优缺点

**优点：**
- 资源和主体完全解耦，支持灵活的共享和转移
- 权限粒度细，能做到用户级别
- 业务表干净，不需要冗余tenant_id字段
- 归属变更不影响业务数据

**缺点：**
- 查询多一次JOIN，性能有一定损耗（通过索引和缓存缓解）
- 归属关系表数据量大（资源数 × 平均归属数），需要定期归档失效记录
- 模型理解成本比tenant_id高，新人上手需要时间

## Q10: 资源归属模型的表结构怎么设计的？

**详细回答：**

### 1. 先说整体ER关系

用一句话概括：**商户包含用户，资源通过归属关系表挂到用户或商户上**。

```
┌─────────────┐         ┌─────────────┐
│   商户表     │ 1    N  │   用户表     │
│  merchant   │────────→│    user     │
└─────────────┘         └─────────────┘
       ↑                       ↑
       │ bindling_type=        │ bindling_type=
       │ MERCHANT              │ USER
       │                       │
┌──────┴───────────────────────┴──────┐
│         资源归属关系表                 │
│       resource_bindling              │
└──────────────────┬──────────────────┘
                   │
                   │ resource_biz_id
                   ↓
┌─────────────────────────────────────┐
│         业务资源表（设备/订单/...）    │
│    device / order / alert / ...     │
└─────────────────────────────────────┘
```

关键点：归属关系表是中间层，它把"资源"和"主体（用户或商户）"连接起来，而不是在业务表上直接加外键。

### 2. 核心表结构

```sql
-- ① 商户表
CREATE TABLE merchant (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(128) NOT NULL,
    status TINYINT DEFAULT 1,       -- 1启用 0停用
    contact_name VARCHAR(64),
    contact_phone VARCHAR(20),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- ② 用户表（每个用户必须归属一个商户）
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    merchant_id BIGINT NOT NULL,     -- 归属商户，不可为空
    username VARCHAR(64) NOT NULL,
    role_type VARCHAR(16) NOT NULL,  -- ADMIN: 商户管理员 / MEMBER: 普通成员
    status TINYINT DEFAULT 1,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_merchant (merchant_id)
);

-- ③ 资源归属关系表（核心！）
CREATE TABLE resource_bindling (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    resource_type VARCHAR(32) NOT NULL,   -- 资源类型：DEVICE / ORDER / ALERT
    resource_biz_id BIGINT NOT NULL,      -- 业务资源ID（如设备表的id）
    bindling_type VARCHAR(16) NOT NULL,   -- 归属主体类型：USER / MERCHANT
    bindling_id BIGINT NOT NULL,          -- 归属主体ID（user_id 或 merchant_id）
    bindling_source VARCHAR(32),          -- 来源：PLATFORM_ASSIGN / PURCHASE / TRANSFER
    bindling_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    status TINYINT DEFAULT 1,             -- 1有效 0失效（软删除）

    -- 核心索引
    INDEX idx_owner (bindling_type, bindling_id, resource_type),
    INDEX idx_resource (resource_type, resource_biz_id),
    UNIQUE INDEX uk_bindling (resource_type, resource_biz_id, bindling_type, bindling_id)
);
```

### 3. 为什么这样设计——逐字段解释

| 字段 | 为什么需要 | 面试怎么说 |
|------|-----------|-----------|
| `resource_type` | 区分不同业务资源，一张归属表管所有类型 | "我们没有给每种资源单独建归属表，统一用resource_type区分，减少表数量" |
| `resource_biz_id` | 指向具体业务表的主键 | "这里存的是设备表、订单表的id，通过resource_type+resource_biz_id定位唯一资源" |
| `bindling_type` | 多态关联，一个字段同时支持挂用户和商户 | "用type+id的组合实现多态，避免建两张关联表" |
| `bindling_id` | 配合bindling_type指向user或merchant | 同上 |
| `bindling_source` | 记录这条归属是怎么来的 | "方便审计追溯，比如区分是平台分配的还是购买的还是转移来的" |
| `status` | 软删除，保留历史 | "转移设备时旧记录标记失效，不物理删除，保留完整的划拨历史" |

### 4. 索引设计思路

三个索引分别服务三种查询场景：

```sql
-- 场景1：查"某用户/商户下所有设备"（最高频）
-- 走 idx_owner (bindling_type, bindling_id, resource_type)
SELECT resource_biz_id FROM resource_bindling
WHERE bindling_type = 'USER' AND bindling_id = 1001 AND resource_type = 'DEVICE';

-- 场景2：查"某设备归属于谁"（划拨/转移时用）
-- 走 idx_resource (resource_type, resource_biz_id)
SELECT bindling_type, bindling_id FROM resource_bindling
WHERE resource_type = 'DEVICE' AND resource_biz_id = 5001;

-- 场景3：唯一索引防重复划拨
-- uk_bindling 保证同一资源不会重复划拨给同一主体
INSERT INTO resource_bindling (...) VALUES (...)
-- 如果重复会报唯一键冲突，业务层捕获后提示"已划拨"
```

### 5. 一个完整的业务场景走一遍

假设：平台把设备D001划拨给商户M100，商户管理员再把D001分配给用户U2001。

```sql
-- Step1: 平台划拨设备给商户
INSERT INTO resource_bindling 
(resource_type, resource_biz_id, bindling_type, bindling_id, bindling_source)
VALUES ('DEVICE', 'D001', 'MERCHANT', 100, 'PLATFORM_ASSIGN');

-- Step2: 商户管理员分配给用户
INSERT INTO resource_bindling 
(resource_type, resource_biz_id, bindling_type, bindling_id, bindling_source)
VALUES ('DEVICE', 'D001', 'USER', 2001, 'MERCHANT_ASSIGN');

-- 此时 resource_bindling 表中 D001 有两条记录：
-- 一条挂在商户M100下，一条挂在用户U2001下

-- 查询效果：
-- 商户管理员（M100的ADMIN）→ 能看到D001（因为商户下有这条记录）
-- 用户U2001 → 能看到D001（因为直接划拨给了他）
-- 用户U2002（同商户的另一个普通用户）→ 看不到D001（没有归属记录）
```

### 6. 追问：查询性能怎么保证？

**索引层面：**
- 最高频的"查我的资源列表"走联合索引，等值查询效率O(logN)
- 唯一索引既防重复又能加速精确查找

**缓存层面：**
- 热点商户的资源ID列表缓存在Redis（Key: `merchant:{id}:devices`，Value: Set<deviceId>）
- 归属变更时通过消息队列异步失效缓存
- 普通用户资源少（通常几十台），不需要缓存，直接查DB

**查询策略：**
- 先查归属表拿到资源ID列表，再用 `IN (ids)` 批量查业务表
- 避免大表JOIN，拆成两次查询反而更快（归属表走索引很快，业务表走主键也很快）
- 分页场景用游标分页（`WHERE id > lastId LIMIT 20`），禁止深度OFFSET

## Q11: RBAC和数据权限是怎么配合工作的？

**详细回答：**

### 1. 先说清楚两者的区别

我们的权限体系分两层，解决的是两个不同的问题：

| | RBAC（功能权限） | 数据权限（资源归属） |
|--|--|--|
| 解决什么问题 | 你**能不能**做这个操作 | 你做了操作之后**能看到哪些数据** |
| 举例 | 能不能访问"设备管理"页面 | 进了页面之后能看到哪些设备 |
| 控制粒度 | 接口级、菜单级、按钮级 | 数据行级 |
| 实现方式 | 网关/注解拦截 | MyBatis拦截器改写SQL |
| 数据模型 | 用户-角色-权限五表 | 资源归属关系表 |

一句话总结：**RBAC管"能不能进门"，数据权限管"进门后能看到什么"**。

### 2. 一个请求的完整鉴权流程

```
用户发起请求：GET /api/device/list
    │
    ▼
┌─────────────────────────────────┐
│ ① 网关层：Token校验              │
│   - 验证JWT签名和有效期           │
│   - 解析出 userId, merchantId,   │
│     roleType 放入请求头转发       │
└───────────────┬─────────────────┘
                ▼
┌─────────────────────────────────┐
│ ② RBAC层：功能权限校验            │
│   - 从Redis取用户权限列表         │
│   - 判断是否有 device:list 权限   │
│   - 没有 → 直接返回403           │
└───────────────┬─────────────────┘
                ▼
┌─────────────────────────────────┐
│ ③ Controller：正常业务逻辑       │
│   - 调用 deviceService.list()   │
└───────────────┬─────────────────┘
                ▼
┌─────────────────────────────────┐
│ ④ 数据权限层：MyBatis拦截器       │
│   - 从ThreadLocal取用户上下文     │
│   - 根据角色自动改写SQL           │
│   - 平台管理员：不加过滤          │
│   - 商户管理员：过滤商户下所有资源  │
│   - 普通用户：只过滤自己名下资源   │
└───────────────┬─────────────────┘
                ▼
        返回过滤后的数据
```

### 3. RBAC部分的实现

**表结构（经典五表）：**

```sql
user           -- 用户表
role           -- 角色表（如：商户管理员、设备运维员、只读用户）
permission     -- 权限表（如：device:list, device:delete, order:create）
user_role      -- 用户-角色关联（多对多）
role_permission -- 角色-权限关联（多对多）
```

**校验流程：**

```java
// 接口上标注需要的权限
@PreAuthorize("hasPermission('device:list')")
@GetMapping("/api/device/list")
public Result<List<Device>> listDevices() { ... }

// 权限校验逻辑（简化版）
public boolean hasPermission(String requiredPermission) {
    Long userId = UserContextHolder.getUserId();
    // 从Redis获取用户权限集合（登录时已加载）
    Set<String> userPermissions = redisTemplate.opsForSet()
        .members("user:permissions:" + userId);
    return userPermissions.contains(requiredPermission);
}
```

**权限缓存策略：**
- 用户登录时，查库加载权限列表写入Redis，过期时间与Token一致
- 角色权限变更时，通过消息广播失效相关用户的缓存
- 下次请求时缓存未命中则重新加载

### 4. 数据权限部分的实现

**核心：MyBatis拦截器 + ThreadLocal上下文**

```java
/**
 * 数据权限拦截器 —— 自动改写SQL注入归属过滤条件
 */
@Intercepts({@Signature(type = Executor.class, method = "query", args = {...})})
public class DataScopeInterceptor implements Interceptor {

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 1. 获取当前用户上下文
        UserContext ctx = UserContextHolder.get();
        
        // 2. 平台管理员不做过滤
        if (ctx.isPlatformAdmin()) {
            return invocation.proceed();
        }
        
        // 3. 检查当前Mapper方法是否需要数据权限过滤
        //    通过自定义注解 @DataScope 标记
        MappedStatement ms = (MappedStatement) invocation.getArgs()[0];
        DataScope dataScope = getDataScopeAnnotation(ms);
        if (dataScope == null) {
            return invocation.proceed();
        }
        
        // 4. 根据角色构建过滤条件
        String originalSql = getBoundSql(invocation).getSql();
        String filteredSql;
        
        if (ctx.isMerchantAdmin()) {
            // 商户管理员：看商户下所有资源
            filteredSql = wrapWithMerchantScope(originalSql, dataScope, ctx);
        } else {
            // 普通用户：只看自己名下资源
            filteredSql = wrapWithUserScope(originalSql, dataScope, ctx);
        }
        
        // 5. 替换SQL执行
        replaceSql(invocation, filteredSql);
        return invocation.proceed();
    }
}
```

**改写后的SQL效果：**

```sql
-- 原始SQL（业务代码写的）
SELECT * FROM device WHERE status = 1

-- 普通用户改写后：
SELECT * FROM device d
WHERE d.status = 1
  AND d.id IN (
    SELECT resource_biz_id FROM resource_bindling
    WHERE resource_type = 'DEVICE'
      AND bindling_type = 'USER'
      AND bindling_id = 1001        -- 当前用户ID
      AND status = 1
  )

-- 商户管理员改写后：
SELECT * FROM device d
WHERE d.status = 1
  AND d.id IN (
    SELECT resource_biz_id FROM resource_bindling
    WHERE resource_type = 'DEVICE'
      AND status = 1
      AND (
        (bindling_type = 'MERCHANT' AND bindling_id = 100)   -- 划拨给商户的
        OR
        (bindling_type = 'USER' AND bindling_id IN           -- 划拨给商户下用户的
          (SELECT id FROM user WHERE merchant_id = 100))
      )
  )
```

### 5. 两者配合的关键设计点

**（1）解耦，互不依赖**

RBAC和数据权限是两个独立的模块，互不感知：
- RBAC不关心数据权限怎么过滤
- 数据权限不关心用户有没有功能权限
- 两者通过ThreadLocal中的UserContext共享用户身份信息

**（2）执行顺序有讲究**

必须先RBAC再数据权限，原因：
- 如果用户连接口权限都没有，直接403，不需要走到数据层
- 减少无效的数据库查询

**（3）注解控制哪些接口需要数据过滤**

```java
// 需要数据权限过滤的Mapper方法加注解
@DataScope(resourceType = "DEVICE", tableAlias = "d")
List<Device> selectDeviceList(@Param("query") DeviceQuery query);

// 不需要过滤的方法不加注解（如统计类、导出类）
List<DeviceStats> selectDeviceStats();
```

这样做的好处：不是所有查询都需要过滤，比如统计"平台总设备数"就不需要按用户过滤。

**（4）ThreadLocal上下文的生命周期**

```java
// 网关Filter中设置
public class UserContextFilter implements Filter {
    @Override
    public void doFilter(request, response, chain) {
        UserContext ctx = parseFromToken(request);
        UserContextHolder.set(ctx);  // 放入ThreadLocal
        try {
            chain.doFilter(request, response);
        } finally {
            UserContextHolder.clear();  // 请求结束必须清除，防止线程复用导致数据串
        }
    }
}
```

### 6. 面试话术建议

> "我们的权限分两层：RBAC控制功能权限，就是你能不能调这个接口；数据权限控制你调了接口之后能看到哪些数据。两者解耦，RBAC在网关/注解层做，数据权限在MyBatis拦截器层做。请求进来先过RBAC，没权限直接403；有权限的话进入业务逻辑，到数据层时拦截器根据用户角色自动改写SQL，商户管理员看商户下所有资源，普通用户只看自己名下的。业务代码完全不需要关心数据过滤，拦截器统一兜底。"

## Q12: 资源划拨和转移是怎么实现的？

**回答思路：**

资源划拨场景：
1. **平台分配**：平台管理员将设备划拨给商户
2. **商户内部分配**：商户管理员将设备划拨给旗下用户
3. **资源转移**：设备从A商户转移到B商户

实现方式：
```java
// 划拨资源给商户
public void bindlingToMerchant(Long resourceId, Long merchantId, String source) {
    ResourceBindling bindling = new ResourceBindling();
    bindling.setResourceId(resourceId);
    bindling.setBindlingType(BindlingType.MERCHANT);
    bindling.setBindlingId(merchantId);
    bindling.setBindlingSource(source);
    resourceBindlingMapper.insert(bindling);
    // 失效缓存
    cacheService.evictMerchantResources(merchantId);
}

// 资源转移（事务保证）
@Transactional
public void transferResource(Long resourceId, Long fromMerchantId, Long toMerchantId) {
    // 1. 删除原归属
    resourceBindlingMapper.deleteByResourceAndOwner(resourceId, BindlingType.MERCHANT, fromMerchantId);
    // 2. 同时删除原商户下用户的归属
    resourceBindlingMapper.deleteByResourceAndMerchantUsers(resourceId, fromMerchantId);
    // 3. 建立新归属
    bindlingToMerchant(resourceId, toMerchantId, "TRANSFER");
    // 4. 失效双方缓存
    cacheService.evictMerchantResources(fromMerchantId);
    cacheService.evictMerchantResources(toMerchantId);
}
```

**追问：并发划拨怎么处理？**

- 资源归属表对 `resource_id + bindling_type + bindling_id` 加唯一索引，防止重复划拨
- 转移操作加分布式锁（基于resource_id），防止同一资源被并发转移到不同商户

## Q13: 如何防止数据越权访问？

**回答思路：**

多层防护：

1. **接口层**：网关校验Token合法性，解析用户身份信息
2. **功能权限层**：RBAC校验用户是否有权调用该接口
3. **数据权限层**：MyBatis拦截器自动注入归属过滤条件，确保用户只能查到自己有权限的资源
4. **业务层兜底**：关键操作（如资源转移、删除）显式校验当前用户是否拥有该资源
5. **审计层**：记录所有资源访问和变更日志，定期审计异常访问模式

具体防护措施：
- 拦截器兜底：即使业务代码忘记加过滤条件，拦截器也会自动补上
- 接口入参校验：用户传入的resource_id，先查归属关系确认有权限再操作
- 日志审计：记录每次资源访问的user_id、resource_id、操作类型，异常告警

**追问：如果有人绕过拦截器直接写SQL呢？**

- 所有数据访问必须走MyBatis Mapper，禁止直接JDBC
- 代码审查时重点关注是否有绕过拦截器的写法
- 定期用SQL审计工具扫描，检查是否有不带归属过滤的查询语句
- 上线前安全测试：模拟跨商户/跨用户访问，验证数据隔离有效性

## Q14（补充）: 归属关系表数据量膨胀到百万级后，性能怎么优化？

> 背景说明：目前平台设备总量在几十万级别，归属关系表还没有遇到明显的性能瓶颈。这个问题是对未来扩展的思考，属于"我们意识到了但还没实际落地"的优化方向。

**回答思路：**

面试时可以这样说：目前我们几十万设备、归属关系表大概在百万行左右，查询性能还OK，靠索引和缓存撑住了。但如果业务继续增长到百万设备级别，归属关系表可能膨胀到千万行，我们内部讨论过几个优化方向：

### 1. 短期优化：缓存 + 索引调优

**现状已有的措施：**
- 联合索引 `(bindling_type, bindling_id, resource_type)` 覆盖"查某用户/商户下所有资源"的场景
- 热点商户的资源ID列表缓存在Redis，归属变更时主动失效

**可进一步优化：**
- 对大商户（设备数>1000）做资源ID的本地缓存（Caffeine），减少Redis网络开销
- 查询时只查资源ID列表，再用IN查询批量获取业务数据，避免大表JOIN
- 分页查询强制走游标分页（基于ID），禁止深度OFFSET分页

### 2. 中期优化：归属关系表分表

当单表超过2000万行时，考虑按 `resource_type` 做水平分表：

```
resource_bindling_device    -- 设备类资源的归属
resource_bindling_order     -- 订单类资源的归属
resource_bindling_alert     -- 告警类资源的归属
```

好处：
- 每张表数据量可控，索引效率高
- 不同资源类型的查询互不影响
- 实现简单，MyBatis拦截器根据resource_type路由到对应表即可

### 3. 中期优化：物化视图 / 宽表冗余

对于"商户管理员查商户下所有设备"这种高频查询，当前SQL需要JOIN + 子查询，数据量大了会慢。可以引入冗余宽表：

```sql
-- 商户资源汇总表（异步维护）
merchant_resource_summary (
    merchant_id BIGINT,
    resource_type VARCHAR(32),
    resource_biz_id BIGINT,
    PRIMARY KEY (merchant_id, resource_type, resource_biz_id)
)
```

- 归属关系变更时，通过消息队列异步更新这张汇总表
- 商户管理员查询直接走汇总表，不需要复杂的JOIN和子查询
- 代价是数据有短暂延迟（毫秒级），对于列表查询可以接受

### 4. 长期优化：归属关系下沉到ES

如果设备量到百万、归属关系到千万，且查询条件复杂（比如"查商户A下所有在线的摄像头设备"），可以考虑：

- 设备信息 + 归属关系同步到Elasticsearch
- 在ES中建立设备文档，字段包含 `owner_user_ids[]` 和 `owner_merchant_ids[]`
- 查询时直接用ES的terms过滤，天然支持多值匹配，性能远优于关系型数据库的JOIN

```json
// ES设备文档结构
{
  "device_id": 12345,
  "device_name": "摄像头-A01",
  "status": "online",
  "owner_merchant_ids": [100, 200],
  "owner_user_ids": [1001, 1002, 2001]
}
```

查询"商户100下所有在线摄像头"：
```json
{
  "query": {
    "bool": {
      "must": [
        { "term": { "owner_merchant_ids": 100 } },
        { "term": { "status": "online" } }
      ]
    }
  }
}
```

### 5. 面试话术建议

> "目前我们设备量在几十万级别，归属关系表靠索引和Redis缓存性能还OK。但我们也意识到如果增长到百万级，JOIN查询会成为瓶颈。内部讨论过几个方向：一是按资源类型分表降低单表数据量；二是用冗余宽表避免复杂JOIN；三是如果查询场景更复杂，可以把归属关系同步到ES，用倒排索引天然解决多对多的过滤问题。目前还没有实际落地，但这是我们的技术储备方向。"

这样回答的好处：
- 诚实说明现状（没有实际做），不会被追问细节时露馅
- 展示了对问题的思考深度和技术视野
- 给出了渐进式的方案（短/中/长期），体现架构演进思维
