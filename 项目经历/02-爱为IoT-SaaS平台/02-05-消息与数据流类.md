## Q25: EventBus的定位是什么？和Kafka/RabbitMQ有什么区别？

**回答思路：**

**一句话定位：** EventBus是进程内的方法调用路由器，Kafka是跨进程的持久化消息管道。它们解决的问题层次完全不同。

**类比：**
- Kafka像邮局——信件投递、存储、确认签收、支持多个收件人各自按自己的节奏取信
- EventBus像办公室里喊一嗓子——同一个进程里，谁关心谁就听到，喊完就没了

**三个维度对比：**

| 维度 | EventBus（本项目） | Kafka / RabbitMQ |
|------|-------------------|-----------------|
| 部署边界 | 单进程内存，和业务代码跑在同一个JVM | 独立集群，跨网络通信 |
| 消息生命周期 | 发完即忘，不落盘，没有重放 | 持久化到磁盘，支持offset回溯 |
| 交付保证 | 无ACK、无重试、subscriber挂了消息就丢 | at-least-once / exactly-once，有dead letter |

**为什么不直接用Kafka：**
> 因为场景不需要。这些模块跑在同一个进程里，走网络绕一圈Kafka再回来，延迟从微秒级变成毫秒级，还多了一个外部依赖要运维。进程内通信用进程内方案，跨进程通信才用Kafka，各司其职。

## Q26: 为什么不用Guava EventBus或Spring ApplicationEvent？

**回答思路：**

核心原因：**我们需要的是一个基于层级topic的路由器，不是一个基于类型的事件广播器。**

**对比Guava EventBus：**

| 维度 | Guava EventBus | 本项目EventBus |
|------|---------------|----------------|
| 路由方式 | 按Java类型匹配 | 按层级topic + 通配符匹配 |
| 通配符 | 不支持 | 支持`*`单层、`**`多层 |
| 共享订阅 | 不支持 | 同组内只投一个，负载均衡语义 |
| 优先级 | 不支持 | 订阅者可设优先级 |
| 取消订阅 | 要手动unregister整个对象 | 返回Disposable，细粒度取消单个订阅 |

> Guava EventBus本质是类型驱动的观察者模式工具。它没有topic概念，没法表达`/device/product-01/*/message/**`这种层级路由。IoT场景里设备有产品、设备ID、消息类型这些层级，用类型匹配根本表达不了。

**对比Spring ApplicationEvent：**

| 维度 | Spring Event | 本项目EventBus |
|------|-------------|----------------|
| 路由方式 | 按事件类型 + 可选SpEL条件 | 按层级topic树 + 通配符 |
| 通配符路由 | 不支持 | 核心能力 |
| 与Spring耦合 | 强绑定Spring容器 | 独立组件，无框架依赖 |
| 动态订阅/取消 | 不方便，listener通常是bean生命周期绑定 | 运行时随时订阅、随时dispose |

> Spring Event的问题：一是路由能力弱，本质还是按类型广播；二是和Spring容器强绑定，订阅者必须是Spring Bean，动态订阅和取消很别扭。我们的场景里设备随时上下线，订阅关系是动态变化的。

## Q27: EventBus的核心架构是怎样的？有哪些核心抽象？

**回答思路：**

**核心对象：**

| 类 | 职责 |
|---|---|
| `EventBus` | 核心接口：publish / subscribe |
| `DefaultEventBus` | 本地实现（去除Broker逻辑） |
| `Subscription` | 订阅信息（subscriberId、topics、features、priority） |
| `TopicPayload` | 携带Topic信息的消息载体 |
| `Payload` | 消息数据抽象（基于byte[]） |
| `Disposable` | 取消订阅接口 |
| `Topic<T>` | 树形Topic数据结构（路由层核心） |
| `Encoder<T>` / `Decoder<T>` | 编解码器接口 |
| `Codecs` | 编解码器注册表 |

**三层职责划分：**
```
TopicUtils — 负责统一规则（标准化、通配符语义）
Topic<T>   — 负责存储和查找（树形索引结构）
DefaultEventBus — 负责在发布/订阅流程里使用它们（投递策略）
```

**核心API：**
```java
// 发布
CompletableFuture<Long> publish(String topic, T payload);

// 订阅
Disposable subscribe(Subscription subscription, Class<T> type, Consumer<T> handler);
```

- publish返回`CompletableFuture<Long>`，异步投递完成后给出成功投递数量
- subscribe返回`Disposable`，细粒度取消单个订阅

## Q28: Topic匹配用了什么数据结构？为什么用树而不是线性扫描？

**回答思路：**

**Topic系统的规则：**
- Topic以`/`分隔层级，如`/device/product-01/device-001/message/property`
- 支持单层通配符`*`（匹配恰好一层）
- 支持多层通配符`**`（匹配零层到多层）

**为什么用树（Trie）而不是线性扫描：**

如果不用树，发布一条消息时要遍历所有订阅模式，逐个调用matches()。订阅一多就是全表扫描。

`Topic<T>`是一个按路径分层的前缀树，把共享前缀组织起来：
```
/device/dev-01/message
/device/dev-02/message
/device/*/message
/device/**
```
这些模式都共享`/device`前缀。树结构能复用这部分路径，匹配时只沿着相关分支走。

**树的内部结构：**
每个节点`Node<T>`有两部分：
- `children`：下一层segment到子节点的映射
- `values`：绑定在当前路径上的订阅者

```
root
└── device
    ├── **        → values: [all-device-subscriber]
    ├── *
    │   └── message → values: [single-level-subscriber]
    └── dev-01
        └── message → values: [exact-subscriber]
```

**match()的核心逻辑——同时探索三类分支：**

对于当前topic的某一层（如`dev-01`），会同时看：
1. 有没有精确子节点`dev-01`
2. 有没有单层通配子节点`*`
3. 有没有多层通配子节点`**`

一次匹配可能有多条路径同时成立，最终收集所有命中的订阅者。

## Q29: `**`多层通配符的匹配为什么复杂？怎么解决性能问题？

**回答思路：**

**为什么`**`复杂：**

`**`可以匹配零层、一层或多层，这意味着它有分支选择：
- 可以先匹配0层，模式向后走一步（"先试试不吞"）
- 也可以吞掉当前层，topic向后走一步，但模式停在原位（"再多吞一层"）

这就引入了回溯/分支，不再是简单的"从左到右一一比较"。

**TopicUtils.matches()中的memo（记忆化搜索）：**

用`(patternIndex, topicIndex)`做缓存键，避免同一组状态被重复计算。这是典型的动态规划写法：
- 不改变语义
- 避免重复计算
- 降低`**`带来的回溯成本
- 让匹配在复杂模式下依然可控

**Topic树match()中的visited：**

`collect()`里用`State(node, index)`记录访问状态。因为`**`会造成"留在当前节点，但topic index继续向前"的搜索路径。没有去重，某些状态组合可能被重复走到，甚至出现无限递归风险。

> memo和visited本质上是同一个思路——控制`**`带来的搜索分支和重复计算。一个用在"模式匹配"，一个用在"树搜索"。

## Q30: TopicUtils做了什么？为什么需要标准化？

**回答思路：**

`TopicUtils`负责Topic字符串本身的规则处理，是路由层的基础能力。

**normalize()——标准化：**
- 去掉首尾空白
- 合并多余的`/`
- 去掉空segment
- 校验通配符是否合法
- 统一成以`/`开头的格式

```
"device//dev-01/message/" → "/device/dev-01/message"
" /device///dev-01/ "     → "/device/dev-01"
"/"                       → "/"
```

**为什么标准化重要：**
- 避免同一个Topic因格式不同被当成不同路径
- 让后续匹配逻辑不必反复处理脏输入
- 订阅时和发布时都使用同一套规范，行为更稳定

**validateSegment()——通配符校验：**

限制通配符只能是完整的`*`或`**`，以下是非法的：
```
abc*、de*ice、***、foo**bar
```
这样做是为了让规则足够清晰，避免把Topic规则做成半个正则表达式系统。

**segments()——拆分层级：**
```
"/device/dev-01/message" → ["device", "dev-01", "message"]
"/" → []
```
根路径`/`表示成空列表，对树结构和匹配算法都很重要。

## Q31: 共享订阅（Shared Subscription）是什么？怎么实现的？

**回答思路：**

**什么是共享订阅：**

当多个订阅者使用相同的subscriberId并启用`shared`特性时，同一条消息只会投递给其中一个订阅者（随机选择）。这是负载均衡语义。

**场景举例：**
> 告警模块部署了3个实例，都订阅`/device/**/event`。如果是广播，每条告警事件会被处理3次。用shared subscription，同组内只有一个实例收到，避免重复处理。

**实现要点：**
- `Subscription.Feature.shared`：标记这个订阅是共享模式
- 同一个subscriberId的多个sink组成一个组
- 投递时，组内随机选一个sink投递，而不是全部广播
- 不同subscriberId之间仍然是广播语义

**投递策略的分层：**
1. `Topic<T>`树负责"找出所有命中的订阅者"
2. `DefaultEventBus`负责"决定怎么投递这些命中项"
   - 按subscriberId分组
   - shared组内随机选一个
   - 非shared的全部投递

## Q32: 订阅优先级怎么实现？同优先级下顺序稳定吗？

**回答思路：**

**优先级机制：**
```java
Subscription.builder()
    .subscriberId("high-priority")
    .topics("/device/**")
    .priority(1)  // 值越小优先级越高，默认Integer.MAX_VALUE
    .build();
```

**投递顺序：**
- 优先级高的订阅者先收到消息
- 同优先级下，按订阅注册顺序（稳定排序）
- 这保证了投递行为是可预测的

**实现方式：**
- Topic树match()返回所有命中的订阅者
- DefaultEventBus按priority排序后再投递
- 排序使用稳定排序算法，同优先级保持注册顺序

**使用场景：**
> 比如安全审计模块需要最先看到所有消息（priority=0），规则引擎其次（priority=10），日志模块最后（priority=100）。这样即使某个低优先级模块处理慢或报错，高优先级模块已经处理完了。

## Q33: 订阅的生命周期怎么管理？取消订阅怎么做？

**回答思路：**

**Disposable模式：**
```java
Disposable sub = eventBus.subscribe(
    Subscription.of("subscriber-1", "/device/**"),
    MyEvent.class,
    event -> { /* handle */ }
);
// 稍后取消
sub.dispose();
```

subscribe返回`Disposable`对象，调用`dispose()`即可取消单个订阅。

**取消订阅的内部实现：**
1. 从Topic树中移除对应的订阅者值
2. 如果某个节点已经没有values也没有children，剪掉这个空节点（树的"垃圾回收"）
3. 避免残留无意义节点占用内存

**并发安全问题——publish和unsubscribe同时发生：**
- 发布时遍历订阅者列表，同时另一个线程在取消订阅
- 解决方案：锁保护订阅关系的修改，但用户回调放在锁外执行
- 这样既保证了订阅关系的一致性，又不会因为用户回调耗时而阻塞其他操作

**异常隔离：**
- 某个订阅者处理消息时报错，不应该影响其他订阅者
- 每个订阅者的回调独立try-catch，异常只记录日志，不中断投递流程

## Q34: 编解码（Codec）怎么设计的？为什么要从EventBus中解耦？

**回答思路：**

**编解码抽象：**

| 接口 | 职责 |
|---|---|
| `Encoder<T>` | 将事件对象编码为Payload（字节数据） |
| `Decoder<T>` | 将Payload解码为事件对象 |
| `Codecs` | 编解码器注册表，按类型查找对应的编解码器 |
| `JsonCodec` | 内置JSON编解码实现（基于Jackson） |

**为什么要解耦：**
- EventBus本体只关心"路由和投递"，不应该绑定具体序列化实现
- 不同的消息类型可能需要不同的编解码方式（JSON、Protobuf、自定义二进制）
- 通过Codecs注册表，按类型自动查找对应的编解码器
- 新增一种序列化格式不需要改EventBus代码

**Payload设计：**
- 基于`byte[]`实现，简单直接
- 与JetLinks原版的区别：原版基于Netty ByteBuf（引用计数），我们简化为byte[]（无引用计数）
- 这样避免了引用计数带来的内存泄漏风险，适合进程内场景

**typed subscribe的实现：**
```java
// 业务可以直接订阅某种事件类型，不用手动解析payload
eventBus.subscribe(subscription, DeviceEvent.class, event -> {
    // event已经是反序列化后的DeviceEvent对象
});
```
EventBus内部通过Codecs找到DeviceEvent对应的Decoder，自动完成反序列化。

## Q35: 你的IoT平台不是集群部署吗？EventBus只是进程内的，跨节点怎么办？

**回答思路：**

> 对，生产环境确实是集群部署，跨节点通信是必须的。但架构上是分层设计的。

**本地EventBus是底层路由引擎**，负责单节点内的topic匹配和投递。它不关心消息从哪来。

**上面有一层Broker机制**，负责跨节点的消息转发。当一个节点收到消息，Broker判断"这条消息在其他节点上有没有人订阅"，如果有，就通过网络转发过去。对端节点收到后，再交给本地EventBus做路由投递。

**职责划分：**

| 层次 | 职责 | 关心的问题 |
|------|------|-----------|
| EventBus（本地层） | 本节点内谁该收到这条消息 | topic匹配、优先级、shared投递、并发安全 |
| Broker（集群层） | 消息要不要转发到其他节点 | 网络通信、序列化、节点发现、重试 |

**架构示意：**
```
┌─────────────────────────────────────────────┐
│                  Node A                      │
│  [模块1] ──publish──→ [EventBus] ──→ [模块2] │
│                          │                  │
│                          ↓                  │
│                      [Broker] ──── 网络 ────→ Node B的Broker
│                                             │        ↓
└─────────────────────────────────────────────┘   Node B的EventBus
                                                       ↓
                                                  [模块3]
```

**分层的好处：** 本地路由逻辑不被网络、序列化、重试这些分布式问题污染。EventBus的topic匹配、优先级、shared subscription这些语义，在单节点和集群模式下完全一致。

**设计预留：** EventBus的subscribe有一个Feature枚举，其中`local`表示只接收本地消息，去掉这个feature就意味着也接收远程转发过来的消息。

## Q36: 如果做一个基于Redis的Broker，你会怎么设计？

**回答思路：**

**核心思路：** Redis Broker的本质是把Redis当作节点之间的消息中转站。每个节点既是publisher也是subscriber。Redis只负责"把消息送到节点"，精确的topic路由仍然由各节点本地的EventBus完成。

**消息流转全景：**
```
Node A publish("/device/dev-01/property", payload)
    │
    ├──→ 本地EventBus投递给Node A的本地订阅者
    │
    └──→ Redis Broker序列化后PUBLISH到Redis Channel
              │
              ↓
         Redis Server广播给所有订阅了该Channel的节点
              │
              ├──→ Node B收到 → 本地EventBus做topic匹配 → 投递
              └──→ Node C收到 → 本地EventBus做topic匹配 → 投递
```

**三个关键设计决策：**

**1. Redis Channel怎么映射Topic？**
- 不能把每个具体topic都映射成一个Redis Channel，否则channel数量爆炸
- 简单方案：用一个统一channel（如`eventbus:broadcast`），消息体里带原始topic，对端收到后由本地EventBus做精确匹配
- 优化方案：按业务域分几个channel（如`eventbus:device`、`eventbus:rule`），减少无关节点的处理量

**2. 消息格式：**

| 字段 | 用途 |
|------|------|
| sourceNodeId | 来源节点标识，避免自己发的消息又被自己消费 |
| topic | 原始topic路径，对端用来做本地路由 |
| payload | 消息体的字节数组 |

**3. 避免自己消费自己的消息：**
```java
public void onMessage(String channel, byte[] data) {
    BrokerMessage msg = deserialize(data);
    if (msg.getSourceNodeId().equals(nodeId)) {
        return; // 忽略自己发出的消息
    }
    localEventBus.publishFromRemote(msg.getTopic(), msg.getPayload());
}
```

**Redis Pub/Sub的局限性：**
- 消息不持久化，没有ACK，节点断线期间的消息会丢失
- 如果业务能容忍偶尔丢消息（如设备状态上报，下一次上报会覆盖），Redis Pub/Sub够用
- 如果不能容忍，可以升级到Redis Streams（支持消费者组和ACK）或直接上Kafka
- 所以Broker接口设计成可插拔的，底层传输根据业务对可靠性的要求来选型

## Q37: DefaultEventBus的发布流程是怎样的？从publish到投递的完整路径？

**回答思路：**

**完整发布流程：**
```
eventBus.publish("/device/dev-01/message", payload)
  ↓
1. TopicPayload先把topic做标准化（normalize）
  ↓
2. 调用topicTree.match(topic)找到所有匹配的订阅者
  ↓
3. 按subscriberId分组
  ↓
4. 对每个组判断是否shared：
   - shared组：随机选一个sink投递
   - 非shared组：全部投递
  ↓
5. 按priority排序，高优先级先投递
  ↓
6. 逐个调用subscriber的回调（try-catch隔离异常）
  ↓
7. 返回CompletableFuture<Long>，值为成功投递数量
```

**关键设计点：**
- Topic树只负责"找出所有命中项"
- DefaultEventBus负责"决定怎么投递这些命中项"（shared、priority、异常隔离）
- 用户回调放在锁外执行，避免阻塞其他操作
- 每个回调独立try-catch，一个报错不影响其他订阅者

## Q38: 并发安全怎么保证？锁的粒度是怎样的？

**回答思路：**

**并发场景：**
- 多个线程同时publish
- publish的同时有线程在subscribe或unsubscribe
- 多个线程同时subscribe不同的topic

**锁的设计原则：**
1. 锁保护订阅关系的修改（Topic树的add/remove）
2. 用户回调放在锁外执行——避免用户代码耗时导致锁持有时间过长
3. 读多写少场景，可以用ReadWriteLock：publish时读锁，subscribe/unsubscribe时写锁

**具体策略：**
- Topic树的`add()`和`remove()`需要加写锁
- Topic树的`match()`可以加读锁（多个publish可以并发匹配）
- 匹配完成后释放锁，再执行用户回调
- 这样publish之间不互斥，只有subscribe/unsubscribe才需要独占

**为什么不能把回调放在锁内：**
- 用户回调可能耗时很长（比如写数据库、调外部接口）
- 如果在锁内执行回调，其他所有publish和subscribe都会被阻塞
- 极端情况下用户回调死锁，整个EventBus就卡死了

## Q39: 举一个完整的EventBus使用场景？

**回答思路：**

**场景：设备属性上报，多模块消费**

```java
// 规则引擎订阅所有设备事件（高优先级）
Disposable ruleSub = eventBus.subscribe(
    Subscription.builder()
        .subscriberId("rule-engine")
        .topics("/device/**/message/**")
        .priority(1)
        .build(),
    DeviceMessage.class,
    msg -> ruleEngine.evaluate(msg)
);

// 告警模块订阅事件类消息（shared，3个实例只投一个）
Disposable alarmSub = eventBus.subscribe(
    Subscription.builder()
        .subscriberId("alarm-service")
        .topics("/device/**/message/event")
        .features(Feature.shared)
        .priority(10)
        .build(),
    DeviceMessage.class,
    msg -> alarmService.check(msg)
);

// 日志模块订阅所有消息（低优先级）
Disposable logSub = eventBus.subscribe(
    Subscription.builder()
        .subscriberId("audit-log")
        .topics("/device/**")
        .priority(100)
        .build(),
    DeviceMessage.class,
    msg -> auditLog.record(msg)
);

// 设备上报属性
eventBus.publish("/device/product-01/dev-001/message/property", propertyPayload);
```

**匹配结果：**
- 规则引擎：`/device/**/message/**`匹配 ✓ → 投递
- 告警模块：`/device/**/message/event`不匹配property ✗ → 不投递
- 日志模块：`/device/**`匹配 ✓ → 投递

**投递顺序：** 规则引擎（priority=1）先收到，日志模块（priority=100）后收到。

**如果发布`/device/product-01/dev-001/message/event`：**
- 规则引擎 ✓、告警模块 ✓（shared，3个实例只投1个）、日志模块 ✓
- 投递顺序：规则引擎 → 告警模块（选一个实例）→ 日志模块

## Q40: 与JetLinks原版的主要差异是什么？你做了哪些简化？

**回答思路：**

| 方面 | JetLinks原版 | 我的简化版 |
|---|---|---|
| **编程模型** | 响应式（Flux/Mono） | `CompletableFuture` + 回调（`Consumer<T>`） |
| **Broker支持** | 支持集群消息转发 | 仅支持本地（预留扩展点） |
| **Payload实现** | 基于Netty ByteBuf（引用计数） | 基于`byte[]`（无引用计数） |
| **线程模型** | Reactor Scheduler | Java `ExecutorService`（可选异步） |
| **Feature集合** | local/broker/shared/sharedOldest/... | 简化为local/shared |
| **序列化** | 自定义 + FastJSON | Jackson JSON |

**为什么不用响应式：**
- 响应式（Reactor）学习曲线陡峭，调试困难
- 进程内EventBus的核心价值在路由和投递语义，不在背压和流控
- 用CompletableFuture + 回调足以表达异步语义，代码更直观
- 保留了JetLinks的核心设计精髓（topic树、shared、priority），去掉了响应式的复杂度

**为什么不实现Broker：**
- 明确的scope控制——聚焦本地路由层的设计和实现
- Broker层涉及网络通信、序列化、节点发现、重试等分布式问题，是另一个维度的复杂度
- 架构上预留了扩展点（Feature.local），后续可以无缝接入Broker
