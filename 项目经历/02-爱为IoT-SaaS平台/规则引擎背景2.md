# 规则引擎实现全程复盘

> 以写这个代码的工程师视角，一步步还原从设计文档到可运行代码的完整思考过程。

---

## 第零步：拿到设计文档，先想清楚"我到底要造什么"

在动手写任何一行代码之前，我把设计文档反复看了几遍，在脑子里形成一个核心画面：

**用户给我一张"规则图"（节点 + 连线），我要把它编译成路由表，然后让数据从入口节点进来，沿着连线一路流到下游节点，每条连线上可以带条件。**

这就是整个引擎要做的全部事情。

想清楚这一点之后，我把工作拆成了 6 个阶段，每个阶段有明确的交付物和验证标准。实现顺序严格按依赖关系排列：

```
模型层 → 数据层 → 条件层 → 执行器层 → 编译层 → 运行时层 → 测试验证
```

为什么是这个顺序？因为每一层只依赖"比自己更底层"的类，从来不会出现循环依赖。这不是巧合，而是按依赖关系严格排列实现顺序的结果。

## 第一步：先定义"规则图长什么样" — 模型层（model）

### 为什么先做这个？

模型层是所有其他层的基础。编译器要编译它，运行时要读它，执行器要从它拿配置。如果模型没定义好，后面所有东西都没法开始。

### 四个纯数据类

| 类 | 职责 |
|---|---|
| `RuleModel` | 整张规则图（包含节点列表 + 连线列表） |
| `RuleNodeModel` | 单个节点（id + executor 名称 + 配置） |
| `RuleLink` | 一条连线（source → target，可带条件） |
| `Condition` | 条件定义（type + configuration） |

### 关键设计决策

**1. executor 字段是字符串标识，不是执行器本身。**

`RuleNodeModel` 里的 `executor` 字段（比如 `"log"`、`"function"`）只是一个查找 key。真正的执行器要到运行时才通过 `TaskExecutorProvider` 创建。这是策略模式的核心——模型只说"我要什么类型的节点"，不管"这个节点怎么实现"。

**2. RuleLink 用 type 字段区分两种边。**

`type` 为 `"output"` 是普通数据边，其他值（如 `"error"`、`"complete"`）是事件边。`isOutput()` 方法通过 `TYPE_OUTPUT.equals(type)` 来判断。这个设计让同一个类能表达两种完全不同的路由语义，而不需要搞两套继承体系。

**3. Condition 只有 type + configuration，非常轻量。**

我故意不在条件里写判断逻辑，判断逻辑交给后面的条件层去做。这样新增一种条件类型时，模型层完全不用改。`Condition.of("field_gt").with("key", "temperature").with("value", 30)` 这种链式写法让构建条件非常直观。

**4. addLink 会自动同步到 source 节点的 outputs/events 列表。**

`RuleModel.addLink()` 除了把边放进全局 `links` 列表，还会根据 `isOutput()` 的结果把边挂到 source 节点的 `outputs` 或 `events` 集合上。这样模型既能从"全图"视角看，也能从"节点局部"视角看。

### 写完之后怎么验证？

写了 `RuleModelTest`，确认能正确构建一个包含节点和连线的规则图，并且 `addLink` 会自动把边挂到 source 节点的 outputs/events 列表上。

## 第二步：定义"数据在节点间怎么传" — 数据层（data）

### 为什么紧接着做这个？

后面的条件层、执行器层、运行时层全都要用到 `RuleData`。它是整个引擎的"血液"——没有它，其他层连方法签名都写不出来。

### RuleData — 统一数据信封

```
RuleData
├── id           — 每份数据包自己的唯一标识（UUID）
├── contextId    — 同一次规则执行的关联 id（UUID）
├── data         — 真正的业务数据（Object）
├── headers      — 元信息（比如错误信息、标记位）
└── sourceNodeId — 这份数据从哪个节点出来的
```

### 为什么不直接传业务对象，而要包一层信封？

三个原因：

1. **条件判断需要读 headers**。没有信封就没地方挂元信息。比如错误传播时需要在 headers 里塞 `_error` 和 `_errorType` 字段。
2. **链路追踪**。`contextId` 让同一次执行链路上的所有数据包可以关联起来，方便调试。
3. **数据隔离**。`copyFor()` 方法让下游节点拿到的是一份新的数据包（新 id），但保留了上游的 `contextId` 和 `headers`。这样既不会污染上游数据，又能保持链路上下文。

### copyFor() 的精确行为

```java
public RuleData copyFor(Object nextData, String nextSourceNodeId) {
    RuleData copied = new RuleData();
    copied.id = UUID.randomUUID().toString();   // 新 id
    copied.contextId = contextId;                // 保留原 contextId
    copied.data = nextData;                      // 使用新的业务数据
    copied.headers = new LinkedHashMap<>(headers); // 复制 headers（浅拷贝）
    copied.sourceNodeId = nextSourceNodeId;      // 更新来源节点
    return copied;
}
```

注意 headers 是浅拷贝——新数据包拿到的是 headers 的副本，往里面加东西不会影响上游。这个细节在错误传播时非常重要。

### 验证

`RuleDataTest` 确认 copy 后 id 不同、contextId 相同、headers 被正确继承。

## 第三步：让引擎能做条件判断 — 条件层（condition）

### 为什么在执行器之前做？

条件判断是边上路由的核心能力。如果没有条件层，后面运行时的路由逻辑就写不了。而且条件层只依赖模型层和数据层，不依赖执行器，所以可以先做。

### 三个类，职责非常清晰

| 类 | 职责 |
|---|---|
| `ConditionEvaluator` | 接口：给我一个 Condition 和一份 RuleData，告诉我条件是否成立 |
| `SimpleConditionEvaluator` | 用 `BiPredicate<Condition, RuleData>` 快速注册一个条件策略 |
| `ConditionEvaluatorRegistry` | 注册表：按 type 查找对应的 evaluator，统一对外入口 |

### 关键设计决策

**1. 条件放在边上而不是节点内。**

这意味着：
- 节点只管"处理数据"
- 边负责"决定数据去哪"
- 修改路由逻辑不需要改节点代码，只需要改边的条件配置

**2. 注册表内部处理 condition 为 null 的情况。**

`ConditionEvaluatorRegistry.evaluate()` 方法在 condition 为 null 时直接返回 true。这样运行时路由代码不需要自己判空，直接调 `conditionRegistry.evaluate(link.getCondition(), data)` 就行。这是一个更干净的设计——把"无条件即放行"的语义封装在注册表内部。

**3. 默认注册了 7 种内置条件。**

| type | 说明 |
|---|---|
| `always_true` | 永远通过，用于默认通路 |
| `header_equals` | 比较 headers 中某个 key 的值 |
| `field_equals` | 比较 data 中某个字段的值 |
| `field_gt` | 字段值 > 配置值 |
| `field_gte` | 字段值 >= 配置值 |
| `field_lt` | 字段值 < 配置值 |
| `field_lte` | 字段值 <= 配置值 |

数值比较类条件内部统一转成 double 再比较，这样 Integer、Long、Double 等类型都能正常处理。如果任意一侧无法转换为数字，则直接抛异常让调用方感知配置错误。

### ObjectNavigator — 条件层的重要支撑工具

`ObjectNavigator` 是 `support` 包下的一个工具类，它让条件判断能统一读取 Map 和 POJO 中的字段：

- 如果是 Map，直接按 key 读取
- 如果是 Java Bean，优先走 getter
- 如果没有 getter，再尝试直接读字段（反射）
- 支持点路径，如 `"device.status"` 会逐级向下读取

它的目标不是做万能表达式引擎，而是给条件判断提供一个足够顺手的字段读取工具。不管业务数据是 `Map<String, Object>` 还是自定义 Java Bean，都能用 `"temperature"` 或 `"device.status"` 这种路径去读值。

## 第四步：定义"节点能做什么" — 执行器层（executor）

### 为什么现在做？

模型定义好了，数据信封有了，条件也能判断了。现在需要定义"节点拿到数据后到底干什么"。执行器层只依赖数据层（处理 `RuleData`）和运行时的 `ExecutionContext`（后面才实现，但接口签名现在就能确定），所以可以在编译层之前先做。

### 三个核心抽象

| 类 | 职责 |
|---|---|
| `TaskExecutor` | 接口：节点执行器，负责处理输入数据 |
| `TaskExecutorProvider` | 接口：工厂，根据节点配置创建执行器实例 |
| `AbstractTaskExecutor` | 基类：统一保存 context + 统一兜底异常处理 |

### AbstractTaskExecutor 的模板方法 — 整个执行器层最精妙的设计

```java
@Override
public void execute(RuleData input, ExecutionContext context) {
    try {
        doExecute(input);      // 子类只管写正常逻辑
    } catch (Exception error) {
        context.onError(error, input);  // 异常自动走 error 事件边
    }
}

protected abstract void doExecute(RuleData input) throws Exception;
```

子类只需要实现 `doExecute()`，完全不用操心异常处理。如果 `doExecute()` 抛了异常，基类会自动把它交给 `context.onError()`，最终转成 error 事件走事件边路由到错误处理节点。

注意 `doExecute` 的签名是 `throws Exception`——这让 `DelayTaskExecutor` 可以直接抛 `InterruptedException` 而不用自己 try-catch。

这就是为什么测试里 `FailingProvider` 只需要 `throw new IllegalStateException("boom")`，错误就能自动流到 errors 收集节点。


### 为什么每个执行器都是 Provider + 内部 Executor 的结构？

同一种类型的节点可能在规则图里出现多次，每次的配置不同。Provider 是单例的（注册一次），但每次 `createExecutor()` 都会创建一个新的 Executor 实例，带着各自的配置。

这是标准的**策略模式 + 工厂模式**组合：
- 策略模式：不同执行器是不同策略
- 工厂模式：Provider 负责创建策略实例
- 注册表模式：引擎维护一个 `Map<String, TaskExecutorProvider>` 来查找

### 6 个内置执行器

| Provider | executor 标识 | 做什么 |
|---|---|---|
| `StartTaskExecutorProvider` | `"start"` | 入口节点，收到什么就透传什么 |
| `LogTaskExecutorProvider` | `"log"` | 打印数据（带 level 配置），然后继续透传 |
| `FunctionTaskExecutorProvider` | `"function"` | 对 Map 数据做 remove / rename / set 变换 |
| `FilterTaskExecutorProvider` | `"filter"` | 节点内部条件过滤，满足才输出 |
| `DelayTaskExecutorProvider` | `"delay"` | 简单 `Thread.sleep` 延迟后继续传递 |
| `NotifyTaskExecutorProvider` | `"notify"` | 通知下发，通过 `NotifySender` 接口解耦实际发送动作 |

几个值得注意的实现细节：

**FunctionTaskExecutor** 在变换前会先复制一份输出 Map（`new LinkedHashMap<>()`），避免直接修改上游传进来的原对象。输出时调用 `context.newRuleData(output, input)` 创建新的 RuleData，保留原上下文但更新 sourceNodeId。

**FilterTaskExecutor** 构造时就把配置中的嵌套 Map 翻译成 `Condition` 对象，执行时通过 `context.testCondition(condition, input)` 复用条件注册表来判断。这样 filter 节点不需要自己实现条件逻辑，直接借用条件层的能力。

**NotifyTaskExecutorProvider** 通过 `NotifySender` 函数式接口解耦发送动作，默认实现只打印到标准输出，生产环境可以注入真正的消息通道。通知发出后继续向下游传递，方便后续节点做审计。

## 第五步：把规则图"编译"成路由表 — 编译层（compiler）

### 为什么要编译？

如果运行时每次路由都要遍历所有 links 去找"从当前节点出发的边"，效率太低。编译阶段把这些信息预处理成 Map，运行时直接 O(1) 查找。

### 三个类

| 类 | 职责 |
|---|---|
| `RuleCompiler` | 输入 RuleModel，输出 CompiledRule |
| `CompiledRule` | 预处理好的运行时快照 |
| `OutputLink` | 编译后的精简边（只保留 targetNodeId + condition） |

### RuleCompiler.compile() 做了这些事

**1. 校验合法性**
- 规则 id 不能为空
- 至少要有一个节点
- 节点 id 不能重复
- 每个节点的 executor 不能为空

**2. 构建 nodeMap**

`Map<nodeId, RuleNodeModel>`，方便后续快速查找。

**3. 构建 outputLinks — 普通边路由表**

`Map<sourceNodeId, List<OutputLink>>`，普通边按源节点分组。运行时只需要 `outputLinks.get(sourceNodeId)` 就能拿到当前节点的所有下游。

**4. 构建 eventLinks — 事件边路由表**

`Map<sourceNodeId, Map<eventName, List<OutputLink>>>`，事件边按源节点 + 事件名二级分组。运行时先按 sourceNodeId 查，再按事件名查，两次 Map.get 就能定位到目标。

**5. 识别起始节点**

优先找 executor 为 `"start"` 的节点。找不到就退化为"所有无入边的节点"——这里用到了编译阶段统计的入度（`inboundCount`）。如果两种方式都找不到起始节点，直接抛异常。

### 关键设计决策

编译阶段统计每个节点的入度（`inboundCount`）。这不是为了拓扑排序，而是为了兜底识别起始节点——如果用户没有显式放 start 节点，那些入度为 0 的节点就是入口。

`CompiledRule` 构造时会复制传入的集合（`new LinkedHashMap<>(nodeMap)` 等），避免外部继续修改影响运行时。`startNodeIds` 更是用了 `List.copyOf()` 生成不可变列表。

### 验证

`RuleCompilerTest` 确认普通边和事件边都能正确编译，以及重复节点 id 会被拒绝。

## 第六步：让一切跑起来 — 运行时层（runtime）

这是最后也是最复杂的一步。前面所有层都是"准备工作"，这一步才是真正把数据驱动起来。

### 5 个类，各有分工

| 类 | 职责 |
|---|---|
| `TaskState` | 枚举：WAITING → RUNNING → STOPPED（注意：没有 PAUSED） |
| `LocalTask` | 运行中的单个节点（executor + context + state） |
| `ExecutionContext` | 节点和运行时之间的桥梁 |
| `LocalRuleRuntime` | 单条规则的运行时实例 |
| `LocalRuleEngine` | 引擎入口，管理所有规则实例 |


### 数据流转的完整路径 — 整个引擎的灵魂

```
用户调用 engine.fireRule("rule-1", rawData)
  ↓
engine 把 rawData 包装成 RuleData，为每个 start 节点 copyFor 一份副本
  ↓
runtime.routeToNode("start", ruleData)
  ↓
LocalTask.execute(ruleData) → executor.execute(ruleData, context)
  ↓
executor 内部调用 context.output(data)  ← 节点不直接调用下游！
  ↓
context.output() → runtime.handleOutput(nodeId, data)
  ↓
runtime 查编译好的 outputLinks，逐条调用 conditionRegistry.evaluate()
  ↓
条件通过 → routeToNode(targetNodeId, data) → 下一个节点继续处理
```

关键要理解的是：

1. **节点不直接调用下游节点**，而是通过 `ExecutionContext.output()` 写出数据
2. **运行时根据编译后的路由表**决定数据去哪
3. **条件判断发生在路由时**，由 `conditionRegistry.evaluate()` 统一处理（包括 null 判空）
4. **错误也是一种事件**，通过事件边流向错误处理节点


### ExecutionContext — 最关键的桥梁

节点执行器永远不知道自己的下游是谁。它只知道"我处理完了，调 `context.output(data)` 把数据扔出去"。至于数据去哪、走哪条边、条件怎么判断，全部由运行时负责。这就是为什么新增一种节点类型完全不需要改引擎核心代码。

ExecutionContext 提供的完整能力：

| 方法 | 作用 |
|---|---|
| `output(data)` | 把数据写到当前节点的普通输出边 |
| `fireEvent(event, data)` | 触发一个事件，走对应的事件边 |
| `onError(error, data)` | 统一错误处理入口 |
| `newRuleData(data)` | 创建一份以当前节点为来源的新数据 |
| `newRuleData(data, parent)` | 基于父数据创建下游数据，保留 contextId 和 headers |
| `testCondition(condition, data)` | 让执行器复用条件注册表（filter 节点用到） |
| `getNodeId()` | 获取当前节点 id |
| `getNodeConfig()` | 获取当前节点配置 |

### onError() 的精确实现

```java
public void onError(Throwable error, RuleData data) {
    RuleData errorData = data == null
        ? newRuleData(null)
        : data.copyFor(data.getData(), nodeId);   // 先 copyFor 创建副本！
    errorData.getHeaders().put("_error", error.getMessage());
    errorData.getHeaders().put("_errorType", error.getClass().getName());
    fireEvent("error", errorData);
}
```

这里有两个容易忽略的细节：

1. **不会污染上游数据**。它先通过 `data.copyFor()` 创建一份副本，再往副本的 headers 里塞 `_error` 和 `_errorType`。原始数据不受影响。
2. **除了 `_error` 还有 `_errorType`**。`_errorType` 存的是异常类的全限定名（如 `java.lang.IllegalStateException`），方便下游节点根据异常类型做不同处理。

错误处理和正常处理用的是同一套路由机制——`fireEvent("error", errorData)` 最终走到 `runtime.handleEvent()`，没有任何特殊逻辑。


### LocalRuleRuntime — 单条规则的运行时

`start()` 方法为每个节点创建独立的执行器和上下文：

```java
public void start() {
    for (RuleNodeModel node : compiledRule.getNodeMap().values()) {
        TaskExecutorProvider provider = providers.get(node.getExecutor());
        TaskExecutor executor = provider.createExecutor(node.getConfiguration());
        ExecutionContext context = new ExecutionContext(
            node.getId(), this, conditionRegistry, node.getConfiguration()
        );
        LocalTask task = new LocalTask(node.getId(), executor, context);
        tasks.put(node.getId(), task);
        task.start();
    }
}
```

`handleOutput()` 和 `handleEvent()` 是两套路由的核心：

- `handleOutput` 查 `compiledRule.getOutputLinks()`，按 sourceNodeId 找到所有普通下游边
- `handleEvent` 查 `compiledRule.getEventLinks()`，按 sourceNodeId + eventName 二级查找

两者都通过 `conditionRegistry.evaluate(link.getCondition(), data)` 统一判断条件，不需要自己判空。

### LocalRuleEngine — 引擎入口

默认构造器会自动注册 6 个内置 Provider：

```java
public LocalRuleEngine() {
    this(new RuleCompiler(), new ConditionEvaluatorRegistry());
    registerProvider(new StartTaskExecutorProvider());
    registerProvider(new LogTaskExecutorProvider());
    registerProvider(new FunctionTaskExecutorProvider());
    registerProvider(new FilterTaskExecutorProvider());
    registerProvider(new DelayTaskExecutorProvider());
    registerProvider(new NotifyTaskExecutorProvider());
}
```

用户也可以通过 `registerProvider()` 注册自定义节点类型，比如测试里的 `CollectingProvider` 和 `FailingProvider`。

`startRule()` 的一个细节：它用 `compiled.getRuleId()` 而不是 `model.getId()` 作为 map key。虽然值相同，但用编译结果的 ruleId 更严谨——表示"这是经过编译校验的合法规则"。另外，重复启动时会先 put 新的再 stop 旧的（`runtimes.put` 在 `previous.stop()` 之前），确保不会出现短暂的"规则不存在"窗口。

`fireRule()` 会为每个起始节点 `copyFor` 一份输入副本，这样多入口规则也能正常工作，各入口拿到的是独立的数据包。


### RuleTrigger — 引擎的外部事件入口

除了手动调用 `fireRule()`，引擎还支持通过 `RuleTrigger` 接口自动驱动规则执行：

```java
public interface RuleTrigger {
    void start(String ruleId, LocalRuleEngine engine);
    void stop();
}
```

触发器负责监听外部事件源（设备上报、消息总线、定时任务等），事件到来时自动调用 `engine.fireRule()` 注入数据。这样规则引擎本身不需要知道"数据从哪来"，触发器也不需要知道"规则内部怎么跑"，两侧通过这个接口解耦。

`LocalRuleEngine.startRule(model, triggers)` 会在运行时就绪后立即启动所有触发器，停止规则时也会一并停止。

## 第七步：写测试 + Demo 验证闭环

### 两个集成测试验证核心场景

**场景 1：`shouldRouteByEdgeConditionAndTransformData`**

```
[start] → [function: set status=alarm] → [high: collect]  (当 temperature=35)
                                        → [normal: collect] (当 temperature=20)
```

注入 temperature=35 和 temperature=20 两份数据，验证：
- 数据经过 function 节点后被加上了 `status: "alarm"` 字段
- temperature=35 的数据走到了 high 桶
- temperature=20 的数据走到了 normal 桶
- 两个桶各只有 1 条数据

这个测试同时验证了条件路由和数据变换两个核心能力。

**场景 2：`shouldRouteErrorEventToDownstreamNode`**

```
[start] → [boom: 抛异常] --error--> [errors: collect]
```

boom 节点的 `doExecute()` 直接 `throw new IllegalStateException("boom")`。验证：
- 错误自动通过 error 事件边流到 errors 收集节点
- errors 桶里有 1 条数据
- 该数据的 headers 里 `_error` 值为 `"boom"`

这个测试验证了 `AbstractTaskExecutor` 的异常兜底 → `context.onError()` 的 copyFor + headers 注入 → `fireEvent("error", ...)` 的事件路由，整条错误传播链路。

### 测试里的两个自定义 Provider

`CollectingProvider`（executor = `"collect"`）：不做任何处理，只把收到的 RuleData 按 bucket 配置分桶存起来，方便断言。

`FailingProvider`（executor = `"boom"`）：`doExecute()` 直接抛异常，用于验证错误传播机制。

这两个 Provider 的存在本身就证明了执行器的可插拔性——只需要实现 `TaskExecutorProvider` 接口，注册进引擎，就能参与规则图的执行。

## 总结：为什么是这个顺序

```
模型层 → 数据层 → 条件层 → 执行器层 → 编译层 → 运行时层 → 测试验证
```

这个顺序的逻辑是**自底向上、依赖驱动**：

| 阶段 | 依赖 | 理由 |
|---|---|---|
| 模型层 | 无 | 纯数据结构，零依赖 |
| 数据层 | 无 | 纯数据结构，零依赖 |
| 条件层 | 模型 + 数据 | 需要读 Condition 和 RuleData |
| 执行器层 | 数据 | 处理 RuleData，通过 ExecutionContext 输出 |
| 编译层 | 模型 | 编译 RuleModel 为路由表 |
| 运行时层 | 前面所有层 | 把一切串起来 |

每一层做完都可以独立测试，不需要等后面的层。你看代码时会发现：每个类的 import 都只引用"比自己更底层"的类，从来不会出现循环依赖。

### 整个引擎的设计精髓，浓缩成三句话

1. **模型只描述"是什么"，不管"怎么执行"** — 策略模式让节点类型可插拔
2. **节点不知道下游是谁，只管往 context 里扔数据** — 运行时负责路由，节点负责处理
3. **错误就是事件，事件就是路由** — 没有特殊的错误处理机制，一切都是边上的条件路由
