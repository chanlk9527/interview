# EventBus 系统设计面试方案

## 1. 题目定位

这道题建议被包装成一道“进程内事件路由组件设计题”，而不是“大规模分布式消息系统设计题”。

这样设计有几个好处：

- 候选人不会一上来联想到 Kafka、RabbitMQ、Pulsar 这类重型系统
- 候选人更容易聚焦在 EventBus 的核心抽象，而不是被存储、ACK、重试、高可用带偏
- 面试官可以更稳定地考察抽象能力、语义设计能力、数据结构能力和工程边界感

这道题的本质，是让候选人设计一个应用进程内的轻量级 Pub/Sub 路由组件。

---

## 2. 面试目标

希望通过这道题考察候选人是否具备以下能力：

- 能否把抽象问题快速落到可沟通的模型上
- 能否先澄清边界，再设计系统
- 能否识别核心对象与职责边界
- 能否为路由和匹配选择合理的数据结构
- 能否定义清晰、一致、可验证的投递语义
- 能否考虑取消订阅、异常隔离、并发安全、扩展性等工程问题

---

## 3. 面试题背景

### 3.1 业务背景

我们在做一个 IoT 平台。系统里有多个模块：

- 设备接入模块
- 规则引擎
- 告警模块
- 日志审计模块
- 设备状态计算模块

这些模块都需要接收设备事件，比如：

- 设备属性上报
- 设备事件上报
- 在线离线状态变化
- 命令执行结果

如果这些模块之间彼此直接调用，会导致强耦合、依赖关系复杂，也不利于模块扩展。

因此，我们想在应用内部设计一个轻量级 EventBus，让模块通过发布-订阅的方式通信。

### 3.2 场景抽象

发布方不需要知道谁会消费消息，只负责把消息发到某个 topic。

订阅方不需要知道消息从哪里来，只需要声明自己关心哪些 topic pattern。

EventBus 负责做两件事：

1. 找到哪些订阅者命中了当前消息
2. 按既定语义把消息投递给它们

---

## 4. 面试官开场说明

下面是一段推荐的开场描述，适合作为面试题背景说明。

> 我们在做一个 IoT 平台，系统里会产生很多设备相关事件。规则引擎、告警、日志、状态计算这些模块都需要接收这些事件，但我们不希望它们彼此直接调用。  
> 所以我们想设计一个应用进程内的轻量级 EventBus。  
> 发布方只负责往某个 topic 发消息，订阅方只负责声明自己关心哪些 topic pattern。  
> 这道题先只考虑单进程、内存内实现，不考虑分布式 broker、持久化、ACK、重试。  
> 你可以把它理解成一个应用内的迷你版 pub/sub 路由器。  
> 请你从需求澄清开始，设计这个 EventBus 的核心抽象、主要 API、数据结构和投递语义。

## 5. 正式题面

可以把题目定义成下面这样。

### 5.1 基础要求

请设计一个本地 EventBus，支持：

- `publish(topic, payload)`
- `subscribe(topicPattern, handler)`
- 返回可取消订阅的句柄

### 5.2 Topic 规则

`topic` 是层级路径，使用 `/` 分隔，例如：

```text
/device/product-01/device-001/message/property
/device/product-01/device-001/message/event
```

需要支持两种通配符：

- `*`：只匹配一层
- `**`：匹配零层到多层

例如：

```text
/device/*/message
/device/**
/device/**/property
```

### 5.3 投递语义

- 默认语义：广播给所有匹配的订阅者
- 订阅者可以取消订阅

### 5.4 进阶要求

如果基础设计讲清楚了，可以继续追问：

- `shared subscription`：同一组订阅者里，一条消息只投给其中一个
- `priority`：高优先级订阅者先收到消息
- `typed subscribe`：业务可以直接订阅某种事件类型，而不是手动解析 payload

## 6. 明确边界

这道题建议面试官主动把边界讲清楚，避免候选人误解题目。

### 6.1 本题明确不要求

- 不做 broker 或跨进程通信
- 不做消息持久化
- 不做 ACK / retry / dead letter
- 不做 exactly-once 之类的强交付语义
- 不要求设计成 Kafka 或 MQ 替代品

### 6.2 本题重点

- 抽象模型
- API 设计
- Topic 路由与匹配
- 订阅生命周期
- shared 与 priority 的语义设计
- 基本并发安全
- 未来如何演进

---

## 7. 如果候选人一时听不懂，该怎么引导

EventBus 对很多候选人来说是抽象概念。面试官最好主动把问题压缩成最小模型。

可以这样解释：

> 先别把它想成 MQ。  
> 你可以先把它理解成应用内的消息路由器。  
> 有人调用 `publish("/device/dev-01/temp", payload)`，凡是订阅了 `/device/**` 或 `/device/*/temp` 这类 pattern 的 handler，都有机会收到这个消息。  
> EventBus 只负责“匹配谁该收到”和“按什么规则投递”。

如果候选人还是抽象不出来，可以进一步举例：

> 例如告警模块订阅 `/device/**/event`，日志模块订阅 `/device/**`，设备状态模块订阅 `/device/*/property`。  
> 当消息发到 `/device/dev-01/event` 时，谁会收到，EventBus 应该怎么判断？

## 8. 候选人常见问题与标准回答

### 8.1 “这是不是 Kafka / RabbitMQ 题？”

推荐回答：

> 不是。先只做单进程、内存内版本。我们暂时不考虑 broker、持久化和集群，只看本地事件总线的设计。

### 8.2 “消息模型是什么？”

推荐回答：

> 最小模型是 `topic + payload`。如果你愿意，可以继续扩展成 typed message 或者引入 codec，但基础版不强制。

### 8.3 “订阅是广播还是队列语义？”

推荐回答：

> 默认是广播，所有命中的订阅者都会收到。只有 shared subscription 才是组内单播。

### 8.4 “要不要异步、线程池？”

推荐回答：

> 基础题先不强制。你可以先把抽象、路由、匹配和投递语义讲清楚。如果设计稳定了，我们再讨论异步化。

### 8.5 “要不要 ACK、重试、顺序保证？”

推荐回答：

> 这题不要求 ACK 和重试。顺序方面，你可以重点讲清楚在单次投递里多个订阅者之间的顺序规则。

### 8.6 “通配符规则有多严格？”

推荐回答：

> `*` 匹配一层，`**` 匹配零到多层。你可以基于这两个规则去定义匹配算法和索引结构。

## 9. 推荐面试推进节奏

下面是一套比较稳的推进方式。

### 第 1 轮：先让候选人复述问题

看点：

- 是否主动确认边界
- 是否能把题目抽象成“本地 pub/sub 路由器”

面试官可引导：

> 你先别急着实现，先讲讲你理解的需求、边界和你打算先设计哪些核心对象。

### 第 2 轮：核心抽象与 API

看点：

- 有没有抽象出 `EventBus`
- 有没有抽象出订阅对象而不是只传裸参数
- 有没有取消订阅能力
- 有没有 topic 与 payload 的消息模型

理想方向：

- `EventBus`
- `Subscription`
- `TopicPayload`
- `Disposable` 或 unsubscribe handle

### 第 3 轮：Topic 匹配与数据结构

看点：

- 是否只想到 `Map<String, List<Handler>>`
- 是否意识到 wildcard 会让线性扫描退化
- 是否想到用层级树或 trie 来存订阅关系

面试官追问：

> 如果订阅量很大，而且 pattern 含有 `*` 和 `**`，你还会用简单线性扫描吗？

### 第 4 轮：语义定义

看点：

- 默认广播语义是否清晰
- shared 是否定义准确
- priority 是否定义准确
- 同优先级下是否有稳定顺序

### 第 5 轮：生命周期与并发

看点：

- 取消订阅怎么做
- 发布时和取消订阅同时发生怎么办
- 锁应该保护哪些部分
- 是否把用户回调放在锁外执行

### 第 6 轮：扩展性

看点：

- 是否想到把编解码从 EventBus 本体中解耦
- 是否能给出后续异步化、监控化、broker 化的演进方向

## 10. 面试官应该期待听到什么

### 10.1 合格答案

- 能先讲清楚边界
- 有基本 API
- 能说明 publish / subscribe / unsubscribe
- 能给出 wildcard 的基本匹配逻辑

### 10.2 较好答案

- 会把订阅配置抽象成 `Subscription`
- 会把消息抽象成 `topic + payload`
- 会想到用 topic tree / trie 做匹配
- 会提到取消订阅和并发问题

### 10.3 很好答案

- 清楚区分 broadcast、shared、priority
- 能说明 shared 组内如何只选一个
- 能说明优先级和稳定顺序
- 会把锁和用户回调分离
- 会考虑异常隔离和监控指标

### 10.4 优秀答案

- 会主动做范围收敛
- 会指出哪些是基础能力，哪些是可扩展能力
- 会设计 codec 抽象但不让 EventBus 依赖具体序列化实现
- 会给出明确演进路线

## 11. 评分维度

### 11.1 需求澄清

- 有没有主动确认范围
- 有没有识别题目不是分布式 MQ

### 11.2 抽象设计

- 核心对象是否自然
- 职责边界是否清晰

### 11.3 数据结构

- 是否能为 topic 匹配选到合适结构
- 是否能解释为什么这么选

### 11.4 语义准确性

- 广播、shared、priority 是否定义准确
- 同优先级顺序是否可解释

### 11.5 工程性

- 取消订阅
- 异常处理
- 并发安全
- 可扩展性

### 11.6 沟通能力

- 能否把抽象问题讲具体
- 能否在被追问时收敛而不是发散

## 12. 面试官追问模板

### 12.1 当候选人一开始就往分布式发散

> 我们先把范围收窄到单进程内存版，把本地语义讲清楚。  
> 等基础版本稳定后，你可以再讲怎么演进到 broker 模型。

### 12.2 当候选人一直停留在 API 表层

> 如果订阅量很大，topic pattern 又带 wildcard，你觉得底层怎么组织订阅关系比较合适？

### 12.3 当候选人没有定义 shared 的语义

> 如果同一个 consumer group 下有多个实例，默认广播还合适吗？  
> 有没有办法让同组内一条消息只投给一个实例？

### 12.4 当候选人没有定义顺序

> 如果多个订阅者都命中，你希望按什么顺序投递？  
> 如果优先级一样，结果要不要稳定？

### 12.5 当候选人忽略并发

> 如果 publish 正在遍历订阅者，同时另一个线程在 unsubscribe，你觉得应该怎么保证语义和性能？

### 12.6 当候选人忘记异常隔离

> 如果某个订阅者处理消息时报错，你希望影响其他订阅者吗？

## 13. 一句话版题目

如果你想给候选人一个更短的版本，可以直接这样说：

> 请设计一个单进程内的 EventBus：模块可以往某个 `topic` 发布消息，其他模块按 `topic pattern` 订阅；支持 `*` 和 `**`，默认广播，可取消订阅。  
> 如果基础版讲清楚了，我们再继续讨论 shared subscription、priority 和 typed subscribe。

## 14. 建议的面试结论标准

### 可以判为“理解较深”

- 主动收敛边界
- 把 topic 路由、匹配、投递分层
- 用合适数据结构支撑 wildcard
- 语义定义清晰且稳定
- 并发边界和错误隔离意识较强

### 可以判为“理解一般”

- 会说 Pub/Sub
- 会设计 API
- 但数据结构和语义定义较模糊
- 对 shared、priority、取消订阅缺乏系统回答

### 可以判为“理解偏弱”

- 长时间无法把题目从 MQ 拉回到本地事件总线
- 不能清楚描述最小模型
- 只会说一堆组件名，但无法落地 API、语义和数据结构
