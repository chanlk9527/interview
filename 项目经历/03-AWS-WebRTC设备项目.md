# AWS WebRTC 设备项目 — 面试题

## 项目概述

基于 AWS Kinesis Video Streams (KVS) with WebRTC 构建 IPC 设备的实时音视频传输系统，实现低延迟直播、双向对讲和历史回放。后端核心职责是信令通道管理、STS 动态鉴权、以及 HLS/DASH 回放地址的生成。

---

## Q1: 这个项目的整体架构是怎样的？你们后端在里面承担什么角色？

**标准回答话术：**

> 整体架构可以分成三个角色：**设备端（Master）、云端（AWS KVS + 我们的后端服务）、客户端（Viewer，也就是用户 APP）**。
>
> WebRTC 的核心特点是**点对点传输**，音视频数据建立连接后直接在设备和 APP 之间流转，不经过我们的业务服务器。但建立这个点对点连接之前，需要一个**信令交换**的过程——设备和 APP 要互相交换 SDP（会话描述）和 ICE Candidate（网络候选地址），这个交换就是通过 AWS KVS 的 Signaling Channel 来完成的。
>
> 我们后端在这里面的角色，主要是三件事：
>
> **第一，信令通道的生命周期管理。** 每个设备对应一个 Signaling Channel，设备激活时我们调 AWS SDK 创建通道，设备注销时删除。同时维护 `deviceId → channelARN` 的映射关系。
>
> **第二，安全鉴权。** 这是后端最核心的职责。设备和 APP 要连接 Signaling Channel，需要 AWS 的访问凭证。但我们不可能把 AWS 的 AK/SK 直接下发给客户端，所以后端通过 **STS（Security Token Service）** 签发临时凭证，凭证里通过 IAM Policy 精确控制权限——比如这个用户只能以 Viewer 身份连接某个特定的 Channel，不能连别人的设备。
>
> **第三，回放地址生成。** KVS 支持把视频流存储下来，用户要看历史录像时，后端调 KVS 的 API 生成带签名的 HLS 播放地址，有效期很短（比如几分钟），过期就失效，防止地址泄露。
>
> 简单说就是：**后端不碰音视频数据本身，但控制着"谁能看、看哪个设备、看多久"这些权限和入口。**

## Q2: STS 动态鉴权具体是怎么做的？为什么不直接用固定的 AK/SK？

**标准回答话术：**

> 固定 AK/SK 的问题很明显：一旦泄露，攻击者就能访问所有设备的视频流，而且你没法针对单个用户做细粒度的权限控制。
>
> 我们的做法是用 **AWS STS 的 AssumeRole** 接口，动态签发临时凭证。整个流程是这样的：
>
> 1. 用户在 APP 上点击某个设备的"实时预览"
> 2. APP 请求我们的后端接口，带上用户 token 和设备 ID
> 3. 后端先做**业务鉴权**：验证这个用户是否有权限查看这个设备（比如设备是不是绑定在他名下、云服务套餐是否过期）
> 4. 鉴权通过后，后端调用 STS AssumeRole，传入一个**会话策略（Session Policy）**，这个策略精确限定了：只能以 Viewer 角色连接这一个 Signaling Channel（通过 Channel ARN 限定），不能做其他任何操作
> 5. STS 返回临时的 AccessKeyId、SecretAccessKey、SessionToken，有效期设为 **15 分钟**（最短可以 15 分钟，最长 12 小时）
> 6. 后端把临时凭证 + Signaling Channel 的 WSS Endpoint 一起返回给 APP
> 7. APP 拿着临时凭证去连接 Signaling Channel，完成信令交换，建立 WebRTC 连接
>
> 这样做的好处是：**每次连接都是独立的临时凭证，权限最小化，过期自动失效**。即使凭证被截获，攻击者也只能在 15 分钟内访问这一个设备，不会影响其他设备。
>
> 设备端（Master）的凭证也是类似的逻辑，但设备端的凭证有效期会长一些（比如 1 小时），因为设备需要持续保持在 Signaling Channel 上等待 Viewer 连入。设备端会在凭证过期前主动向后端刷新。

## Q3: WebRTC 的信令交换流程能详细说一下吗？

**标准回答话术：**

> WebRTC 建立连接需要经过**信令交换**，核心就是交换两样东西：**SDP（Session Description Protocol）** 和 **ICE Candidate**。
>
> SDP 描述的是媒体能力——我支持什么编码格式（H.264/VP8）、音频用什么采样率、传输用什么协议。ICE Candidate 描述的是网络路径——我的 IP 地址、端口、是局域网地址还是公网地址、有没有经过 TURN 中继。
>
> 具体流程是这样的：
>
> 1. **设备（Master）启动后**，用后端下发的临时凭证连接到 Signaling Channel，以 Master 身份注册，进入等待状态
> 2. **用户打开 APP 点实时预览**，APP 先请求后端拿到临时凭证和 WSS 地址，然后以 Viewer 身份连接到同一个 Signaling Channel
> 3. **Viewer 发起 Offer**：APP 创建本地的 SDP Offer（描述自己的媒体能力），通过 Signaling Channel 发给 Master
> 4. **Master 回复 Answer**：设备收到 Offer 后，创建 SDP Answer 回传给 Viewer
> 5. **双方交换 ICE Candidate**：同时，双方各自通过 STUN 服务器探测自己的公网地址，把发现的 Candidate 通过 Signaling Channel 发给对方
> 6. **ICE 连通性检查**：双方尝试用收到的 Candidate 建立直连。如果能直连就走 P2P；如果双方都在 NAT 后面直连不通，就走 **TURN 中继服务器**转发
> 7. **连接建立**：DTLS 握手完成后，音视频数据开始通过 SRTP 加密传输
>
> AWS KVS 提供了 STUN 和 TURN 服务器，我们不需要自己搭建。后端在这个过程中不参与信令的具体转发，信令是通过 AWS 的 Signaling Channel（WebSocket）直接中转的。后端的角色就是**在连接建立之前，确保双方都拿到了合法的凭证和正确的 Channel 地址**。

## Q4: 实时直播和历史回放是怎么切换的？

**标准回答话术：**

> 实时直播和历史回放走的是**完全不同的技术路径**，但对用户来说是无缝切换的。
>
> **实时直播**走的是 WebRTC，就是前面说的信令交换 → P2P 连接 → SRTP 传输。延迟可以做到亚秒级（Sub-second），适合实时监控和双向对讲。
>
> **历史回放**走的是 HLS/DASH 流媒体协议。KVS 支持把 WebRTC 传输的视频流同时存储到云端（通过配置 Media Storage），存储格式是 MKV 片段。用户要看回放时，后端调用 KVS 的 `GetHLSStreamingSessionURL` API，传入时间范围，KVS 会返回一个带签名的 HLS 播放地址。APP 拿到这个地址后用标准的 HLS 播放器就能播放。
>
> 后端在这里做的事情：
> - 根据用户请求的时间段，调 KVS API 生成 HLS URL
> - URL 设置较短的有效期（5~10 分钟），防止泄露
> - 做权限校验，确保用户只能回放自己设备的录像
> - 如果用户的云存储套餐过期了，直接拒绝回放请求
>
> 切换的体验就是：用户在 APP 上看实时画面时走 WebRTC，往回拖时间轴就自动切到 HLS 回放，松手回到当前时间又切回 WebRTC。这个切换逻辑主要在 APP 端实现，后端只需要提供两种模式的入口接口。

## Q5: 如果设备在 NAT 后面，WebRTC 怎么穿透的？

**标准回答话术：**

> 这是 WebRTC 的 **ICE 框架**解决的问题，核心靠两个东西：**STUN** 和 **TURN**。
>
> **STUN（Session Traversal Utilities for NAT）** 的作用是让设备和 APP 发现自己的公网 IP 和端口。因为在 NAT 后面的设备只知道自己的内网地址，不知道 NAT 给它映射的公网地址是什么。通过向 STUN 服务器发一个请求，STUN 服务器会告诉你"我看到你的地址是 x.x.x.x:yyyy"，这就是一个 **Server Reflexive Candidate**。
>
> 拿到公网地址后，双方尝试直连。如果 NAT 类型比较友好（比如 Full Cone NAT），直连就能通，这是最理想的情况，延迟最低。
>
> 但如果双方都在**对称型 NAT（Symmetric NAT）** 后面，STUN 穿透就会失败，因为对称型 NAT 对每个目标地址都映射不同的端口，STUN 探测到的端口和实际通信时用的端口不一样。这时候就要走 **TURN（Traversal Using Relays around NAT）** 中继——音视频数据先发到 TURN 服务器，再由 TURN 转发给对方。代价是延迟会增加（多了一跳），而且 TURN 服务器的带宽成本比较高。
>
> AWS KVS 自带 STUN 和 TURN 服务，我们不需要自己部署。后端在生成临时凭证的时候，会同时调用 `GetIceServerConfig` API 拿到 STUN/TURN 服务器的地址和临时密码，一起返回给客户端。
>
> 从我们的实际数据来看，大部分场景（大概 70%~80%）STUN 穿透就够了，只有少数复杂网络环境才需要走 TURN。

## Q6: 多个用户同时观看同一个设备怎么处理？

**标准回答话术：**

> AWS KVS 的 Signaling Channel 支持**一个 Master 对多个 Viewer** 的模式，一个设备最多可以同时被 10 个 Viewer 观看（这是 AWS 的限制）。
>
> 每个 Viewer 和 Master 之间是独立的 WebRTC 连接，信令交换也是独立的。也就是说，如果有 3 个用户同时看一个摄像头，设备端需要同时维护 3 路 WebRTC 连接，编码 3 份视频流。
>
> 这对设备端的性能是有压力的，尤其是 IPC 这种嵌入式设备，CPU 和带宽都有限。我们的处理方式是：
>
> - **后端做并发控制**：维护每个设备当前的 Viewer 数量，超过阈值（比如 3 个）就拒绝新的观看请求，返回"设备忙"的提示
> - **优先级管理**：设备主人的观看请求优先级最高，分享给其他人的链接优先级低，资源紧张时优先保障主人的连接
> - **超时自动断开**：如果 Viewer 长时间没有交互（比如 APP 切到后台），后端会通知设备端主动断开这路连接，释放资源

## Q7: 这个项目中遇到过什么比较棘手的问题？

**标准回答话术：**

> 印象比较深的有两个：
>
> **第一个是凭证过期导致的连接中断。** 早期我们给 Viewer 签发的 STS 临时凭证有效期设的是 15 分钟，但用户看直播经常超过 15 分钟。凭证过期后 Signaling Channel 的 WebSocket 连接会被 AWS 断开，用户端就黑屏了。
>
> 解决方案是在 APP 端做了**凭证预刷新机制**：在凭证过期前 2 分钟，APP 自动向后端请求新的凭证，用新凭证重新建立 Signaling 连接和 WebRTC 连接。这个过程对用户来说会有一个很短暂的画面闪烁（几百毫秒），但比直接黑屏好得多。后来我们把 Viewer 的凭证有效期调长到 1 小时，刷新频率就低多了。
>
> **第二个是 TURN 中继的成本问题。** 有一段时间发现 AWS 的 KVS 账单突然涨了不少，排查下来是因为某些地区的用户网络环境差，大量走了 TURN 中继，TURN 的流量费比 P2P 贵很多。我们的应对是在后端加了监控，统计每个设备的连接方式（P2P vs TURN），对于长期走 TURN 的设备，在 APP 端提示用户优化网络环境。同时和 AWS 确认了 TURN 的计费模型，在成本预算里做了相应的预留。

## Q8: 为什么选 AWS KVS 而不是自己搭建 WebRTC 服务？

**标准回答话术：**

> 自建 WebRTC 服务的工作量非常大。你需要自己搭建 **Signaling Server**（信令服务器）、**STUN/TURN Server**（穿透和中继服务器），还要处理 TURN 的全球部署和带宽成本。光一个 TURN 服务器的运维和带宽费用就不低，而且要保证全球各地的用户都能就近接入。
>
> AWS KVS 把这些基础设施全包了——Signaling Channel、STUN、TURN 都是托管服务，按用量计费，不需要我们运维。而且 KVS 还自带视频存储能力，直播流可以直接存到云端，回放时生成 HLS 地址就行，不需要我们自己搭流媒体存储。
>
> 对我们来说，核心价值在 IPC 设备的业务功能（智能识别、告警、云存储套餐），WebRTC 只是一个传输通道。把传输层交给 AWS 托管，我们专注在业务层，这个投入产出比是最合理的。
>
> 当然也有代价：对 AWS 有依赖，KVS 的一些限制（比如单 Channel 最多 10 个 Viewer）我们只能接受，没法自己改。如果未来设备量大到一定程度，成本上可能需要重新评估。
