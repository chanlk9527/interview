# 运营报表后台系统

## 一、项目背景与业务概述

### 1.1 项目定位

运营报表后台系统是公司内部的运营管理平台（B端后台），服务于运营团队、产品团队和客服团队。核心目标是：
- **精细化运营**：通过用户画像和规则引擎，对不同用户群体进行精准的活动推送与云服务营销，提升付费转化率
- **数据驱动决策**：构建多维度数据报表，为运营策略调整提供数据支撑
- **售后支撑**：提供一站式客服工作台，提升客诉处理效率

### 1.2 业务背景

公司主营IPC（网络摄像头）产品，盈利模式为 **硬件 + 云服务增值订阅**。用户购买摄像头后，可以通过APP内商城购买云存储、AI识别等增值服务套餐。运营后台的核心使命就是：
- 通过营销活动提升云服务的付费转化率和续费率
- 通过数据分析发现业务增长点和问题
- 通过售后工具高效处理用户问题，降低退订率

## 二、精细化运营与营销系统

### 2.1 活动管理模块

运营人员可以在后台创建和管理营销活动，活动创建流程分为两步：**基本信息配置** → **活动内容配置**。

#### 2.1.1 基本信息配置

| 配置项 | 说明 |
|--------|------|
| **产品策略** | 开关控制，决定是否启用产品维度的策略（如针对特定产品线的活动） |
| **选择商户** | 支持多商户体系（如Foscam等不同品牌），活动可按商户维度隔离 |
| **活动名称** | 活动的标识名称 |
| **活动时间段** | 活动的生效时间范围，支持时区配置（UTC+8），面向全球用户需考虑时区差异 |
| **优先级** | 数字越大，展示位置越靠前。用于解决多个活动同时生效时的展示冲突 |
| **广告类型** | 分为 **普通广告**（品牌宣传、功能介绍）和 **云服务促销广告**（直接关联云服务套餐购买，如折扣、优惠券） |
| **目标人群** | 支持 **指定用户** 模式，通过"添加规则"配置用户筛选条件（基于用户画像） |
| **多设备购买策略** | 控制用户是否可以为多台设备重复参与同一活动（支持/不支持） |
| **备注** | 活动备注信息，最多200字符 |

#### 2.1.2 目标人群规则引擎（核心亮点）

"添加规则"是活动精准投放的核心能力。系统设计了一套 **多维度、可组合的规则引擎**，运营人员通过配置化方式圈选目标用户，无需开发介入。规则体系分为 **五大规则组**，每组内条件取 AND 关系，组间取 AND 关系，实现精细化人群圈选。

##### A. 触发规则（用户基础属性）

用户维度的基础筛选条件，决定活动"给谁看"：

| 规则字段 | 操作符 | 说明 |
|----------|--------|------|
| 注册时间 | 大于/小于/区间 | 按注册时间范围筛选，如"注册超过30天的老用户" |
| 是否首次注册绑定设备 | 是/否 | 区分新用户（刚绑定第一台设备）和老用户 |
| 产品型号 | 包含/不包含 | 指定设备型号，如只针对某款摄像头用户 |
| 目标APP版本 | 大于/小于/等于 | 控制活动只在特定APP版本展示（兼容性考虑） |
| 国家 | 包含/不包含 | 按国家/地区投放，支持多选 |
| 语言 | 包含/不包含 | 按APP语言设置筛选 |
| 设备数量 | 大于/小于/等于/区间 | 如"绑定3台以上设备的用户"推送多设备套餐 |
| 云服务状态 | 枚举选择 | 试用中/付费中/已过期/从未开通等 |
| 用户来源渠道 | 包含/不包含 | 区分不同获客渠道的用户 |
| 平台 | iOS/Android | 按操作系统筛选 |
| 是否VIP用户 | 是/否 | 区分高价值用户 |

##### B. 设备规则

设备维度的筛选条件，精确到设备级别：

| 规则字段 | 操作符 | 说明 |
|----------|--------|------|
| 设备型号/系列 | 包含/不包含 | 精确到具体型号或产品系列 |
| 固件版本 | 大于/小于/等于 | 如只对新固件用户开放某功能的促销 |
| 设备在线状态 | 在线/离线 | 只对活跃设备用户推送 |
| 设备绑定时间 | 大于/小于/区间 | 如"绑定超过7天但未开通云服务" |
| 设备能力集 | 包含 | 如设备是否支持AI识别、是否支持云存储 |

##### C. 授权类型相关的活动规则（云服务订阅状态）

与云服务订阅/授权状态相关的规则，这是 **转化率提升的关键维度**：

| 规则字段 | 操作符 | 说明 |
|----------|--------|------|
| 是否有试用资格 | 是/否 | 筛选从未试用过的用户，推送免费试用 |
| 当前订阅套餐类型 | 包含/不包含 | 如只针对基础套餐用户推送升级优惠 |
| 订阅到期时间 | 大于/小于/区间 | 如"7天内到期"的用户推送续费提醒 |
| 是否自动续费 | 是/否 | 未开启自动续费的用户推送续费引导 |
| 历史订阅次数 | 大于/小于/等于 | 区分首次付费和多次续费用户 |
| 试用剩余天数 | 大于/小于 | 试用即将结束时推送付费转化 |
| 授权来源 | 枚举 | 区分自购、赠送、活动获得等来源 |

##### D. 交易规则（订单与支付维度）

基于用户的交易行为进行筛选：

| 规则字段 | 操作符 | 说明 |
|----------|--------|------|
| 累计消费金额 | 大于/小于/区间 | 区分高价值和低价值用户 |
| 最近一次购买时间 | 大于/小于 | 如"超过90天未购买"的沉默用户 |
| 历史购买次数 | 大于/小于/等于 | 区分新客和老客 |
| 支付渠道 | 包含/不包含 | 如只针对Apple IAP用户的专属活动 |
| 是否有退款记录 | 是/否 | 避免对频繁退款用户投放高价值优惠券 |

##### E. 模板规则（活动模板匹配）

控制活动与已有模板/历史活动的关联关系：

| 规则字段 | 操作符 | 说明 |
|----------|--------|------|
| 是否参与过指定活动 | 是/否 | 避免重复推送，或做二次触达 |
| 是否领取过指定优惠券 | 是/否 | 防止同类优惠券重复发放 |
| 历史活动参与次数 | 大于/小于 | 控制活动触达频率，避免过度打扰 |

##### 规则组合示例

```
触发规则:
  注册时间 > 30天 AND 云服务状态 = 已过期 AND 国家 IN (US, UK, DE)
  
AND 设备规则:
  设备型号 IN (C2, R2, R4) AND 设备在线状态 = 在线
  
AND 授权规则:
  历史订阅次数 >= 1 AND 是否自动续费 = 否
  
AND 交易规则:
  累计消费金额 > $10 AND 最近购买时间 > 90天前

→ 命中人群：曾经付费但已流失的高价值海外用户
→ 营销策略：推送"老用户回归专享5折续费券"
```

##### 规则引擎技术实现（面试重点）

**1. 规则存储模型**

采用 **规则组 + 规则项** 的两层结构存储：

```sql
-- 规则组表（一个活动可以有多个规则组）
CREATE TABLE activity_rule_group (
    id BIGINT PRIMARY KEY,
    activity_id BIGINT NOT NULL,        -- 关联活动
    group_type VARCHAR(32) NOT NULL,     -- 规则组类型：trigger/device/license/trade/template
    logic_op VARCHAR(8) DEFAULT 'AND',   -- 组内逻辑：AND/OR
    INDEX idx_activity_id (activity_id)
);

-- 规则项表（每个规则组下的具体条件）
CREATE TABLE activity_rule_item (
    id BIGINT PRIMARY KEY,
    group_id BIGINT NOT NULL,            -- 关联规则组
    field_code VARCHAR(64) NOT NULL,     -- 规则字段编码：register_time, country, device_model...
    operator VARCHAR(16) NOT NULL,       -- 操作符：eq, neq, gt, lt, in, not_in, between
    field_value TEXT NOT NULL,            -- 值（JSON格式，支持单值/多值/区间）
    INDEX idx_group_id (group_id)
);
```

**2. 规则匹配引擎 — 两种模式**

系统支持两种匹配模式，适用于不同场景：

**模式一：推式匹配（Push模式）— 活动生效时批量圈人**

活动生效时，系统主动扫描用户库，筛选出符合规则的目标用户：

```
活动生效 → 解析规则组 → 动态拼接SQL/ES查询 → 批量查询命中用户 → 写入推送队列
```

- 将规则条件动态翻译为SQL WHERE子句或ES Query DSL
- 适用于 Push推送、邮件营销等需要批量触达的场景
- 大用户量下分批查询 + MQ异步推送，避免DB压力

```java
// 伪代码：规则条件动态拼接SQL
public String buildWhereClause(List<RuleItem> rules) {
    return rules.stream()
        .map(rule -> {
            switch (rule.getOperator()) {
                case "gt": return rule.getFieldCode() + " > " + rule.getValue();
                case "in": return rule.getFieldCode() + " IN (" + rule.getValue() + ")";
                case "between": return rule.getFieldCode() + " BETWEEN " + rule.getMin() + " AND " + rule.getMax();
                // ...
            }
        })
        .collect(Collectors.joining(" AND "));
}
```

**模式二：拉式匹配（Pull模式）— 用户请求时实时判定**

用户打开APP时，实时判断该用户命中了哪些活动：

```
用户打开APP → 请求活动列表 → 加载所有生效中的活动规则 → 逐条匹配用户属性 → 返回命中的活动
```

- 适用于 Banner广告、开屏广告、H5广告、APP活动中心等展示类场景
- 用户属性从缓存（Redis）中获取，规则在内存中执行匹配
- 活动规则缓存到本地/Redis，避免每次请求查DB

```java
// 伪代码：内存规则匹配
public boolean matchUser(UserProfile user, ActivityRule rule) {
    // 逐个规则组匹配，组间AND
    return rule.getGroups().stream().allMatch(group -> {
        // 组内规则项匹配
        return group.getItems().stream().allMatch(item -> {
            Object userValue = user.getAttribute(item.getFieldCode());
            return evaluate(userValue, item.getOperator(), item.getFieldValue());
        });
    });
}
```

**3. 性能优化策略**

| 优化点 | 方案 |
|--------|------|
| **规则缓存** | 活动规则加载到Redis/本地缓存，活动变更时通过消息通知刷新 |
| **用户画像预计算** | 用户核心属性（设备数、订阅状态、消费金额等）异步预计算并写入用户画像表，避免实时关联查询 |
| **批量匹配分片** | Push模式下，按用户ID分片并行查询，每批1000~5000条，通过MQ分发 |
| **活动预筛** | Pull模式下，先按简单条件（国家、平台）快速过滤掉不可能命中的活动，减少精确匹配次数 |
| **命中结果缓存** | 用户命中的活动列表缓存到Redis，设置短TTL（如5分钟），避免重复计算 |

**4. 多活动冲突处理**

当一个用户同时命中多个活动时：
- **优先级排序**：按活动优先级字段排序，数字越大越靠前
- **互斥规则**：同类型活动（如两个开屏广告）只展示优先级最高的一个
- **频控策略**：限制用户单日/单周被推送的活动次数，避免过度打扰

#### 2.1.3 活动内容与触达渠道

活动创建的第二步是配置活动内容，系统支持 **6种触达渠道**，每种渠道可独立开关：

| 渠道 | 说明 | 特点 |
|------|------|------|
| **消息推送（Push）** | 直接推送系统消息到用户手机端 | 触达率高，实时性强 |
| **邮件营销** | 以邮件形式向用户发送广告内容 | 内容可自定义，对用户干扰度低 |
| **APP活动中心** | APP内置活动中心展示平台 | 图文形式展示，用户主动浏览 |
| **H5广告** | APP首页和IPC直播页的广告 | 支持自定义H5网页，内容丰富多样 |
| **Banner广告** | 常规横幅广告 | 广告曝光度好，吸引用户点击，但对用户干扰强 |
| **开屏广告** | APP启动时弹出的广告 | 干扰用户体验，但广告效果较好 |

每种渠道可以独立配置内容（图片、文案、跳转链接等），运营人员可以根据活动目标灵活组合多种渠道。

### 2.2 活动生命周期管理

```
创建(草稿) → 审核 → 待生效 → 生效中 → 已结束
                ↓
              驳回
```

- **草稿**：运营人员创建活动，可反复编辑
- **审核**：提交后由运营主管审核，防止误操作
- **待生效**：审核通过后，等待活动开始时间
- **生效中**：活动时间段内，系统自动按规则投放
- **已结束**：活动到期自动结束，也支持手动提前终止

## 三、优惠券系统

### 3.1 业务场景

优惠券是云服务促销广告的核心营销工具，与商城支付系统紧密配合。主要应用场景：

- **新用户首购优惠**：注册后发放体验券，引导首次付费
- **续费挽留**：订阅即将到期时发放续费折扣券
- **升级促销**：引导用户从基础套餐升级到高级套餐
- **节日/活动促销**：黑五、双十一等节点的限时优惠
- **售后补偿**：客诉处理时发放补偿券，提升用户满意度

### 3.2 优惠券类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **折扣券** | 按比例折扣 | 8折续费券、首月5折券 |
| **满减券** | 满足金额条件后减免 | 满$10减$3 |
| **固定金额券** | 直接抵扣固定金额 | $2抵扣券 |
| **免费体验券** | 免费使用指定时长的云服务 | 7天云存储免费体验 |
| **首期特惠券** | 首期订阅特价，后续恢复原价 | 首月$0.99（原价$5.99） |

### 3.3 优惠券核心属性

```
优惠券模板（CouponTemplate）
├── 基本信息：名称、类型、面值/折扣率
├── 使用条件：
│   ├── 适用套餐：指定SKU / 全部套餐
│   ├── 适用支付方式：单次购买 / 订阅
│   ├── 最低消费金额（满减场景）
│   └── 是否可与其他优惠叠加
├── 发放规则：
│   ├── 发放方式：系统自动发放 / 运营手动发放 / 活动领取
│   ├── 发放数量上限
│   └── 每人限领数量
├── 有效期规则：
│   ├── 固定时间段（2024-01-01 ~ 2024-01-31）
│   └── 领取后N天有效
└── 状态：草稿 / 已发布 / 已停用
```

### 3.4 优惠券生命周期

```
创建模板 → 发布 → 发放到用户 → 用户领取/系统发放 → 使用(核销) → 过期/作废
```

**核销流程**（与商城支付系统对接）：
1. 用户在APP商城选择云服务套餐
2. 进入支付页面，系统自动匹配可用优惠券并展示
3. 用户选择优惠券，系统计算优惠后金额
4. 用户确认支付，调用支付网关（Stripe/PayPal等）
5. 支付成功后，优惠券状态标记为"已使用"，关联订单号
6. 若订单退款，优惠券根据策略决定是否退回

### 3.5 优惠券与活动的关联

优惠券通过运营活动进行分发，典型流程：
1. 运营人员创建优惠券模板
2. 创建营销活动，广告类型选择"云服务促销广告"
3. 活动内容关联优惠券模板
4. 配置目标人群规则（如：试用期即将结束的用户）
5. 选择触达渠道（如：Push推送 + APP活动中心）
6. 活动生效后，系统自动向目标用户发放优惠券并推送通知

## 四、多维数据分析平台（报表中心）

### 4.1 核心报表维度

| 报表类别 | 核心指标 |
|----------|----------|
| **设备报表** | 设备日活/月活、新增激活数、在线率、设备型号分布、固件版本分布 |
| **用户报表** | 注册用户数、日活/月活、留存率（次日/7日/30日）、地区分布 |
| **订单报表** | 订单量、营收金额、ARPU（每用户平均收入）、支付渠道分布 |
| **套餐报表** | 各套餐销量、套餐分布、续费率、升级/降级率、退订率 |
| **活动报表** | 活动曝光量、点击率、转化率、优惠券发放/使用/过期数量、ROI |
| **客服报表** | 工单量、处理时长、退款金额、客诉类型分布 |

### 4.2 报表输出形式

- **实时看板**：关键指标实时刷新，用于日常监控
- **日报/月报**：定时生成，支持邮件自动推送给运营团队
- **自定义查询**：支持按时间范围、商户、地区等维度筛选
- **数据导出**：支持导出Excel，供运营团队二次分析

## 五、全链路售后支撑体系（客服工作台）

### 5.1 设备运维

- **OTA升级管理**：远程推送固件升级，支持灰度发布
- **设备解绑**：协助用户解绑设备（如二手转让场景）
- **厂测数据清洗**：清理出厂测试阶段产生的脏数据
- **设备诊断**：查看设备在线状态、网络质量、最近上报日志

### 5.2 订单售后

- **全额退款**：订单全额退款，同步取消云服务权益
- **部分退款**：按比例退款（如按剩余天数折算）
- **取消订阅**：取消自动续费，当前周期权益保留至到期
- **补偿发券**：向用户发放补偿优惠券
- **订单查询**：按用户ID/订单号/设备ID等多维度查询订单详情

## 六、技术要点（面试可能追问）

### 6.1 活动投放的并发与一致性
- 活动生效时需要批量推送给大量用户，如何避免瞬时流量冲击？→ MQ削峰 + 分批推送
- 优惠券发放如何防止超发？→ Redis原子操作（DECR）控制库存 + 数据库兜底校验
- 多个活动同时命中同一用户，如何决定展示哪个？→ 优先级排序 + 互斥规则

### 6.2 报表数据的实时性与准确性
- 实时指标：基于Redis计数器（如日活UV用HyperLogLog）
- 离线报表：定时任务聚合MySQL数据，生成报表快照
- 大数据量查询优化：预聚合 + 分区表 + 读写分离

### 6.3 优惠券防刷与安全
- 每人限领限制：Redis + 用户ID做幂等校验
- 优惠券码防猜测：使用随机码 + 校验位
- 核销幂等：订单号 + 优惠券ID联合唯一索引，防止重复核销
- 退款场景的券退回策略：根据业务规则决定是否退回（如全额退款退回，部分退款不退回）
