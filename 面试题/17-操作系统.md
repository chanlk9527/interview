# 17 - 操作系统

---

## 一、进程与线程

1. 进程和线程的区别？

**回答：**

进程是操作系统**资源分配的基本单位**，线程是**CPU 调度的基本单位**。

| 对比维度 | 进程 | 线程 |
|---------|------|------|
| 资源 | 拥有独立的地址空间、文件描述符、堆栈等 | 共享所属进程的地址空间和资源，仅有独立的栈和寄存器 |
| 开销 | 创建/销毁/切换开销大（需要切换页表、刷新 TLB） | 创建/销毁/切换开销小（共享地址空间，无需切换页表） |
| 通信 | 需要 IPC 机制（管道、共享内存、Socket 等） | 可直接读写共享变量（但需要同步机制） |
| 隔离性 | 进程间相互隔离，一个进程崩溃不影响其他进程 | 同一进程内的线程共享内存，一个线程崩溃可能导致整个进程崩溃 |
| 并发性 | 多进程可以利用多核 CPU | 多线程同样可以利用多核 CPU，且切换更轻量 |

**结合 Java 来说：**
- JVM 本身是一个进程，Java 程序中的线程对应操作系统的内核线程（1:1 模型）
- Java 线程共享堆内存，每个线程有独立的虚拟机栈、程序计数器、本地方法栈
- 多线程编程需要关注线程安全问题，使用 synchronized、ReentrantLock 等同步机制

2. 进程间通信（IPC）的方式？管道、消息队列、共享内存、信号量、Socket？

**回答：**

进程间通信（Inter-Process Communication）主要有以下几种方式：

| IPC 方式 | 原理 | 特点 | 适用场景 |
|---------|------|------|---------|
| **管道（Pipe）** | 内核中的一块缓冲区，半双工通信，数据单向流动 | 只能用于有亲缘关系的进程（父子进程）；命名管道（FIFO）可用于无亲缘关系进程 | Shell 中的 `cmd1 \| cmd2` |
| **消息队列** | 内核中的消息链表，进程可以向队列发送/接收消息 | 消息有类型，可以按类型读取；有大小限制；生命周期随内核 | 进程间传递结构化数据 |
| **共享内存** | 多个进程映射同一块物理内存到各自的地址空间 | **速度最快**，无需内核拷贝；但需要配合信号量等同步机制 | 大量数据交换，如数据库的 Buffer Pool |
| **信号量（Semaphore）** | 一个计数器，用于控制多个进程对共享资源的访问 | 本质是同步机制，不是数据传输；P 操作减一，V 操作加一 | 配合共享内存使用，控制并发访问 |
| **信号（Signal）** | 软件层面的中断，通知进程某个事件发生 | 异步通信，信息量少（只有信号编号） | `kill -9` 杀进程、`Ctrl+C` 发送 SIGINT |
| **Socket** | 基于网络协议栈的通信方式，支持不同主机间通信 | 全双工，可跨网络；也可用 Unix Domain Socket 做本机通信 | 分布式系统、微服务间通信 |

**面试加分点：**
- 共享内存 + 信号量是经典组合：共享内存负责数据传输，信号量负责同步
- Redis、Nginx 等高性能中间件大量使用共享内存
- 微服务架构中，服务间通信本质上是基于 Socket 的 RPC 或 HTTP 调用

3. 线程间同步的方式？互斥锁、条件变量、信号量、读写锁？

**回答：**

线程共享进程的地址空间，访问共享资源时需要同步机制来保证数据一致性：

| 同步方式 | 原理 | 特点 | Java 对应 |
|---------|------|------|----------|
| **互斥锁（Mutex）** | 同一时刻只有一个线程能持有锁，其他线程阻塞等待 | 最基本的同步原语；保证互斥访问 | `synchronized`、`ReentrantLock` |
| **条件变量（Condition）** | 线程在条件不满足时等待，条件满足时被唤醒 | 必须配合互斥锁使用；避免忙等待 | `Object.wait()/notify()`、`Condition.await()/signal()` |
| **信号量（Semaphore）** | 维护一个计数器，允许多个线程同时访问（计数信号量） | 互斥锁是信号量的特例（计数为1） | `java.util.concurrent.Semaphore` |
| **读写锁（RWLock）** | 读锁共享、写锁独占；读读并发、读写/写写互斥 | 适合读多写少场景，提升并发度 | `ReentrantReadWriteLock` |
| **自旋锁（SpinLock）** | 获取不到锁时不阻塞，而是循环忙等待 | 适合锁持有时间极短的场景；避免线程切换开销 | CAS 操作（`AtomicInteger` 等） |

**操作系统层面 vs Java 层面的对应关系：**
- `synchronized` 底层依赖操作系统的 **pthread_mutex**（重量级锁状态）
- 轻量级锁、自适应自旋等是 JVM 层面的优化，尽量减少直接阻塞到操作系统互斥锁
- `LockSupport.park()/unpark()` 底层调用操作系统的 **futex**（Linux）

4. 协程是什么？和线程的区别？Java 的虚拟线程（Virtual Thread）？

**回答：**

**协程（Coroutine）** 是一种用户态的轻量级线程，由程序自身调度，不需要操作系统内核参与切换。

| 对比维度 | 线程 | 协程 |
|---------|------|------|
| 调度方 | 操作系统内核调度（抢占式） | 用户程序调度（协作式） |
| 切换开销 | 需要内核态/用户态切换，保存/恢复寄存器、切换栈（微秒级） | 仅保存/恢复少量寄存器和栈指针（纳秒级） |
| 内存占用 | 默认栈大小 1MB（Linux）或 512KB | 通常几 KB，可动态增长 |
| 并发数量 | 受限于系统资源，通常几千到几万 | 可以轻松创建数十万甚至百万个 |
| 阻塞行为 | 一个线程阻塞不影响其他线程 | 一个协程阻塞会阻塞其所在的载体线程（需要配合非阻塞 IO） |

**Java 虚拟线程（Virtual Thread，JDK 21 正式发布）：**
- 本质上是 JVM 层面实现的协程，采用 **M:N 模型**（M 个虚拟线程映射到 N 个平台线程）
- 虚拟线程在遇到阻塞操作（如 IO）时，JVM 会自动将其从载体线程（Carrier Thread）上卸载，让载体线程去执行其他虚拟线程
- 使用方式：`Thread.ofVirtual().start(() -> { ... })` 或 `Executors.newVirtualThreadPerTaskExecutor()`
- 适合 **IO 密集型** 场景（如大量 HTTP 请求、数据库查询），不适合 CPU 密集型
- 注意事项：虚拟线程中不要使用 `synchronized`（会 pin 住载体线程），应改用 `ReentrantLock`

**Go 的 goroutine 也是类似的 M:N 模型，** 是协程在工程实践中最成功的案例之一。

5. 进程调度算法？FCFS、SJF、优先级、时间片轮转、多级反馈队列？

**回答：**

| 调度算法 | 原理 | 优点 | 缺点 |
|---------|------|------|------|
| **FCFS（先来先服务）** | 按到达顺序排队执行 | 简单公平 | 短作业可能等很久（护航效应） |
| **SJF（短作业优先）** | 优先执行预计运行时间最短的进程 | 平均等待时间最短 | 长作业可能饥饿；难以预估运行时间 |
| **优先级调度** | 按优先级高低调度，优先级高的先执行 | 重要任务优先处理 | 低优先级进程可能饥饿（可用老化机制解决） |
| **时间片轮转（RR）** | 每个进程分配一个时间片，用完就切换到下一个 | 响应时间好，适合交互式系统 | 时间片太小→切换开销大；太大→退化为 FCFS |
| **多级反馈队列（MLFQ）** | 多个优先级队列，新进程进入最高优先级队列；用完时间片降级，等待久了升级 | 兼顾响应时间和吞吐量，自适应 | 实现复杂 |

**Linux 的实际调度策略：**
- Linux 使用 **CFS（Completely Fair Scheduler，完全公平调度器）**
- CFS 基于红黑树，按虚拟运行时间（vruntime）排序，vruntime 最小的进程优先调度
- 实时进程使用 SCHED_FIFO 或 SCHED_RR 策略，优先级高于普通进程
- nice 值范围 -20 到 19，值越小优先级越高

6. 上下文切换的开销？

**回答：**

上下文切换（Context Switch）是指 CPU 从一个进程/线程切换到另一个进程/线程时，保存当前状态并恢复目标状态的过程。

**切换过程中需要做的事情：**
1. **保存当前上下文：** 寄存器（PC、SP、通用寄存器等）、程序状态字
2. **切换内核栈：** 每个进程/线程有独立的内核栈
3. **切换地址空间（进程切换才需要）：** 更新页表基址寄存器（CR3），导致 TLB 失效
4. **恢复目标上下文：** 加载目标进程/线程的寄存器和状态

**开销分析：**

| 类型 | 直接开销 | 间接开销 |
|------|---------|---------|
| **线程切换** | 保存/恢复寄存器（约 1-10 微秒） | CPU Cache 部分失效（Cache Miss 增加） |
| **进程切换** | 保存/恢复寄存器 + 切换页表 | TLB 全部失效 + CPU Cache 大量失效 |

**间接开销往往比直接开销更大：**
- TLB 失效后，后续的内存访问都需要重新查页表（TLB Miss 惩罚约 10-100 个时钟周期）
- CPU Cache（L1/L2/L3）中的热点数据失效，需要重新从内存加载（Cache Miss 惩罚约 100-300 个时钟周期）

**Java 开发中的关联：**
- 线程数不是越多越好，过多线程会导致频繁上下文切换，反而降低性能
- 线程池大小的经验公式：CPU 密集型 = CPU 核数 + 1；IO 密集型 = CPU 核数 × 2（或根据 IO 等待比例计算）
- 虚拟线程的优势之一就是减少上下文切换开销（用户态切换，不经过内核）

## 二、内存管理

1. 虚拟内存是什么？为什么需要虚拟内存？

**回答：**

**虚拟内存** 是操作系统提供的一种内存管理技术，为每个进程提供一个独立的、连续的虚拟地址空间，通过页表映射到物理内存。

**为什么需要虚拟内存：**

1. **进程隔离：** 每个进程有独立的虚拟地址空间，无法直接访问其他进程的内存，提高安全性和稳定性
2. **地址空间连续：** 程序看到的是连续的虚拟地址，实际物理内存可以是不连续的碎片，由 MMU 负责映射
3. **内存超卖：** 虚拟地址空间可以大于物理内存，不常用的页面可以换出到磁盘（Swap），实现"以小博大"
4. **简化编程：** 程序员不需要关心物理内存的分配和布局，链接器可以使用固定的虚拟地址
5. **共享内存：** 多个进程的虚拟页面可以映射到同一物理页面（如共享库 libc.so）

**地址转换过程：**
```
虚拟地址 → MMU 查页表 → 物理地址
         ↓ (页表未命中)
       缺页中断 → 从磁盘加载页面到物理内存 → 更新页表 → 重新执行指令
```

**Java 开发中的关联：**
- JVM 的堆内存（-Xmx）是在虚拟地址空间中分配的，并非立即占用物理内存
- Linux 的 overcommit 机制允许分配超过物理内存的虚拟内存（OOM Killer 在物理内存不足时杀进程）
- 容器环境中要注意 JVM 感知的内存限制与 cgroup 限制的关系

2. 分页和分段的区别？

**回答：**

| 对比维度 | 分页（Paging） | 分段（Segmentation） |
|---------|---------------|---------------------|
| 划分方式 | 将地址空间划分为**固定大小**的页（通常 4KB） | 将地址空间按**逻辑含义**划分为段（代码段、数据段、栈段等） |
| 大小 | 页大小固定 | 段大小不固定，由程序逻辑决定 |
| 地址结构 | 页号 + 页内偏移 | 段号 + 段内偏移 |
| 碎片问题 | 有内部碎片（最后一页可能未填满） | 有外部碎片（段之间可能有空隙） |
| 对程序员 | 透明，程序员无感知 | 可见，程序员可以按逻辑组织 |
| 共享与保护 | 以页为粒度，不够灵活 | 以段为粒度，可以对不同段设置不同权限（代码段只读等） |

**现代操作系统的做法——段页式：**
- Linux 实际上采用的是**分页为主**的方案，段机制被弱化（所有段的基址都设为 0，等于没分段）
- x86-64 架构下，段寄存器基本只用于兼容，实际内存管理完全靠分页
- 多级页表（Linux 使用四级/五级页表）解决页表本身占用内存过大的问题

3. 页面置换算法？FIFO、LRU、LFU、Clock？

**回答：**

当物理内存不足时，需要将某些页面换出到磁盘，腾出空间给新页面。选择哪个页面换出，就是页面置换算法要解决的问题。

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **OPT（最优）** | 淘汰未来最长时间不会被访问的页面 | 缺页率最低（理论最优） | 无法实现，需要预知未来（仅作对比基准） |
| **FIFO（先进先出）** | 淘汰最早进入内存的页面 | 实现简单，用队列即可 | 可能淘汰常用页面；存在 **Belady 异常**（增加物理页面数反而缺页率上升） |
| **LRU（最近最少使用）** | 淘汰最长时间未被访问的页面 | 性能接近 OPT，符合时间局部性 | 精确实现开销大（需要记录每次访问时间戳或维护链表） |
| **LFU（最不经常使用）** | 淘汰访问次数最少的页面 | 考虑了访问频率 | 早期高频访问的页面难以被淘汰；需要维护计数器 |
| **Clock（时钟算法）** | 页面组成环形链表，每个页面有访问位；指针扫描时，访问位为1则清零跳过，为0则淘汰 | LRU 的近似实现，开销小 | 精度不如 LRU |

**与 Java/中间件的关联：**
- Redis 的内存淘汰策略中 `allkeys-lru` 使用的是**近似 LRU**（随机采样 + 淘汰最旧的）
- Redis 4.0 引入了 LFU 策略（`allkeys-lfu`）
- MySQL InnoDB 的 Buffer Pool 使用**改进版 LRU**（分为 young 和 old 两个区域，防止全表扫描污染缓存）
- `LinkedHashMap` 的 `accessOrder=true` 模式就是一个 LRU 缓存的实现基础

4. 内存映射（mmap）是什么？

**回答：**

**mmap（Memory Mapped File）** 是一种将文件或设备映射到进程虚拟地址空间的技术。映射后，进程可以像访问内存一样直接读写文件，由操作系统负责在内存和磁盘之间同步数据。

**工作原理：**
```
进程虚拟地址空间                    物理内存                    磁盘文件
┌──────────┐                  ┌──────────┐              ┌──────────┐
│ mmap 区域 │ ──页表映射──→   │ 物理页面  │ ←──缺页加载──→ │ 文件内容  │
└──────────┘                  └──────────┘              └──────────┘
```

1. 调用 mmap 时，只是建立虚拟地址到文件的映射关系，并不立即加载数据
2. 首次访问映射区域时触发**缺页中断**，操作系统将文件对应部分加载到物理内存
3. 后续访问直接读写内存，操作系统在适当时机将修改写回磁盘

**mmap vs 传统 read/write：**

| 对比 | mmap | read/write |
|------|------|-----------|
| 数据拷贝 | 文件数据直接映射到用户空间，**减少一次内核缓冲区到用户缓冲区的拷贝** | 需要从内核缓冲区拷贝到用户缓冲区 |
| 系统调用 | 首次缺页后，后续访问无需系统调用 | 每次读写都需要系统调用 |
| 适用场景 | 大文件随机访问、进程间共享内存 | 小文件顺序读写 |

**实际应用：**
- **RocketMQ** 的 CommitLog 使用 mmap 实现高性能消息存储（MappedByteBuffer）
- **Kafka** 的索引文件使用 mmap
- Java NIO 的 `FileChannel.map()` 返回 `MappedByteBuffer`，底层就是 mmap
- 共享库（.so 文件）的加载也是通过 mmap 实现的

5. 用户态和内核态的区别？系统调用的过程？

**回答：**

**用户态（User Mode）和内核态（Kernel Mode）** 是 CPU 的两种运行级别，用于保护操作系统内核不被用户程序破坏。

| 对比维度 | 用户态 | 内核态 |
|---------|--------|--------|
| 权限 | 只能访问用户空间的内存，不能直接操作硬件 | 可以访问所有内存空间，可以操作硬件 |
| 指令 | 只能执行非特权指令 | 可以执行所有指令（包括特权指令如操作 IO、修改页表） |
| 地址空间 | 0 ~ 3GB（32位系统）/ 低 128TB（64位） | 3GB ~ 4GB（32位）/ 高 128TB（64位） |
| 稳定性 | 用户程序崩溃不影响内核 | 内核崩溃导致系统宕机（Kernel Panic） |

**系统调用（System Call）的过程：**

```
用户程序调用 read()
    ↓
C 库函数（glibc）封装，将系统调用号放入寄存器（如 eax）
    ↓
执行 int 0x80（x86）或 syscall 指令（x86-64）→ 触发软中断
    ↓
CPU 切换到内核态，保存用户态上下文（寄存器、栈指针等）
    ↓
根据系统调用号查 sys_call_table，找到对应的内核函数（如 sys_read）
    ↓
执行内核函数，完成实际操作
    ↓
恢复用户态上下文，切换回用户态，返回结果
```

**触发用户态→内核态切换的场景：**
1. **系统调用：** 程序主动请求内核服务（read、write、fork 等）
2. **异常：** 缺页中断、除零错误、非法内存访问等
3. **外部中断：** 硬件中断（磁盘 IO 完成、网络数据到达、时钟中断等）

**Java 开发中的关联：**
- Java 的 `new` 操作通常不需要系统调用（JVM 在用户态的堆中分配）
- 但文件 IO、网络 IO、线程创建等操作都需要系统调用
- 减少系统调用次数是性能优化的重要手段（如 BufferedInputStream 减少 read 调用次数，epoll 减少 IO 相关的系统调用）

## 三、IO 模型

1. 同步/异步、阻塞/非阻塞的区别？

**回答：**

这两组概念描述的是不同维度的事情，经常被混淆：

**阻塞 vs 非阻塞——关注的是调用者等待结果时的状态：**
- **阻塞：** 调用者发起请求后，线程被挂起，一直等到结果返回才继续执行
- **非阻塞：** 调用者发起请求后，立即返回（可能返回"未就绪"），线程可以去做其他事情，需要轮询或回调获取结果

**同步 vs 异步——关注的是结果通知的方式：**
- **同步：** 调用者主动等待或轮询结果（由调用者自己去获取结果）
- **异步：** 调用者发起请求后就不管了，结果就绪后由内核/框架通过回调、事件等方式通知调用者

**组合理解：**

| 组合 | 说明 | 举例 |
|------|------|------|
| 同步阻塞 | 发起调用后一直等，直到返回结果 | Java BIO：`inputStream.read()` |
| 同步非阻塞 | 发起调用立即返回，需要不断轮询检查结果 | Java NIO：`channel.read()` 返回 0 表示未就绪 |
| 异步非阻塞 | 发起调用立即返回，结果就绪后通过回调通知 | Java AIO：`AsynchronousSocketChannel`，Linux AIO |
| 异步阻塞 | 理论存在但实际很少用（没有意义） | — |

**一个经典的比喻（烧水）：**
- 同步阻塞：站在水壶旁边等水开
- 同步非阻塞：去做其他事，但每隔一会儿来看看水开了没
- 异步非阻塞：水壶有报警器，水开了自动响铃通知你

2. Linux 的五种 IO 模型？

**回答：**

网络 IO 操作分为两个阶段：① 等待数据就绪（数据从网卡到内核缓冲区）② 数据拷贝（从内核缓冲区拷贝到用户缓冲区）。五种模型的区别在于这两个阶段是否阻塞。

| IO 模型 | 阶段一（等待数据） | 阶段二（拷贝数据） | 说明 |
|---------|------------------|------------------|------|
| **阻塞 IO（BIO）** | 阻塞 | 阻塞 | 最简单，线程一直等到数据拷贝完成。对应 Java 的 `Socket.getInputStream().read()` |
| **非阻塞 IO** | 非阻塞（轮询） | 阻塞 | 调用立即返回，需要不断轮询检查数据是否就绪；就绪后拷贝阶段仍然阻塞 |
| **IO 多路复用** | 阻塞在 select/poll/epoll | 阻塞 | 一个线程监控多个 fd，有数据就绪时再进行读写。对应 Java NIO 的 `Selector` |
| **信号驱动 IO** | 非阻塞（信号通知） | 阻塞 | 注册信号处理函数，数据就绪时内核发送 SIGIO 信号；实际使用较少 |
| **异步 IO（AIO）** | 非阻塞 | 非阻塞 | 两个阶段都不阻塞，数据拷贝完成后内核通知应用。Linux 的 AIO 实现不够成熟，实际用得少 |

**重点掌握前三种，尤其是 IO 多路复用：**
- 前四种本质上都是**同步 IO**（阶段二都需要进程参与拷贝）
- 只有异步 IO 是真正的**异步**（内核完成所有工作后通知进程）
- 实际工程中，**IO 多路复用 + 非阻塞 IO** 是主流方案（Netty、Nginx、Redis 都是这个模型）

3. IO 多路复用？select、poll、epoll 的区别？（和 Netty 章节关联）

**回答：**

IO 多路复用允许一个线程同时监控多个文件描述符（fd），当任意一个 fd 就绪时通知程序处理。

| 对比维度 | select | poll | epoll |
|---------|--------|------|-------|
| 数据结构 | fd_set（位图） | pollfd 数组（链表） | 红黑树 + 就绪链表 |
| fd 数量限制 | 默认 1024（FD_SETSIZE） | 无硬限制（但性能随 fd 数量下降） | 无限制（受系统资源约束） |
| fd 传递方式 | 每次调用都要将全部 fd 集合从用户态拷贝到内核态 | 同 select | fd 只需注册一次（epoll_ctl），无需重复拷贝 |
| 就绪检测 | 内核遍历所有 fd 检查就绪状态，O(n) | 同 select，O(n) | 内核通过回调将就绪 fd 加入就绪链表，O(1) 通知 |
| 返回方式 | 返回就绪 fd 数量，需要遍历找出哪些就绪 | 同 select | 直接返回就绪 fd 列表，无需遍历 |
| 触发模式 | 水平触发（LT） | 水平触发（LT） | 支持水平触发（LT）和**边缘触发（ET）** |

**epoll 的两种触发模式：**
- **LT（Level Triggered，水平触发）：** fd 就绪后，如果没有处理完，下次 epoll_wait 还会通知。编程简单，不容易丢数据
- **ET（Edge Triggered，边缘触发）：** fd 状态变化时只通知一次，必须一次性读完所有数据。性能更高，但编程复杂，需配合非阻塞 IO

**epoll 的核心 API：**
```c
epoll_create()   // 创建 epoll 实例（内核中的红黑树 + 就绪链表）
epoll_ctl()      // 注册/修改/删除监控的 fd
epoll_wait()     // 等待就绪事件，返回就绪 fd 列表
```

**与 Netty 的关联：**
- Netty 的 `NioEventLoop` 底层就是封装了 epoll（Linux）/ kqueue（macOS）
- Netty 默认使用 LT 模式，也提供了 `EpollEventLoopGroup` 直接使用 Linux epoll（ET 模式）
- Netty 解决了 JDK NIO 的 epoll 空轮询 Bug（Selector.select() 在没有事件时也返回）

4. 零拷贝的原理？sendfile、mmap？

**回答：**

**零拷贝（Zero-Copy）** 是指在数据传输过程中，减少或避免 CPU 参与的数据拷贝次数，从而提升 IO 性能。

**传统 IO 的数据流转（read + write 发送文件到网络）：**
```
磁盘 → 内核页缓存（DMA 拷贝）→ 用户缓冲区（CPU 拷贝）→ Socket 缓冲区（CPU 拷贝）→ 网卡（DMA 拷贝）
```
共 4 次拷贝 + 4 次上下文切换（2 次 read + 2 次 write）

**优化方案一：mmap + write**
```
磁盘 → 内核页缓存（DMA 拷贝）→ [用户空间直接映射，无需拷贝] → Socket 缓冲区（CPU 拷贝）→ 网卡（DMA 拷贝）
```
- 3 次拷贝 + 4 次上下文切换
- mmap 将内核页缓存映射到用户空间，省去了内核→用户的拷贝
- RocketMQ 使用此方案（MappedByteBuffer）

**优化方案二：sendfile（Linux 2.1+）**
```
磁盘 → 内核页缓存（DMA 拷贝）→ Socket 缓冲区（CPU 拷贝）→ 网卡（DMA 拷贝）
```
- 3 次拷贝 + 2 次上下文切换（只有一次系统调用）
- 数据不经过用户空间，直接在内核中从页缓存拷贝到 Socket 缓冲区

**优化方案三：sendfile + DMA gather copy（Linux 2.4+，需要网卡支持 scatter-gather）**
```
磁盘 → 内核页缓存（DMA 拷贝）→ 网卡（DMA gather 拷贝，只传递 fd 和 offset 给 Socket 缓冲区）
```
- 2 次拷贝（都是 DMA，CPU 零拷贝）+ 2 次上下文切换
- 这才是真正意义上的"零拷贝"

**Java 中的零拷贝：**

| Java API | 底层实现 | 使用者 |
|----------|---------|--------|
| `MappedByteBuffer`（FileChannel.map） | mmap | RocketMQ |
| `FileChannel.transferTo()` | sendfile | Kafka、Netty 的 `DefaultFileRegion` |

**Kafka 高吞吐的秘诀之一就是 sendfile：** 消费者拉取消息时，数据直接从磁盘页缓存发送到网卡，不经过用户空间。

## 五、线上问题排查

1. CPU 飙高怎么排查？（top → 定位线程 → jstack）

**回答：**

**排查步骤：**

```
第一步：top 找到 CPU 占用最高的 Java 进程
$ top                          # 按 P 键按 CPU 排序，记下 PID（如 12345）

第二步：top -Hp 找到该进程中 CPU 占用最高的线程
$ top -Hp 12345                # 记下线程 PID（如 12378）

第三步：将线程 PID 转为十六进制
$ printf "%x\n" 12378          # 输出 305a

第四步：jstack 导出线程堆栈，搜索该线程
$ jstack 12345 | grep "305a" -A 30    # 找到对应线程的堆栈信息
```

**常见原因及对应堆栈特征：**

| 原因 | 堆栈特征 | 解决方案 |
|------|---------|---------|
| 死循环 | 线程一直在某个方法中，状态为 RUNNABLE | 检查循环退出条件 |
| 频繁 GC | 大量 GC 线程占用 CPU | `jstat -gcutil <pid> 1000` 查看 GC 频率；检查内存泄漏或调整堆大小 |
| 正则回溯 | 卡在 `java.util.regex` 相关方法 | 优化正则表达式，避免灾难性回溯 |
| 锁竞争激烈 | 大量线程 BLOCKED，等待同一把锁 | 减小锁粒度，使用无锁数据结构 |
| 序列化/反序列化 | 卡在 JSON 解析等方法 | 检查是否有超大对象，考虑换更高效的序列化框架 |

**一行命令版（生产环境快速排查）：**
```bash
# 找到 CPU 最高的 Java 线程并打印堆栈
top -Hp $(jps | grep -v Jps | awk '{print $1}') -bn1 | head -20
```

2. 内存泄漏怎么排查？（free → jmap → MAT）

**回答：**

**排查步骤：**

```
第一步：确认系统内存状况
$ free -h                      # 查看系统整体内存使用
$ top                          # 按 M 键按内存排序，找到内存占用高的进程

第二步：查看 JVM 内存使用情况
$ jstat -gcutil <pid> 1000     # 每秒打印 GC 统计，观察老年代（O）是否持续增长
$ jmap -heap <pid>             # 查看堆内存各区域使用情况

第三步：查看对象分布（初步定位）
$ jmap -histo:live <pid> | head -30    # 查看存活对象数量 Top 30（会触发 Full GC）

第四步：导出堆转储文件（详细分析）
$ jmap -dump:live,format=b,file=heap.hprof <pid>    # 导出 hprof 文件

第五步：使用 MAT（Eclipse Memory Analyzer）分析
- 打开 heap.hprof 文件
- 查看 Leak Suspects Report（泄漏嫌疑报告）
- 查看 Dominator Tree（支配树）找到占用内存最大的对象
- 通过 GC Roots 引用链找到泄漏的根因
```

**常见内存泄漏场景：**

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| 集合类持续增长 | 往 static Map/List 中不断添加数据，没有清理 | 使用弱引用、设置上限、定期清理 |
| 连接未关闭 | 数据库连接、HTTP 连接、IO 流未关闭 | 使用 try-with-resources |
| ThreadLocal 泄漏 | 线程池中的线程复用，ThreadLocal 未 remove | 使用完毕后调用 `remove()` |
| 监听器/回调未注销 | 注册了事件监听器但没有取消注册 | 在对象销毁时取消注册 |
| 类加载器泄漏 | 热部署时旧的 ClassLoader 无法被回收 | 检查是否有静态引用指向旧类 |

**建议：** 生产环境提前配置 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/`，OOM 时自动导出堆转储。

3. 磁盘 IO 高怎么排查？（iostat → iotop）

**回答：**

**排查步骤：**

```
第一步：确认磁盘 IO 状况
$ iostat -x 1                  # 每秒采样，查看各磁盘的 IO 指标
# 关注指标：
#   %util > 80%  → 磁盘繁忙
#   await > 10ms → IO 等待时间过长
#   r/s, w/s     → 每秒读写次数

第二步：找到 IO 最高的进程
$ iotop -o                     # 只显示有 IO 活动的进程，按 IO 排序
# 或者
$ pidstat -d 1                 # 每秒显示各进程的 IO 统计

第三步：确认进程在读写什么文件
$ lsof -p <pid>                # 查看进程打开的所有文件
$ strace -p <pid> -e trace=read,write -c    # 统计进程的 IO 系统调用
```

**常见原因及解决方案：**

| 原因 | 表现 | 解决方案 |
|------|------|---------|
| 日志写入过多 | 大量 write 操作，日志文件增长快 | 调整日志级别（DEBUG→INFO）；异步写日志（Logback AsyncAppender） |
| 频繁 Full GC | GC 日志写入频繁 | 排查内存问题，减少 GC 频率 |
| 数据库大量随机读 | MySQL 的 iowait 高 | 优化慢查询，增加 Buffer Pool，使用 SSD |
| Swap 频繁读写 | si/so 指标高 | 增加物理内存，或关闭 Swap（`swapoff -a`） |
| 大文件排序/临时文件 | /tmp 目录 IO 高 | 增加内存让排序在内存中完成 |

4. 网络问题怎么排查？（ping → telnet → tcpdump → wireshark）

**回答：**

**分层排查思路（从底层到应用层逐步缩小范围）：**

```
第一步：ping — 检查网络连通性（ICMP 层）
$ ping 目标IP                  # 能 ping 通说明网络层没问题
# ping 不通可能是：网络不通、防火墙禁止 ICMP、目标主机宕机

第二步：telnet — 检查端口可达性（TCP 层）
$ telnet 目标IP 端口            # 能连上说明 TCP 端口正常
# 连不上可能是：服务未启动、防火墙规则、端口未监听

第三步：curl — 检查应用层响应（HTTP 层）
$ curl -v http://目标IP:端口/api    # 查看 HTTP 请求/响应详情
# 关注：响应状态码、响应时间、是否超时

第四步：tcpdump — 抓包分析（定位具体问题）
$ tcpdump -i eth0 host 目标IP and port 端口 -w capture.pcap
# 抓取的 pcap 文件可以用 Wireshark 打开分析

第五步：Wireshark — 图形化分析抓包数据
# 分析 TCP 三次握手是否正常
# 查看是否有重传（Retransmission）、RST、超时等异常
```

**常见网络问题及排查方法：**

| 问题 | 排查方法 | 可能原因 |
|------|---------|---------|
| 连接超时 | tcpdump 看是否有 SYN 重传 | 防火墙拦截、目标端口未监听、网络丢包 |
| 连接被拒绝 | 看是否收到 RST 包 | 服务未启动、端口未监听 |
| 响应慢 | `ss -ti` 查看 TCP 连接的 RTT；tcpdump 分析延迟 | 网络延迟高、服务端处理慢、带宽不足 |
| 连接数过多 | `ss -s` 查看各状态连接数 | TIME_WAIT 堆积、连接池配置不当、连接泄漏 |
| DNS 解析慢 | `dig` 或 `nslookup` 测试 DNS | DNS 服务器响应慢，可配置本地 hosts 或换 DNS |

**Java 应用常见网络问题：**
- `Connection refused` → 目标服务未启动或端口错误
- `Connection timed out` → 网络不通或防火墙拦截
- `Read timed out` → 服务端处理超时，调整客户端超时参数
- `Too many open files` → fd 耗尽，调整 `ulimit -n` 和连接池大小
