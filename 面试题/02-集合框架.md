# 02 - 集合框架

---

## 一、集合选型与复杂度

1. `List`、`Set`、`Queue`、`Deque`、`Map` 分别适合表达什么业务语义？不要只从数据结构角度回答。
2. 面对“读多写少、写多读少、有序、去重、TopN、范围查询”这些需求，你会如何选择集合？
3. 时间复杂度相同的集合，为什么实际性能可能差很多？请从内存布局、缓存局部性、扩容、装箱角度解释。
4. 不可变集合（`List.of`、`Set.of`、`Map.of`）适合哪些场景？和 `Collections.unmodifiableXxx` 有什么差别？
5. 集合中存放可变对象会带来哪些问题？尤其是作为 `HashMap` key 或 `HashSet` 元素时。

## 二、List 与数组

1. `ArrayList` 和 `LinkedList` 如何选型？为什么很多场景下 `LinkedList` 并没有想象中快？
2. `ArrayList` 的扩容、拷贝、预估容量会如何影响批量写入性能？
3. `Arrays.asList()`、`List.of()`、`Collections.singletonList()` 返回的集合有什么限制？
4. `CopyOnWriteArrayList` 适合什么样的读写比例？它在内存和一致性上有什么代价？
5. 业务代码中什么时候应该返回数组、不可变 List、可变 List 或 Stream？

## 三、Map 与哈希设计

1. `HashMap` 的 put/get 大致流程是什么？哈希扰动、桶、链表、红黑树分别解决什么问题？
2. 为什么 `HashMap` 容量通常是 2 的幂？负载因子和初始容量如何影响性能和内存？
3. 设计 key 时如何保证 `equals()` 和 `hashCode()` 稳定？可变 key 会造成什么线上问题？
4. `LinkedHashMap` 如何支持访问顺序？用它实现 LRU 缓存时还缺少哪些生产级能力？
5. `TreeMap`、`HashMap`、`LinkedHashMap` 在排序、范围查询、迭代顺序上的差异是什么？
6. `WeakHashMap`、`IdentityHashMap`、`EnumMap` 分别适合什么特殊场景？

## 四、并发集合

1. `ConcurrentHashMap` 如何在并发读写中平衡性能和一致性？哪些复合操作仍然需要额外设计？
2. `computeIfAbsent`、`merge`、`putIfAbsent` 适合解决哪些并发初始化或计数问题？使用时有什么坑？
3. `Collections.synchronizedMap`、`ConcurrentHashMap`、`CopyOnWriteArrayList` 的适用场景有什么不同？
4. 高并发计数时为什么常用 `LongAdder`？它和 `AtomicLong` 在读一致性上有什么差异？
5. 本地缓存为什么通常不建议自己用 Map 从零实现？Caffeine 这类库补齐了哪些能力？

## 五、Queue、Deque 与阻塞队列

1. `ArrayDeque`、`LinkedList`、`PriorityQueue` 分别适合哪些队列/栈/堆场景？
2. `PriorityQueue` 为什么不是线程安全的？用它实现延时任务还需要考虑什么？
3. `ArrayBlockingQueue`、`LinkedBlockingQueue`、`SynchronousQueue`、`DelayQueue` 的差异是什么？
4. 阻塞队列在生产者-消费者模型中如何体现背压？队列长度应该如何设置？
5. 线程池队列选择会如何影响吞吐、延迟、内存和拒绝策略？

## 六、迭代、Stream 与工程实践

1. fail-fast 机制能保证并发安全吗？它真正解决的是什么问题？
2. 遍历集合时删除元素有哪些安全写法？为什么增强 for 中直接 remove 容易出错？
3. Stream API 的 `map`、`flatMap`、`collect`、`groupingBy` 适合哪些数据转换？何时不该强行 Stream 化？
4. 并行流为什么容易踩线程池和阻塞 IO 的坑？什么场景才值得使用？
5. 集合数据量很大时，如何避免一次性加载、重复拷贝和中间集合膨胀？
