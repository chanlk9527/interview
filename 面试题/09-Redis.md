# 09 - Redis

---

## 一、数据结构

1. Redis 的五种基础数据类型？各自的使用场景？

**答：**

Redis 五种基础数据类型及典型使用场景：

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| **String** | 最基本的 key-value 类型，值可以是字符串、整数或浮点数，最大 512MB | 缓存对象（JSON序列化）、计数器（INCR）、分布式锁（SETNX）、Session 共享 |
| **Hash** | 键值对集合，类似 Java 的 HashMap | 存储对象的多个字段（用户信息、商品详情），避免序列化/反序列化开销，可单独修改某个字段 |
| **List** | 双向链表，有序可重复 | 消息队列（LPUSH + BRPOP）、最新消息列表（朋友圈时间线）、分页查询 |
| **Set** | 无序集合，元素唯一 | 标签系统、共同好友（交集 SINTER）、抽奖（SRANDMEMBER/SPOP）、去重 |
| **Sorted Set (ZSet)** | 有序集合，每个元素关联一个 score 用于排序 | 排行榜（ZRANGEBYSCORE）、延迟队列（score 存时间戳）、带权重的优先级队列 |

面试加分点：回答时可以结合自己项目中的实际使用场景，比如"我们项目中用 String 做设备在线状态缓存，用 ZSet 做设备告警排行"。

2. Redis 底层数据结构？SDS、ziplist、quicklist、skiplist、hashtable、intset？

**答：**

Redis 对外暴露的是五种数据类型，但底层实际使用了多种高效数据结构来实现：

| 底层结构 | 说明 | 被谁使用 |
|----------|------|----------|
| **SDS（Simple Dynamic String）** | 动态字符串，记录长度、剩余空间，二进制安全，支持预分配和惰性释放 | String 的底层实现 |
| **ziplist（压缩列表）** | 连续内存块存储，节省内存，适合少量小元素 | Hash（元素少时）、ZSet（元素少时）、List（Redis 3.2 前） |
| **quicklist（快速列表）** | ziplist + 双向链表的组合，兼顾内存和性能 | List（Redis 3.2+）的底层实现 |
| **skiplist（跳表）** | 多层有序链表，支持 O(logN) 查找、范围查询 | ZSet 的底层实现（元素多时） |
| **hashtable（哈希表）** | 数组 + 链表，渐进式 rehash | Hash（元素多时）、Set（元素为字符串时） |
| **intset（整数集合）** | 有序整数数组，紧凑存储 | Set（元素全为整数且数量少时） |
| **listpack（Redis 7.0+）** | ziplist 的替代品，解决了连锁更新问题 | 替代 ziplist 用于 Hash、ZSet 的紧凑编码 |

数据类型与底层结构的对应关系：
- **String** → SDS / int（纯整数时直接用 int 编码）
- **List** → quicklist
- **Hash** → ziplist/listpack（小数据量） → hashtable（大数据量）
- **Set** → intset（纯整数且少量） → hashtable
- **ZSet** → ziplist/listpack（少量） → skiplist + hashtable

3. String 的底层实现？SDS 和 C 字符串的区别？

**答：**

**String 的三种编码方式：**
- **int**：值为纯整数且范围在 long 内，直接用整数存储
- **embstr**：字符串长度 ≤ 44 字节，SDS 和 redisObject 分配在一块连续内存中，只需一次内存分配
- **raw**：字符串长度 > 44 字节，SDS 和 redisObject 分开分配，需两次内存分配

**SDS 与 C 字符串的区别：**

| 对比项 | C 字符串 | SDS |
|--------|----------|-----|
| 获取长度 | O(N) 遍历 | O(1) 直接读 len 字段 |
| 缓冲区溢出 | 不检查，可能溢出 | 自动扩容，杜绝溢出 |
| 内存分配 | 每次修改都要重新分配 | 空间预分配 + 惰性释放，减少分配次数 |
| 二进制安全 | 以 `\0` 判断结尾，不能存二进制 | 以 len 判断长度，可以存任意二进制数据（图片、序列化对象等） |
| 兼容性 | — | 末尾仍保留 `\0`，兼容部分 C 函数 |

SDS 空间预分配策略：
- 修改后 len < 1MB → 分配 2 × len + 1 字节
- 修改后 len ≥ 1MB → 分配 len + 1MB + 1 字节

4. Hash 的底层实现？什么时候从 ziplist 转为 hashtable？

**答：**

**Hash 的两种底层编码：**
- **ziplist（Redis 7.0 前）/ listpack（Redis 7.0+）**：当数据量小时使用，field 和 value 紧挨着存储在连续内存中，内存紧凑
- **hashtable**：当数据量大时使用，数组 + 链表结构，查找 O(1)

**转换条件（满足任一即转为 hashtable）：**
1. Hash 中的 field 数量超过 `hash-max-ziplist-entries`（默认 512）
2. Hash 中任意 field 或 value 的长度超过 `hash-max-ziplist-value`（默认 64 字节）

**hashtable 的渐进式 rehash：**
- Redis 的 hashtable 使用两个哈希表 ht[0] 和 ht[1]
- 扩容时不会一次性迁移所有数据（避免阻塞），而是在每次 CRUD 操作时顺带迁移一部分
- 迁移期间，查找会先查 ht[0] 再查 ht[1]，新增只写 ht[1]
- 全部迁移完成后，ht[1] 变为 ht[0]，释放旧表

**触发扩容的条件：**
- 负载因子 ≥ 1 且没有在执行 BGSAVE/BGREWRITEAOF → 扩容
- 负载因子 ≥ 5 → 强制扩容（即使在持久化）

5. Sorted Set 为什么用跳表而不是红黑树？

**答：**

Redis 作者 antirez 选择跳表而非红黑树，主要原因：

1. **范围查询更高效**：ZSet 大量使用 ZRANGEBYSCORE 等范围操作，跳表找到起始节点后沿链表顺序遍历即可，非常自然；红黑树需要中序遍历，实现更复杂
2. **实现简单**：跳表的插入、删除逻辑比红黑树简单很多，红黑树需要复杂的旋转和变色操作来维持平衡，跳表只需随机层数 + 指针调整
3. **内存局部性相当**：两者都是指针结构，内存局部性差不多
4. **时间复杂度相同**：跳表的查找、插入、删除平均都是 O(logN)，和红黑树一致
5. **可调节性**：通过调整层数概率参数（Redis 用 p=0.25，最大 32 层），可以灵活平衡时间和空间

antirez 原话大意：跳表不比平衡树差，但实现起来更简单、更直观，调试也更方便。

**补充**：ZSet 实际上同时使用了 skiplist + hashtable 两个结构：
- skiplist 负责按 score 排序和范围查询
- hashtable 负责按 member 查找 score，O(1) 复杂度

6. Redis 的新数据类型？Stream、HyperLogLog、Bitmap、GEO？

**答：**

| 类型 | 说明 | 典型场景 |
|------|------|----------|
| **Bitmap** | 基于 String 的位操作，每个 bit 存 0/1 | 用户签到（每天一个 bit）、布隆过滤器、活跃用户统计（BITCOUNT） |
| **HyperLogLog** | 基数估算算法，固定占用 12KB 内存 | 统计 UV（独立访客数），允许约 0.81% 的误差，适合海量去重计数 |
| **GEO** | 基于 ZSet 实现，将经纬度编码为 GeoHash 作为 score | 附近的人/店铺（GEORADIUS）、距离计算（GEODIST） |
| **Stream（Redis 5.0+）** | 类似 Kafka 的消息流结构，支持消费者组、消息确认、持久化 | 消息队列（替代 List 的 LPUSH/BRPOP 方案），支持多消费者组独立消费、消息回溯、ACK 机制 |

**Stream 的核心特性：**
- 消息 ID 自动生成（时间戳-序号），天然有序
- 支持消费者组（Consumer Group），不同组独立消费
- 支持 ACK 机制，未确认的消息会进入 pending 列表
- 支持 XREAD 阻塞读取
- 相比 List 做队列，Stream 更可靠，但仍不如 Kafka/RocketMQ 适合高吞吐场景

## 二、持久化

1. RDB 持久化的原理？触发方式？优缺点？

**答：**

**原理：**
RDB（Redis Database）是将某一时刻的内存数据生成快照（snapshot），以二进制文件（dump.rdb）的形式保存到磁盘。

**触发方式：**
- **手动触发**：
  - `SAVE`：阻塞主线程，直到快照完成（生产环境禁用）
  - `BGSAVE`：fork 子进程生成快照，主线程继续处理请求
- **自动触发**：
  - 配置 `save m n`（m 秒内有 n 次修改则触发 BGSAVE）
  - 主从全量同步时，主节点自动执行 BGSAVE
  - 执行 `SHUTDOWN` 且未开启 AOF 时
  - 执行 `FLUSHALL` 时（生成空的 RDB 文件）

**BGSAVE 的 Copy-On-Write（写时复制）：**
- fork 出子进程后，父子进程共享内存页
- 当主进程修改某个数据时，操作系统才会复制该内存页给子进程
- 这样子进程拿到的是 fork 那一刻的一致性快照

**优点：**
- RDB 文件紧凑，适合备份和灾难恢复
- 恢复速度快（直接加载二进制文件）
- 对性能影响小（子进程处理，不阻塞主线程）

**缺点：**
- 可能丢失最后一次快照之后的数据（两次快照之间宕机）
- fork 子进程时如果数据量大，可能短暂阻塞（fork 本身的耗时）
- 数据量大时 COW 可能导致内存占用翻倍

---

2. AOF 持久化的原理？三种写回策略？AOF 重写？

**答：**

**原理：**
AOF（Append Only File）以追加的方式记录每一条写命令到 AOF 文件中。Redis 重启时通过重放这些命令来恢复数据。

**写回流程：**
命令执行 → 写入 AOF 缓冲区（aof_buf） → 根据策略写入磁盘

**三种写回策略（appendfsync）：**

| 策略 | 说明 | 数据安全 | 性能 |
|------|------|----------|------|
| **always** | 每次写命令都 fsync 到磁盘 | 最多丢 1 条命令 | 最差，每次都有磁盘 IO |
| **everysec**（默认） | 每秒 fsync 一次 | 最多丢 1 秒数据 | 折中，推荐使用 |
| **no** | 由操作系统决定何时 fsync | 可能丢较多数据 | 最好，但不可控 |

**AOF 重写（BGREWRITEAOF）：**
- **为什么要重写**：AOF 文件会越来越大（比如对同一个 key 执行了 100 次 SET，其实只需保留最后一次）
- **原理**：fork 子进程，根据当前内存数据生成最精简的命令集（不是分析旧 AOF 文件），写入新 AOF 文件
- **重写期间的增量数据**：主进程在重写期间的新写命令会同时写入 AOF 缓冲区和 AOF 重写缓冲区，子进程完成后将重写缓冲区的内容追加到新文件，最后原子替换旧文件
- **自动触发条件**：`auto-aof-rewrite-percentage 100`（文件比上次重写后增长 100%）且 `auto-aof-rewrite-min-size 64mb`（文件至少 64MB）

---

3. RDB 和 AOF 的区别？如何选择？

**答：**

| 对比项 | RDB | AOF |
|--------|-----|-----|
| 持久化方式 | 内存快照（二进制） | 追加写命令（文本） |
| 数据安全 | 可能丢失两次快照间的数据 | 最多丢 1 秒数据（everysec） |
| 文件大小 | 紧凑，体积小 | 通常比 RDB 大 |
| 恢复速度 | 快（直接加载） | 慢（需要重放命令） |
| 性能影响 | fork 时可能短暂阻塞 | 写入和 fsync 有持续开销 |
| 适用场景 | 备份、灾难恢复、冷备 | 数据安全要求高的场景 |

**如何选择：**
- **对数据安全要求高**：开启 AOF（everysec），可接受最多丢 1 秒数据
- **纯缓存场景**：可以只用 RDB，甚至不开持久化
- **推荐做法**：RDB + AOF 都开启。AOF 保证数据安全，RDB 用于快速恢复和备份
- Redis 4.0+ 推荐使用混合持久化

---

4. 混合持久化是什么？怎么配置？

**答：**

**混合持久化（Redis 4.0+）：**
在 AOF 重写时，将当前内存数据以 RDB 格式写入 AOF 文件的开头，后续的增量命令仍以 AOF 格式追加。这样 AOF 文件 = RDB 头部 + AOF 尾部。

**优势：**
- 兼具 RDB 的快速恢复和 AOF 的数据安全
- 重启时先加载 RDB 部分（快），再重放 AOF 增量部分（少量命令）
- 恢复速度远快于纯 AOF

**配置：**
```
aof-use-rdb-preamble yes  # Redis 5.0+ 默认开启
```

**文件结构：**
```
[RDB 格式的全量数据] + [AOF 格式的增量命令]
```

---

5. Redis 重启时如何恢复数据？

**答：**

恢复优先级：
1. **如果开启了 AOF**：优先加载 AOF 文件（数据更完整）
2. **如果只开启了 RDB**：加载 dump.rdb 文件
3. **如果都没开**：无数据恢复

具体流程：
```
Redis 启动
  → 检查是否开启 AOF
    → 是：加载 AOF 文件（如果是混合持久化，先加载 RDB 头部，再重放 AOF 尾部）
    → 否：检查是否存在 dump.rdb
      → 是：加载 RDB 文件
      → 否：以空数据启动
```

**注意事项：**
- AOF 文件损坏时可以用 `redis-check-aof --fix` 修复
- RDB 文件损坏时可以用 `redis-check-rdb` 检查
- 即使同时存在 RDB 和 AOF 文件，开启了 AOF 的情况下也只会加载 AOF

## 三、缓存问题

1. 缓存穿透是什么？怎么解决？（布隆过滤器、空值缓存）

**答：**

**缓存穿透**：查询一个缓存和数据库中都不存在的数据，每次请求都会打到数据库。恶意攻击者可以利用这一点，用大量不存在的 key 发起请求，压垮数据库。

**解决方案：**

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **缓存空值** | 查询数据库为空时，缓存一个空值（如 `""`），设置较短 TTL（如 2-5 分钟） | 实现简单 | 浪费内存存空值；短时间内数据库有了该数据，缓存仍返回空（可通过写入时删除缓存解决） |
| **布隆过滤器** | 在缓存前加一层布隆过滤器，将所有合法 key 预加载进去，请求先过布隆过滤器，不存在则直接拒绝 | 内存占用小，拦截效果好 | 有误判率（可能放过少量不存在的 key）；不支持删除（可用布谷鸟过滤器替代）；需要维护数据同步 |
| **参数校验** | 在接口层做基本校验，比如 ID 不能为负数、格式校验等 | 简单有效 | 只能拦截明显非法请求 |

**实际项目中推荐组合使用**：接口参数校验 + 布隆过滤器 + 缓存空值。

---

2. 缓存击穿是什么？怎么解决？（互斥锁、逻辑过期）

**答：**

**缓存击穿**：某个热点 key 在缓存中过期的瞬间，大量并发请求同时打到数据库，导致数据库压力骤增。与缓存穿透的区别是：数据确实存在，只是缓存刚好过期。

**解决方案：**

**方案一：互斥锁（强一致）**
```
查缓存 → 未命中 → 尝试获取分布式锁（SETNX）
  → 获取成功 → 查数据库 → 写缓存 → 释放锁
  → 获取失败 → 休眠重试（或返回默认值）
```
- 优点：保证只有一个线程重建缓存，数据一致性好
- 缺点：其他线程需要等待，可能有性能问题；需要处理锁超时和死锁

**方案二：逻辑过期（高可用）**
- 缓存不设置 TTL（永不过期），但在 value 中存一个逻辑过期时间
- 读取时判断是否逻辑过期，如果过期则开启异步线程去重建缓存，当前请求返回旧数据
- 优点：不会阻塞请求，性能好
- 缺点：短时间内可能返回过期数据，牺牲了一致性

**方案三：热点 key 永不过期**
- 对于确定的热点 key，不设置过期时间，由后台任务定期更新
- 适合数据变化不频繁的场景

**选择建议**：对一致性要求高用互斥锁，对可用性要求高用逻辑过期。

---

3. 缓存雪崩是什么？怎么解决？（随机过期时间、多级缓存）

**答：**

**缓存雪崩**：大量缓存 key 在同一时间过期，或者 Redis 服务宕机，导致大量请求直接打到数据库，数据库瞬间压力过大甚至崩溃。

**两种场景及解决方案：**

**场景一：大量 key 同时过期**
| 方案 | 说明 |
|------|------|
| **随机过期时间** | 在基础 TTL 上加一个随机值（如 TTL = 基础时间 + random(0, 300)秒），打散过期时间 |
| **多级缓存** | 本地缓存（Caffeine/Guava）+ Redis + 数据库，即使 Redis 缓存失效，本地缓存仍可兜底 |
| **互斥锁** | 同缓存击穿方案，控制重建缓存的并发数 |
| **缓存预热** | 系统启动或低峰期提前加载热点数据到缓存 |

**场景二：Redis 服务宕机**
| 方案 | 说明 |
|------|------|
| **高可用架构** | 使用 Redis Sentinel 或 Redis Cluster，避免单点故障 |
| **限流降级** | 在应用层做限流（如 Sentinel、Resilience4j），数据库扛不住时返回降级结果 |
| **多级缓存** | 本地缓存兜底，Redis 挂了仍能提供部分服务 |

---

4. 缓存和数据库的一致性问题？Cache Aside、Read/Write Through、Write Behind？

**答：**

**三种经典缓存模式：**

**1. Cache Aside（旁路缓存）— 最常用**
- 读：先读缓存 → 命中则返回；未命中 → 读数据库 → 写入缓存
- 写：先更新数据库 → 再删除缓存
- 优点：实现简单，应用最广泛
- 缺点：存在短暂不一致窗口（删缓存前有请求读到旧值）

**2. Read/Write Through（读写穿透）**
- 应用只和缓存交互，缓存层负责和数据库同步
- Read Through：缓存未命中时，缓存层自动从数据库加载
- Write Through：写操作时，缓存层同步写入数据库
- 优点：应用代码简单，一致性较好
- 缺点：需要缓存中间件支持（如 Guava LoadingCache），写操作延迟较高（同步写 DB）

**3. Write Behind / Write Back（异步写回）**
- 写操作只更新缓存，缓存层异步批量写入数据库
- 优点：写性能极高
- 缺点：数据可能丢失（缓存挂了还没来得及写 DB）；实现复杂；一致性最弱

**实际项目中**：绝大多数场景使用 Cache Aside 模式。

---

5. 先更新数据库还是先删缓存？延迟双删？

**答：**

**四种组合分析：**

| 方案 | 问题 |
|------|------|
| 先更新 DB，再更新缓存 | ❌ 两个并发写请求可能导致缓存和 DB 数据不一致（后写 DB 的先更新了缓存） |
| 先更新缓存，再更新 DB | ❌ 更新 DB 失败则缓存是脏数据；且同样有并发问题 |
| **先删缓存，再更新 DB** | ⚠️ 并发问题：线程 A 删缓存 → 线程 B 读缓存未命中 → 读 DB 旧值写入缓存 → 线程 A 更新 DB → 缓存是旧值 |
| **先更新 DB，再删缓存**（推荐） | ✅ 问题概率极低：只有在"读请求缓存未命中 + 读 DB + 写缓存"这个窗口内恰好有写请求更新了 DB 并删了缓存，才会不一致。由于写 DB 通常比读 DB 慢，这个时序很难出现 |

**推荐方案：先更新数据库，再删除缓存。**

**延迟双删（针对"先删缓存再更新 DB"的补救）：**
```
1. 删除缓存
2. 更新数据库
3. 延迟一段时间（如 500ms ~ 1s）
4. 再次删除缓存
```
- 第二次删除是为了清除在步骤 2 期间被其他线程写入的旧缓存
- 延迟时间需要大于一次读请求的耗时
- 缺点：延迟时间不好确定，且仍有极端情况不一致

**更可靠的方案：基于 binlog 的异步删除**
- 通过 Canal 监听 MySQL binlog，解析出变更的数据，异步删除对应缓存
- 优点：解耦，可靠性高，不依赖延迟时间
- 缺点：引入额外组件，有一定延迟

---

6. 热 Key 问题怎么处理？

**答：**

**什么是热 Key**：某个 key 被大量并发访问（如热门商品、明星微博），导致该 key 所在的 Redis 节点压力过大，甚至成为瓶颈。

**如何发现热 Key：**
- `redis-cli --hotkeys`（需开启 LFU 淘汰策略）
- `MONITOR` 命令（生产慎用，性能开销大）
- 业务层统计（在代码中记录 key 的访问频率）
- 代理层统计（如 Twemproxy、Codis 的统计功能）

**解决方案：**

| 方案 | 说明 |
|------|------|
| **本地缓存** | 在应用层加 Caffeine/Guava 本地缓存，热 key 直接从本地读取，不打 Redis。设置较短 TTL 保证一致性 |
| **key 分散（读写分离/分片）** | 将热 key 复制多份，如 `key_1`、`key_2`...`key_N`，读请求随机访问其中一个，分散到不同 Redis 节点 |
| **读写分离** | 主节点写，多个从节点读，分散读压力 |
| **热 key 永不过期 + 异步更新** | 避免缓存击穿，后台线程定期刷新 |

**实际推荐**：本地缓存是最有效的方案，配合 key 分散使用效果更好。

---

7. 大 Key 问题怎么排查和解决？

**答：**

**什么是大 Key**：
- String 类型：value 大于 10KB（有些标准是 1MB）
- Hash/List/Set/ZSet：元素数量超过 5000，或总大小超过 10MB

**大 Key 的危害：**
- 内存不均：Cluster 模式下导致数据倾斜，某个节点内存远大于其他节点
- 阻塞：大 key 的读写、删除都可能阻塞主线程（如 DEL 一个几百万元素的 Hash）
- 网络带宽：大 value 传输占用大量带宽
- 主从同步延迟：大 key 的同步耗时长

**如何排查：**
- `redis-cli --bigkeys`：扫描每种类型中最大的 key（只能找到每种类型 top1）
- `MEMORY USAGE key`：查看某个 key 的内存占用
- `redis-rdb-tools`：离线分析 RDB 文件，找出所有大 key
- 监控系统：定期扫描或通过慢查询日志发现

**解决方案：**

| 方案 | 说明 |
|------|------|
| **拆分** | 将大 Hash 拆成多个小 Hash（如按 ID 取模分片：`user:{id%100}:{id}`） |
| **压缩** | 对 value 进行 gzip/snappy 压缩后存储 |
| **异步删除** | 使用 `UNLINK` 代替 `DEL`（Redis 4.0+），后台线程异步释放内存，不阻塞主线程 |
| **设置合理过期** | 避免大 key 长期占用内存 |
| **lazy-free** | 开启 `lazyfree-lazy-expire yes` 等配置，让过期删除也走异步 |

## 四、高可用架构

1. Redis 主从复制的原理？全量同步和增量同步？

**答：**

**主从复制的作用**：数据备份、读写分离（主写从读）、高可用的基础。

**全量同步（首次连接或 repl_backlog 溢出时）：**
```
1. 从节点发送 PSYNC ? -1（首次）或 PSYNC runid offset
2. 主节点判断需要全量同步，返回 FULLRESYNC runid offset
3. 主节点执行 BGSAVE 生成 RDB 文件
4. 主节点将 RDB 发送给从节点（期间的写命令存入 repl_backlog 缓冲区）
5. 从节点清空旧数据，加载 RDB
6. 主节点将 repl_backlog 中积压的命令发送给从节点
7. 从节点执行这些命令，完成同步
```

**增量同步（断线重连时）：**
```
1. 从节点发送 PSYNC runid offset（携带自己的复制偏移量）
2. 主节点检查 offset 是否在 repl_backlog 缓冲区范围内
   → 在范围内：返回 CONTINUE，只发送 offset 之后的增量命令
   → 不在范围内（缓冲区已被覆盖）：触发全量同步
```

**repl_backlog_buffer**：
- 环形缓冲区，默认 1MB，记录主节点最近的写命令
- 如果从节点断线时间过长，缓冲区被覆盖，就只能全量同步
- 生产环境建议调大（如 256MB），减少全量同步概率

**命令传播**：全量同步完成后，主节点持续将写命令异步发送给从节点（异步复制，所以主从之间是最终一致性）。

---

2. 哨兵（Sentinel）模式的原理？故障转移流程？

**答：**

**Sentinel 的作用**：监控主从节点状态、自动故障转移、通知客户端新主节点地址。

**监控机制：**
- 每个 Sentinel 每秒向主从节点发送 PING，检测是否存活
- **主观下线（SDOWN）**：单个 Sentinel 认为节点不可达（超过 `down-after-milliseconds` 未响应）
- **客观下线（ODOWN）**：Sentinel 之间互相确认，超过 `quorum` 个 Sentinel 都认为主节点下线，才判定为客观下线（只针对主节点）

**故障转移流程：**
```
1. 主节点被判定为客观下线
2. Sentinel 之间选举出一个 Leader（Raft 算法）来执行故障转移
3. Leader Sentinel 从从节点中选出新主节点，选择依据（优先级从高到低）：
   a. slave-priority 最小的（手动设置的优先级）
   b. 复制偏移量最大的（数据最新）
   c. runid 最小的（兜底）
4. 向选中的从节点发送 SLAVEOF NO ONE，使其成为新主节点
5. 向其他从节点发送 SLAVEOF 新主节点，让它们复制新主
6. 将旧主节点标记为从节点（恢复后自动成为新主的从节点）
7. 通知客户端新主节点地址
```

**Sentinel 集群建议**：至少 3 个 Sentinel 节点，quorum 设为 2，保证高可用。

---

3. Redis Cluster 的原理？哈希槽（Hash Slot）分配？

**答：**

**Redis Cluster 核心思想**：将数据分散到多个节点，实现水平扩展和高可用。

**哈希槽（Hash Slot）：**
- Redis Cluster 将整个数据空间划分为 **16384 个哈希槽**（0 ~ 16383）
- 每个 key 通过 `CRC16(key) % 16384` 计算出所属的槽
- 每个主节点负责一部分槽，如 3 个主节点：节点 A（0-5460）、节点 B（5461-10922）、节点 C（10923-16383）

**为什么是 16384 个槽？**
- 节点间通过 Gossip 协议交换槽信息，每个节点用 bitmap 表示自己负责的槽
- 16384 个槽的 bitmap 只需 2KB（16384/8），网络开销小
- 官方建议集群最多 1000 个节点，16384 个槽完全够用

**客户端路由：**
- 客户端可以向任意节点发送请求
- 如果 key 不在该节点负责的槽上，返回 `MOVED slot ip:port` 重定向
- 客户端缓存槽与节点的映射关系，后续直接访问正确节点
- 扩缩容期间可能返回 `ASK` 重定向（临时重定向，不更新本地缓存）

**Hash Tag**：通过 `{tag}` 语法让多个 key 落在同一个槽，如 `user:{1000}.name` 和 `user:{1000}.age` 都按 `1000` 计算槽，支持同槽的多 key 操作。

---

4. Redis Cluster 的节点通信？Gossip 协议？

**答：**

**通信方式**：Redis Cluster 节点之间通过 **Gossip 协议** 进行去中心化通信，使用独立的集群总线端口（服务端口 + 10000，如 6379 → 16379）。

**Gossip 协议的消息类型：**

| 消息类型 | 说明 |
|----------|------|
| **PING** | 每个节点每秒随机选择几个节点发送 PING，携带自身状态和部分已知节点信息 |
| **PONG** | 收到 PING 后回复 PONG，携带自身最新状态 |
| **MEET** | 新节点加入集群时，向已有节点发送 MEET，被接受后通过 Gossip 传播到整个集群 |
| **FAIL** | 某节点被判定为故障时，向整个集群广播 FAIL 消息 |

**Gossip 的特点：**
- **去中心化**：不需要中心节点，每个节点都是对等的
- **最终一致**：信息通过多轮传播最终到达所有节点，但有一定延迟
- **容错性好**：部分节点故障不影响信息传播

**故障检测：**
- 节点 A 向节点 B 发送 PING，超过 `cluster-node-timeout` 未收到 PONG → A 将 B 标记为 PFAIL（主观下线）
- A 通过 Gossip 传播 B 的 PFAIL 状态
- 当超过半数主节点都将 B 标记为 PFAIL → B 被标记为 FAIL（客观下线），触发故障转移

---

5. Redis Cluster 如何处理节点故障？

**答：**

**故障转移流程（主节点故障）：**
```
1. 集群中超过半数主节点将故障节点标记为 PFAIL
2. 故障节点被标记为 FAIL，广播到整个集群
3. 故障主节点的从节点发起选举：
   a. 从节点将 currentEpoch +1，向所有主节点发送 FAILOVER_AUTH_REQUEST
   b. 主节点投票（每个 epoch 只投一票），复制偏移量最大的从节点优先发起选举，更容易获得投票
   c. 从节点获得超过半数主节点的投票 → 当选新主节点
4. 新主节点执行 SLAVEOF NO ONE
5. 新主节点接管故障节点的所有哈希槽
6. 广播 PONG 通知集群更新配置
```

**集群不可用的情况：**
- 某个主节点故障且没有可用的从节点 → 该节点负责的槽无法服务
- 默认配置 `cluster-require-full-coverage yes`：任何槽不可用则整个集群不可用
- 设为 `no`：只有故障槽不可用，其他槽正常服务（推荐生产环境设为 no）

**从节点迁移**：如果某个主节点的所有从节点都挂了，其他有多余从节点的主节点会自动迁移一个从节点过来，提高容错能力。

---

6. Redis 集群方案的对比？主从 vs 哨兵 vs Cluster？

**答：**

| 对比项 | 主从复制 | 哨兵模式 | Redis Cluster |
|--------|----------|----------|---------------|
| 高可用 | ❌ 主节点故障需手动切换 | ✅ 自动故障转移 | ✅ 自动故障转移 |
| 数据分片 | ❌ 不支持 | ❌ 不支持 | ✅ 16384 哈希槽分片 |
| 水平扩展 | ❌ 写能力受限于单主 | ❌ 写能力受限于单主 | ✅ 多主节点，写能力线性扩展 |
| 读写分离 | ✅ 主写从读 | ✅ 主写从读 | ✅ 但通常不做读写分离 |
| 复杂度 | 低 | 中 | 高 |
| 适用场景 | 数据备份、读多写少 | 中小规模，需要自动故障转移 | 大规模数据，需要水平扩展 |

**选择建议：**
- 数据量小、并发不高 → 主从 + 哨兵足够
- 数据量大、需要水平扩展写能力 → Redis Cluster
- 对一致性要求极高 → 考虑其他方案（如 etcd、ZooKeeper）

**Cluster 的限制：**
- 不支持跨槽的多 key 操作（除非用 Hash Tag）
- 不支持 SELECT 切换数据库（只有 db0）
- 事务只能在同一个节点上执行

## 五、分布式锁

1. Redis 实现分布式锁的方式？SETNX + EXPIRE？

**答：**

**基本实现：**
```redis
SET lock_key unique_value NX EX 30
```
- `NX`：key 不存在时才设置（互斥）
- `EX 30`：设置 30 秒过期（防止死锁）
- `unique_value`：使用 UUID 等唯一值，释放锁时校验，防止误删别人的锁

**释放锁（Lua 脚本保证原子性）：**
```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```
必须用 Lua 脚本，因为 GET + DEL 不是原子操作，可能出现：A 判断是自己的锁 → 锁过期 → B 获取锁 → A 删除了 B 的锁。

**为什么不用 SETNX + EXPIRE 两条命令？**
- 不是原子操作，SETNX 成功后如果宕机，EXPIRE 没执行，锁永远不释放（死锁）
- Redis 2.6.12+ 支持 `SET key value NX EX seconds` 一条命令搞定

**存在的问题：**
- 锁过期但业务没执行完 → 其他线程获取到锁 → 并发问题
- 解决：看门狗机制自动续期（Redisson 的方案）

---

2. Redisson 分布式锁的原理？看门狗机制？

**答：**

**Redisson 加锁原理（基于 Lua 脚本）：**
- 使用 Hash 结构存储锁：`key = 锁名`，`field = 客户端ID（UUID:threadId）`，`value = 重入次数`
- 支持可重入：同一个线程多次加锁，value 递增；解锁时递减，减到 0 才真正释放
- 加锁、解锁都通过 Lua 脚本保证原子性

**看门狗（Watchdog）机制：**
- 如果加锁时没有指定 leaseTime（过期时间），Redisson 默认设置 30 秒过期
- 同时启动一个后台定时任务（看门狗），每 10 秒（lockWatchdogTimeout / 3）检查一次
- 如果锁还被持有，自动续期到 30 秒
- 业务执行完释放锁时，看门狗停止
- 如果持有锁的 JVM 宕机，看门狗也随之停止，锁会在 30 秒后自动过期，不会死锁

**如果指定了 leaseTime**：不会启动看门狗，锁到期自动释放。

**Redisson 的其他锁类型：**
- `RLock`：可重入锁
- `RFairLock`：公平锁（按请求顺序获取）
- `RReadWriteLock`：读写锁
- `RSemaphore`：信号量
- `RCountDownLatch`：闭锁

---

3. RedLock 算法是什么？有什么争议？

**答：**

**背景**：单 Redis 实例做分布式锁，如果主节点宕机，从节点可能还没同步到锁数据就被提升为主节点，导致锁丢失。RedLock 就是为了解决这个问题。

**RedLock 算法流程（假设 N=5 个独立 Redis 实例）：**
```
1. 获取当前时间 T1
2. 依次向 5 个 Redis 实例发送加锁请求（SET NX EX），每个请求设置较短超时
3. 获取当前时间 T2，计算加锁耗时 = T2 - T1
4. 如果在超过半数实例（≥3）上加锁成功，且加锁耗时 < 锁的过期时间 → 加锁成功
5. 锁的实际有效时间 = 过期时间 - 加锁耗时
6. 如果加锁失败 → 向所有实例发送解锁请求
```

**争议（Martin Kleppmann vs antirez 的论战）：**

Martin Kleppmann 的质疑：
- **时钟跳跃问题**：RedLock 依赖各节点时钟大致同步，如果某个节点时钟跳跃（NTP 调整），锁可能提前过期
- **GC 停顿问题**：客户端获取锁后发生长时间 GC，锁过期了但客户端不知道，仍然操作共享资源
- **本质问题**：如果需要强一致的锁，应该用 ZooKeeper 或 etcd 这类基于共识算法的系统

antirez 的回应：
- 时钟跳跃可以通过运维手段避免（禁止大幅度 NTP 调整）
- GC 问题是所有分布式锁都有的，不是 RedLock 特有的

**实际建议**：
- 大多数场景用单 Redis 实例 + Redisson 就够了
- 对正确性要求极高的场景（如金融），用 ZooKeeper/etcd
- RedLock 实现复杂，收益有限，实际使用较少

---

4. Redis 分布式锁和 ZooKeeper 分布式锁的对比？

**答：**

| 对比项 | Redis 分布式锁 | ZooKeeper 分布式锁 |
|--------|---------------|-------------------|
| 实现方式 | SET NX EX + Lua 脚本 | 临时顺序节点 + Watch 机制 |
| 一致性 | AP 模型，异步复制可能丢锁 | CP 模型，ZAB 协议保证强一致 |
| 可靠性 | 主从切换可能丢锁 | 节点宕机临时节点自动删除，不会死锁 |
| 性能 | 高（内存操作，QPS 可达 10w+） | 较低（需要写入多数节点达成共识） |
| 锁释放 | 需要主动释放或等待过期 | 客户端断开连接，临时节点自动删除 |
| 公平性 | 默认非公平（Redisson 有公平锁实现） | 天然公平（顺序节点，按序获取） |
| 看门狗/续期 | Redisson 提供看门狗自动续期 | 通过 Session 心跳维持，无需额外续期 |
| 惊群效应 | 无（各自尝试 SETNX） | 无（Watch 前一个节点，链式通知） |
| 运维复杂度 | 低（通常已有 Redis） | 高（需要独立部署 ZK 集群） |

**选择建议：**
- **追求性能、已有 Redis 基础设施** → Redis + Redisson，适合大多数业务场景
- **追求强一致性、对正确性要求极高** → ZooKeeper / etcd，适合金融、库存等关键场景
- **实际项目中**：90% 的场景用 Redis 分布式锁就够了，因为锁冲突导致的短暂不一致可以通过业务层幂等来兜底

## 六、性能优化与运维

1. Redis 为什么这么快？单线程模型？IO 多路复用？

**答：**

Redis 单机 QPS 可达 10w+，主要原因：

1. **纯内存操作**：数据存储在内存中，读写速度是磁盘的几个数量级以上
2. **单线程模型（避免锁竞争）**：
   - 核心的命令执行是单线程的，避免了多线程的上下文切换和锁竞争开销
   - 单线程不代表慢，瓶颈不在 CPU，而在内存和网络 IO
3. **IO 多路复用（epoll）**：
   - 使用 epoll/kqueue 等 IO 多路复用技术，单线程可以同时监听大量客户端连接
   - 有事件就绪时才处理，不需要为每个连接创建线程
   - 事件循环：注册事件 → 等待就绪 → 处理就绪事件 → 循环
4. **高效的数据结构**：SDS、跳表、压缩列表等都是针对性能优化设计的
5. **单线程避免了不必要的开销**：无锁、无上下文切换、无线程创建销毁

**注意**：Redis 并非完全单线程：
- 主线程：处理客户端请求、执行命令
- 后台线程：BGSAVE（RDB持久化）、BGREWRITEAOF、异步删除（UNLINK）、lazy-free 等

2. Redis 6.0 的多线程是怎么回事？和单线程有什么区别？

**答：**

**背景**：随着网络硬件性能提升，Redis 的性能瓶颈从内存转移到了网络 IO。大量请求时，单线程处理网络读写成为瓶颈。

**Redis 6.0 多线程模型：**
- **多线程只用于网络 IO（读取请求 + 写回响应）**
- **命令执行仍然是单线程**，不需要加锁，保持了线程安全

**处理流程：**
```
1. 主线程通过 epoll 接收客户端连接和请求
2. 将读取到的请求分配给多个 IO 线程并行解析
3. IO 线程完成读取和解析后，主线程单线程执行命令
4. 主线程将响应结果分配给多个 IO 线程并行写回客户端
```

**配置：**
```
io-threads-do-reads yes   # 开启多线程读
io-threads 4              # IO 线程数（建议 CPU 核数的一半，不超过 8）
```

**与单线程的对比：**

| 对比项 | Redis 6.0 前 | Redis 6.0+ |
|--------|-------------|------------|
| 网络 IO | 单线程 | 多线程 |
| 命令执行 | 单线程 | 单线程（不变） |
| 线程安全 | 天然安全 | 天然安全（命令执行仍单线程） |
| 性能提升 | — | 网络 IO 密集场景提升约 1-2 倍 |

**默认是关闭的**，大多数场景单线程已经够用，只有在网络 IO 成为瓶颈时才需要开启。

3. Redis 的内存淘汰策略有哪些？LRU vs LFU？

**答：**

当 Redis 内存使用达到 `maxmemory` 限制时，触发内存淘汰。

**8 种淘汰策略（Redis 4.0+）：**

| 策略 | 说明 |
|------|------|
| **noeviction**（默认） | 不淘汰，写操作直接返回错误（OOM），读操作正常 |
| **allkeys-lru** | 从所有 key 中淘汰最近最少使用的 |
| **allkeys-lfu** | 从所有 key 中淘汰最不经常使用的（Redis 4.0+） |
| **allkeys-random** | 从所有 key 中随机淘汰 |
| **volatile-lru** | 从设置了过期时间的 key 中淘汰最近最少使用的 |
| **volatile-lfu** | 从设置了过期时间的 key 中淘汰最不经常使用的 |
| **volatile-random** | 从设置了过期时间的 key 中随机淘汰 |
| **volatile-ttl** | 从设置了过期时间的 key 中淘汰 TTL 最小的（即将过期的） |

**LRU vs LFU：**

| 对比项 | LRU（Least Recently Used） | LFU（Least Frequently Used） |
|--------|---------------------------|------------------------------|
| 淘汰依据 | 最近最少使用（按时间） | 最不经常使用（按频率） |
| 适用场景 | 通用场景 | 有明显冷热数据的场景 |
| 问题 | 偶尔被访问一次的冷数据不会被淘汰 | 新 key 频率低容易被误淘汰（Redis 用衰减计数器解决） |

**Redis 的近似 LRU**：Redis 不是严格 LRU（太耗内存），而是随机采样 N 个 key（默认 5 个，`maxmemory-samples`），淘汰其中最久未使用的。样本数越大越接近真实 LRU，但性能开销越大。

**推荐**：缓存场景一般用 `allkeys-lru` 或 `allkeys-lfu`。

4. Redis 的过期删除策略？惰性删除 + 定期删除？

**答：**

Redis 使用 **惰性删除 + 定期删除** 两种策略配合：

**1. 惰性删除（Lazy Expiration）：**
- 不主动删除过期 key，而是在每次访问 key 时检查是否过期
- 如果过期则删除并返回空
- 优点：对 CPU 友好，不浪费时间扫描
- 缺点：如果过期 key 一直没被访问，就一直占用内存（内存泄漏）

**2. 定期删除（Periodic Expiration）：**
- Redis 每秒执行 10 次（由 `hz` 配置，默认 10）过期扫描：
  ```
  1. 从设置了过期时间的 key 中随机取 20 个
  2. 删除其中已过期的 key
  3. 如果过期 key 占比超过 25%，重复步骤 1
  4. 每次扫描有时间上限（默认 25ms），避免阻塞主线程
  ```
- 优点：兼顾 CPU 和内存，定期清理过期 key
- 缺点：仍可能有部分过期 key 残留

**为什么不用定时删除？**
- 定时删除：为每个 key 设置一个定时器，到期立即删除
- 虽然内存最友好，但大量定时器会消耗大量 CPU，影响 Redis 性能
- 所以 Redis 选择了惰性 + 定期的折中方案

**兜底**：如果惰性删除和定期删除都没清理掉，还有内存淘汰策略作为最后防线。

5. Pipeline 的作用和使用场景？

**答：**

**问题**：Redis 客户端每次发送命令都要经历：发送命令 → 等待响应 → 发送下一条命令。网络往返时间（RTT）成为瓶颈，尤其是需要执行大量命令时。

**Pipeline 原理**：将多条命令一次性发送给 Redis 服务端，服务端依次执行后将所有结果一次性返回。减少了网络往返次数。

**对比：**
```
普通模式（100 条命令）：100 次 RTT
Pipeline 模式（100 条命令）：1 次 RTT
```

**使用场景：**
- 批量写入数据（如初始化缓存、数据迁移）
- 批量读取数据
- 不需要前一条命令结果来决定后一条命令的场景

**注意事项：**
- Pipeline 不是原子操作，中间可能穿插其他客户端的命令
- 不要一次 Pipeline 太多命令（如几万条），会占用大量内存缓冲区，建议分批（如每批 500-1000 条）
- Pipeline 中的命令如果有一条失败，其他命令仍会执行（不像事务可以回滚）
- 在 Redis Cluster 中，Pipeline 的命令必须落在同一个节点上（或使用客户端自动分组）

6. Redis 事务和 Lua 脚本的区别？

**答：**

**Redis 事务（MULTI/EXEC）：**
```redis
MULTI          -- 开启事务
SET key1 v1    -- 命令入队
SET key2 v2    -- 命令入队
EXEC           -- 执行所有命令
```
- 命令入队时不执行，EXEC 时按顺序执行
- 不支持回滚：某条命令执行失败，其他命令仍会执行（不像 MySQL 事务）
- WATCH 实现乐观锁：WATCH key → MULTI → 操作 → EXEC，如果 key 在 WATCH 后被修改，EXEC 返回 nil

**Lua 脚本（EVAL）：**
```redis
EVAL "redis.call('SET', KEYS[1], ARGV[1]); redis.call('SET', KEYS[2], ARGV[2])" 2 key1 key2 v1 v2
```

**对比：**

| 对比项 | 事务（MULTI/EXEC） | Lua 脚本 |
|--------|-------------------|----------|
| 原子性 | 命令按序执行，但不保证原子性（某条失败不回滚） | 整个脚本原子执行，执行期间不会被其他命令打断 |
| 条件逻辑 | 不支持（命令入队时不知道前一条的结果） | 支持 if/else、循环等逻辑 |
| 性能 | 多次网络交互（MULTI + 命令 + EXEC） | 一次网络交互，服务端执行 |
| 错误处理 | 某条失败继续执行 | 脚本报错则整体失败 |
| 使用场景 | 简单的批量操作 | 需要原子性 + 条件逻辑的场景（如分布式锁、限流） |

**实际项目中**：Lua 脚本用得更多，因为它真正保证了原子性，且支持复杂逻辑。Redis 事务因为不支持回滚，实用性有限。

7. Redis 的内存优化手段？

**答：**

| 优化手段 | 说明 |
|----------|------|
| **选择合适的数据结构** | 能用 Hash 存对象就不要用 String 存 JSON（Hash 的 ziplist 编码更省内存） |
| **控制 key 和 value 大小** | key 尽量短（如 `u:1000:n` 代替 `user:1000:name`）；value 避免存大对象 |
| **利用紧凑编码** | 调整 `hash-max-ziplist-entries/value`、`zset-max-ziplist-entries/value` 等参数，让更多数据使用 ziplist/listpack 编码 |
| **设置合理的过期时间** | 避免无用数据长期占用内存 |
| **使用内存淘汰策略** | 配置 `maxmemory` 和合适的淘汰策略 |
| **共享整数对象** | Redis 默认缓存 0-9999 的整数对象，避免重复创建 |
| **压缩存储** | 对大 value 在应用层做 gzip/snappy 压缩后再存入 Redis |
| **使用 Bitmap/HyperLogLog** | 特定场景下比 Set 等结构省内存（如签到用 Bitmap，UV 统计用 HyperLogLog） |
| **避免大 Key** | 拆分大 Hash/List/Set，减少单 key 内存占用 |
| **定期清理** | 使用 `SCAN` 命令定期扫描清理无用 key（不要用 KEYS *） |

**内存分析工具：**
- `INFO memory`：查看内存使用概况
- `MEMORY USAGE key`：查看单个 key 的内存占用
- `MEMORY DOCTOR`：内存诊断建议
- `redis-rdb-tools`：离线分析 RDB 文件中的内存分布

2. 如何用 Redis 实现排行榜？

**答：**

使用 **Sorted Set（ZSet）**，score 作为排序依据。

**核心命令：**
```redis
# 添加/更新分数
ZADD leaderboard 100 "user:1001"
ZADD leaderboard 200 "user:1002"
ZINCRBY leaderboard 50 "user:1001"    # 加 50 分

# 查询排行榜（从高到低，Top 10）
ZREVRANGE leaderboard 0 9 WITHSCORES

# 查询某用户排名（从高到低，0 开始）
ZREVRANK leaderboard "user:1001"

# 查询某用户分数
ZSCORE leaderboard "user:1001"

# 查询某分数范围内的用户
ZREVRANGEBYSCORE leaderboard 200 100 WITHSCORES
```

**实际设计考虑：**
- **分数相同时的排序**：ZSet 中 score 相同按 member 字典序排序。如果需要按时间排序，可以将 score 设计为 `分数 * 10^13 + (MAX_TIMESTAMP - 时间戳)`，分数高的在前，同分时先达到的在前
- **分页**：使用 `ZREVRANGE key start stop` 实现分页
- **多维度排行榜**：不同维度用不同的 ZSet key（如日榜、周榜、月榜）
- **定期清理**：过期的排行榜数据定期删除，避免内存浪费

3. 如何用 Redis 实现限流？（滑动窗口、令牌桶）

**答：**

**方案一：固定窗口计数器（String + INCR）**
```redis
# 每分钟限制 100 次
INCR rate:user:1001:202604241530    # 当前分钟的计数器
EXPIRE rate:user:1001:202604241530 60
# 判断计数是否超过 100
```
- 简单但有临界问题：窗口切换瞬间可能出现 2 倍流量

**方案二：滑动窗口（ZSet）**
```lua
-- Lua 脚本实现滑动窗口限流
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])    -- 窗口大小（毫秒）
local limit = tonumber(ARGV[3])

-- 移除窗口外的记录
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
-- 统计窗口内的请求数
local count = redis.call('ZCARD', key)
if count < limit then
    redis.call('ZADD', key, now, now .. math.random())
    redis.call('EXPIRE', key, math.ceil(window / 1000))
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```
- 精确控制窗口内的请求数，无临界问题
- 缺点：每个请求都要存一条记录，内存占用较大

**方案三：令牌桶（基于 Lua 脚本）**
```
原理：以固定速率往桶里放令牌，请求来时取一个令牌，桶空则拒绝
- 用 String 存储：当前令牌数 + 上次补充时间
- 每次请求时计算应该补充多少令牌，然后尝试消耗一个
- 支持突发流量（桶满时可以一次性消耗多个令牌）
```

**方案四：漏桶（固定速率处理）**
- 请求进入队列，以固定速率处理
- 可以用 List + 定时消费实现
- 适合需要平滑流量的场景

**推荐**：一般场景用滑动窗口，需要允许突发流量用令牌桶。

4. 如何用 Redis 实现延迟队列？

**答：**

**核心思路**：使用 ZSet，score 存储任务的执行时间戳，member 存储任务内容。

**实现步骤：**

**1. 添加延迟任务：**
```redis
# 30 秒后执行的任务
ZADD delay_queue 1745481630 '{"taskId":"001","action":"closeOrder","orderId":"12345"}'
```

**2. 消费者轮询取出到期任务（Lua 脚本保证原子性）：**
```lua
-- 取出 score ≤ 当前时间的任务
local tasks = redis.call('ZRANGEBYSCORE', KEYS[1], 0, ARGV[1], 'LIMIT', 0, 1)
if #tasks > 0 then
    redis.call('ZREM', KEYS[1], tasks[1])
    return tasks[1]
else
    return nil
end
```

**3. 消费者循环：**
```java
while (true) {
    String task = evalLuaScript(delayQueueKey, System.currentTimeMillis());
    if (task != null) {
        processTask(task);  // 处理任务
    } else {
        Thread.sleep(100);  // 没有到期任务，休眠一会
    }
}
```

**典型场景：**
- 订单超时未支付自动关闭
- 延迟通知/提醒
- 定时重试

**注意事项：**
- 多个消费者时，Lua 脚本的 ZRANGEBYSCORE + ZREM 保证不会重复消费
- 轮询间隔影响精度，100ms 轮询则最大延迟 100ms
- 任务处理失败需要有重试机制（重新 ZADD 回去）
- 对可靠性要求高的场景，建议用专业的消息队列（RocketMQ 延迟消息、RabbitMQ 死信队列）

5. 如何用 Redis 实现分布式 Session？

**答：**

**问题**：多实例部署时，用户请求可能被负载均衡到不同服务器，而 Session 默认存在本地内存中，导致用户在不同服务器间切换时丢失登录状态。

**解决方案：将 Session 存储到 Redis 中，所有服务器共享。**

**Spring Boot 集成方式（Spring Session + Redis）：**

1. 引入依赖：
```xml
<dependency>
    <groupId>org.springframework.session</groupId>
    <artifactId>spring-session-data-redis</artifactId>
</dependency>
```

2. 配置：
```yaml
spring:
  session:
    store-type: redis
    timeout: 1800  # Session 过期时间 30 分钟
  redis:
    host: localhost
    port: 6379
```

3. 启用：
```java
@EnableRedisHttpSession(maxInactiveIntervalInSeconds = 1800)
```

**原理：**
- Spring Session 通过 Filter 拦截请求，将 HttpSession 的读写操作代理到 Redis
- Session 数据以 Hash 结构存储：`spring:session:sessions:{sessionId}`
- 每次请求自动续期

**存储结构：**
```
Key: spring:session:sessions:abc-123-def
Hash:
  sessionAttr:user → {序列化的用户对象}
  creationTime → 1745481630000
  lastAccessedTime → 1745481930000
  maxInactiveInterval → 1800
```

**其他方案：**
- 不用 Session，改用 JWT Token（无状态，不需要服务端存储）
- Token + Redis：Token 做身份标识，用户信息存 Redis（兼具无状态和服务端控制能力，如主动踢人下线）
