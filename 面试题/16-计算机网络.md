# 16 - 计算机网络

## 一、网络分层模型

1. OSI 七层模型和 TCP/IP 四层模型？

**回答：**

OSI 七层模型偏理论，主要用于理解网络通信的分层职责；TCP/IP 四层模型偏工程实践，是互联网实际使用的协议体系。

| OSI 七层模型 | TCP/IP 四层模型 | 主要作用 | 常见协议/技术 |
|-------------|----------------|----------|---------------|
| 应用层 | 应用层 | 面向用户和应用程序，定义数据格式和交互规则 | HTTP、HTTPS、DNS、FTP、SMTP、WebSocket |
| 表示层 | 应用层 | 数据编码、压缩、加密解密 | TLS/SSL、JPEG、JSON、XML |
| 会话层 | 应用层 | 建立、管理和释放会话 | RPC 会话、登录态、Session |
| 传输层 | 传输层 | 端到端通信，负责进程间数据传输 | TCP、UDP |
| 网络层 | 网络层 | 主机到主机的寻址和路由 | IP、ICMP、ARP、路由协议 |
| 数据链路层 | 网络接口层 | 相邻节点之间的数据帧传输、差错检测 | Ethernet、MAC、PPP |
| 物理层 | 网络接口层 | 比特流在物理介质上传输 | 网线、光纤、无线电信号 |

**面试表达：**
- OSI 七层更适合描述职责边界，TCP/IP 四层更贴近实际落地。
- 应用开发最常接触应用层、传输层和网络层，比如 HTTP、TCP、IP、DNS、TLS。
- 排查网络问题时可以按分层思路从下往上看：网络是否通、端口是否通、协议是否正确、应用是否正常。

2. 每一层的作用和常见协议？

**回答：**

| 层级 | 作用 | 常见协议/设备 | 面试重点 |
|------|------|---------------|----------|
| 物理层 | 负责 0/1 比特流在物理介质上传输 | 网线、光纤、无线 | 关注信号、电压、速率 |
| 数据链路层 | 把比特流组织成帧，负责同一链路内的 MAC 寻址和差错检测 | Ethernet、MAC、交换机 | 交换机工作在二层，通过 MAC 地址转发 |
| 网络层 | 负责 IP 寻址和路由选择，实现跨网络通信 | IP、ICMP、路由器 | 路由器工作在三层，通过 IP 地址转发 |
| 传输层 | 负责端到端进程通信，提供可靠或不可靠传输 | TCP、UDP | TCP 可靠、面向连接；UDP 简单、低延迟 |
| 应用层 | 定义具体应用的数据格式和交互语义 | HTTP、DNS、FTP、SMTP、MQTT | 后端开发最常接触 |

**一次 HTTP 请求的分层过程：**

```text
应用层：浏览器构造 HTTP 请求
传输层：TCP 建立连接，并把 HTTP 数据切分成 TCP 段
网络层：IP 根据目标地址进行路由转发
数据链路层：以太网帧在相邻节点之间传输
物理层：数据变成电信号/光信号/无线信号发送
```

**补充：**
- DNS 虽然是应用层协议，但它为网络访问提供域名到 IP 的解析。
- TLS 位于应用层和传输层之间，通常理解为给应用层协议提供加密通道。
- ARP 用于局域网内根据 IP 找 MAC，常被归到网络层和数据链路层之间。

## 二、TCP

1. TCP 三次握手的过程？为什么是三次而不是两次？

**回答：**

TCP 三次握手的目标是：确认双方的收发能力正常，并同步双方的初始序列号。

```text
第一次：客户端 → 服务端：SYN，seq = x
      客户端进入 SYN_SENT，表示我要建立连接。

第二次：服务端 → 客户端：SYN + ACK，seq = y，ack = x + 1
      服务端进入 SYN_RCVD，表示我收到了你的请求，也同意建立连接。

第三次：客户端 → 服务端：ACK，ack = y + 1
      客户端和服务端进入 ESTABLISHED，连接建立完成。
```

**为什么不是两次：**

如果只有两次握手，服务端收到客户端的 SYN 后就认为连接建立，但客户端未必真的收到服务端的确认。比如一个历史的、延迟到达的 SYN 报文被服务端收到，服务端会误以为客户端要建立新连接，从而分配资源，导致半连接浪费。

三次握手的第三次 ACK 可以让服务端确认：
- 客户端的发送能力正常。
- 客户端的接收能力正常。
- 客户端确实收到了服务端的 SYN + ACK。
- 双方的初始序列号都已完成同步。

**面试加分点：**
- TCP 连接本质是双方维护状态，而不是网络中真的有一条物理连接。
- 三次握手不是为了加密，而是为了可靠地建立连接状态。
- 握手阶段可能遭遇 SYN Flood 攻击，可以通过 SYN Cookie、调大半连接队列、限流等方式缓解。

2. TCP 四次挥手的过程？为什么需要 TIME_WAIT？

**回答：**

TCP 是全双工协议，连接关闭时，两个方向的数据流需要分别关闭，所以通常需要四次挥手。

```text
第一次：客户端 → 服务端：FIN
      客户端表示自己没有数据要发了，进入 FIN_WAIT_1。

第二次：服务端 → 客户端：ACK
      服务端确认收到关闭请求，客户端进入 FIN_WAIT_2。
      此时服务端可能还有数据要发。

第三次：服务端 → 客户端：FIN
      服务端数据发送完毕，也请求关闭连接。

第四次：客户端 → 服务端：ACK
      客户端确认服务端的 FIN，进入 TIME_WAIT。
      等待 2MSL 后真正关闭。
```

**为什么需要 TIME_WAIT：**

1. **保证最后一个 ACK 能到达服务端**
   - 如果客户端最后的 ACK 丢失，服务端会重发 FIN。
   - 客户端处于 TIME_WAIT 时还能再次发送 ACK。

2. **防止历史报文污染新连接**
   - TCP 报文在网络中可能延迟。
   - 等待 2MSL 可以让旧连接中的报文在网络中自然消失，避免相同四元组的新连接收到旧数据。

**为什么是 2MSL：**
- MSL 是报文最大生存时间。
- 一来一回最多经历 2 个 MSL，因此等待 2MSL 可以覆盖旧报文和重传报文的生命周期。

**补充：**
- 主动关闭连接的一方会进入 TIME_WAIT。
- 服务端 TIME_WAIT 很多，通常说明服务端在主动关闭连接，或者短连接过多。
- 可以通过连接池、长连接、合理的 keep-alive、反向代理复用连接来减少 TIME_WAIT 压力。

3. TCP 如何保证可靠传输？（序列号、确认应答、超时重传、滑动窗口）

**回答：**

TCP 通过一组机制共同保证可靠传输：

| 机制 | 作用 |
|------|------|
| 序列号 | 给每个字节编号，接收方可以按序重组数据，发现乱序和丢包 |
| 确认应答 ACK | 接收方告诉发送方自己已经收到哪些数据 |
| 超时重传 | 发送方在超时时间内未收到 ACK，就重传数据 |
| 快速重传 | 收到多个重复 ACK 时，不等超时，立即重传疑似丢失的报文 |
| 滑动窗口 | 允许发送方连续发送多个报文，提高吞吐量 |
| 流量控制 | 接收方通过窗口大小告诉发送方还能接收多少数据 |
| 拥塞控制 | 根据网络拥塞程度动态调整发送速率 |
| 校验和 | 校验 TCP 头和数据，发现传输错误 |

**可靠传输过程：**

```text
发送方发送 seq = 1000，长度 500 的数据
接收方收到后返回 ACK = 1500
表示 1500 之前的数据都已经收到，下次希望从 1500 开始接收
```

**重点理解：**
- TCP 的 ACK 是累计确认，不是每个包都必须单独确认。
- TCP 保证的是字节流可靠有序，不保证消息边界。
- 应用层如果需要消息边界，需要自己设计协议，例如长度字段、分隔符、固定长度等。

4. TCP 的流量控制？滑动窗口机制？

**回答：**

TCP 的流量控制是为了解决发送方发得太快、接收方处理不过来的问题。核心机制是**滑动窗口**。

接收方会在 TCP 报文头中通过 `Window` 字段告诉发送方自己的接收窗口大小，也就是还能接收多少字节。发送方根据这个窗口控制未确认数据的数量。

```text
发送窗口 = min(接收窗口 rwnd, 拥塞窗口 cwnd)
```

其中：
- `rwnd`：接收窗口，由接收方决定，用于流量控制。
- `cwnd`：拥塞窗口，由发送方根据网络状况维护，用于拥塞控制。

**滑动窗口的好处：**
- 不需要发一个包等一个 ACK，可以批量发送，提高吞吐量。
- ACK 返回后，窗口向前滑动，继续发送新数据。
- 接收方缓存不足时，可以把窗口调小，甚至通告为 0。

**零窗口问题：**

如果接收方窗口为 0，发送方会停止发送。但为了避免双方一直等待，TCP 有**零窗口探测**机制，发送方会定期发送探测报文，询问接收方窗口是否恢复。

**面试加分点：**
- 流量控制关注接收方能力，避免把接收方打爆。
- 拥塞控制关注网络能力，避免把网络打爆。
- 实际发送速度同时受接收窗口和拥塞窗口限制。

---

5. TCP 的拥塞控制？慢启动、拥塞避免、快重传、快恢复？

**回答：**

TCP 拥塞控制是为了解决网络中间链路拥塞的问题。发送方通过拥塞窗口 `cwnd` 控制发送速率。

| 阶段 | 触发条件 | 行为 |
|------|----------|------|
| 慢启动 | 连接刚建立或超时后重新开始 | `cwnd` 从小开始指数增长 |
| 拥塞避免 | `cwnd` 达到慢启动阈值 `ssthresh` | `cwnd` 线性增长 |
| 快重传 | 收到 3 个重复 ACK | 不等超时，立即重传丢失报文 |
| 快恢复 | 快重传之后 | 将 `ssthresh` 降低，`cwnd` 不直接回到 1，而是进入较温和的恢复 |

**过程理解：**

1. **慢启动**
   - 初始 `cwnd` 较小。
   - 每经过一个 RTT，`cwnd` 近似翻倍。
   - 虽然叫慢启动，但增长速度其实很快，只是从小窗口开始探测。

2. **拥塞避免**
   - 当 `cwnd >= ssthresh` 后，进入线性增长。
   - 每个 RTT 大约增加 1 个 MSS，避免增长过快导致拥塞。

3. **超时重传**
   - 说明网络可能严重拥塞。
   - 通常将 `ssthresh = cwnd / 2`，然后 `cwnd` 降到较小值重新慢启动。

4. **快重传 + 快恢复**
   - 收到 3 个重复 ACK，说明部分报文丢失，但网络仍有数据在流动。
   - 不必像超时那样激进地从头开始，降低窗口后继续传输。

**总结：**
- 慢启动和拥塞避免是发送速率增长策略。
- 快重传和快恢复是丢包后的恢复策略。
- 拥塞控制是 TCP 能在互联网大规模运行的重要原因。

6. TCP 和 UDP 的区别？各自的使用场景？

**回答：**

| 对比维度 | TCP | UDP |
|----------|-----|-----|
| 是否连接 | 面向连接，需要三次握手 | 无连接，直接发送 |
| 可靠性 | 可靠传输，保证有序、不丢、不重 | 不保证可靠，不保证顺序 |
| 数据形式 | 字节流，没有消息边界 | 数据报，有消息边界 |
| 拥塞控制 | 有 | 无 |
| 流量控制 | 有 | 无 |
| 头部开销 | 至少 20 字节 | 8 字节 |
| 速度 | 相对较慢，但稳定可靠 | 开销小，延迟低 |

**TCP 适用场景：**
- HTTP/HTTPS
- 文件传输
- 数据库连接
- RPC 调用
- 对可靠性要求高的业务系统

**UDP 适用场景：**
- DNS 查询
- 视频直播、音视频通话
- 游戏同步
- IoT 设备上报
- QUIC/HTTP3 底层

**面试表达：**

TCP 适合可靠性优先的场景，UDP 适合实时性优先、允许少量丢包或应用层自己做可靠性控制的场景。比如视频通话中，丢一帧可以接受，但因为重传导致整体卡顿反而体验更差。

7. TCP 粘包问题？（和 Netty 章节关联）

**回答：**

TCP 是面向字节流的协议，它只保证字节有序到达，不保证应用层消息边界。因此发送方多次 `write` 的数据，接收方可能一次读到，也可能一次只读到一部分，这就是常说的粘包/拆包。

**产生原因：**
- TCP 字节流没有消息边界。
- Nagle 算法可能合并小包发送。
- 接收方缓冲区一次读取多个应用层消息。
- 发送的数据大于 MSS 或接收缓冲区，导致拆包。

**解决方案：**

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| 固定长度 | 每个消息固定 N 字节，不足补齐 | 协议简单、字段长度固定 |
| 分隔符 | 用 `\n`、特殊字符作为消息结束标识 | 文本协议，如 Redis 协议部分场景 |
| 长度字段 | 消息头中记录消息体长度 | 最常用，适合二进制协议 |
| 应用层协议 | HTTP、WebSocket、MQTT 等协议自带边界 | 直接使用成熟协议 |

**Netty 中的处理：**
- `FixedLengthFrameDecoder`：固定长度拆包。
- `DelimiterBasedFrameDecoder`：按分隔符拆包。
- `LineBasedFrameDecoder`：按换行符拆包。
- `LengthFieldBasedFrameDecoder`：按长度字段拆包，最常用。

**面试加分点：**
- 粘包不是 TCP 的 bug，而是 TCP 字节流语义的正常表现。
- UDP 是数据报协议，有消息边界，所以一般不会出现 TCP 意义上的粘包问题。

8. TCP 的 Keep-Alive 机制？

**回答：**

TCP Keep-Alive 是 TCP 层的保活机制，用于检测一条长时间空闲的连接是否仍然可用。

**工作方式：**
- 连接空闲一段时间后，内核发送 Keep-Alive 探测包。
- 如果对端正常，会返回 ACK。
- 如果连续多次没有响应，内核认为连接失效并关闭连接。

**Linux 常见参数：**

| 参数 | 含义 |
|------|------|
| `tcp_keepalive_time` | 连接空闲多久后开始探测 |
| `tcp_keepalive_intvl` | 探测包发送间隔 |
| `tcp_keepalive_probes` | 最大探测次数 |

**TCP Keep-Alive 和应用层心跳的区别：**

| 对比 | TCP Keep-Alive | 应用层心跳 |
|------|----------------|------------|
| 所在层级 | 传输层 | 应用层 |
| 感知能力 | 只能判断连接是否大致存活 | 可以判断业务是否正常 |
| 默认时间 | 通常较长，可能是小时级 | 可根据业务设置为秒级 |
| 灵活性 | 依赖操作系统参数 | 应用可自定义内容和策略 |

**实际项目中：**
- 长连接系统一般会同时使用 TCP Keep-Alive 和应用层心跳。
- TCP Keep-Alive 兜底检测死连接，应用层心跳用于业务级保活、超时踢出、连接状态维护。
- 在 Netty 中常用 `IdleStateHandler` 实现读写空闲检测和心跳。

## 三、HTTP

1. HTTP 1.0、1.1、2.0、3.0 的区别？

**回答：**

| 版本 | 核心特点 | 主要问题/改进 |
|------|----------|---------------|
| HTTP/1.0 | 短连接为主，每次请求通常新建 TCP 连接 | 连接建立成本高 |
| HTTP/1.1 | 默认长连接、支持管线化、Host 头、分块传输、缓存增强 | 队头阻塞仍然明显 |
| HTTP/2 | 二进制分帧、多路复用、Header 压缩、服务端推送 | TCP 层丢包仍会导致整个连接阻塞 |
| HTTP/3 | 基于 QUIC/UDP，内置 TLS 1.3，多路复用更彻底 | 部署和网络设备兼容性要求更高 |

**HTTP/1.1 的改进：**
- 默认 `Connection: keep-alive`，复用 TCP 连接。
- 引入 `Host` 头，支持虚拟主机。
- 支持 `Transfer-Encoding: chunked` 分块传输。
- 缓存控制更完善，如 `Cache-Control`、`ETag`。

**HTTP/2 的改进：**
- 使用二进制帧，不再是纯文本协议。
- 多个请求可以复用一个 TCP 连接并发传输。
- HPACK 压缩请求头，减少重复 Header 开销。
- 解决了 HTTP/1.1 应用层队头阻塞，但 TCP 丢包仍会影响整个连接。

**HTTP/3 的改进：**
- 基于 QUIC，底层使用 UDP。
- QUIC 在用户态实现可靠传输、拥塞控制和多路复用。
- 某个流丢包不会阻塞其他流，缓解 TCP 层队头阻塞。
- 握手更快，结合 TLS 1.3 可减少连接建立耗时。

2. HTTP 的请求方法？GET 和 POST 的区别？

**回答：**

常见 HTTP 方法：

| 方法 | 含义 |
|------|------|
| GET | 获取资源 |
| POST | 创建资源或提交数据 |
| PUT | 整体更新资源 |
| PATCH | 局部更新资源 |
| DELETE | 删除资源 |
| HEAD | 只获取响应头，不返回响应体 |
| OPTIONS | 查询服务器支持的方法，常用于 CORS 预检 |

**GET 和 POST 的区别：**

| 对比维度 | GET | POST |
|----------|-----|------|
| 语义 | 获取资源 | 提交数据/创建资源 |
| 参数位置 | 通常放在 URL Query 中 | 通常放在请求体 Body 中 |
| 幂等性 | 通常幂等 | 通常不幂等 |
| 缓存 | 更容易被浏览器/CDN 缓存 | 默认不缓存 |
| 长度限制 | 受浏览器、代理、服务器 URL 长度限制 | 理论上由服务器配置限制 |
| 安全性 | 参数暴露在 URL 中，不适合敏感数据 | 参数在 Body 中，但仍需 HTTPS 才安全 |

**注意：**
- GET 也可以带 Body，但不推荐，很多中间件不会按预期处理。
- POST 不是天然安全，明文 HTTP 下 Body 也能被抓包看到。
- 安全性要靠 HTTPS、鉴权、签名、权限控制，而不是靠 GET/POST 本身。

---

3. HTTP 状态码？常见的 200、301、302、304、400、401、403、404、500、502、503？

**回答：**

HTTP 状态码按首位数字分为五类：

| 分类 | 含义 |
|------|------|
| 1xx | 信息提示，请求已收到，继续处理 |
| 2xx | 成功 |
| 3xx | 重定向或缓存相关 |
| 4xx | 客户端错误 |
| 5xx | 服务端错误 |

**常见状态码：**

| 状态码 | 含义 | 常见场景 |
|--------|------|----------|
| 200 | OK，请求成功 | 查询成功、接口正常返回 |
| 201 | Created，资源创建成功 | 新增资源成功 |
| 204 | No Content，成功但无响应体 | 删除成功、不需要返回内容 |
| 301 | 永久重定向 | 域名永久迁移，HTTP 跳 HTTPS |
| 302 | 临时重定向 | 临时跳转、登录后跳转 |
| 304 | Not Modified | 协商缓存命中，资源未修改 |
| 400 | Bad Request | 参数格式错误、JSON 解析失败 |
| 401 | Unauthorized | 未登录或 Token 无效 |
| 403 | Forbidden | 已认证但没有权限 |
| 404 | Not Found | 资源不存在、路径错误 |
| 405 | Method Not Allowed | 请求方法不支持 |
| 429 | Too Many Requests | 请求过多，被限流 |
| 500 | Internal Server Error | 服务端代码异常 |
| 502 | Bad Gateway | 网关收到上游服务异常响应 |
| 503 | Service Unavailable | 服务不可用、过载、维护中 |
| 504 | Gateway Timeout | 网关等待上游服务超时 |

**面试区分：**
- 401 是未认证，重点是“你是谁”还没确认。
- 403 是无权限，重点是“你是谁”已确认，但不能访问。
- 502 通常是网关和上游服务之间出问题。
- 503 通常是服务暂时不可用，比如服务过载、发布中、熔断降级。

4. HTTP 的长连接和短连接？Connection: keep-alive？

**回答：**

**短连接**是每次 HTTP 请求都新建 TCP 连接，请求完成后关闭连接。HTTP/1.0 默认短连接。

**长连接**是多个 HTTP 请求复用同一个 TCP 连接，减少频繁建立和关闭连接的成本。HTTP/1.1 默认长连接。

| 对比 | 短连接 | 长连接 |
|------|--------|--------|
| TCP 连接 | 每次请求新建 | 多次请求复用 |
| 开销 | 握手和挥手成本高 | 成本低 |
| 性能 | QPS 高时较差 | 适合高频请求 |
| 资源占用 | 连接用完即释放 | 空闲连接也占用资源 |

**`Connection: keep-alive` 的作用：**
- 客户端告诉服务端希望复用连接。
- 服务端同意后，请求结束不立即关闭 TCP 连接。
- 后续请求可以继续使用同一条连接。

**注意：**
- 长连接不是永久连接，服务端通常会配置 idle timeout 和最大请求数。
- 长连接需要配合连接池使用，比如 Java HTTP 客户端、数据库连接池、RPC 连接池。
- 如果连接池配置不合理，可能出现连接泄漏、空闲连接过多、服务端连接数耗尽。

5. Cookie 和 Session 的区别？

**回答：**

Cookie 和 Session 都常用于保存用户状态，但位置和安全边界不同。

| 对比 | Cookie | Session |
|------|--------|---------|
| 存储位置 | 客户端浏览器 | 服务端 |
| 数据内容 | 可存少量数据，通常存 SessionId 或 Token | 存用户登录态、权限等服务端状态 |
| 安全性 | 客户端可见，需防篡改、防窃取 | 相对安全，但需要保护 SessionId |
| 容量 | 较小，通常几 KB | 由服务端存储决定 |
| 扩展性 | 无状态能力更好 | 分布式场景需要共享 Session |

**典型流程：**

```text
用户登录成功
服务端创建 Session，并生成 SessionId
服务端通过 Set-Cookie 把 SessionId 写给浏览器
浏览器后续请求自动带上 Cookie
服务端根据 SessionId 找到用户登录态
```

**Cookie 安全属性：**
- `HttpOnly`：禁止 JavaScript 读取，降低 XSS 窃取风险。
- `Secure`：只在 HTTPS 下传输。
- `SameSite`：限制跨站携带 Cookie，降低 CSRF 风险。
- `Max-Age/Expires`：控制过期时间。

**分布式场景：**
- 可以用 Redis 集中存储 Session。
- 也可以改用 JWT 做无状态认证。
- 后端系统中更常见的是 Token/JWT + Redis 黑名单/续期机制的组合。

---

6. HTTP 缓存机制？强缓存和协商缓存？

**回答：**

HTTP 缓存分为**强缓存**和**协商缓存**。

**强缓存：**

浏览器直接使用本地缓存，不向服务器发送请求。

常见响应头：

| Header | 含义 |
|--------|------|
| `Cache-Control: max-age=3600` | 资源 3600 秒内有效 |
| `Cache-Control: no-cache` | 不直接使用强缓存，需要协商 |
| `Cache-Control: no-store` | 不允许缓存 |
| `Expires` | 过期时间，HTTP/1.0 字段，优先级低于 Cache-Control |

**协商缓存：**

浏览器向服务端发请求，由服务端判断资源是否变化。如果没变，返回 304，浏览器使用本地缓存。

常见 Header：

| 首次响应 | 再次请求 | 含义 |
|----------|----------|------|
| `ETag` | `If-None-Match` | 基于资源唯一标识判断是否变化 |
| `Last-Modified` | `If-Modified-Since` | 基于资源最后修改时间判断是否变化 |

**优先级：**
- 强缓存优先于协商缓存。
- `Cache-Control` 优先于 `Expires`。
- `ETag` 通常优先于 `Last-Modified`，因为修改时间精度有限，而且文件内容没变但修改时间可能变化。

**实际使用：**
- HTML 通常使用协商缓存或不强缓存，避免版本更新不及时。
- JS/CSS/图片等静态资源通常文件名带 hash，然后设置长时间强缓存。
- CDN 也会基于 HTTP 缓存头决定是否回源。

## 四、HTTPS

1. HTTPS 的工作原理？SSL/TLS 握手过程？

**回答：**

HTTPS = HTTP + TLS。HTTP 负责应用层语义，TLS 负责身份认证、密钥协商、数据加密和完整性校验。

**HTTPS 解决的问题：**
- 加密：防止内容被窃听。
- 完整性：防止内容被篡改。
- 身份认证：确认访问的服务端是真的目标网站。

**TLS 1.2 简化握手过程：**

```text
1. ClientHello
   客户端发送支持的 TLS 版本、加密套件、随机数 client_random。

2. ServerHello
   服务端选择 TLS 版本和加密套件，返回 server_random。

3. Certificate
   服务端发送数字证书，客户端验证证书链、域名、有效期、签名。

4. Key Exchange
   双方通过 RSA 或 ECDHE 等方式协商预主密钥。

5. 生成会话密钥
   双方根据 client_random、server_random、预主密钥生成对称加密密钥。

6. Finished
   双方切换到加密通信，并校验握手过程未被篡改。
```

**为什么 HTTPS 既用非对称加密又用对称加密：**
- 非对称加密用于身份认证和密钥协商，解决“如何安全交换密钥”。
- 对称加密用于实际数据传输，性能更高。

**TLS 1.3 的改进：**
- 减少握手轮次，连接建立更快。
- 移除不安全的旧算法。
- 默认使用前向安全的密钥交换方式。

---

2. 对称加密和非对称加密的区别？

**回答：**

| 对比 | 对称加密 | 非对称加密 |
|------|----------|------------|
| 密钥 | 加密和解密使用同一个密钥 | 公钥加密、私钥解密，或私钥签名、公钥验签 |
| 性能 | 快，适合大量数据加密 | 慢，适合密钥交换和签名 |
| 密钥分发 | 难点在于如何安全传递密钥 | 公钥可以公开，私钥必须保密 |
| 常见算法 | AES、ChaCha20 | RSA、ECC |
| 典型用途 | HTTPS 数据传输、数据库字段加密 | TLS 握手、数字签名、证书认证 |

**在 HTTPS 中的配合：**

```text
非对称加密/密钥交换：协商会话密钥
对称加密：使用会话密钥加密 HTTP 请求和响应数据
摘要/认证标签：保证数据完整性，防篡改
```

**补充：**
- RSA 既可用于加密，也可用于签名。
- ECDHE 常用于密钥交换，具备前向安全性。
- AES-GCM 是常见的认证加密模式，既加密又校验完整性。

---

3. 数字证书的作用？CA 证书链？

**回答：**

数字证书的核心作用是：把“域名/主体”和“公钥”可信地绑定起来，防止中间人伪造服务端。

**证书中通常包含：**
- 域名或主体信息。
- 服务端公钥。
- 证书有效期。
- 签发者信息。
- 签名算法。
- CA 对证书内容的数字签名。

**客户端验证证书时会检查：**
- 证书是否过期。
- 域名是否匹配。
- 证书是否被吊销。
- 证书链是否能追溯到系统信任的根 CA。
- 证书签名是否正确。

**CA 证书链：**

```text
服务器证书
   ↑ 由中间 CA 签发
中间 CA 证书
   ↑ 由根 CA 签发
根 CA 证书
   ↑ 预置在操作系统/浏览器信任库中
```

**为什么需要证书链：**
- 根 CA 私钥非常重要，通常离线保存，不直接签发大量网站证书。
- 中间 CA 负责日常签发，降低根 CA 私钥暴露风险。
- 客户端只要信任根 CA，就可以通过证书链信任服务器证书。

**面试加分点：**
- 证书不是用来加密全部数据的，而是用来证明公钥可信。
- HTTPS 防中间人攻击的关键是证书校验。如果客户端跳过证书校验，HTTPS 安全性会大幅下降。
- 移动端可以使用证书固定（Certificate Pinning）进一步降低伪造证书风险。

---

4. mTLS（双向 TLS）是什么？你项目中怎么用的？（GDPR 项目）

**回答：**

mTLS 是 Mutual TLS，双向 TLS。普通 HTTPS 只校验服务端证书，客户端确认“我访问的服务端是真的”。mTLS 则要求服务端也校验客户端证书，确认“访问我的客户端也是真的”。

| 对比 | 普通 TLS | mTLS |
|------|----------|------|
| 服务端身份认证 | 有 | 有 |
| 客户端身份认证 | 通常通过账号、Token、Cookie | 通过客户端证书 |
| 安全强度 | 较高 | 更高，适合服务间调用、设备接入、高敏系统 |

**mTLS 握手重点：**

```text
1. 客户端校验服务端证书。
2. 服务端发送 CertificateRequest，要求客户端提供证书。
3. 客户端发送自己的客户端证书。
4. 服务端校验客户端证书链、有效期、吊销状态和身份映射。
5. 双方协商会话密钥，建立加密通道。
```

**结合 GDPR 项目可以这样回答：**

在海外业务 GDPR 合规项目里，数据传输安全是重点。我们基础层面要求全链路 TLS 1.2+，前端到后端、服务间调用、设备到云端都要走 HTTPS 或 TLS 加密通道。对于涉及敏感数据处理的内部服务调用，可以使用 mTLS 做双向认证，避免只有 Token 泄露就能调用核心接口。

落地上可以分成几步：
- 建立内部 CA 或使用企业证书体系，为服务签发客户端证书和服务端证书。
- 网关或服务框架开启客户端证书校验。
- 将证书中的 CN/SAN 映射到服务身份，比如 `user-service`、`privacy-service`。
- 对敏感接口同时做证书身份校验、Token/权限校验、审计日志记录。
- 证书定期轮换，吊销异常证书。

**项目表达：**

我们当时的安全设计不只依赖 TLS，还做了证书固定、敏感字段二次加密、访问控制和审计日志。mTLS 更适合用在服务间高敏链路或设备可信接入场景，它解决的是“调用方身份可信”的问题，是 GDPR 中完整性和保密性要求的一部分技术落地。

## 五、WebSocket

1. WebSocket 和 HTTP 的区别？

**回答：**

WebSocket 是一种在单个 TCP 连接上进行全双工通信的协议，适合服务端主动推送消息。HTTP 是请求-响应模型，通常由客户端发起请求，服务端返回响应。

| 对比 | HTTP | WebSocket |
|------|------|-----------|
| 通信模式 | 请求-响应 | 全双工，双方都可主动发送 |
| 连接 | 可短连接，也可长连接 | 建立后保持长连接 |
| 服务端推送 | 原生不擅长，需要轮询/SSE 等方式 | 原生支持 |
| 协议开销 | 每次请求都有较多 Header | 握手后帧头较小 |
| 适用场景 | 普通 API、网页资源加载 | IM、实时通知、行情、协同编辑、设备状态推送 |

**关系：**
- WebSocket 握手阶段依赖 HTTP。
- 握手成功后通过 `101 Switching Protocols` 升级协议。
- 升级后就不再按 HTTP 请求-响应模式通信，而是 WebSocket 帧通信。

---

2. WebSocket 的握手过程？

**回答：**

WebSocket 通过 HTTP 发起协议升级。

**客户端请求示例：**

```http
GET /ws HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: xxxxx
Sec-WebSocket-Version: 13
```

**服务端响应示例：**

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: yyyyy
```

**关键点：**
- `Upgrade: websocket` 表示希望升级到 WebSocket 协议。
- `Connection: Upgrade` 表示当前连接需要协议升级。
- `Sec-WebSocket-Key` 是客户端生成的随机值。
- 服务端根据固定算法生成 `Sec-WebSocket-Accept`，客户端用它确认服务端支持 WebSocket。
- 返回 101 后，连接进入 WebSocket 帧通信阶段。

**安全版本：**
- `ws://` 类似 HTTP 明文。
- `wss://` 类似 HTTPS，基于 TLS 加密，生产环境应使用 `wss://`。

---

3. WebSocket 的使用场景？

**回答：**

WebSocket 适合实时性强、服务端需要主动推送的场景。

常见场景：
- 即时通讯：单聊、群聊、在线状态。
- 实时通知：订单状态、审批消息、系统告警。
- 金融行情：股票、汇率、数字资产价格推送。
- 协同编辑：多人文档、白板、代码协作。
- 在线游戏：状态同步、房间消息。
- IoT 平台：设备在线状态、设备事件、云端指令下发结果。

**后端设计关注点：**
- 连接管理：用户 ID 和连接 ID 的映射。
- 心跳保活：及时清理死连接。
- 鉴权：握手阶段校验 Token。
- 集群路由：用户连接在不同节点时，需要通过 Redis、MQ、网关等方式转发消息。
- 限流和背压：防止慢客户端拖垮服务端。

**面试加分点：**
- WebSocket 解决的是实时双向通信问题，不适合替代所有 HTTP API。
- 大规模 WebSocket 系统的难点不在协议本身，而在连接管理、状态同步、消息路由和服务端资源控制。

---

4. WebSocket 和长轮询的对比？

**回答：**

| 对比 | 长轮询 | WebSocket |
|------|--------|-----------|
| 原理 | 客户端发起请求，服务端有数据才返回；返回后客户端再发下一次请求 | 建立长连接后，双方可随时发送消息 |
| 实时性 | 较好，但依赖请求周期 | 更好，服务端可立即推送 |
| 连接开销 | 每轮都有 HTTP 请求/响应开销 | 握手后开销小 |
| 兼容性 | 基于 HTTP，兼容性好 | 需要代理、网关支持协议升级 |
| 实现复杂度 | 简单 | 连接管理更复杂 |
| 适用场景 | 低频通知、兼容性要求高 | 高频实时通信、双向通信 |

**长轮询流程：**

```text
客户端发起请求
服务端没有数据就挂起请求
有数据或超时后返回
客户端收到响应后立即发起下一次请求
```

**选择建议：**
- 消息频率低、兼容性优先，可以用长轮询。
- 实时性要求高、双向通信频繁，优先 WebSocket。
- 只需要服务端单向推送，也可以考虑 SSE（Server-Sent Events）。

## 六、DNS 与 CDN

1. DNS 的解析过程？递归查询和迭代查询？

**回答：**

DNS 的作用是把域名解析成 IP 地址。

**一次典型 DNS 解析过程：**

```text
1. 浏览器缓存
2. 操作系统缓存 / hosts 文件
3. 本地 DNS 服务器（运营商 DNS 或公共 DNS）
4. 根 DNS 服务器
5. 顶级域 DNS 服务器，例如 .com
6. 权威 DNS 服务器，例如 example.com 的 DNS
7. 返回 IP，并按 TTL 缓存
```

**递归查询：**
- 客户端把域名交给本地 DNS。
- 本地 DNS 负责一路查到底，最后把结果返回给客户端。
- 对客户端来说，只发一次请求。

**迭代查询：**
- DNS 服务器不会替你查到底，而是告诉你下一步应该问谁。
- 例如根 DNS 告诉你去问 `.com`，`.com` 再告诉你去问 `example.com` 的权威 DNS。

**实际过程：**
- 客户端到本地 DNS 通常是递归查询。
- 本地 DNS 到根、顶级域、权威 DNS 通常是迭代查询。

**常见记录类型：**

| 记录 | 含义 |
|------|------|
| A | 域名解析到 IPv4 |
| AAAA | 域名解析到 IPv6 |
| CNAME | 别名记录 |
| MX | 邮件服务器 |
| TXT | 文本记录，常用于域名验证 |
| NS | 指定权威 DNS 服务器 |

**面试加分点：**
- DNS 结果会被多级缓存，TTL 会影响域名切换生效时间。
- DNS 劫持、污染、解析慢都可能导致服务访问异常。
- 排查时可用 `nslookup`、`dig`、`ping`、`curl -v`。

---

2. CDN 的原理？如何加速内容分发？

**回答：**

CDN（Content Delivery Network）是内容分发网络。核心思想是把静态资源或可缓存内容分发到离用户更近的边缘节点，让用户就近访问，减少源站压力和网络延迟。

**CDN 访问流程：**

```text
用户访问 static.example.com
DNS/CNAME 将请求调度到合适的 CDN 边缘节点
边缘节点检查本地是否有缓存
如果命中，直接返回给用户
如果未命中，回源站拉取资源
边缘节点缓存资源，并返回给用户
```

**CDN 如何加速：**
- 就近访问，减少跨地域网络延迟。
- 边缘缓存，减少源站访问压力。
- 使用更优的网络链路和调度策略。
- 支持压缩、HTTP/2/HTTP/3、TLS 加速。
- 抵御部分流量洪峰和 DDoS 攻击。

**缓存策略：**
- 静态资源使用文件 hash，例如 `app.abc123.js`，可以设置长缓存。
- HTML 通常缓存时间短，避免页面版本不一致。
- 图片、视频、大文件适合 CDN 分发。
- 资源更新时可以通过刷新/预热 CDN 缓存。

**面试加分点：**
- CDN 不只是“缓存”，还包括智能调度、链路优化、边缘安全能力。
- 动态接口也可以通过边缘加速、动态路由优化提升访问速度，但不能像静态资源一样随便缓存。
- 设计缓存时要关注一致性、过期时间、回源压力和缓存击穿。

## 七、网络安全

1. 常见的网络攻击？XSS、CSRF、SQL 注入、DDoS？

**回答：**

| 攻击 | 原理 | 危害 |
|------|------|------|
| XSS | 攻击者把恶意脚本注入页面，用户浏览时执行 | 窃取 Cookie、冒充用户操作、篡改页面 |
| CSRF | 利用用户已登录态，诱导用户浏览器向目标网站发请求 | 在用户不知情的情况下执行转账、修改资料等操作 |
| SQL 注入 | 把恶意 SQL 拼接进查询语句 | 数据泄露、数据篡改、绕过登录 |
| DDoS | 大量请求或流量打满服务/带宽/连接资源 | 服务不可用 |
| 中间人攻击 | 攻击者拦截并篡改通信 | 窃听、篡改数据 |
| 重放攻击 | 捕获合法请求后重复发送 | 重复下单、重复支付、伪造操作 |
| 暴力破解 | 反复尝试密码或验证码 | 账号被盗 |

**区分 XSS 和 CSRF：**
- XSS 是攻击者让脚本在用户浏览器里执行。
- CSRF 是攻击者借用户的登录态发起跨站请求。
- XSS 可能进一步导致 CSRF，因为脚本可以读取页面内容、发请求。

---

2. 如何防范这些攻击？

**回答：**

| 攻击 | 防范措施 |
|------|----------|
| XSS | 输入校验、输出转义、富文本白名单、CSP、Cookie 设置 HttpOnly |
| CSRF | CSRF Token、SameSite Cookie、校验 Referer/Origin、关键操作二次确认 |
| SQL 注入 | 使用预编译参数化查询、ORM 参数绑定、最小权限账号、禁止拼接 SQL |
| DDoS | CDN/WAF、高防 IP、限流、熔断、黑白名单、弹性扩容 |
| 中间人攻击 | HTTPS、证书校验、证书固定、HSTS |
| 重放攻击 | 时间戳、Nonce、签名、请求幂等控制 |
| 暴力破解 | 登录限流、验证码、MFA、密码错误锁定、异常登录告警 |

**后端接口通用安全设计：**
- 所有敏感接口必须 HTTPS。
- 认证使用 Token/JWT/Session，授权使用 RBAC 或权限点。
- 参数校验放在服务端，不能只依赖前端。
- 对写操作做幂等控制，防止重复提交。
- 敏感操作记录审计日志。
- 对公网接口加限流和防刷策略。

**结合项目表达：**

在 GDPR 合规项目里，安全不仅是防攻击，也包括数据传输加密、存储加密、最小权限、审计日志和用户权利响应。网络安全是数据隐私合规的底座。

3. OAuth 2.0 的授权流程？（结合 Alexa/Google Assistant 项目）

**回答：**

OAuth 2.0 是授权框架，核心目标是：用户不需要把账号密码交给第三方应用，而是通过授权码或 Token 授权第三方访问自己的资源。

**核心角色：**

| 角色 | 含义 |
|------|------|
| Resource Owner | 资源拥有者，通常是用户 |
| Client | 第三方客户端，例如 Alexa Skill、Google Assistant Action |
| Authorization Server | 授权服务器，负责登录和发 Token |
| Resource Server | 资源服务器，提供用户数据或设备控制接口 |

**授权码模式流程：**

```text
1. 用户在第三方应用中点击账号绑定。
2. 第三方跳转到我们的授权服务器登录页。
3. 用户登录并同意授权。
4. 授权服务器重定向回第三方 redirect_uri，并带上 authorization code。
5. 第三方后端用 code + client_id + client_secret 向授权服务器换取 access_token。
6. 第三方使用 access_token 调用资源服务器 API。
7. access_token 过期后，可以用 refresh_token 换取新的 access_token。
```

**为什么推荐授权码模式：**
- Token 不直接暴露在浏览器地址栏中。
- 第三方后端用 `client_secret` 换 Token，安全性更高。
- 可以配合 PKCE 防止授权码被截获后滥用。

**结合 Alexa/Google Assistant 项目可以这样回答：**

智能音箱要控制用户的 IoT 设备时，首先要做账号绑定。用户在 Alexa 或 Google Assistant 里绑定我们的账号，跳转到我们的授权页面登录并授权。授权成功后，Alexa/Google 拿到 access token，后续用户说“打开客厅摄像头”或“关闭灯”时，平台会带着 token 调用我们的云端接口。我们服务端校验 token 后，找到对应用户和设备，再下发控制指令。

**落地关注点：**
- `redirect_uri` 必须严格校验，防止授权码泄露。
- `access_token` 有较短有效期。
- `refresh_token` 要安全存储并支持撤销。
- 授权范围 `scope` 要最小化，例如只允许设备控制，不允许访问不必要的数据。
- 账号解绑时要吊销 Token，避免第三方继续访问。

4. JWT 的原理和使用？

**回答：**

JWT（JSON Web Token）是一种自包含 Token，常用于无状态认证和服务间身份传递。

**JWT 结构：**

```text
Header.Payload.Signature
```

| 部分 | 内容 |
|------|------|
| Header | Token 类型、签名算法，如 HS256、RS256 |
| Payload | 业务声明，如 userId、tenantId、scope、exp |
| Signature | 对 Header 和 Payload 的签名，防止篡改 |

**验证过程：**

```text
客户端登录成功后，服务端签发 JWT
客户端后续请求在 Authorization: Bearer <token> 中携带 JWT
服务端校验签名、过期时间、issuer、audience、权限范围
校验通过后，从 payload 中取出用户身份和权限上下文
```

**优点：**
- 无状态，服务端不必存 Session。
- 适合分布式系统、微服务、网关鉴权。
- Payload 自包含，可以减少查库次数。

**缺点和注意事项：**
- JWT 一旦签发，在过期前默认无法主动失效。
- Payload 只是 Base64Url 编码，不是加密，不能放密码、手机号等敏感信息。
- Token 泄露后风险较高，需要 HTTPS、短过期时间、刷新机制。
- 权限变化不能实时生效，除非引入黑名单、版本号或短有效期。

**常见方案：**
- Access Token 短有效期，例如 15 分钟到 2 小时。
- Refresh Token 长有效期，用于续签。
- 服务端保存 Refresh Token 或 Token 版本号，支持主动撤销。
- 网关统一校验 JWT，把用户上下文透传给后端服务。

**面试加分点：**
- JWT 解决的是认证信息传递和无状态校验问题，不等于完整的权限系统。
- 真正的授权仍然需要结合 RBAC、数据权限、租户隔离等业务规则。
- 对安全要求高的系统，可以使用非对称签名 RS256，私钥只在认证服务保存，其他服务用公钥验签。
