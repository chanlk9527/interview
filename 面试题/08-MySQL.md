## 一、存储引擎

1. InnoDB 和 MyISAM 的区别？为什么推荐 InnoDB？

**回答：**

| 对比维度 | InnoDB | MyISAM |
|---------|--------|--------|
| **事务支持** | ✅ 支持 ACID 事务 | ❌ 不支持 |
| **锁粒度** | 行级锁（并发高） | 表级锁（并发低） |
| **外键** | ✅ 支持 | ❌ 不支持 |
| **崩溃恢复** | ✅ 通过 Redo Log 实现 crash-safe | ❌ 崩溃后数据可能丢失 |
| **索引结构** | 聚簇索引（数据和主键索引存在一起） | 非聚簇索引（索引和数据文件分离） |
| **全文索引** | MySQL 5.6+ 支持 | ✅ 原生支持 |
| **COUNT(*)** | 需要遍历（因为 MVCC，不同事务看到的行数不同） | 有专门的计数器，直接返回，速度快 |
| **存储文件** | `.ibd` 文件（数据+索引） | `.MYD`（数据）+ `.MYI`（索引） |

**为什么推荐 InnoDB？**

- MySQL 5.5 之后 InnoDB 成为**默认存储引擎**
- 支持事务和行级锁，适合**高并发 OLTP 场景**
- 支持 crash-safe，通过 Redo Log + 两阶段提交保证数据不丢失
- 支持 MVCC，读写不冲突，并发性能好
- 支持外键约束，保证数据完整性

> **面试加分点：** MyISAM 适合读多写少、不需要事务的场景（如数据仓库、日志表），但在实际生产中几乎都用 InnoDB。

2. InnoDB 的架构？Buffer Pool、Redo Log、Undo Log、Binlog？

**回答：**

InnoDB 架构分为**内存结构**和**磁盘结构**两大部分：

**内存结构：**

- **Buffer Pool（缓冲池）**：InnoDB 最核心的内存区域，默认 128MB（生产环境通常设为物理内存的 60%~80%）。缓存数据页和索引页，减少磁盘 I/O。采用改进的 LRU 算法管理页面淘汰（分为 young 区和 old 区，防止全表扫描污染热数据）。
- **Change Buffer（写缓冲）**：针对非唯一二级索引的写操作，先缓存变更，等到下次读取该页时再合并（merge），减少随机磁盘 I/O。
- **Log Buffer（日志缓冲）**：Redo Log 的内存缓冲区，事务提交时根据 `innodb_flush_log_at_trx_commit` 策略刷盘。
- **Adaptive Hash Index（自适应哈希索引）**：InnoDB 自动为热点页建立哈希索引，加速等值查询。

**磁盘结构（关键日志）：**

| 日志 | 层级 | 作用 | 写入方式 |
|------|------|------|---------|
| **Redo Log** | InnoDB 引擎层 | 保证**持久性（Durability）**，crash-safe 的关键。记录物理修改（"某页某偏移写了什么"） | 循环写，固定大小文件组 |
| **Undo Log** | InnoDB 引擎层 | 保证**原子性（Atomicity）**，支持事务回滚和 MVCC。记录逻辑修改的逆操作 | 存储在 undo tablespace |
| **Binlog** | Server 层 | 用于**主从复制**和**数据恢复**。记录逻辑操作（SQL 语句或行变更） | 追加写，不会覆盖 |

**三者协作关系：**
- 事务执行时：修改 Buffer Pool 中的数据页 → 写 Undo Log（用于回滚）→ 写 Redo Log（prepare 状态）→ 写 Binlog → Redo Log 改为 commit 状态（两阶段提交）
- 崩溃恢复时：通过 Redo Log 恢复已提交但未刷盘的数据，通过 Undo Log 回滚未提交的事务

> **面试加分点：** `innodb_flush_log_at_trx_commit` 设为 1 表示每次事务提交都刷盘（最安全），设为 0 或 2 可以提升性能但有丢数据风险。

3. 一条 SQL 的执行流程？（连接器 → 分析器 → 优化器 → 执行器）

**回答：**

一条 SQL 语句在 MySQL 中的执行流程如下：

```
客户端 → 连接器 → [查询缓存(8.0已移除)] → 分析器 → 优化器 → 执行器 → 存储引擎
```

**1. 连接器（Connector）**
- 负责建立连接、身份认证、权限校验
- 使用 TCP 三次握手建立连接，验证用户名密码
- 查询该用户的权限并缓存，后续操作都基于此权限（即使中途修改权限，已建立的连接不受影响）
- 长连接可能导致内存占用过大（OOM），可以定期断开或使用 `mysql_reset_connection` 重置

**2. 查询缓存（Query Cache）— MySQL 8.0 已移除**
- 以 SQL 语句为 key，查询结果为 value 缓存
- 命中率极低（任何表更新都会清空该表所有缓存），弊大于利，8.0 直接删除

**3. 分析器（Parser）**
- **词法分析**：将 SQL 字符串拆分为 token（识别关键字、表名、列名等）
- **语法分析**：根据语法规则构建语法树（AST），检查 SQL 语法是否正确
- 报错示例：`You have an error in your SQL syntax`

**4. 优化器（Optimizer）**
- 决定使用哪个索引、JOIN 的表连接顺序、是否使用覆盖索引等
- 基于**成本模型（Cost-Based Optimizer）**选择执行计划
- 可以通过 `EXPLAIN` 查看优化器选择的执行计划

**5. 执行器（Executor）**
- 先检查当前用户对该表是否有执行权限
- 调用存储引擎的接口，按照执行计划逐行（或批量）读取/写入数据
- 对于更新语句，还会写 Redo Log 和 Binlog

**更新语句的额外流程：**
```
执行器 → 从存储引擎读取目标行 → 修改数据 → 写 Undo Log 
→ 更新 Buffer Pool 中的数据页 → 写 Redo Log（prepare）
→ 写 Binlog → Redo Log（commit）
```

> **面试加分点：** 面试官可能追问"为什么权限检查在执行器而不是优化器？"——因为有些操作（如触发器、视图）需要在执行阶段才能确定涉及的表和列。

## 二、索引

1. 索引的数据结构？为什么用 B+ 树而不是 B 树、Hash、红黑树？

**回答：**

MySQL InnoDB 的索引默认使用 **B+ 树**数据结构。

**B+ 树的特点：**
- 非叶子节点只存储索引键（不存数据），一个节点能存更多 key，树更矮
- 所有数据都存储在叶子节点，叶子节点之间通过**双向链表**连接
- 树高度通常为 3~4 层，一次查询只需 3~4 次磁盘 I/O

**为什么不用其他数据结构？**

| 数据结构 | 不适合的原因 |
|---------|------------|
| **B 树** | 非叶子节点也存数据，导致每个节点能存的 key 更少，树更高，磁盘 I/O 更多；不支持高效的范围查询（叶子节点没有链表连接） |
| **Hash 索引** | 只支持等值查询（=、IN），不支持范围查询（>、<、BETWEEN）、排序（ORDER BY）和最左前缀匹配；存在哈希冲突问题 |
| **红黑树（二叉树）** | 树的高度太高（二叉结构），数据量大时需要大量磁盘 I/O。例如 100 万数据，红黑树高度约 20，B+ 树高度只有 3 |
| **跳表** | 每层都是链表结构，不利于磁盘的顺序读取和页缓存 |

**关键数字感知：**
- InnoDB 一个页（Page）大小为 **16KB**
- 假设主键为 bigint（8 字节），指针 6 字节，一个非叶子节点可存约 **1170 个 key**
- 三层 B+ 树可存储：1170 × 1170 × 16 ≈ **2000 万+** 行数据
- 这意味着 2000 万行数据只需 **3 次磁盘 I/O** 即可定位

> **面试加分点：** MySQL 的 Memory 引擎默认使用 Hash 索引；InnoDB 还有自适应哈希索引（AHI），会自动为热点数据建立 Hash 索引加速等值查询。
2. 聚簇索引和非聚簇索引（二级索引）的区别？

**回答：**

| 对比维度 | 聚簇索引（Clustered Index） | 非聚簇索引 / 二级索引（Secondary Index） |
|---------|--------------------------|--------------------------------------|
| **定义** | 数据行和索引存储在一起，叶子节点就是数据行 | 叶子节点存储的是索引列的值 + 主键值 |
| **数量** | 每张表只能有**一个** | 可以有**多个** |
| **选择规则** | 优先选主键；无主键则选第一个唯一非空索引；都没有则 InnoDB 自动生成隐藏的 `ROW_ID` 作为聚簇索引 | 用户创建的普通索引、唯一索引、联合索引都是二级索引 |
| **查询效率** | 直接获取整行数据，无需回表 | 如果查询列不在索引中，需要**回表**（用主键值到聚簇索引中再查一次） |
| **数据顺序** | 数据物理存储顺序与索引顺序一致 | 索引顺序与数据物理存储顺序无关 |

**图示理解：**

```
聚簇索引（主键索引）：
[非叶子节点: 主键值] → [叶子节点: 完整的数据行]

二级索引（如 name 索引）：
[非叶子节点: name值] → [叶子节点: name值 + 主键值]
                                    ↓ 回表
                        聚簇索引 → 完整数据行
```

**为什么推荐使用自增主键？**
- 自增主键保证数据按顺序插入，减少页分裂（Page Split）
- 主键越小，二级索引占用空间越小（因为每个二级索引叶子节点都要存主键值）
- UUID 作为主键会导致随机插入、频繁页分裂、索引体积膨胀

> **面试加分点：** MyISAM 的索引都是非聚簇的，主键索引和二级索引的叶子节点存储的都是数据行的物理地址（指针），这是 InnoDB 和 MyISAM 索引结构的本质区别。
3. 回表查询是什么？如何避免回表？覆盖索引？

**回答：**

**回表查询：**
当通过二级索引查询时，如果需要的列不在索引中，就需要拿到主键值后再到聚簇索引中查找完整数据行，这个过程叫**回表**。回表意味着多一次 B+ 树查找，性能开销较大。

```sql
-- 假设有索引 idx_name(name)
SELECT * FROM user WHERE name = '张三';
-- 执行过程：
-- 1. 在 idx_name 索引中找到 name='张三' 的叶子节点，拿到主键 id=5
-- 2. 回表：用 id=5 到聚簇索引中查找完整数据行
```

**覆盖索引（Covering Index）：**
如果查询的所有列都包含在索引中，就不需要回表，直接从索引中返回数据。EXPLAIN 中 Extra 列会显示 **`Using index`**。

```sql
-- 假设有联合索引 idx_name_age(name, age)
SELECT name, age FROM user WHERE name = '张三';
-- 索引中已经包含 name 和 age，无需回表 ✅

SELECT name, age, email FROM user WHERE name = '张三';
-- email 不在索引中，需要回表 ❌
```

**避免回表的方法：**

1. **使用覆盖索引**：将查询需要的列都加入联合索引（但不要过度，索引也有维护成本）
2. **减少 SELECT * 的使用**：只查询需要的列，增加命中覆盖索引的概率
3. **索引下推（ICP）**：MySQL 5.6+ 的优化，在索引遍历过程中先过滤不满足条件的记录，减少回表次数

> **面试加分点：** 覆盖索引是最常用的 SQL 优化手段之一。在设计联合索引时，可以把高频查询的列加入索引来实现覆盖索引，但要权衡索引大小和写入性能。
4. 联合索引的最左前缀原则？

**回答：**

**最左前缀原则：** 联合索引 `(a, b, c)` 在 B+ 树中是按照 a → b → c 的顺序排列的。查询时必须从最左列开始匹配，才能使用索引。

**能命中索引的情况：**

```sql
-- 联合索引 idx_abc(a, b, c)

WHERE a = 1                      -- ✅ 命中 a
WHERE a = 1 AND b = 2            -- ✅ 命中 a, b
WHERE a = 1 AND b = 2 AND c = 3  -- ✅ 命中 a, b, c（全部）
WHERE a = 1 AND c = 3            -- ✅ 命中 a（c 无法使用，因为 b 断了）
WHERE b = 2 AND a = 1            -- ✅ 命中 a, b（优化器会自动调整顺序）
WHERE a > 1 AND b = 2            -- ✅ a 走范围查询，b 在 MySQL 8.0+ 可通过索引跳跃扫描优化
```

**不能命中索引的情况：**

```sql
WHERE b = 2                      -- ❌ 缺少最左列 a
WHERE b = 2 AND c = 3            -- ❌ 缺少最左列 a
WHERE c = 3                      -- ❌ 缺少最左列 a
```

**范围查询对后续列的影响：**

```sql
WHERE a = 1 AND b > 2 AND c = 3
-- a 等值匹配 ✅，b 范围查询 ✅，c 无法使用索引 ❌
-- 因为 b 是范围查询后，c 在 B+ 树中不再有序
-- MySQL 8.0 的索引跳跃扫描（Index Skip Scan）可能优化此场景
```

**联合索引设计原则：**
1. **等值查询的列放前面，范围查询的列放后面**
2. **区分度高的列放前面**（区分度 = 不同值的数量 / 总行数）
3. **高频查询的列优先考虑**
4. 尽量让索引覆盖常用查询，减少回表

> **面试加分点：** MySQL 8.0 引入了 Index Skip Scan 优化，即使缺少最左列，在某些条件下（最左列区分度低）也能使用联合索引。
5. 索引下推（Index Condition Pushdown）是什么？

**回答：**

**索引下推（ICP，Index Condition Pushdown）** 是 MySQL 5.6 引入的优化，核心思想是将部分 WHERE 条件的判断从 Server 层**下推**到存储引擎层，在索引遍历过程中提前过滤不满足条件的记录，减少回表次数。

**举例说明：**

```sql
-- 联合索引 idx_name_age(name, age)
SELECT * FROM user WHERE name LIKE '张%' AND age = 25;
```

**没有 ICP（MySQL 5.6 之前）：**
1. 存储引擎通过索引找到所有 `name LIKE '张%'` 的记录（假设 100 条）
2. 将 100 条记录全部回表，返回完整数据行给 Server 层
3. Server 层再逐行判断 `age = 25`，过滤出最终结果（假设 5 条）
4. 回表了 100 次

**有 ICP（MySQL 5.6+）：**
1. 存储引擎通过索引找到所有 `name LIKE '张%'` 的记录
2. 因为 age 也在联合索引中，存储引擎直接在索引中判断 `age = 25`
3. 只有满足 `age = 25` 的记录才回表（只回表 5 次）
4. 回表次数从 100 次降到 5 次

**EXPLAIN 中的体现：**
- Extra 列显示 **`Using index condition`** 表示使用了索引下推

**ICP 的适用条件：**
- 只适用于二级索引（非聚簇索引）
- 联合索引中，范围查询列之后的列虽然无法用于索引查找，但可以用于 ICP 过滤
- InnoDB 和 MyISAM 都支持

> **面试加分点：** ICP 的本质是"能在索引层面判断的条件就不要等到回表后再判断"，是一种减少回表 I/O 的优化策略。
6. 前缀索引的使用场景？

**回答：**

**前缀索引：** 对字符串列的前 N 个字符建立索引，而不是对整个字符串建索引，以减少索引占用的空间。

```sql
-- 对 email 列的前 6 个字符建索引
ALTER TABLE user ADD INDEX idx_email(email(6));
```

**使用场景：**
- 字符串列很长（如 email、URL、地址），完整建索引太占空间
- 前 N 个字符已经有足够的区分度

**如何确定前缀长度？**

```sql
-- 计算完整列的区分度
SELECT COUNT(DISTINCT email) / COUNT(*) FROM user;
-- 结果：0.98

-- 计算不同前缀长度的区分度
SELECT 
  COUNT(DISTINCT LEFT(email, 4)) / COUNT(*) AS L4,
  COUNT(DISTINCT LEFT(email, 6)) / COUNT(*) AS L6,
  COUNT(DISTINCT LEFT(email, 8)) / COUNT(*) AS L8
FROM user;
-- 选择区分度接近完整列（如达到 95% 以上）的最短前缀
```

**前缀索引的缺点：**
1. **无法使用覆盖索引**：因为索引中只存了前缀，必须回表获取完整值进行精确匹配
2. **无法用于 ORDER BY 和 GROUP BY**：前缀不能保证完整值的排序
3. **无法用于范围查询的优化**

**替代方案：**
- 如果是倒序区分度更高的字段（如身份证号后几位区分度高），可以存储**反转值**再建前缀索引
- 或者新增一个 **hash 列**，存储 `CRC32(email)` 的值，对 hash 列建索引

> **面试加分点：** 前缀索引是空间和查询效率的权衡。实际使用中，email 字段用前缀索引比较常见，但要注意它不能做覆盖索引。
7. 索引失效的常见场景？（函数、隐式转换、OR、LIKE '%xx'、不等于等）

**回答：**

| 场景 | 示例 | 原因 |
|------|------|------|
| **对索引列使用函数** | `WHERE YEAR(create_time) = 2024` | 函数改变了索引列的值，无法匹配 B+ 树。改为：`WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'` |
| **对索引列做运算** | `WHERE id + 1 = 10` | 同上，改为 `WHERE id = 9` |
| **隐式类型转换** | `WHERE phone = 13800138000`（phone 是 varchar） | MySQL 会将 varchar 转为数字比较，相当于对索引列加了函数 `CAST(phone AS signed)` |
| **隐式字符集转换** | 两表 JOIN 时字符集不同 | 会触发 `CONVERT()` 函数，导致索引失效 |
| **LIKE 左模糊** | `WHERE name LIKE '%三'` | B+ 树按前缀排序，左模糊无法利用索引。`LIKE '张%'` 可以走索引 |
| **OR 条件** | `WHERE a = 1 OR b = 2` | 如果 a 和 b 没有各自的索引，则全表扫描。如果都有索引，MySQL 可能用 index_merge 优化 |
| **不等于（!= / <>）** | `WHERE status != 1` | 不等于需要扫描大量数据，优化器可能认为全表扫描更快 |
| **IS NOT NULL** | `WHERE name IS NOT NULL` | 如果大部分数据都不为 NULL，优化器可能选择全表扫描 |
| **NOT IN / NOT EXISTS** | `WHERE id NOT IN (1,2,3)` | 类似不等于，可能导致全表扫描 |
| **联合索引不满足最左前缀** | 索引 `(a,b,c)`，查询 `WHERE b = 1` | 跳过了最左列 a |

**注意：** 索引是否失效最终取决于**优化器的成本估算**。即使写法"正确"，如果优化器认为全表扫描更快（如表数据量很小、查询结果占比很大），也可能不走索引。

**排查方法：**
```sql
EXPLAIN SELECT * FROM user WHERE ...;
-- 关注 type、key、rows、Extra 字段
```

> **面试加分点：** MySQL 8.0 支持函数索引（Functional Index），可以对 `YEAR(create_time)` 这样的表达式直接建索引：`ALTER TABLE t ADD INDEX idx_year ((YEAR(create_time)));`
8. 如何分析和优化慢查询？EXPLAIN 各字段的含义？

**回答：**

**慢查询分析步骤：**

1. **开启慢查询日志**
```sql
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过1秒记录
-- 或在 my.cnf 中配置
```

2. **使用 EXPLAIN 分析执行计划**

| 字段 | 含义 | 关注点 |
|------|------|--------|
| **id** | 查询序号，id 越大越先执行 | 子查询、UNION 时会有多个 id |
| **select_type** | 查询类型 | SIMPLE（简单查询）、PRIMARY（主查询）、SUBQUERY（子查询）、DERIVED（派生表） |
| **table** | 访问的表 | |
| **type** | 访问类型（**最重要**） | 性能从好到差：`system > const > eq_ref > ref > range > index > ALL`。至少要达到 **range** 级别 |
| **possible_keys** | 可能使用的索引 | |
| **key** | 实际使用的索引 | NULL 表示没走索引 |
| **key_len** | 使用的索引长度 | 可以判断联合索引用了几个列 |
| **ref** | 索引的哪一列被使用 | const（常量）、字段名 |
| **rows** | 预估扫描行数 | 越小越好 |
| **filtered** | 经过条件过滤后的行百分比 | 越大越好 |
| **Extra** | 额外信息（**很重要**） | 见下表 |

**Extra 常见值：**

| Extra 值 | 含义 | 好坏 |
|----------|------|------|
| `Using index` | 覆盖索引，无需回表 | ✅ 好 |
| `Using index condition` | 索引下推（ICP） | ✅ 好 |
| `Using where` | Server 层进行了额外过滤 | ⚠️ 一般 |
| `Using filesort` | 需要额外排序（未利用索引排序） | ❌ 需优化 |
| `Using temporary` | 使用了临时表（常见于 GROUP BY、DISTINCT） | ❌ 需优化 |
| `Using join buffer` | JOIN 时没有用到索引，使用了 Block Nested Loop | ❌ 需优化 |

**type 字段详解：**
- **const**：通过主键或唯一索引等值查询，最多返回一行
- **eq_ref**：JOIN 时，被驱动表通过主键/唯一索引关联，每次只匹配一行
- **ref**：通过普通索引等值查询，可能返回多行
- **range**：索引范围扫描（>、<、BETWEEN、IN）
- **index**：全索引扫描（遍历整个索引树）
- **ALL**：全表扫描，最差

**优化手段：**
1. 添加合适的索引（覆盖索引、联合索引）
2. 避免索引失效的写法
3. 优化 SQL 结构（子查询改 JOIN、减少 SELECT *）
4. 对大表进行分页优化（深度分页问题）
5. 使用 `FORCE INDEX` 强制指定索引（慎用）

> **面试加分点：** 除了 EXPLAIN，还可以用 `EXPLAIN ANALYZE`（MySQL 8.0.18+）查看实际执行时间和行数，比 EXPLAIN 的预估值更准确。
9. 什么时候不适合建索引？

**回答：**

1. **数据量很小的表**：全表扫描可能比走索引更快（走索引还有回表开销）
2. **区分度很低的列**：如性别（只有男/女）、状态（0/1），索引过滤效果差，优化器大概率不会选择该索引
3. **频繁更新的列**：每次 INSERT/UPDATE/DELETE 都需要维护索引的 B+ 树结构，写入性能下降
4. **WHERE 条件中不会用到的列**：索引是为查询服务的，不查询的列建索引纯属浪费
5. **有大量重复值的列**：和区分度低类似，索引效果差
6. **表的写操作远多于读操作**：索引加速读但拖慢写，写多读少的场景索引收益低

**区分度的经验值：**
```sql
SELECT COUNT(DISTINCT column) / COUNT(*) FROM table;
-- 一般区分度 > 0.1（10%）才值得建索引
-- 主键的区分度为 1（100%），性别的区分度约 0.5 但实际只有两个值
```

**索引不是越多越好：**
- 每个索引都占用磁盘空间
- 每次写操作都要维护所有相关索引
- 一般建议单表索引不超过 **5~6 个**，联合索引字段不超过 **5 个**

> **面试加分点：** 可以提到"索引的维护成本"——插入数据时可能触发页分裂（Page Split），删除数据时可能导致页合并（Page Merge），这些都是额外开销。

## 三、事务

1. 事务的 ACID 特性？各自是如何保证的？

**回答：**

| 特性 | 含义 | 实现机制 |
|------|------|---------|
| **A - 原子性（Atomicity）** | 事务中的操作要么全部成功，要么全部回滚，不存在中间状态 | **Undo Log**：记录数据修改前的逆操作，回滚时执行逆操作恢复数据 |
| **C - 一致性（Consistency）** | 事务执行前后，数据库从一个一致状态转换到另一个一致状态 | 由 A、I、D 三个特性共同保证，加上数据库的约束（主键、外键、唯一索引、NOT NULL 等） |
| **I - 隔离性（Isolation）** | 并发事务之间互不干扰 | **MVCC**（多版本并发控制）+ **锁机制**（行锁、间隙锁、临键锁） |
| **D - 持久性（Durability）** | 事务一旦提交，数据就永久保存，即使系统崩溃也不丢失 | **Redo Log**：事务提交时先写 Redo Log 并刷盘（WAL 机制），崩溃恢复时重放 Redo Log |

**深入理解一致性：**
- 一致性是事务的**最终目标**，A、I、D 是实现一致性的**手段**
- 例如转账：A 扣 100，B 加 100。原子性保证不会只执行一半，隔离性保证并发转账不会出错，持久性保证提交后不丢失
- 一致性还依赖应用层的逻辑正确性（数据库只能保证技术层面的一致性）

**WAL（Write-Ahead Logging）机制：**
- 先写日志，再写数据页
- Redo Log 是顺序写磁盘（速度快），数据页是随机写磁盘（速度慢）
- 通过 WAL 将随机写转换为顺序写，大幅提升写入性能

> **面试加分点：** 面试官可能追问"一致性和其他三个特性的关系"——一致性是目的，AID 是手段。数据库通过 AID 保证技术层面的一致性，但业务层面的一致性需要应用代码保证。
2. 四种事务隔离级别？各自解决了什么问题？

**回答：**

**三种并发问题：**
- **脏读**：读到其他事务**未提交**的数据（事务回滚后数据就是脏的）
- **不可重复读**：同一事务内两次读取同一行，结果不同（被其他事务 UPDATE 了）
- **幻读**：同一事务内两次范围查询，结果集行数不同（被其他事务 INSERT/DELETE 了）

**四种隔离级别：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|---------|------|-----------|------|---------|
| **读未提交（Read Uncommitted）** | ❌ 可能 | ❌ 可能 | ❌ 可能 | 直接读取最新数据，无任何隔离 |
| **读已提交（Read Committed, RC）** | ✅ 解决 | ❌ 可能 | ❌ 可能 | MVCC：每次 SELECT 都生成新的 ReadView |
| **可重复读（Repeatable Read, RR）** | ✅ 解决 | ✅ 解决 | ⚠️ 大部分解决 | MVCC：事务开始时生成 ReadView，整个事务复用。加上间隙锁防止幻读 |
| **串行化（Serializable）** | ✅ 解决 | ✅ 解决 | ✅ 解决 | 读加共享锁，写加排他锁，完全串行执行 |

**MySQL 默认隔离级别：** 可重复读（RR）

**RC 和 RR 的核心区别（MVCC 角度）：**
- **RC**：每次 SELECT 都创建新的 ReadView → 能看到其他事务已提交的最新数据 → 不可重复读
- **RR**：只在事务第一次 SELECT 时创建 ReadView，后续复用 → 看到的始终是事务开始时的快照 → 可重复读

**为什么互联网公司常用 RC 而不是默认的 RR？**
- RC 下锁的范围更小（没有间隙锁），并发性能更好
- RC 下半一致性读（semi-consistent read）可以减少锁等待
- 配合 Binlog 的 ROW 格式，RC 也能保证主从一致性

> **面试加分点：** Oracle、PostgreSQL 的默认隔离级别是 RC，MySQL 选择 RR 是历史原因（早期 Binlog 的 Statement 格式在 RC 下会导致主从不一致）。
3. MVCC 的实现原理？ReadView、Undo Log 版本链？

**回答：**

**MVCC（Multi-Version Concurrency Control，多版本并发控制）** 是 InnoDB 实现 RC 和 RR 隔离级别的核心机制，让读写操作不冲突，提升并发性能。

**实现依赖三个组件：**

**1. 隐藏字段（每行数据都有）**
- `trx_id`：最近修改该行的事务 ID
- `roll_pointer`：指向 Undo Log 中该行的上一个版本，形成版本链

**2. Undo Log 版本链**
每次修改一行数据，旧版本会写入 Undo Log，通过 roll_pointer 串成链表：

```
当前数据行 (trx_id=103)
    ↓ roll_pointer
旧版本1 (trx_id=102)
    ↓ roll_pointer
旧版本2 (trx_id=101)
    ↓ roll_pointer
最初版本 (trx_id=100)
```

**3. ReadView（读视图）**
ReadView 是事务在执行快照读时生成的"可见性判断依据"，包含四个关键字段：

| 字段 | 含义 |
|------|------|
| `m_ids` | 生成 ReadView 时，当前所有**活跃（未提交）事务**的 ID 列表 |
| `min_trx_id` | m_ids 中的最小值 |
| `max_trx_id` | 生成 ReadView 时，系统应该分配给下一个事务的 ID（即当前最大事务 ID + 1） |
| `creator_trx_id` | 创建该 ReadView 的事务 ID |

**可见性判断规则（对版本链中的每个版本）：**

```
if (trx_id == creator_trx_id)
    → 可见（自己修改的）

if (trx_id < min_trx_id)
    → 可见（该版本在 ReadView 创建前已提交）

if (trx_id >= max_trx_id)
    → 不可见（该版本在 ReadView 创建后才开始）

if (min_trx_id <= trx_id < max_trx_id)
    → 如果 trx_id 在 m_ids 中 → 不可见（该事务还未提交）
    → 如果 trx_id 不在 m_ids 中 → 可见（该事务已提交）
```

如果当前版本不可见，就沿着 roll_pointer 找上一个版本，直到找到可见的版本。

**RC 和 RR 的区别就在于 ReadView 的生成时机：**
- **RC**：每次 SELECT 都生成新的 ReadView
- **RR**：只在第一次 SELECT 时生成 ReadView，后续复用

> **面试加分点：** MVCC 只在快照读（普通 SELECT）时生效。当前读（SELECT ... FOR UPDATE、INSERT、UPDATE、DELETE）不走 MVCC，而是通过加锁实现隔离。
4. 当前读和快照读的区别？

**回答：**

| 对比维度 | 快照读（Snapshot Read） | 当前读（Current Read） |
|---------|----------------------|---------------------|
| **定义** | 读取的是数据的历史版本（MVCC 快照） | 读取的是数据的最新版本 |
| **实现** | 通过 MVCC 的 ReadView + Undo Log 版本链 | 通过加锁（共享锁或排他锁） |
| **SQL 语句** | 普通 `SELECT` | `SELECT ... FOR UPDATE`（加排他锁）<br>`SELECT ... LOCK IN SHARE MODE`（加共享锁）<br>`INSERT`、`UPDATE`、`DELETE` |
| **是否加锁** | 不加锁（读写不冲突） | 加锁（可能阻塞其他事务） |
| **并发性能** | 高（不阻塞写操作） | 较低（可能产生锁等待） |

**示例：**

```sql
-- 事务A
BEGIN;
SELECT * FROM user WHERE id = 1;  -- 快照读，读到 name='张三'

-- 事务B（此时提交了修改）
UPDATE user SET name = '李四' WHERE id = 1;
COMMIT;

-- 事务A（继续）
SELECT * FROM user WHERE id = 1;           -- 快照读，RR下仍然读到 '张三'
SELECT * FROM user WHERE id = 1 FOR UPDATE; -- 当前读，读到最新值 '李四'
```

**为什么 UPDATE/DELETE 是当前读？**
- 因为修改数据必须基于最新版本，否则会覆盖其他事务的修改
- 例如：事务 A 和事务 B 同时修改同一行，如果基于快照读的旧版本修改，就会丢失更新

> **面试加分点：** 在 RR 隔离级别下，快照读通过 MVCC 避免幻读，当前读通过 Next-Key Lock（临键锁）避免幻读。两种读的防幻读机制不同。
5. 可重复读（RR）隔离级别下能完全避免幻读吗？

**回答：**

**不能完全避免。** RR 隔离级别在大多数场景下可以避免幻读，但存在特殊情况会出现幻读。

**RR 防止幻读的两种机制：**
1. **快照读**：通过 MVCC 的 ReadView 机制，始终读取事务开始时的快照，自然不会看到其他事务新插入的行
2. **当前读**：通过 Next-Key Lock（临键锁 = 记录锁 + 间隙锁）锁住范围，阻止其他事务在该范围内插入新行

**会出现幻读的特殊场景：**

```sql
-- 场景：快照读和当前读混用

-- 事务A
BEGIN;
SELECT * FROM user WHERE id > 5;  -- 快照读，返回空（假设没有 id>5 的数据）

-- 事务B
INSERT INTO user(id, name) VALUES(10, '张三');
COMMIT;

-- 事务A
UPDATE user SET name = '李四' WHERE id = 10;  -- 当前读！能更新成功（因为读到了最新数据）
SELECT * FROM user WHERE id > 5;  -- 快照读，但现在能看到 id=10 的行了！
-- 因为事务A自己修改了这行，trx_id 变成了自己的事务ID，MVCC判断为可见
COMMIT;
```

**原因分析：**
- 事务 A 第一次快照读看不到 id=10 的行（MVCC 过滤）
- 事务 A 执行 UPDATE 是当前读，基于最新数据操作，成功修改了 id=10 的行
- 修改后该行的 trx_id 变成了事务 A 自己的 ID
- 事务 A 再次快照读时，MVCC 判断 `trx_id == creator_trx_id`，可见
- 于是出现了"幻读"——前后两次查询结果集不同

**如何完全避免幻读？**
1. 第一次查询就使用当前读：`SELECT ... FOR UPDATE`，加上 Next-Key Lock 锁住范围
2. 使用 Serializable 隔离级别（但性能差）

> **面试加分点：** 这是一个高频追问点。关键在于理解"快照读和当前读混用"会打破 MVCC 的一致性视图，因为当前读会修改行的 trx_id。

## 四、锁机制

1. MySQL 的锁分类？全局锁、表级锁、行级锁？

**回答：**

**按锁粒度分类：**

**1. 全局锁**
- `FLUSH TABLES WITH READ LOCK (FTWRL)`：对整个数据库加读锁，所有表只读
- 使用场景：全库逻辑备份（mysqldump）
- 缺点：加锁期间所有写操作阻塞，业务停摆
- 更好的方案：使用 `mysqldump --single-transaction`，利用 MVCC 的一致性快照备份（仅限 InnoDB）

**2. 表级锁**

| 锁类型 | 说明 |
|--------|------|
| **表锁** | `LOCK TABLES t READ/WRITE`，粒度大，并发低，一般不用 |
| **元数据锁（MDL）** | 自动加锁。对表做 CRUD 时加 MDL 读锁，对表做 DDL（ALTER TABLE）时加 MDL 写锁。防止 DDL 和 DML 并发冲突 |
| **意向锁（IS/IX）** | InnoDB 自动加的表级锁，用于快速判断表中是否有行锁，避免逐行检查。IS（意向共享锁）、IX（意向排他锁） |
| **AUTO-INC 锁** | 自增主键插入时的特殊表锁，保证自增值连续。MySQL 5.1.22+ 引入轻量级互斥锁优化 |

**3. 行级锁（InnoDB 特有）**

| 锁类型 | 说明 |
|--------|------|
| **记录锁（Record Lock）** | 锁住索引上的一条记录 |
| **间隙锁（Gap Lock）** | 锁住索引记录之间的间隙，防止其他事务在间隙中插入新记录（防幻读） |
| **临键锁（Next-Key Lock）** | 记录锁 + 间隙锁，锁住记录本身及其前面的间隙。InnoDB 在 RR 隔离级别下默认使用 |
| **插入意向锁（Insert Intention Lock）** | 一种特殊的间隙锁，INSERT 操作在等待间隙锁释放时加的锁，不同事务插入同一间隙的不同位置不会互相阻塞 |

**按锁模式分类：**
- **共享锁（S 锁 / 读锁）**：`SELECT ... LOCK IN SHARE MODE`（8.0 推荐 `FOR SHARE`），多个事务可以同时持有
- **排他锁（X 锁 / 写锁）**：`SELECT ... FOR UPDATE`、INSERT、UPDATE、DELETE，独占，阻塞其他事务的读写

> **面试加分点：** MDL 锁是一个容易被忽视的坑。如果有长事务持有 MDL 读锁，此时执行 ALTER TABLE 会被阻塞，而后续所有对该表的 CRUD 也会被阻塞（因为 MDL 写锁在等待队列中），可能导致数据库连接打满。
2. InnoDB 的行锁类型？记录锁、间隙锁、临键锁（Next-Key Lock）？

**回答：**

InnoDB 的行锁是**加在索引上**的，不是加在数据行上。如果没有索引，行锁会退化为表锁。

**1. 记录锁（Record Lock）**
- 锁住索引上的**一条具体记录**
- 等值查询命中唯一索引/主键时，Next-Key Lock 退化为记录锁

```sql
-- id 是主键
SELECT * FROM user WHERE id = 5 FOR UPDATE;
-- 只锁住 id=5 这一条记录
```

**2. 间隙锁（Gap Lock）**
- 锁住索引记录之间的**间隙**（开区间），不锁记录本身
- 目的：防止其他事务在间隙中插入新记录（防幻读）
- **只在 RR 隔离级别下存在**，RC 下没有间隙锁

```sql
-- 假设表中 id 有 5, 10, 15
SELECT * FROM user WHERE id = 8 FOR UPDATE;
-- id=8 不存在，加间隙锁 (5, 10)，阻止在此范围内插入
```

**3. 临键锁（Next-Key Lock）**
- **记录锁 + 间隙锁**，锁住记录本身及其前面的间隙（左开右闭区间）
- InnoDB 在 RR 隔离级别下的**默认行锁类型**

```sql
-- 假设表中 id 有 5, 10, 15
SELECT * FROM user WHERE id >= 10 FOR UPDATE;
-- 加锁范围：(5, 10]、(10, 15]、(15, +∞)
```

**Next-Key Lock 的退化规则：**

| 场景 | 退化结果 |
|------|---------|
| 等值查询，命中唯一索引 | 退化为**记录锁**（只锁这一条记录） |
| 等值查询，未命中记录 | 退化为**间隙锁**（锁住该值所在的间隙） |
| 等值查询，非唯一索引，命中记录 | Next-Key Lock + 间隙锁（锁住记录及下一个间隙） |
| 范围查询 | 保持 Next-Key Lock |

**4. 插入意向锁（Insert Intention Lock）**
- INSERT 操作在等待间隙锁释放时加的锁
- 多个事务向同一间隙的不同位置插入数据时，不会互相阻塞
- 提升了并发插入的性能

> **面试加分点：** 行锁是加在索引上的，如果 WHERE 条件没有走索引，InnoDB 会对聚簇索引的所有记录加 Next-Key Lock，效果等同于表锁。这是"行锁升级为表锁"的常见原因。
3. 什么情况下行锁会升级为表锁？

**回答：**

严格来说，InnoDB **没有真正的"锁升级"机制**（不像 SQL Server 那样会自动将行锁升级为表锁）。但以下场景的效果等同于表锁：

**1. WHERE 条件没有使用索引**
```sql
-- name 列没有索引
SELECT * FROM user WHERE name = '张三' FOR UPDATE;
-- InnoDB 会对聚簇索引的所有记录加 Next-Key Lock
-- 效果等同于锁住整张表
```
原因：行锁是加在索引上的，没有索引就只能在聚簇索引上逐行加锁。

**2. 索引失效导致全表扫描**
```sql
-- phone 是 varchar 类型，有索引
SELECT * FROM user WHERE phone = 13800138000 FOR UPDATE;
-- 隐式类型转换导致索引失效，退化为全表扫描，锁住所有行
```

**3. 全表更新/删除**
```sql
UPDATE user SET status = 0;  -- 没有 WHERE 条件，锁住所有行
DELETE FROM user WHERE status = 0;  -- 如果 status 没有索引，锁住所有行
```

**4. 锁住大量行**
虽然不是真正的表锁，但如果 WHERE 条件匹配了大量行，加的行锁数量很多，效果上接近表锁，并发性能急剧下降。

**如何避免：**
1. 确保 WHERE 条件走索引（用 EXPLAIN 确认）
2. 避免隐式类型转换
3. 缩小锁的范围，尽量精确匹配
4. 事务尽量短，减少持锁时间

> **面试加分点：** 注意区分"行锁升级为表锁"和"MDL 锁"。DDL 操作（ALTER TABLE）加的是 MDL 写锁，这是真正的表级锁，和行锁升级是两回事。
4. 死锁的产生条件？如何排查和解决死锁？

**回答：**

**死锁的四个必要条件（操作系统通用）：**
1. **互斥**：资源同一时刻只能被一个事务持有
2. **持有并等待**：事务持有已获得的锁，同时等待获取其他锁
3. **不可剥夺**：已获得的锁不能被强制释放
4. **循环等待**：事务之间形成环形等待链

**MySQL 中死锁的典型场景：**

```sql
-- 场景1：两个事务以不同顺序锁定行
-- 事务A
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;  -- 锁住 id=1
UPDATE account SET balance = balance + 100 WHERE id = 2;  -- 等待 id=2 的锁

-- 事务B
BEGIN;
UPDATE account SET balance = balance - 50 WHERE id = 2;   -- 锁住 id=2
UPDATE account SET balance = balance + 50 WHERE id = 1;   -- 等待 id=1 的锁 → 死锁！
```

```sql
-- 场景2：间隙锁导致的死锁
-- 事务A：SELECT ... WHERE id = 8 FOR UPDATE; (加间隙锁 (5,10))
-- 事务B：SELECT ... WHERE id = 9 FOR UPDATE; (加间隙锁 (5,10))
-- 事务A：INSERT INTO t VALUES(7, ...); (等待事务B的间隙锁)
-- 事务B：INSERT INTO t VALUES(8, ...); (等待事务A的间隙锁) → 死锁！
```

**InnoDB 的死锁处理：**
- **死锁检测**：默认开启（`innodb_deadlock_detect = ON`），使用等待图（wait-for graph）算法检测
- **处理方式**：检测到死锁后，回滚持有最少行级锁的事务（代价最小的事务）
- **超时机制**：`innodb_lock_wait_timeout`（默认 50 秒），等待超时自动回滚

**排查死锁：**
```sql
-- 查看最近一次死锁信息
SHOW ENGINE INNODB STATUS;  -- 关注 LATEST DETECTED DEADLOCK 部分

-- 查看当前锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
SELECT * FROM performance_schema.data_lock_waits;  -- MySQL 8.0+
```

**解决和预防死锁：**
1. **固定加锁顺序**：所有事务按相同顺序访问表和行（如按 id 升序）
2. **缩短事务**：减少持锁时间，降低死锁概率
3. **降低隔离级别**：RC 没有间隙锁，减少间隙锁导致的死锁
4. **合理使用索引**：避免行锁退化为表锁
5. **一次性锁定所需资源**：避免持有并等待

> **面试加分点：** 高并发场景下，死锁检测本身也有性能开销（O(n²)）。如果并发量极高，可以考虑关闭死锁检测（`innodb_deadlock_detect = OFF`），依赖超时机制处理，但需要业务层做好重试。
5. 乐观锁和悲观锁的区别？在 MySQL 中如何实现？

**回答：**

| 对比维度 | 悲观锁 | 乐观锁 |
|---------|--------|--------|
| **思想** | 假设一定会冲突，先加锁再操作 | 假设不会冲突，操作时再检查 |
| **加锁时机** | 读取数据时就加锁 | 不加锁，更新时检查版本 |
| **适用场景** | 写多读少，冲突频繁 | 读多写少，冲突较少 |
| **性能** | 加锁有开销，可能阻塞 | 无锁开销，但冲突时需要重试 |
| **实现方式** | 数据库锁机制 | 应用层实现（版本号/CAS） |

**悲观锁在 MySQL 中的实现：**

```sql
-- 共享锁（读锁）
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;  -- 或 FOR SHARE (8.0+)

-- 排他锁（写锁）
SELECT * FROM user WHERE id = 1 FOR UPDATE;

-- 使用流程
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;  -- 加排他锁，其他事务阻塞
UPDATE account SET balance = balance - 100 WHERE id = 1;
COMMIT;  -- 释放锁
```

**乐观锁在 MySQL 中的实现：**

**方式一：版本号（推荐）**
```sql
-- 表中增加 version 字段
-- 1. 读取数据和版本号
SELECT id, balance, version FROM account WHERE id = 1;
-- 假设读到 balance=1000, version=1

-- 2. 更新时检查版本号
UPDATE account SET balance = 900, version = version + 1 
WHERE id = 1 AND version = 1;
-- 如果 affected_rows = 0，说明被其他事务修改过，需要重试
```

**方式二：CAS（Compare And Swap）**
```sql
-- 用旧值作为条件
UPDATE account SET balance = 900 
WHERE id = 1 AND balance = 1000;
-- 如果 balance 已被修改，条件不匹配，更新失败
```

**方式二的 ABA 问题：**
- 值从 A → B → A，CAS 检查通过但实际已被修改过
- 版本号方案不存在 ABA 问题（版本号只增不减）

**选择建议：**
- 秒杀、库存扣减等高并发场景 → 乐观锁（减少锁等待）
- 转账、订单等强一致性场景 → 悲观锁（保证数据安全）
- 乐观锁冲突率高时，大量重试反而比悲观锁更慢

> **面试加分点：** 乐观锁不是数据库层面的锁，而是应用层的并发控制策略。MySQL 的 MVCC 本质上也是一种乐观并发控制思想。
6. 意向锁的作用？

**回答：**

**意向锁（Intention Lock）** 是 InnoDB 自动加的**表级锁**，用于快速判断表中是否已有行锁，避免加表锁时逐行检查。

**两种意向锁：**
- **意向共享锁（IS）**：事务准备给某行加共享锁（S 锁）前，先在表级加 IS 锁
- **意向排他锁（IX）**：事务准备给某行加排他锁（X 锁）前，先在表级加 IX 锁

**为什么需要意向锁？**

假设没有意向锁：
```
事务A：对 id=5 的行加了排他锁（X 锁）
事务B：想对整张表加表锁（LOCK TABLES t WRITE）
→ 事务B 需要逐行检查是否有行锁 → 效率极低（O(n)）
```

有了意向锁：
```
事务A：对 id=5 加行级 X 锁前，先在表级加 IX 锁
事务B：想加表级 X 锁，发现表上已有 IX 锁 → 直接阻塞，无需逐行检查（O(1)）
```

**意向锁的兼容性矩阵：**

| | IS | IX | S（表锁） | X（表锁） |
|---|---|---|---|---|
| **IS** | ✅ 兼容 | ✅ 兼容 | ✅ 兼容 | ❌ 冲突 |
| **IX** | ✅ 兼容 | ✅ 兼容 | ❌ 冲突 | ❌ 冲突 |
| **S** | ✅ 兼容 | ❌ 冲突 | ✅ 兼容 | ❌ 冲突 |
| **X** | ❌ 冲突 | ❌ 冲突 | ❌ 冲突 | ❌ 冲突 |

**关键点：**
- 意向锁之间**互相兼容**（IS 和 IX 不冲突），因为它们只是"声明意图"
- 意向锁只和**表级锁**冲突，不和行级锁冲突
- 意向锁是 InnoDB **自动加的**，不需要手动操作

> **面试加分点：** 意向锁的本质是一种"协调机制"，让表级锁和行级锁能够共存。它解决的是"如何高效判断表中是否有行锁"的问题。

## 五、日志系统

1. Redo Log 和 Binlog 的区别？各自的作用？

**回答：**

| 对比维度 | Redo Log | Binlog |
|---------|----------|--------|
| **所属层级** | InnoDB 存储引擎层 | MySQL Server 层 |
| **作用** | 保证 crash-safe（崩溃恢复） | 主从复制、数据恢复（备份恢复） |
| **记录内容** | 物理日志：某个数据页的某个偏移量做了什么修改 | 逻辑日志：SQL 语句（Statement）或行变更（Row） |
| **写入方式** | **循环写**：固定大小的文件组，写满后从头覆盖 | **追加写**：文件写满后切换到新文件，不会覆盖 |
| **写入时机** | 事务执行过程中持续写入（WAL） | 事务提交时一次性写入 |
| **文件大小** | 固定大小（如 4 个 1GB 的文件） | 不固定，持续增长 |
| **是否所有引擎都有** | 仅 InnoDB | 所有存储引擎都有 |

**Redo Log 的作用详解：**
- 实现 WAL（Write-Ahead Logging）：先写日志再写数据页
- 将随机写（数据页）转换为顺序写（日志），提升写入性能
- 崩溃恢复时，重放 Redo Log 恢复已提交但未刷盘的数据
- 有两个关键指针：write pos（当前写入位置）和 checkpoint（已刷盘位置），两者之间是可用空间

**Binlog 的作用详解：**
- **主从复制**：主库写 Binlog → 从库的 I/O 线程拉取 → 写入 Relay Log → SQL 线程重放
- **数据恢复**：通过 `mysqlbinlog` 工具回放 Binlog，恢复到指定时间点
- **审计**：记录所有数据变更操作

**为什么需要两个日志？**
- Redo Log 是 InnoDB 引擎特有的，用于 crash-safe
- Binlog 是 Server 层的，所有引擎通用，用于复制和恢复
- 历史原因：MySQL 最初没有 InnoDB，只有 Binlog。InnoDB 作为插件引入后，自带了 Redo Log

> **面试加分点：** 两者必须通过**两阶段提交（2PC）**保持一致，否则崩溃恢复后可能出现主从数据不一致。
2. 两阶段提交（2PC）是什么？为什么需要？

**回答：**

**两阶段提交（2PC）** 是 MySQL 内部为了保证 Redo Log 和 Binlog 数据一致性而采用的机制。

**提交流程：**

```
1. Prepare 阶段：将 Redo Log 写入磁盘，状态标记为 prepare
2. Commit 阶段：
   a. 将 Binlog 写入磁盘
   b. 将 Redo Log 状态改为 commit
```

**为什么需要两阶段提交？**

假设不用 2PC，考虑两种异常场景：

**场景一：先写 Redo Log，后写 Binlog**
```
1. 写 Redo Log 成功 ✅
2. 写 Binlog 前崩溃 ❌
→ 主库通过 Redo Log 恢复了数据（有这条修改）
→ 从库没有 Binlog，不会重放（没有这条修改）
→ 主从数据不一致！
```

**场景二：先写 Binlog，后写 Redo Log**
```
1. 写 Binlog 成功 ✅
2. 写 Redo Log 前崩溃 ❌
→ 主库没有 Redo Log，数据恢复后没有这条修改
→ 从库通过 Binlog 重放了这条修改
→ 主从数据不一致！
```

**有了 2PC 后的崩溃恢复逻辑：**

| 崩溃时机 | Redo Log 状态 | Binlog 状态 | 恢复策略 |
|---------|--------------|------------|---------|
| Prepare 之后、Binlog 之前 | prepare | 无 | **回滚事务** |
| Binlog 之后、Commit 之前 | prepare | 有（完整） | **提交事务**（用 Binlog 中的 XID 匹配 Redo Log） |
| Commit 之后 | commit | 有 | 正常，无需处理 |

**关键机制：XID**
- 每个事务都有一个全局唯一的 XID
- Redo Log 和 Binlog 中都记录了 XID
- 崩溃恢复时，通过 XID 判断 Binlog 中是否有完整的事务记录

**MySQL 8.0 的优化——组提交（Group Commit）：**
- 多个事务的 Redo Log 和 Binlog 合并为一次 fsync，减少磁盘 I/O
- 分为三个阶段：flush → sync → commit，每个阶段都可以批量处理

> **面试加分点：** 两阶段提交的本质是分布式事务的思想——Redo Log 和 Binlog 可以看作两个"参与者"，MySQL 作为"协调者"保证它们的一致性。
3. Undo Log 的作用？和 MVCC 的关系？

**回答：**

**Undo Log 的两大作用：**

**1. 事务回滚（保证原子性）**
- Undo Log 记录的是数据修改前的逆操作
- INSERT → 记录 DELETE
- DELETE → 记录 INSERT
- UPDATE → 记录反向 UPDATE（旧值）
- 事务回滚时，执行 Undo Log 中的逆操作，恢复数据到修改前的状态

**2. MVCC 多版本读（保证隔离性）**
- 每次修改数据行时，旧版本写入 Undo Log
- 通过 `roll_pointer` 将同一行的多个版本串成**版本链**
- 快照读时，根据 ReadView 的可见性规则，沿版本链找到对当前事务可见的版本

**Undo Log 与 MVCC 的关系：**

```
数据行 (trx_id=103, roll_pointer → )
    ↓
Undo Log 版本1 (trx_id=102, roll_pointer → )
    ↓
Undo Log 版本2 (trx_id=101, roll_pointer → )
    ↓
Undo Log 版本3 (trx_id=100, roll_pointer = NULL)
```

- 事务执行快照读时，从最新版本开始，根据 ReadView 判断可见性
- 如果不可见，沿 roll_pointer 找上一个版本，直到找到可见版本
- Undo Log 就是 MVCC 版本链的**物理存储**

**Undo Log 的类型：**
- **insert undo log**：INSERT 操作产生，事务提交后可以立即删除（因为 INSERT 的数据对其他事务不可见，不需要 MVCC）
- **update undo log**：UPDATE/DELETE 操作产生，需要保留到没有事务需要读取该版本时才能删除

**Undo Log 的清理：**
- 由 **purge 线程**负责清理不再需要的 Undo Log
- 如果有长事务一直不提交，Undo Log 无法清理，会导致 undo tablespace 膨胀
- 这也是为什么要避免长事务的原因之一

> **面试加分点：** Undo Log 本身的修改也会产生 Redo Log（保证 Undo Log 的持久性）。三种日志的关系：Redo Log 保护 Undo Log，Undo Log 支撑 MVCC 和回滚。
4. Binlog 的三种格式？Statement、Row、Mixed？

**回答：**

| 格式 | 记录内容 | 优点 | 缺点 |
|------|---------|------|------|
| **Statement** | 记录原始 SQL 语句 | 日志量小，节省磁盘和网络带宽 | 某些函数（NOW()、RAND()、UUID()）在从库重放时结果不同，导致主从不一致 |
| **Row** | 记录每一行数据的具体变更（修改前后的值） | 不会出现主从不一致问题，最安全 | 日志量大（批量操作时每行都记录），占用更多磁盘和带宽 |
| **Mixed** | 默认用 Statement，遇到可能不一致的语句自动切换为 Row | 兼顾日志量和安全性 | 判断逻辑复杂，某些边界情况可能判断不准 |

**Statement 格式不安全的例子：**

```sql
-- 主库执行
DELETE FROM user WHERE create_time < '2024-01-01' LIMIT 100;
-- 如果主库和从库选择了不同的索引，删除的 100 行可能不同！

-- 使用了不确定函数
INSERT INTO log(time) VALUES(NOW());
-- 主库和从库执行时间不同，NOW() 返回值不同
```

**推荐配置：**
- MySQL 5.7.7+ 默认使用 **Row** 格式
- 生产环境推荐使用 **Row** 格式，保证主从一致性
- 配合 `binlog_row_image = MINIMAL` 可以减少 Row 格式的日志量（只记录修改的列，而非整行）

**查看和设置：**
```sql
SHOW VARIABLES LIKE 'binlog_format';
SET GLOBAL binlog_format = 'ROW';
```

**Row 格式的额外好处：**
- 可以用于数据恢复：通过 `mysqlbinlog --base64-output=decode-rows -v` 查看具体变更
- 可以实现"闪回"：将 Row 格式的 INSERT 反转为 DELETE，UPDATE 的新旧值互换

> **面试加分点：** 使用 RC 隔离级别时必须配合 Row 格式的 Binlog，否则 Statement 格式在 RC 下可能导致主从不一致（这也是 MySQL 默认用 RR 的历史原因之一）。

## 六、SQL 优化

1. 大表查询优化的常见手段？

**回答：**

**1. 索引优化（最重要）**
- 为 WHERE、JOIN、ORDER BY、GROUP BY 涉及的列建立合适的索引
- 使用联合索引，遵循最左前缀原则
- 利用覆盖索引避免回表
- 用 EXPLAIN 分析执行计划，确保走索引

**2. SQL 语句优化**
- 避免 `SELECT *`，只查需要的列
- 避免索引失效的写法（函数、隐式转换、左模糊等）
- 子查询改为 JOIN（减少临时表）
- 用 `EXISTS` 替代 `IN`（当子查询结果集大时）
- 避免在 WHERE 中对索引列做运算

**3. 分页优化**
- 深度分页使用延迟关联或游标分页（详见下一题）

**4. 数据量层面**
- **分库分表**：水平拆分，将数据分散到多个表/库
- **归档历史数据**：将冷数据迁移到归档表，保持主表数据量可控
- **分区表**：按时间、范围等分区，查询时只扫描相关分区

**5. 缓存层面**
- 热点数据放 Redis 缓存，减少数据库查询
- 使用本地缓存（Caffeine/Guava）缓存高频不变数据

**6. 架构层面**
- **读写分离**：写操作走主库，读操作走从库，分散压力
- **使用搜索引擎**：复杂查询（全文搜索、多条件筛选）交给 Elasticsearch
- **预计算/物化视图**：提前计算聚合结果，查询时直接读取

**7. 数据库配置优化**
- 调大 Buffer Pool（`innodb_buffer_pool_size`，建议物理内存的 60%~80%）
- 调整 Redo Log 文件大小，减少 checkpoint 频率
- 合理设置连接池大小

> **面试加分点：** 优化的优先级一般是：SQL 和索引优化 > 缓存 > 读写分离 > 分库分表。分库分表是最后的手段，引入的复杂度最高。
2. 深度分页（LIMIT 100000, 10）的优化方案？

**回答：**

**问题：** `SELECT * FROM user ORDER BY id LIMIT 100000, 10` 需要扫描前 100010 行，丢弃前 100000 行，只返回 10 行。偏移量越大，性能越差。

**方案一：延迟关联（推荐）**

先通过子查询在索引上定位 ID，再用 ID 回表取数据：

```sql
-- 优化前（慢）
SELECT * FROM user ORDER BY id LIMIT 100000, 10;

-- 优化后（快）
SELECT u.* FROM user u
INNER JOIN (
    SELECT id FROM user ORDER BY id LIMIT 100000, 10
) tmp ON u.id = tmp.id;
```

原理：子查询 `SELECT id FROM user ORDER BY id LIMIT 100000, 10` 走覆盖索引（只读 id），不需要回表，速度快。外层查询只需要对 10 个 id 回表。

**方案二：游标分页 / 书签法（最推荐）**

记住上一页的最后一条记录的 ID，下一页从该 ID 开始查：

```sql
-- 第一页
SELECT * FROM user WHERE id > 0 ORDER BY id LIMIT 10;
-- 假设最后一条 id = 10

-- 第二页
SELECT * FROM user WHERE id > 10 ORDER BY id LIMIT 10;
-- 假设最后一条 id = 20

-- 第 N 页
SELECT * FROM user WHERE id > {last_id} ORDER BY id LIMIT 10;
```

优点：无论翻到多深，性能都一样（直接从索引定位，不需要扫描前面的行）。
缺点：不支持跳页（只能上一页/下一页），适合"加载更多"的场景。

**方案三：使用搜索引擎**
- 将数据同步到 Elasticsearch，利用 ES 的 `search_after` 实现深度分页
- 适合复杂查询 + 深度分页的场景

**方案四：业务限制**
- 限制用户最多翻到第 100 页（Google 搜索也是这么做的）
- 提供筛选条件缩小结果集，而不是无限翻页

**各方案对比：**

| 方案 | 性能 | 是否支持跳页 | 适用场景 |
|------|------|------------|---------|
| 延迟关联 | 较好 | ✅ 支持 | 后台管理系统 |
| 游标分页 | 最好 | ❌ 不支持 | APP 瀑布流、加载更多 |
| ES | 好 | 部分支持 | 搜索场景 |
| 业务限制 | - | - | 通用 |

> **面试加分点：** 游标分页要求排序字段必须有索引且值唯一（或组合唯一），否则可能漏数据或重复数据。
3. COUNT(*) 和 COUNT(1) 和 COUNT(列名) 的区别？

**回答：**

| 写法 | 含义 | 是否统计 NULL | 性能 |
|------|------|-------------|------|
| `COUNT(*)` | 统计总行数 | ✅ 包含 NULL 行 | InnoDB 会选择最小的索引树遍历，**最优** |
| `COUNT(1)` | 统计总行数（对每行返回常量 1 再计数） | ✅ 包含 NULL 行 | 和 COUNT(*) **性能相同** |
| `COUNT(列名)` | 统计该列**非 NULL** 的行数 | ❌ 不包含 NULL | 需要判断是否为 NULL，可能需要回表 |
| `COUNT(DISTINCT 列名)` | 统计该列不重复且非 NULL 的值的个数 | ❌ 不包含 NULL | 需要去重，性能最差 |

**COUNT(*) 的优化：**
- InnoDB 中 COUNT(*) 需要遍历索引树逐行计数（因为 MVCC，不同事务看到的行数可能不同，无法像 MyISAM 那样直接返回）
- InnoDB 会自动选择**最小的索引树**来遍历（二级索引通常比聚簇索引小，因为不包含完整数据行）
- 所以 `COUNT(*)` ≈ `COUNT(1)` > `COUNT(主键)` > `COUNT(普通列)`

**COUNT(*) 和 COUNT(1) 真的一样吗？**
- MySQL 官方文档明确说明：`COUNT(*)` 和 `COUNT(1)` 没有性能差异
- MySQL 优化器会将 `COUNT(1)` 优化为和 `COUNT(*)` 相同的执行方式
- 推荐使用 `COUNT(*)`，语义更清晰

**大表 COUNT 的优化方案：**

1. **使用近似值**：`EXPLAIN SELECT COUNT(*) FROM user;` 中的 rows 字段是估算值，适合不需要精确值的场景
2. **缓存计数**：用 Redis 维护计数器，增删时同步更新
3. **单独维护计数表**：在一张统计表中维护行数，通过事务保证一致性
4. **使用 `information_schema`**：`SELECT TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_NAME = 'user';`（不精确，误差可达 40%~50%）

> **面试加分点：** MyISAM 有专门的行数计数器，`COUNT(*)` 不带 WHERE 条件时直接返回，O(1) 复杂度。InnoDB 因为 MVCC 做不到这一点。
4. JOIN 的原理？Nested Loop Join、Hash Join？

**回答：**

**1. Simple Nested Loop Join（简单嵌套循环）**
- 最朴素的算法：外层表每一行都去内层表全表扫描匹配
- 时间复杂度：O(M × N)，性能极差
- MySQL 实际不会使用这种算法

**2. Index Nested Loop Join（索引嵌套循环，NLJ）**
- 外层表（驱动表）每一行，通过**索引**去内层表（被驱动表）查找匹配行
- 时间复杂度：O(M × logN)（索引查找是 logN）
- **要求被驱动表的 JOIN 列上有索引**

```sql
-- user 表 10 行（驱动表），order 表 10000 行（被驱动表）
-- order.user_id 上有索引
SELECT * FROM user u JOIN order o ON u.id = o.user_id;
-- 扫描 user 10 行，每行通过索引查 order → 10 × 3次IO ≈ 30 次
```

**3. Block Nested Loop Join（块嵌套循环，BNL）— MySQL 8.0.18 前**
- 当被驱动表没有索引时使用
- 将驱动表的数据批量加载到 **Join Buffer**（内存），然后扫描被驱动表时一次性与 Buffer 中的多行比较
- 减少了被驱动表的扫描次数
- Join Buffer 大小由 `join_buffer_size` 控制（默认 256KB）

**4. Hash Join — MySQL 8.0.18+**
- 替代了 BNL，用于**没有索引**的等值 JOIN
- 原理：
  1. Build 阶段：将较小的表（驱动表）构建为内存中的 Hash 表
  2. Probe 阶段：扫描较大的表，对每行计算 Hash 值，在 Hash 表中查找匹配
- 时间复杂度：O(M + N)，比 BNL 快很多
- 如果内存不够，会使用磁盘临时文件（Grace Hash Join）

**驱动表的选择：**
- MySQL 优化器会自动选择**小表作为驱动表**（减少外层循环次数）
- 可以用 `STRAIGHT_JOIN` 强制指定驱动表顺序（慎用）
- "小表"不是指行数少，而是指参与 JOIN 的数据量小（经过 WHERE 过滤后的结果集）

**优化建议：**
1. **被驱动表的 JOIN 列一定要有索引**（走 NLJ 而不是 Hash Join）
2. 小表驱动大表
3. 避免过多表 JOIN（一般不超过 3~4 张表）

> **面试加分点：** MySQL 8.0.20 彻底移除了 BNL，全部用 Hash Join 替代。EXPLAIN 中 Extra 显示 `Using join buffer (hash join)` 表示使用了 Hash Join。
5. 子查询 vs JOIN 的性能对比？

**回答：**

**一般结论：JOIN 性能优于子查询，但不绝对。**

**子查询的问题：**

```sql
-- 相关子查询（Correlated Subquery）— 性能差
SELECT * FROM user u 
WHERE u.id IN (SELECT user_id FROM order WHERE amount > 100);
-- 对于 user 表的每一行，都要执行一次子查询（早期 MySQL 版本）
```

- 早期 MySQL（5.5 及之前）对 IN 子查询的优化很差，会将外层表的每一行代入子查询执行
- 可能产生临时表（`Using temporary`），无法使用索引
- 相关子查询（子查询引用了外层表的列）每次都要重新执行

**JOIN 的优势：**

```sql
-- 改写为 JOIN
SELECT DISTINCT u.* FROM user u 
INNER JOIN order o ON u.id = o.user_id 
WHERE o.amount > 100;
-- 可以利用索引，优化器可以选择最优的驱动表和连接方式
```

- 优化器可以选择最优的执行计划（驱动表、索引等）
- 可以利用 Index NLJ 或 Hash Join
- 不需要创建临时表

**但 MySQL 5.6+ 做了大量优化：**

| 优化 | 说明 |
|------|------|
| **半连接优化（Semi-Join）** | 将 IN 子查询自动转换为 JOIN |
| **物化（Materialization）** | 将子查询结果物化为临时表，建立索引后再 JOIN |
| **EXISTS 转换** | 将 IN 子查询转换为 EXISTS |
| **首次匹配（FirstMatch）** | 找到第一个匹配就停止 |

**所以在 MySQL 5.6+ 中，很多场景下子查询和 JOIN 性能差距不大。**

**仍然推荐 JOIN 的原因：**
1. 语义更清晰，可读性更好
2. 优化器对 JOIN 的优化更成熟
3. 避免依赖优化器的自动转换（不同版本行为可能不同）

**子查询更合适的场景：**
- 只需要判断存在性时，`EXISTS` 比 JOIN 更高效（找到一条就返回）
- 子查询结果集很小且可以被缓存时

> **面试加分点：** 可以用 `EXPLAIN` 查看优化器是否将子查询转换为了 JOIN（看 select_type 是否为 SIMPLE 而非 SUBQUERY）。
6. ORDER BY 的优化？filesort 和 index sort？

**回答：**

MySQL 的 ORDER BY 有两种排序方式：

**1. Index Sort（索引排序）— 最优**
- 如果 ORDER BY 的列有索引，且查询可以利用该索引的有序性，直接按索引顺序读取，无需额外排序
- EXPLAIN 中 Extra **不会**出现 `Using filesort`

```sql
-- id 是主键（天然有序）
SELECT * FROM user ORDER BY id LIMIT 10;  -- 直接走索引，无需排序

-- 联合索引 idx_age_name(age, name)
SELECT age, name FROM user WHERE age = 25 ORDER BY name;  
-- age 等值过滤后，name 在索引中已有序，覆盖索引 + 无需排序 ✅
```

**2. Filesort（文件排序）— 需要优化**
- 无法利用索引排序时，MySQL 需要在内存（sort_buffer）或磁盘上进行排序
- EXPLAIN 中 Extra 显示 `Using filesort`

Filesort 有两种算法：

| 算法 | 过程 | 适用场景 |
|------|------|---------|
| **全字段排序（单路排序）** | 将查询需要的所有字段放入 sort_buffer 排序，排序后直接返回 | 字段总长度 < `max_length_for_sort_data`（默认 4096 字节） |
| **rowid 排序（双路排序）** | 只将排序字段 + 主键放入 sort_buffer 排序，排序后再回表取其他字段 | 字段总长度 > `max_length_for_sort_data` |

- sort_buffer 大小由 `sort_buffer_size` 控制（默认 256KB）
- 如果数据量超过 sort_buffer，会使用磁盘临时文件（归并排序），性能急剧下降

**优化方法：**

1. **利用索引避免 filesort**
```sql
-- 为 ORDER BY 的列建索引
-- WHERE 条件列和 ORDER BY 列建联合索引
ALTER TABLE user ADD INDEX idx_age_create(age, create_time);
SELECT * FROM user WHERE age = 25 ORDER BY create_time;  -- 无需 filesort
```

2. **减少排序数据量**
- 避免 `SELECT *`，只查需要的列（减少 sort_buffer 中的数据量）
- 加 LIMIT 限制返回行数

3. **调整参数**
- 增大 `sort_buffer_size`（但不宜过大，每个连接独占）
- 调整 `max_length_for_sort_data` 控制排序算法选择

**ORDER BY 使用索引的条件：**
- ORDER BY 列是索引的最左前缀
- WHERE 条件使用了索引的前缀列（等值），ORDER BY 使用了后续列
- 不能混合 ASC 和 DESC（MySQL 8.0 支持降序索引可以解决）
- 不能有非索引列参与排序

> **面试加分点：** MySQL 8.0 支持**降序索引**（`CREATE INDEX idx ON t(a ASC, b DESC)`），解决了之前 `ORDER BY a ASC, b DESC` 无法走索引的问题。

## 七、分库分表

1. 什么时候需要分库分表？垂直拆分 vs 水平拆分？

**回答：**

**什么时候需要分库分表？**

| 信号 | 说明 |
|------|------|
| 单表数据量超过 **2000 万~5000 万** 行 | B+ 树层级增加，查询性能下降 |
| 单库数据量超过 **500GB~1TB** | 磁盘 I/O 成为瓶颈 |
| 数据库 QPS 超过 **5000~10000** | 单库连接数和 CPU 成为瓶颈 |
| SQL 优化、索引优化、缓存都已做到位 | 分库分表是最后的手段 |

**垂直拆分：**

**垂直分库：** 按业务模块拆分，不同业务的表放到不同的数据库
```
原来：一个库包含 user、order、product、payment 表
拆分后：
  - user_db: user、user_address
  - order_db: order、order_item
  - product_db: product、category
  - payment_db: payment、refund
```
优点：业务解耦，不同库可以独立扩展
缺点：跨库 JOIN 困难，分布式事务复杂

**垂直分表：** 将一张宽表按列拆分，常用列和不常用列分开
```
原来：user(id, name, age, avatar, bio, ...)
拆分后：
  - user(id, name, age)          -- 热数据，高频访问
  - user_detail(user_id, avatar, bio, ...)  -- 冷数据，低频访问
```
优点：减少单行数据量，一页能存更多行，提升缓存命中率

**水平拆分：**

**水平分表：** 将同一张表的数据按某个规则分散到多张表
```
user → user_0, user_1, user_2, user_3
-- 按 id % 4 分片
```

**水平分库：** 将数据分散到多个数据库实例
```
db_0.user, db_1.user, db_2.user, db_3.user
-- 按 id % 4 分到不同库
```

**垂直 vs 水平对比：**

| 维度 | 垂直拆分 | 水平拆分 |
|------|---------|---------|
| 拆分方式 | 按业务/列拆分 | 按行拆分 |
| 解决的问题 | 业务耦合、单行数据过大 | 单表数据量过大 |
| 复杂度 | 较低 | 较高（分片路由、跨片查询） |
| 适用阶段 | 优先考虑 | 垂直拆分后仍不够时 |

> **面试加分点：** 分库分表的优先级：SQL 优化 → 缓存 → 读写分离 → 垂直拆分 → 水平拆分。能不分就不分，分了之后的运维复杂度会大幅增加。
2. 分片键如何选择？

**回答：**

分片键（Sharding Key）是决定数据分布到哪个分片的字段，选择好坏直接影响系统性能。

**选择原则：**

| 原则 | 说明 |
|------|------|
| **查询频率高** | 大部分查询都会带上该字段作为条件，避免跨分片查询 |
| **数据分布均匀** | 避免数据倾斜（某些分片数据量远大于其他分片） |
| **尽量避免跨片操作** | 同一业务的相关数据尽量在同一分片 |

**常见分片策略：**

**1. Hash 取模**
```
分片号 = hash(sharding_key) % 分片数
```
- 优点：数据分布均匀
- 缺点：扩容时需要数据迁移（可以用一致性哈希缓解）

**2. 范围分片**
```
id 1~1000万 → 分片0
id 1000万~2000万 → 分片1
```
- 优点：扩容方便（新增分片即可），范围查询友好
- 缺点：可能数据倾斜（新数据集中在最新分片，形成热点）

**3. 按时间分片**
```
2024年数据 → 分片0
2025年数据 → 分片1
```
- 适合日志、订单等有时间属性的数据
- 缺点：最新分片是热点

**实际案例：**

| 业务 | 推荐分片键 | 原因 |
|------|-----------|------|
| 用户表 | user_id | 大部分查询按用户维度 |
| 订单表 | user_id（而非 order_id） | 查询场景多为"某用户的订单"，同一用户的订单在同一分片 |
| 商品表 | product_id | 按商品维度查询 |
| 日志表 | 时间 + user_id | 按时间范围 + 用户维度查询 |

**分片键选择的常见坑：**
1. **选了低区分度的字段**（如 status），导致数据严重倾斜
2. **选了非查询条件字段**，导致大量查询需要广播到所有分片
3. **没有考虑 JOIN 场景**，关联表的分片键不一致导致跨片 JOIN

> **面试加分点：** 如果业务有多个查询维度（如订单既按 user_id 查，也按 order_id 查），可以建立**映射表**（order_id → user_id 的索引表）或使用**基因法**（将 user_id 的分片信息嵌入 order_id 中）。
3. 分库分表后的问题？（跨库 JOIN、分布式事务、全局 ID、数据迁移）

**回答：**

**1. 跨库 JOIN**

问题：数据分散在不同库，无法直接 JOIN。

解决方案：
- **冗余字段**：将常用的关联字段冗余到当前表，避免 JOIN
- **应用层组装**：分别查询两个库，在应用层用代码做关联
- **广播表/全局表**：将小表（如字典表、配置表）在每个分片都存一份
- **绑定表**：相关联的表使用相同的分片键和分片规则，保证关联数据在同一分片

**2. 分布式事务**

问题：一个业务操作涉及多个库，如何保证事务一致性？

解决方案：
- **XA 事务（强一致）**：两阶段提交，性能差，一般不用
- **TCC（Try-Confirm-Cancel）**：业务层实现补偿，侵入性强
- **Saga 模式**：长事务拆分为多个本地事务 + 补偿操作
- **本地消息表 + MQ（最终一致性）**：最常用，将分布式事务转化为本地事务 + 异步消息
- **Seata**：阿里开源的分布式事务框架，支持 AT、TCC、Saga 等模式

**3. 全局唯一 ID**

问题：分库分表后，各表的自增 ID 会重复。

解决方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **UUID** | 简单，无需中心化 | 无序（导致页分裂）、太长（36 字符） |
| **数据库自增（号段模式）** | 有序，简单 | 依赖数据库，有单点风险 |
| **Redis INCR** | 有序，性能高 | 依赖 Redis，持久化风险 |
| **雪花算法（Snowflake）** | 有序、高性能、去中心化 | 依赖时钟，时钟回拨会重复 |
| **Leaf（美团）** | 号段模式 + 雪花算法，生产级方案 | 需要额外部署 |

**雪花算法结构（64 位）：**
```
0 | 41位时间戳 | 10位机器ID | 12位序列号
  | (69年)     | (1024台)   | (4096/ms)
```

**4. 数据迁移（扩容）**

问题：分片数量变化时，数据需要重新分布。

解决方案：
- **停机迁移**：最简单，但业务需要停服
- **双写迁移**：新老库同时写入，后台迁移历史数据，迁移完成后切换读
- **一致性哈希**：扩容时只需迁移部分数据（相邻节点之间）
- **预分片**：一开始就分足够多的逻辑分片（如 1024 个），物理上先合并，扩容时只需迁移逻辑分片

**5. 排序、聚合、分页**

问题：跨分片的 ORDER BY、GROUP BY、COUNT 需要在应用层或中间件层做归并。

解决方案：
- 中间件（ShardingSphere）会将 SQL 发送到所有分片，收集结果后在内存中归并排序
- 深度分页问题更严重（每个分片都要返回大量数据）

> **面试加分点：** 分库分表引入的复杂度非常高，所以业界有"能不分就不分"的共识。很多场景下，通过 TiDB 等分布式数据库可以避免手动分库分表。
4. 常用的分库分表中间件？ShardingSphere、MyCat？

**回答：**

**两种架构模式：**

| 模式 | 代表 | 原理 | 部署方式 |
|------|------|------|---------|
| **Client 端（JDBC 代理）** | ShardingSphere-JDBC | 以 JAR 包形式嵌入应用，在 JDBC 层拦截 SQL 进行路由 | 无需额外部署 |
| **Proxy 端（数据库代理）** | ShardingSphere-Proxy、MyCat | 独立部署的代理服务，应用连接代理，代理转发到真实数据库 | 需要额外部署代理服务 |

**ShardingSphere（推荐）：**

Apache 顶级项目，目前最主流的分库分表方案。

- **ShardingSphere-JDBC**：
  - 轻量级 Java 框架，以 JAR 包引入
  - 在应用层完成 SQL 解析、路由、改写、执行、结果归并
  - 无网络开销，性能最好
  - 只支持 Java 应用

- **ShardingSphere-Proxy**：
  - 独立部署的数据库代理
  - 对应用透明，支持任何语言
  - 有额外的网络开销

- 核心功能：分库分表、读写分离、分布式事务、数据加密、影子库

**MyCat：**
- 基于 Proxy 模式的数据库中间件
- 对应用完全透明（应用以为连的是 MySQL）
- 社区活跃度下降，新项目不太推荐

**对比：**

| 维度 | ShardingSphere-JDBC | ShardingSphere-Proxy | MyCat |
|------|-------------------|---------------------|-------|
| 架构 | Client 端 | Proxy 端 | Proxy 端 |
| 性能 | ⭐⭐⭐ 最好 | ⭐⭐ 有网络开销 | ⭐⭐ 有网络开销 |
| 语言限制 | 仅 Java | 无限制 | 无限制 |
| 运维复杂度 | 低（无需额外部署） | 高（需要部署和维护代理） | 高 |
| 社区活跃度 | ⭐⭐⭐ 活跃 | ⭐⭐⭐ 活跃 | ⭐ 下降 |
| 功能丰富度 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |

**新趋势——分布式数据库：**
- **TiDB**：兼容 MySQL 协议，自动分片，无需手动分库分表
- **OceanBase**：蚂蚁金服开源，兼容 MySQL/Oracle
- **CockroachDB**：兼容 PostgreSQL 协议
- 这些分布式数据库从根本上解决了分库分表的问题

> **面试加分点：** 如果是新项目且数据量预期很大，可以直接考虑 TiDB 等分布式数据库，避免分库分表带来的复杂度。如果是存量系统改造，ShardingSphere-JDBC 是最平滑的方案。
5. 读写分离的实现方案？主从复制的原理？主从延迟怎么处理？

**回答：**

**主从复制的原理：**

```
主库（Master）                          从库（Slave）
    │                                      │
    ├─ 1. 写操作写入 Binlog                │
    │                                      │
    ├─ 2. Binlog Dump 线程 ──────────────→ I/O 线程
    │     将 Binlog 发送给从库              │ 接收 Binlog，写入 Relay Log
    │                                      │
    │                                      ├─ 3. SQL 线程
    │                                      │     读取 Relay Log，重放 SQL
    │                                      │     将数据写入从库
```

**三个关键线程：**
1. **Binlog Dump 线程**（主库）：读取主库 Binlog，发送给从库
2. **I/O 线程**（从库）：接收主库的 Binlog，写入本地 Relay Log
3. **SQL 线程**（从库）：读取 Relay Log，重放 SQL 语句，写入从库数据

**复制模式：**

| 模式 | 说明 | 数据安全 | 性能 |
|------|------|---------|------|
| **异步复制** | 主库写完 Binlog 就返回成功，不等从库确认 | 可能丢数据 | 最好 |
| **半同步复制** | 主库等待**至少一个从库**确认收到 Binlog 后才返回 | 较安全 | 较好 |
| **全同步复制** | 主库等待**所有从库**都执行完才返回 | 最安全 | 最差 |
| **GTID 复制** | 基于全局事务 ID，自动定位复制位点 | 同上 | 运维更方便 |

MySQL 5.7+ 默认使用**半同步复制**（需要安装插件）。

**读写分离的实现方案：**

**1. 应用层实现**
- 代码中手动指定数据源：写操作用主库，读操作用从库
- 使用 Spring 的 `AbstractRoutingDataSource` 动态切换数据源
- 优点：灵活，无额外组件
- 缺点：侵入业务代码

**2. 中间件实现**
- ShardingSphere-JDBC / ShardingSphere-Proxy
- MyCat、ProxySQL、MySQL Router
- 优点：对应用透明
- 缺点：增加运维复杂度

**主从延迟的原因：**
1. 从库的 SQL 线程是**单线程**重放（MySQL 5.6 之前），主库并发写入快，从库重放慢
2. 从库机器性能差于主库
3. 主库有大事务（如大批量 UPDATE），从库重放耗时长
4. 网络延迟

**主从延迟的处理方案：**

| 方案 | 说明 |
|------|------|
| **强制走主库** | 对实时性要求高的读操作（如刚下单后查订单），直接读主库 |
| **延迟判断** | 查询前检查 `SHOW SLAVE STATUS` 中的 `Seconds_Behind_Master`，延迟大则走主库 |
| **等待 GTID** | MySQL 5.7+ 的 `WAIT_FOR_EXECUTED_GTID_SET()`，等待从库追上指定的 GTID |
| **半同步复制** | 保证至少一个从库收到了 Binlog，减少延迟窗口 |
| **并行复制** | MySQL 5.7+ 支持多线程并行重放 Relay Log（基于组提交的并行复制） |
| **缓存兜底** | 写操作后将数据写入 Redis，读操作优先读缓存 |

**MySQL 5.7+ 并行复制：**
```sql
-- 从库配置
slave_parallel_type = LOGICAL_CLOCK  -- 基于组提交的并行
slave_parallel_workers = 8           -- 并行线程数
```

> **面试加分点：** MySQL 8.0.26+ 引入了 `WRITESET` 并行复制，基于行级别的冲突检测，并行度更高，主从延迟更小。实际生产中，"强制走主库"是最简单有效的方案，配合缓存可以覆盖大部分场景。
