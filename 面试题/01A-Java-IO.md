# 01A - Java IO

---

## 一、IO 基础与资源管理

1. 字节流、字符流、缓冲流分别解决什么问题？在文件、网络、日志、接口响应中如何选择？
   - 参考答案：字节流面向原始 byte，适合图片、视频、压缩包、网络报文等二进制数据；字符流在字节流之上处理字符集解码，适合纯文本；缓冲流通过内存缓冲减少系统调用次数，适合频繁小块读写。
   - 选型时先看数据类型：二进制用 `InputStream`/`OutputStream`，文本用 `Reader`/`Writer` 并明确字符集。文件大多加 `BufferedInputStream`、`BufferedReader` 或 NIO；网络 IO 通常交给 Netty、HTTP 客户端或 Servlet 容器；日志用成熟日志框架的异步和缓冲能力；接口响应要结合内容类型、大小、是否流式下载来选择字节输出或 JSON 序列化。
2. 字符编码为什么经常导致乱码？服务间传输、文件导入、数据库读写中如何统一编码策略？
   - 参考答案：乱码的根因是“写入时用一种编码，读取时按另一种编码解释”。比如 UTF-8 字节被当成 GBK 解析，或接口没有声明 `Content-Type;charset`，都会导致字符映射错误。
   - 工程上应统一默认使用 UTF-8：HTTP 明确请求和响应字符集，文件导入要求模板编码或自动探测后校验，数据库连接、表、字段统一 `utf8mb4`，日志和消息队列也保持一致。对外部历史文件或第三方接口，要在边界层转换成系统内部统一编码，再进入业务逻辑。
3. `try-with-resources`、`AutoCloseable`、`flush()`、`close()` 分别在资源释放中承担什么责任？
   - 参考答案：`AutoCloseable` 定义了资源可关闭的契约，`try-with-resources` 负责在代码块结束时自动调用 `close()`，并能保留关闭阶段的 suppressed exception。`flush()` 是把用户态或框架缓冲区里的数据尽快刷到底层目标，不代表释放资源；`close()` 是释放文件句柄、Socket、缓冲区等资源，通常会先隐式 `flush()`。
   - 面试中可以强调：写入流在关键边界可以主动 `flush()`，但资源生命周期要靠 `close()` 或 `try-with-resources` 兜底，避免文件句柄泄漏和数据没落盘。
4. Java IO 中的装饰器模式带来了什么灵活性？过度包装时又会带来哪些可读性问题？
   - 参考答案：Java IO 通过装饰器模式把能力叠加起来，例如 `FileInputStream` 提供文件字节读取，`BufferedInputStream` 增加缓冲，`InputStreamReader` 增加字符集解码，`BufferedReader` 增加按行读取。这样不用为每种组合都创建一个新类，扩展很灵活。
   - 问题是包装层太多时，代码会变得难读，资源关闭顺序和异常来源也不直观。实践中应把常见组合封装成工具方法或使用 `Files.newBufferedReader()` 等高层 API，并在变量命名里体现最终语义。
5. 大文件读取为什么不能简单 `readAllBytes()`？如何根据文件大小、处理方式和内存限制选择方案？
   - 参考答案：`readAllBytes()` 会一次性把文件内容放进堆内存，大文件容易触发 OOM，也会造成长时间 GC 和请求线程阻塞。它只适合小文件、配置文件、测试数据这类可控场景。
   - 大文件应按处理方式选方案：文本逐行用 `BufferedReader` 或 `Files.lines()` 并及时关闭；二进制传输用固定大小 byte buffer 分块；文件复制或下载优先考虑 `FileChannel.transferTo()`；需要随机访问或超大文件索引时再评估 `MappedByteBuffer`。核心原则是限制单次内存占用，并把处理过程设计成流式。
6. 临时文件、文件权限、路径拼接在安全上有哪些风险？如何防止路径穿越和敏感文件泄露？
   - 参考答案：主要风险包括用户传入 `../` 造成路径穿越、临时文件权限过宽、文件名可预测导致覆盖或窃取、下载接口暴露任意路径、上传文件伪装后缀或覆盖敏感文件。
   - 防护上要把用户输入当成不可信数据：路径用 `Path.resolve()` 后 `normalize()`，再校验结果必须位于允许的根目录下；临时文件用 `Files.createTempFile()` 生成随机名并设置合理权限；上传文件重新生成服务端文件名，校验大小、类型和扩展名；下载接口只通过业务 ID 映射文件，不直接接受绝对路径。

### 基础补充题

1. `InputStream` 和 `OutputStream` 的基本读写流程是什么？
   - 参考答案：读流程通常是创建输入流、准备 byte 数组缓冲区、循环调用 `read(buffer)`，根据返回的有效字节数处理数据，最后关闭流。写流程是创建输出流、调用 `write()` 写入 byte 或 byte 数组，必要时 `flush()`，最后 `close()`。
2. `Reader`/`Writer` 和字节流相比最大的区别是什么？
   - 参考答案：`Reader`/`Writer` 面向字符，会涉及字符集编码和解码；字节流面向原始 byte，不理解字符含义。处理文本时用字符流更自然，处理图片、压缩包、协议报文等二进制数据必须用字节流。
3. 为什么关闭流时通常只需要关闭最外层包装流？
   - 参考答案：包装流的 `close()` 通常会向内层委托关闭，被包装的底层流也会一起释放。关闭最外层可以保证缓冲层先刷出数据，再关闭底层资源；如果手动逐层关闭，代码冗余且容易顺序错误。
4. `flush()` 和 `close()` 的区别是什么？
   - 参考答案：`flush()` 只把缓冲区数据刷到底层目标，流仍然可以继续使用；`close()` 会释放资源，之后通常不能再读写。写网络响应、日志或文件时，`flush()` 关注数据及时性，`close()` 关注资源生命周期。

## 二、文件 API 与大文件处理

1. `Path`、`Files`、`File` 的定位是什么？新代码中为什么更推荐 `java.nio.file` API？
   - 参考答案：`File` 是早期 API，既表示路径又提供少量文件操作，很多方法失败时只返回 `false`，错误信息不足。`Path` 专注表示路径，`Files` 提供创建、复制、移动、权限、属性、流式读写等静态工具方法。
   - 新代码更推荐 `java.nio.file`，因为它异常更明确、路径处理更现代、支持符号链接和文件属性，也更容易和 `FileChannel`、`WatchService` 等 NIO 能力配合。只有遇到老库接口要求 `File` 时，再通过 `path.toFile()` 转换。
2. 大文件逐行处理、分块上传、断点续传、批量导入分别适合哪些读写模式？
   - 参考答案：逐行处理适合日志、CSV、文本报表，用 `BufferedReader` 按行消费，避免全量加载。分块上传适合大视频、大固件包等文件，服务端按 chunk 保存并校验分片。断点续传需要记录文件偏移、分片编号、校验值和上传状态，支持从失败位置继续。
   - 批量导入要把“读取、校验、转换、入库”拆成批次，常见做法是流式解析文件，每批几百到几千行提交一次，失败行记录下来，避免一个坏数据拖垮整批任务。
3. `MappedByteBuffer` 适合什么场景？它在内存释放、文件锁定、地址空间上有哪些坑？
   - 参考答案：`MappedByteBuffer` 把文件映射到进程地址空间，适合大文件随机访问、索引文件、日志存储、需要减少用户态拷贝的高性能读写场景。它不适合普通小文件处理，因为复杂度和排查成本更高。
   - 主要坑是释放不受普通对象生命周期直接控制，依赖 GC 触发清理；Windows 上映射未释放时可能导致文件无法删除或移动；32 位或地址空间紧张时容易映射失败；过度映射还可能造成缺页、抖动和直接内存压力。生产使用要控制映射窗口大小，并做好异常和资源监控。
4. 文件锁 `FileLock` 能解决哪些跨进程并发问题？它有哪些平台相关限制？
   - 参考答案：`FileLock` 可以在多个进程同时读写同一文件时做互斥或协调，例如单机任务防重、日志切割、共享文件写入、导入任务锁定。它分为共享锁和排他锁，但实际支持程度依赖操作系统和文件系统。
   - 限制是它通常是 advisory lock，需要各进程都遵守；网络文件系统上语义可能不稳定；锁粒度、是否允许重入、共享锁支持在不同平台上有差异。关键业务不能只依赖文件锁，必要时用数据库唯一约束、分布式锁或任务状态表兜底。
5. `WatchService` 文件监听适合哪些场景？为什么不能把它当成强一致事件系统？
   - 参考答案：`WatchService` 适合监听配置目录、导入目录、临时文件生成、开发工具热加载等本地文件变化场景，可以感知创建、修改、删除事件。
   - 它不能当强一致事件系统，因为事件可能合并、丢失、重复或乱序，不同平台实现也有差异；文件写入过程中收到事件时，文件可能还没写完。可靠做法是监听后再扫描目录、校验文件完整性，并用状态记录实现幂等处理。
6. 线上文件处理任务如何做限速、重试、幂等和失败恢复？
   - 参考答案：限速可以控制并发数、队列长度、单任务读写速率和下游调用 QPS，避免打满磁盘、网络或数据库。重试要区分临时失败和永久失败，设置最大次数、退避策略和死信或人工处理入口。
   - 幂等通常靠任务 ID、文件 hash、分片编号、导入批次号、数据库唯一约束和状态机保证。失败恢复要记录处理进度，比如已完成分片、已导入行号、批次游标，重启后从安全检查点继续，而不是从头盲目重跑。

### 基础补充题

1. `File` 和 `Path` 表示文件路径时有什么基本区别？
   - 参考答案：`File` 是旧 API，表示文件或目录路径，也带一些文件操作方法；`Path` 是 NIO 的路径抽象，只负责路径表达和解析，配合 `Files` 完成具体操作。新代码通常优先使用 `Path`。
2. `Files.exists()`、`Files.isDirectory()`、`Files.size()` 常用于哪些判断？
   - 参考答案：`Files.exists()` 判断路径是否存在，`Files.isDirectory()` 判断是否目录，`Files.size()` 获取文件大小。常用于上传前校验、下载前校验、目录创建判断、批处理任务过滤文件等场景。
3. 逐行读取文本文件常见写法有哪些？为什么要指定字符集？
   - 参考答案：常见写法有 `Files.newBufferedReader(path, charset)`、`Files.lines(path, charset)`、`BufferedReader.readLine()`。指定字符集是为了让字节到字符的解码规则明确，避免不同机器默认编码不同导致乱码。
4. 复制、移动、删除文件时常见的异常情况有哪些？
   - 参考答案：常见异常包括文件不存在、目标已存在、权限不足、目录非空、文件被占用、磁盘空间不足、跨文件系统移动失败、符号链接处理不符合预期。生产代码要捕获具体异常并给出可定位的日志。

## 三、网络 IO 模型

1. BIO、NIO、AIO 的核心差异是什么？实际项目中为什么更多使用 Netty、Servlet 容器或成熟客户端库？
   - 参考答案：BIO 是阻塞 IO，通常一个连接或请求占用一个线程，模型简单但高并发连接成本高。NIO 是非阻塞 IO，通过 `Selector` 让少量线程管理大量连接，适合高并发网络服务。AIO 是异步 IO，操作系统完成后回调通知，编程模型更复杂，Java 生态里使用相对少。
   - 实际项目更多用 Netty、Servlet 容器、HTTP 客户端库，是因为它们已经封装了线程模型、编解码、连接管理、超时、SSL、背压和异常处理。直接写底层 IO 容易遗漏边界条件，除非是在做框架或特殊协议。
2. Socket 读写中的粘包、半包、超时、连接关闭分别如何处理？
   - 参考答案：TCP 是字节流协议，没有天然消息边界，所以会出现粘包和半包。常见解决方案是固定长度、分隔符、长度字段、或使用 HTTP/WebSocket 等已有协议；解码器要能缓存未完整的数据，等完整帧到达再处理。
   - 超时要设置连接超时、读超时、写超时和业务处理超时，避免线程或连接无限等待。连接关闭要区分正常 EOF、异常断开、半关闭和心跳超时，及时释放资源，并让上层业务具备重试或失败感知能力。
3. Selector、Channel、Buffer 的协作模型是什么？为什么直接写 NIO 代码容易出错？
   - 参考答案：`Channel` 代表可读写的数据通道，`Buffer` 是读写数据的中间容器，`Selector` 负责监听多个 Channel 的连接、读、写等就绪事件。线程从 Selector 拿到就绪事件后，对对应 Channel 执行非阻塞读写，并管理 Buffer 状态。
   - 直接写 NIO 容易错在 Buffer 的 `flip()`/`clear()`/`compact()` 状态切换、半包粘包、写半截、空轮询、连接关闭、异常释放和线程模型上。生产中通常用 Netty，让框架处理这些细节。
4. 堆内 Buffer 和直接内存 Buffer 如何取舍？直接内存泄漏或占用过高怎么排查？
   - 参考答案：堆内 Buffer 分配和回收成本低，受 GC 管理，适合普通业务处理和小数据。直接内存 Buffer 位于堆外，和底层 IO 交互时可减少一次拷贝，适合高吞吐网络和文件传输，但分配成本高、释放不如堆内直观。
   - 排查直接内存问题要看 `-XX:MaxDirectMemorySize`、Netty allocator 指标、NMT、JFR、堆外内存曲线、容器内存限制和是否存在 ByteBuf 未释放。使用 Netty 时尤其要关注引用计数对象是否 `release()`，打开 leak detector 可以帮助定位。
5. Reactor 模式如何支撑高并发连接？单 Reactor、多 Reactor、主从 Reactor 的区别是什么？
   - 参考答案：Reactor 模式用事件循环监听 IO 就绪事件，再分发给处理器处理，避免为每个连接创建独立线程。它适合大量连接、单次处理较短、主要瓶颈在 IO 的场景。
   - 单 Reactor 是一个线程负责 accept、读写和分发，简单但容易成为瓶颈。多 Reactor 通常多个事件循环分别处理不同连接，提高并发能力。主从 Reactor 中主 Reactor 负责接收连接，从 Reactor 负责已建立连接的读写，Netty 的 boss/worker 线程组就是典型体现。
6. Linux `select`、`poll`、`epoll` 的差异会如何影响 Java 网络框架的设计？
   - 参考答案：`select` 和 `poll` 都需要扫描文件描述符集合，连接数很大时效率下降；`select` 还有描述符数量限制。`epoll` 通过内核事件通知机制减少无效扫描，更适合海量连接场景。
   - Java NIO 的 Selector 会映射到底层平台能力，Netty 在 Linux 上还提供 native epoll transport，以获得更好的性能和更多网络参数控制。框架设计上会围绕事件就绪、边缘触发/水平触发、连接分配和事件循环线程模型做优化。

### 基础补充题

1. 什么是 Socket？客户端和服务端建立连接的基本流程是什么？
   - 参考答案：Socket 是应用程序使用网络连接的编程抽象，可以理解为 IP、端口和协议之上的通信端点。服务端监听端口，客户端发起连接，TCP 三次握手成功后双方通过输入输出流或 Channel 读写数据，结束时关闭连接。
2. 阻塞式 Socket 读写为什么需要设置超时时间？
   - 参考答案：阻塞读写在对端不发送数据、网络异常或连接半开时可能一直等待，占住线程和连接资源。设置超时可以让调用及时失败，释放资源，并触发重试、降级或告警。
3. NIO 的 `Buffer` 中 `capacity`、`position`、`limit` 含义是什么？
   - 参考答案：`capacity` 是缓冲区总容量，创建后通常不变；`position` 是当前读写位置；`limit` 是当前可读或可写的边界。写入后调用 `flip()` 会把 limit 设为当前 position，再把 position 归零，方便读取刚写入的数据。
4. 为什么网络传输中需要处理拆包和粘包？
   - 参考答案：因为 TCP 只保证字节流有序可靠，不保证一次 `write()` 对应一次 `read()`。发送端的多个消息可能被合并，单个消息也可能被拆成多次接收，所以应用层必须定义消息边界并做编解码。

## 四、零拷贝与数据传输

1. 传统文件传输经过哪些用户态和内核态拷贝？零拷贝减少了什么成本？
   - 参考答案：传统文件传输通常是磁盘读到内核页缓存，再从内核拷贝到用户态缓冲区，应用调用 `write()` 后又从用户态拷贝回内核 Socket 缓冲区，最后由网卡发送。这中间既有数据拷贝，也有用户态和内核态切换。
   - 零拷贝的目标不是绝对没有拷贝，而是减少数据在用户态和内核态之间来回搬运，降低 CPU 消耗、内存带宽占用和上下文切换成本。对大文件传输、静态资源下载、日志转发这类场景收益更明显。
2. `FileChannel.transferTo/transferFrom`、`sendfile`、`mmap` 分别适合哪些传输场景？
   - 参考答案：`transferTo()`/`transferFrom()` 是 Java 对高效通道传输的封装，常用于文件到 Socket、文件到文件的大块传输，底层可能利用 `sendfile`。`sendfile` 适合把文件内容直接发送到网络连接，比如静态文件下载。
   - `mmap` 适合把文件映射到内存后做随机访问或高频读取，例如索引文件、存储引擎日志、只读数据文件。它更像是内存映射访问，不等同于所有传输场景都该用它。
3. Kafka、Netty、静态文件下载为什么会关注零拷贝？它对业务接口一定有收益吗？
   - 参考答案：Kafka 需要把磁盘日志高吞吐发送给消费者，Netty 经常承载高并发网络传输，静态文件下载则是典型的文件到网络连接搬运，所以它们都关注减少拷贝和 CPU 开销。
   - 但业务接口不一定有收益。如果响应需要 JSON 序列化、加密、压缩、字段过滤、审计改写，就必须进入应用层处理，零拷贝空间有限。对于小响应或瓶颈在数据库、下游服务的接口，优化协议和查询通常比追求零拷贝更有效。
4. 零拷贝和 TLS 加密、压缩、限速、审计日志之间有什么冲突或限制？
   - 参考答案：TLS 加密、压缩、内容改写和审计日志通常需要读取或转换数据内容，这会把数据重新拉回应用层或加密层处理，削弱 `sendfile` 这类零拷贝能力。限速也可能要求分段发送和调度，不能简单一次性把文件交给内核。
   - 所以设计下载链路时要区分“原样传输”和“需要加工”。原样静态文件最适合零拷贝；需要加密、压缩、水印、鉴权审计的场景，要优先保证正确性和可观测性，再评估局部优化。
5. 做大文件下载接口时，如何设计 Range、缓存头、限流、断连处理和观测指标？
   - 参考答案：大文件下载要支持 HTTP Range，返回 `206 Partial Content`、`Content-Range` 和正确的 `Content-Length`，方便断点续传。缓存头可根据文件是否可变设置 `ETag`、`Last-Modified`、`Cache-Control`，减少重复下载。
   - 限流要控制单用户、单 IP、全局带宽和并发连接数；断连时要及时停止读取并释放文件句柄；观测上关注下载耗时、吞吐、失败率、客户端取消、Range 命中、磁盘 IO、网络带宽和热点文件分布。

### 基础补充题

1. 什么是用户态和内核态？普通文件传输为什么会发生多次拷贝？
   - 参考答案：用户态是应用程序运行的空间，内核态是操作系统内核运行并管理硬件资源的空间。普通文件传输中，数据会从磁盘到内核缓冲区，再到应用缓冲区，再回到内核 Socket 缓冲区，所以会发生多次拷贝和状态切换。
2. `transferTo()` / `transferFrom()` 常用于什么场景？
   - 参考答案：它们常用于 Channel 之间的大块数据传输，比如文件复制、文件下载、日志文件发送。相比应用层循环读写，可能利用操作系统能力减少拷贝并提升吞吐。
3. 零拷贝是不是完全没有数据拷贝？为什么？
   - 参考答案：不是。零拷贝通常指减少或避免数据在用户态和内核态之间的拷贝，底层从磁盘到内存、从内存到网卡等过程仍然存在数据搬运，只是路径更短、CPU 参与更少。
4. 静态文件下载、日志传输等场景为什么适合关注零拷贝？
   - 参考答案：这些场景的数据通常不需要业务层逐字节修改，只是从磁盘原样传到网络或另一个文件，瓶颈容易落在 IO 和 CPU 拷贝上。因此减少拷贝能直接提升吞吐并降低机器资源消耗。

## 五、序列化与数据格式

1. Java 原生序列化为什么在现代服务中应谨慎使用？性能、兼容性和安全风险分别是什么？
   - 参考答案：Java 原生序列化使用方便，但现代服务中要谨慎。性能上，它生成的数据体积较大、反射和对象图处理成本高；兼容性上，类名、字段、`serialVersionUID`、继承结构变化都可能导致反序列化失败；安全上，反序列化不可信数据可能触发 gadget chain，造成远程代码执行。
   - 工程实践中，内部缓存或短期本地存储也要限制使用范围；跨服务通信更推荐 JSON、Protobuf、Avro 等明确协议，并对反序列化类型做白名单和依赖治理。
2. JSON、Protobuf、Avro、Kryo、Hessian 在可读性、跨语言、Schema 演进、性能上如何取舍？
   - 参考答案：JSON 可读性最好、调试方便、跨语言强，但体积和解析性能一般，适合 HTTP API、配置和日志。Protobuf 体积小、性能好、跨语言强，依赖 `.proto` Schema，适合 RPC 和高吞吐消息。Avro 对 Schema 演进支持好，常用于数据平台和消息流。
   - Kryo 性能高但跨语言和长期兼容性弱，更适合 Java 内部临时序列化。Hessian 使用简单、跨语言有限，在一些老 RPC 体系里常见。选型时不要只看性能，还要看是否需要可读、跨语言、长期存储、字段演进和生态支持。
3. 反序列化漏洞的本质是什么？如何通过白名单、禁用危险类型、依赖治理降低风险？
   - 参考答案：反序列化漏洞的本质是把不可信输入恢复成对象时，触发了类加载、构造、`readObject()`、setter、代理或第三方库 gadget 的副作用，攻击者借此执行非预期逻辑甚至命令。
   - 防护要从边界开始：不反序列化不可信 Java 原生对象；必须使用时限制类型白名单，禁用危险多态类型和默认类型推断；升级存在 gadget 风险的依赖；对外接口使用结构化协议；在网关、WAF、依赖扫描和安全测试中加入反序列化风险检查。
4. 接口字段新增、删除、重命名时，如何保证序列化格式的前后兼容？
   - 参考答案：字段新增通常要给默认值，并让老客户端忽略未知字段；字段删除要先停止使用，再灰度下线，避免仍有消费者依赖；字段重命名不要直接改名，最好新增新字段、双写双读一段时间，再废弃旧字段。
   - 对 Protobuf 这类协议，要保留旧字段编号，不复用 tag；对 JSON，要保持字段含义稳定，避免同名字段语义变化。接口版本、兼容性测试、灰度发布和消费者治理比单纯改 DTO 更重要。
5. RPC、MQ、缓存、数据库 JSON 字段分别对序列化方案有什么不同要求？
   - 参考答案：RPC 更关注低延迟、跨语言、Schema 明确和兼容演进，常用 Protobuf、JSON 或框架默认协议。MQ 更关注消息长期兼容、消费者异步升级、可追踪和可重放，Schema 演进规则要更严格。
   - 缓存序列化关注体积、速度、反序列化兼容和缓存升级策略，不能让类结构小改动导致大面积缓存失效。数据库 JSON 字段关注可读性、可查询性、长期存储兼容和数据迁移，字段含义必须稳定，避免把它当成随意变化的对象快照。

### 基础补充题

1. 什么是序列化和反序列化？它们分别发生在什么场景？
   - 参考答案：序列化是把对象转换成字节、字符串或其他可传输可存储格式；反序列化是把这些数据恢复成对象。它们常发生在 RPC、HTTP JSON、消息队列、缓存、文件存储和数据库 JSON 字段中。
2. JSON 相比 Java 原生序列化有什么优点？
   - 参考答案：JSON 可读性好、跨语言友好、便于调试和排查，对类名和 Java 运行时依赖更少。它更适合作为接口和消息格式，但性能和体积不一定优于二进制协议。
3. `serialVersionUID` 的作用是什么？
   - 参考答案：`serialVersionUID` 用来标识可序列化类的版本。反序列化时，如果数据中的版本号和当前类不一致，通常会抛出 `InvalidClassException`，用于避免不兼容的类结构被错误恢复。
4. 字段新增或删除时，为什么要考虑序列化兼容性？
   - 参考答案：因为生产环境里常有新旧版本服务、历史消息、旧缓存和存量数据同时存在。字段变化如果不兼容，可能导致老消费者解析失败、新服务读不了旧数据，或字段默认值引发业务错误。

## 六、流式处理与背压

1. 为什么高吞吐文件或网络处理需要考虑背压？生产者过快会导致哪些资源问题？
   - 参考答案：背压解决的是生产速度大于消费速度时，系统如何让上游降速或排队受控。高吞吐文件和网络处理如果没有背压，数据会不断堆积在线程池队列、内存缓冲、Socket 缓冲、磁盘临时文件或数据库连接上。
   - 生产者过快会导致 OOM、GC 飙升、队列延迟扩大、磁盘打满、连接耗尽、下游雪崩。工程上要通过有界队列、限流、批量大小、并发控制、超时取消和 Reactive Streams 请求量控制来传递压力。
2. `InputStream`/`OutputStream`、Iterator、Stream API、Reactive Streams 在流式处理上的差异是什么？
   - 参考答案：`InputStream`/`OutputStream` 是字节层面的顺序读写，适合文件、网络和二进制传输。`Iterator` 是对象层面的拉取式遍历，适合逐条消费集合或游标数据。Java Stream API 偏集合和函数式处理，支持惰性中间操作，但不等同于异步背压模型。
   - Reactive Streams 是带背压协议的异步流，消费者通过 request 数量告诉生产者能处理多少。它适合高并发 IO 和异步链路，但调试、线程切换和上下文传播更复杂，普通阻塞流程不一定需要上它。
3. 上传文件边读边处理时，如何避免把完整文件放进内存？
   - 参考答案：上传接口要使用流式读取或框架的 multipart 临时文件机制，不要把文件转成完整 byte 数组。处理时固定缓冲区分块读取，边校验、边计算 hash、边写对象存储或临时文件。
   - 如果是导入类文件，应使用流式解析器，比如 SAX、逐行 CSV、EasyExcel 流式读取等，每批数据校验和入库。还要限制文件大小、单用户并发、临时目录容量，并在异常时清理临时文件。
4. 下载或转发大响应时，如何处理客户端慢连接、下游超时和取消？
   - 参考答案：客户端慢连接会让服务端写缓冲堆积，要设置写超时、连接空闲超时、发送缓冲限制和全局并发限制。转发大响应时不要先完整读入内存，而是边读下游边写上游，并在两端之间传递取消信号。
   - 下游超时要区分连接超时、读取超时和整体请求超时；客户端取消时应停止下游读取、关闭流并记录取消原因。指标上要观察慢下载数量、写超时、客户端断开、下游耗时和传输中断点。
5. WebFlux、Netty、Servlet 异步 IO 的适用边界是什么？什么时候传统阻塞模型反而更简单可靠？
   - 参考答案：WebFlux 和 Netty 适合高并发连接、长连接、流式响应、网关转发和大量等待 IO 的场景，前提是链路中的数据库、Redis、HTTP 客户端等也能配合异步或至少不会阻塞事件循环。Servlet 异步 IO 适合在传统 Servlet 体系中释放请求线程，处理长请求或异步回调。
   - 如果业务是典型 CRUD、团队主要熟悉 Spring MVC、并发量可通过线程池和连接池支撑，传统阻塞模型往往更简单可靠。盲目引入响应式会增加调试、事务、上下文、错误处理和阻塞调用混用的成本。

### 基础补充题

1. 什么是流式处理？它和一次性加载全部数据有什么区别？
   - 参考答案：流式处理是数据一边产生一边分批消费，不要求全部数据先加载完成。它和一次性加载相比，内存占用更稳定，也更适合大文件、长连接和实时数据处理。
2. 上传文件时为什么通常要分块读取？
   - 参考答案：分块读取可以控制内存占用，避免大文件一次性进入堆内存。它还方便边读边校验、边写存储、显示进度、失败重试和限速。
3. 生产者速度大于消费者速度时会出现什么问题？
   - 参考答案：数据会在队列、缓冲区、内存、磁盘或连接中堆积，导致延迟升高、内存膨胀、GC 压力、超时、丢弃甚至系统崩溃。严重时还会把压力传递给下游，形成级联故障。
4. 背压的核心目的是什么？
   - 参考答案：背压的核心目的是让消费者的处理能力能够反向约束生产者，避免无限制堆积。它不是单纯提高吞吐，而是在吞吐、延迟和资源占用之间保持系统稳定。

## 七、高阶 API 与工程库选型

1. 实际项目中什么时候优先使用 `Files`、Spring `Resource`、Commons IO 这类高阶 API，而不是直接操作 `InputStream`/`OutputStream`？
   - 参考答案：当需求是复制、移动、读取小文本、目录遍历、资源加载、临时文件创建这类标准操作时，应优先用高阶 API。比如 `Files.copy()`、`Files.newBufferedReader()`、`Files.walk()`、Spring `Resource`、Apache Commons IO 的 `FileUtils`/`IOUtils` 都能减少重复样板代码和资源关闭错误。
   - 但高阶 API 不是无脑替代。要看它是否会一次性读入内存、是否递归遍历过深、是否隐藏异常细节、是否适合大文件。面试时可以总结为：业务代码尽量用更安全的抽象，性能敏感或协议特殊的地方再下沉到底层流、Channel 或框架扩展点。
2. 文件上传在 Spring MVC、Servlet 或对象存储场景中通常用哪些封装？为什么不建议自己解析 multipart 请求？
   - 参考答案：普通 Web 项目里通常使用 Spring `MultipartFile`、Servlet `Part` 或框架提供的 multipart resolver；云存储场景更多使用 S3、MinIO、OSS、COS 等 SDK 的上传 API、分片上传和预签名 URL。它们已经处理了边界解析、临时文件、大小限制、流关闭和异常封装。
   - 自己解析 multipart 容易在边界字符串、文件名编码、内存阈值、临时文件清理、超大文件、恶意请求和并发限制上踩坑。更好的做法是配置框架的最大文件大小、临时目录和超时策略，业务层只处理校验、存储、元数据落库和失败补偿。
3. 文件下载接口中，什么时候使用 `ResponseEntity<Resource>`、`StreamingResponseBody`、`ResourceRegion` 或对象存储预签名 URL？
   - 参考答案：小文件或普通附件下载可以用 `ResponseEntity<Resource>`，让框架处理响应头和资源输出；大文件或边生成边下载适合 `StreamingResponseBody`，避免完整内容进内存；需要 Range 断点续传、视频拖动播放时可用 `ResourceRegion` 或容器的 Range 支持。
   - 如果文件已经在对象存储中，很多时候更推荐返回预签名 URL，让客户端直连对象存储下载，应用服务只做鉴权和授权生成，避免业务服务器承担大流量转发。只有需要审计、脱敏、水印、动态打包或内网隔离时，才让应用服务代理下载。
4. CSV、Excel、XML、JSON 大文件导入为什么更推荐成熟解析库？如何选择解析模式？
   - 参考答案：这些格式都有复杂边界：CSV 有引号、换行和转义；Excel 有单元格类型、公式、日期和样式；XML 有命名空间和实体安全；JSON 有嵌套结构和类型转换。成熟库能处理格式细节并提供更稳定的错误定位。
   - 选型上，CSV 可用 Apache Commons CSV 或 OpenCSV；Excel 小文件可用 Apache POI，超大文件更偏向 EasyExcel 或 POI SAX/SXSSF 模式；XML 大文件用 SAX/StAX，避免 DOM 全量加载；JSON 大数组导入用 Jackson Streaming API。核心是使用流式解析、批量校验和批量入库。
5. 目录复制、压缩解压、文件清理这类工具型操作，使用第三方库时要注意哪些安全和稳定性问题？
   - 参考答案：Commons IO、Guava、Hutool 等工具库可以减少样板代码，但目录递归和压缩解压不能只看调用是否简单。要注意符号链接循环、目录层级过深、文件数量过多、权限不足、磁盘空间不足、异常中断后的半成品清理。
   - 解压尤其要防 Zip Slip 路径穿越，不能相信压缩包内的文件名。解压目标路径要 `normalize()` 后校验仍在目标目录下，还要限制总大小、文件数量和单文件大小，防止压缩炸弹把磁盘打满。
6. 调用外部 HTTP 接口时，为什么通常使用 OkHttp、Apache HttpClient、Spring `WebClient` 或 OpenFeign，而不是原生 Socket？
   - 参考答案：HTTP 客户端库已经封装了连接池、DNS、TLS、代理、重定向、Cookie、压缩、超时、重试、拦截器和指标采集。业务代码直接使用原生 Socket 不仅要自己实现 HTTP 协议细节，还很容易遗漏连接复用、异常关闭和慢请求治理。
   - 选型时，阻塞式服务常用 Apache HttpClient、OkHttp 或 OpenFeign；响应式链路可用 `WebClient`。无论选哪个，都要显式配置连接超时、读写超时、连接池大小、最大空闲连接、重试条件和熔断限流策略。
7. 对接对象存储时，为什么优先使用官方 SDK、TransferManager 或预签名 URL，而不是自己实现分片上传和断点续传？
   - 参考答案：对象存储 SDK 通常已经封装了鉴权签名、分片上传、并发上传、失败重试、ETag 校验、区域 endpoint、临时凭证刷新和错误码映射。自己实现这些能力成本高，而且容易在签名过期、分片合并、重复提交、网络抖动和一致性边界上出问题。
   - 常见设计是：服务端负责生成上传任务、临时凭证或预签名 URL，客户端直传对象存储；上传完成后回调或主动通知服务端落库。后端需要校验文件大小、类型、hash、归属权限和生命周期策略，而不是把所有文件流量都压在应用服务器上。
8. 自定义 TCP 协议、长连接或高并发网关场景中，为什么优先使用 Netty/gRPC/RSocket 等框架，而不是直接写 Java NIO？
   - 参考答案：Netty 这类框架已经解决了事件循环、线程模型、ByteBuf 池化、拆包粘包、编解码、SSL、心跳、半包缓存、背压和异常释放等难点。gRPC 提供了 HTTP/2、多路复用、IDL、代码生成和流式调用，适合服务间强契约通信。
   - 直接写 NIO 适合学习或非常小的实验，但生产级长连接服务要考虑连接生命周期、协议升级、限流、指标、优雅关闭和内存泄漏排查。框架能把注意力从底层细节转到协议设计和业务稳定性上。
9. 日志、图片、视频、PDF 等文件处理为什么通常依赖专门库？自己按字节处理会有什么问题？
   - 参考答案：这些文件格式都有复杂结构和元数据。图片处理会涉及编码格式、EXIF、压缩质量、缩略图和内存占用；视频处理涉及容器格式、编码、关键帧和转码；PDF 涉及字体、页面、加密和文本抽取；日志采集还涉及滚动、压缩、偏移和多行合并。
   - 因此图片常用 Thumbnailator、ImageIO 或更专业的图像库，视频常接 FFmpeg，PDF 常用 PDFBox、iText 等，日志采集用 Filebeat、Fluent Bit 或日志框架能力。自己按字节处理不仅效率低，也很难保证格式正确性和异常文件的安全边界。
10. 使用高阶 IO 库时，如何评估它是否真的比低层 API 更合适？
   - 参考答案：可以从五个维度评估：是否覆盖格式和协议细节，是否支持流式处理和大文件，是否有成熟的资源释放和异常模型，是否便于配置超时、限流、重试和观测，是否有活跃维护和安全更新。
   - 高阶库的价值是降低出错概率和维护成本，但要警惕“方便方法”隐藏全量加载、默认无限重试、默认无超时、吞掉异常或引入过重依赖。最佳实践是业务层使用高阶抽象，关键路径压测验证，必要时读源码或文档确认资源和内存行为。
