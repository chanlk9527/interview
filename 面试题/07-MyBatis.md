# 07 - MyBatis

---

## 一、核心原理与边界

1. MyBatis 解决的是 SQL 映射问题还是 ORM 问题？它和 JPA/Hibernate 的核心取舍是什么？
2. Mapper 接口、XML/注解 SQL、`SqlSession`、Executor、StatementHandler、ResultSetHandler 如何串起一次查询？
3. `#{}` 和 `${}` 的差异是什么？除了 SQL 注入，还会影响哪些预编译和缓存行为？
4. 动态 SQL 如何保持可读性？复杂条件过多时，应该继续写 XML 还是拆到应用层/查询构建器？
5. resultMap 如何处理字段映射、类型转换、枚举、JSON 字段和嵌套对象？

## 二、查询设计与性能

1. 一对一、一对多映射如何避免笛卡尔积、重复对象和 N+1 查询？
2. 分页查询为什么不能只关心 `limit offset`？深分页、排序稳定性、总数统计有什么风险？
3. 批量插入、批量更新、批量查询如何设计才能兼顾性能、事务大小和数据库压力？
4. 大结果集导出如何做流式读取？`fetchSize`、游标、事务边界和连接占用要注意什么？
5. 慢 SQL 排查时，MyBatis 层能提供哪些线索？SQL 日志、参数、执行耗时、Mapper 方法如何关联？

## 三、缓存与一致性

1. 一级缓存的作用范围是什么？为什么同一个事务内可能读到缓存结果？
2. 二级缓存为什么生产环境要谨慎？跨 Mapper、事务提交、脏读风险和失效策略分别是什么？
3. MyBatis 与 Redis/Caffeine 等业务缓存如何配合？缓存 key、失效、击穿和一致性如何设计？
4. 查询结果对象被修改后，缓存和后续读取可能出现什么副作用？

## 四、插件与扩展

1. MyBatis 插件能拦截哪些核心组件？分页、审计字段、租户隔离、SQL 监控分别适合在哪层做？
2. PageHelper 或类似分页插件的原理是什么？在复杂 SQL、count 查询、排序字段上有哪些坑？
3. MyBatis-Plus 提升了哪些开发效率？Wrapper、自动填充、逻辑删除、租户插件会带来哪些隐式行为？
4. 类型处理器 TypeHandler 适合解决什么问题？枚举、加密字段、JSON 字段如何设计？
5. 插件链顺序会如何影响 SQL 改写结果？如何测试和观测插件行为？

## 五、工程实践

1. Mapper XML 应该如何组织，才能避免 SQL 复制粘贴和业务逻辑散落？
2. 表结构变更后，如何保证 Mapper、DTO、resultMap、测试数据同步演进？
3. 多数据源、读写分离、分库分表场景下，MyBatis 层应该负责什么，不应该负责什么？
4. 如何做 SQL 层面的单元测试或集成测试？Testcontainers、内存数据库、真实数据库各有什么取舍？
5. MyBatis 在高并发服务中最常见的坑是什么？连接池、事务、慢 SQL、N+1、批量操作分别如何预防？
