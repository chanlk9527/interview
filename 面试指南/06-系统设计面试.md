# 系统设计面试

## 一、系统设计面试的本质

### 1.1 面试官到底在考什么

系统设计题不是让你背一个标准答案，而是看你能不能在不完整信息下，做出合理的架构判断。

面试官重点看以下能力：

| 能力 | 面试中的表现 | 加分点 |
|------|--------------|--------|
| 需求分析 | 能先确认边界，而不是上来画图 | 主动区分核心需求和非核心需求 |
| 容量意识 | 能估算QPS、存储量、连接数 | 数字不一定精确，但量级要合理 |
| 架构拆解 | 能把系统拆成接入、业务、存储、异步、监控等模块 | 数据流清晰 |
| 技术取舍 | 能说明为什么选A而不是B | 能讲出优缺点和适用场景 |
| 工程落地 | 考虑幂等、限流、降级、监控、灰度 | 不是只画理想架构 |
| 业务理解 | 设计能服务业务目标 | 能结合成本、效率、稳定性做权衡 |

### 1.2 不同候选人的差距

| 层级 | 常见表现 |
|------|----------|
| 初级 | 罗列中间件：Redis、MQ、MySQL、ES都放上去，但说不清为什么 |
| 中级 | 能画出基本架构，能讲核心流程，但对容量和异常场景考虑不足 |
| 高级 | 先澄清需求，再根据规模做取舍，能深入关键模块和故障场景 |
| 架构型 | 能把业务约束、技术方案、团队成本、演进路线结合起来讲 |

对Java后端、IoT、视频云平台方向来说，系统设计通常会围绕：**设备接入、长连接、消息通道、流媒体服务、设备影子、OTA、高并发数据写入、多租户权限与设备安全**。

## 二、系统设计标准答题框架

### 2.1 六步法

推荐使用以下框架，适合大部分系统设计题：

```
Step 1：需求澄清
Step 2：容量估算
Step 3：整体架构
Step 4：核心模块设计
Step 5：高可用与异常处理
Step 6：总结取舍与演进方向
```

### 2.2 时间分配

如果系统设计题给你20-30分钟，可以这样分配：

| 阶段 | 建议时间 | 目标 |
|------|----------|------|
| 需求澄清 | 2-3分钟 | 明确边界，防止答偏 |
| 容量估算 | 2-4分钟 | 判断系统量级 |
| 整体架构 | 5分钟 | 画出核心组件和数据流 |
| 核心模块 | 8-12分钟 | 深入1-2个关键模块 |
| 高可用与优化 | 3-5分钟 | 展示工程经验 |
| 总结 | 1分钟 | 收束方案，说明取舍 |

### 2.3 口头开场模板

```
我先确认一下需求边界，再做一个简单容量估算，然后画整体架构。
之后我会重点展开核心链路，比如接入层、消息可靠性、存储设计和高可用。
如果时间允许，再补充监控、限流、降级和后续演进。
```

这个开场的好处是：你把节奏掌握在自己手里，也让面试官知道你不是在背答案。

## 三、Step 1：需求澄清

### 3.1 不要急着设计

系统设计题最常见的扣分点是：面试官刚说“设计一个消息推送系统”，候选人马上开始说“用Redis、Kafka、MySQL、Netty”。这样会显得缺少产品和工程意识。

先问清楚以下问题：

| 维度 | 要问的问题 | 为什么重要 |
|------|------------|------------|
| 核心功能 | 这个系统最核心的功能是什么？ | 决定MVP范围 |
| 用户规模 | 用户/设备量级多大？ | 决定是否需要分布式架构 |
| 读写比例 | 读多还是写多？ | 决定缓存和存储设计 |
| 实时性 | 延迟要求是毫秒、秒级还是分钟级？ | 决定同步/异步链路 |
| 一致性 | 强一致还是最终一致可以接受？ | 决定事务方案 |
| 可用性 | 可用性目标是99.9%还是99.99%？ | 决定容灾复杂度 |
| 安全性 | 是否涉及鉴权、权限、数据隔离？ | 决定安全边界 |
| 成本约束 | 是否允许引入复杂中间件？ | 决定落地可行性 |

### 3.2 IoT场景需求澄清清单

针对视频云平台、IoT设备接入类题目，可以重点问：

- 设备规模：十万级、百万级还是千万级设备？
- 连接方式：MQTT、HTTP、WebSocket、WebRTC分别承担什么角色？
- 设备是否长期在线？是否有弱网、断线重连、离线消息？
- 设备身份如何认证？证书、Token、设备三元组还是临时凭证？
- 消息类型有哪些？控制指令、告警、状态上报、OTA任务、影子同步？
- 消息是否要求顺序？是否允许重复？是否必须到达？
- 是否要支持多租户、设备分享、家庭成员权限？
- 是否有海外部署、跨区域访问、GDPR等合规要求？

### 3.3 澄清后的假设

如果面试官不想细化需求，你可以主动做假设：

> 我先做一个合理假设：系统服务百万级IPC设备，设备通过MQTT或WebSocket保持长连接，云端需要支持设备认证、状态上报、控制指令下发、离线消息和基础监控。消息要求秒级到达，允许至少一次投递，但业务层要做幂等。

这样既推进了面试，也把后续设计的前提讲清楚。

## 四、Step 2：容量估算

### 4.1 为什么要估算

容量估算不是数学考试，而是为了判断架构复杂度。10万设备和1000万设备的设计不一样，100 QPS和10万 QPS的技术选型也不一样。

估算时抓住五类数字：

```
用户/设备规模 → 请求量/消息量 → 峰值QPS → 存储量 → 带宽/连接数
```

### 4.2 常用估算公式

| 指标 | 公式 |
|------|------|
| 日请求量 | DAU × 人均日请求次数 |
| 平均QPS | 日请求量 / 86400 |
| 峰值QPS | 平均QPS × 3到10倍 |
| 日存储量 | 日数据条数 × 单条数据大小 |
| 年存储量 | 日存储量 × 365 × 副本数 |
| 长连接内存 | 在线连接数 × 单连接内存占用 |

### 4.3 示例：IoT消息推送系统估算

假设系统有500万台活跃设备，每台设备每天平均接收12条推送：

```
日消息量 = 500万 × 12 = 6000万
平均QPS = 6000万 / 86400 ≈ 700
峰值QPS = 700 × 3 ≈ 2100
```

如果每条消息平均1KB，保存一年，3副本：

```
日存储量 = 6000万 × 1KB ≈ 60GB
年存储量 = 60GB × 365 × 3 ≈ 65TB
```

这时可以得出设计结论：

- 需要消息队列削峰，不能全部同步处理
- 历史消息不能只放MySQL单表，需要按时间或设备分区
- 热消息可放Redis，冷数据落MySQL/ES/对象存储
- 需要监控消息堆积、消费延迟、到达率

### 4.4 示例：设备长连接估算

假设在线设备100万，每个连接占用约10KB内存：

```
连接内存 = 100万 × 10KB ≈ 10GB
```

考虑JVM堆外内存、业务对象、连接管理Map、线程栈、操作系统资源，单机不应该把资源打满。可以按单机承载5万到10万连接估算：

```
100万连接 / 10万 = 10台接入节点
```

再考虑冗余和故障转移，可以规划为15-20台接入节点。

## 五、Step 3：整体架构设计

### 5.1 架构图的基本层次

一个清晰的后端系统架构一般包含以下层次：

| 层次 | 常见组件 | 说明 |
|------|----------|------|
| 接入层 | Nginx、API Gateway、Netty、MQTT Broker、WebSocket Gateway | 处理连接、协议、鉴权、限流 |
| 业务层 | 用户服务、设备服务、消息服务、OTA服务、影子服务 | 承载核心业务逻辑 |
| 异步层 | Kafka、RabbitMQ、ActiveMQ、RocketMQ | 解耦、削峰、重试 |
| 缓存层 | Redis、Caffeine | 热数据、会话、分布式锁 |
| 存储层 | MySQL、MongoDB、ES、TDengine、对象存储 | 结构化、搜索、时序、文件数据 |
| 运维层 | Prometheus、Grafana、ELK、链路追踪 | 监控、日志、告警、排障 |

### 5.2 口头描述架构的顺序

按数据流讲，最容易让面试官跟上：

```
客户端/设备
  → 接入网关
  → 鉴权与协议解析
  → 业务服务
  → 消息队列/缓存
  → 存储与下游服务
  → 监控告警
```

示例：

> 设备首先通过MQTT或WebSocket连接到接入网关，网关负责TLS、设备身份认证、心跳和连接管理。认证通过后，上行消息进入消息队列，由设备服务、规则引擎或告警服务异步消费。下行控制指令先写入消息服务并生成消息ID，如果设备在线就通过连接路由直接下发，如果离线则写入离线消息表，等待设备重连后补发。

### 5.3 架构图中要主动标注的内容

- 数据流向：请求从哪里来，到哪里结束
- 同步链路和异步链路：哪些需要实时响应，哪些可以异步
- 关键存储：哪些数据在MySQL，哪些在Redis，哪些在ES/时序库
- 扩展点：哪个模块可以水平扩展
- 故障边界：某个下游挂了，主链路是否还能工作

## 六、Step 4：核心模块设计

系统设计题不要平均用力。画完整体架构后，选择1-2个最核心模块深入讲。

### 6.1 接入层设计

IoT/视频云平台的接入层通常是重点。

#### 关键问题

| 问题 | 设计思路 |
|------|----------|
| 如何支持多协议 | 抽象统一消息模型，MQTT/HTTP/WebSocket适配成内部事件 |
| 如何做设备认证 | 设备ID + 密钥/证书/Token，接入时校验签名和有效期 |
| 如何管理长连接 | 连接注册表：deviceId → gatewayId/channelId |
| 如何处理心跳 | IdleStateHandler或Broker心跳机制，超时关闭连接 |
| 如何处理重连 | 断线清理旧连接，新连接覆盖连接路由 |
| 如何水平扩展 | 接入节点无状态化，连接路由写Redis或注册中心 |

#### 回答模板

> 接入层我会分成协议适配和连接管理两部分。协议适配负责把MQTT、HTTP、WebSocket统一转换成内部消息模型；连接管理维护deviceId到具体接入节点和channel的映射。这样业务层不用关心底层协议，只处理统一事件。

### 6.2 消息可靠性设计

消息系统的追问非常高频，尤其是推送、控制指令、OTA任务。

#### 可靠性链路

```
生产者写消息
  → 生成全局唯一messageId
  → 持久化消息
  → 投递到MQ
  → 消费者ACK
  → 设备ACK
  → 更新消息状态
```

#### 常见机制

| 场景 | 方案 |
|------|------|
| 消息丢失 | 生产者确认、MQ持久化、消费ACK、失败重试 |
| 重复消费 | messageId幂等表、Redis去重、业务状态机 |
| 消息乱序 | 按deviceId分区、单设备顺序队列、消息序号 |
| 消息积压 | 扩容消费者、批量消费、降级非核心消息 |
| 设备离线 | 离线消息表、TTL过期、重连补发 |
| 下游故障 | 重试队列、死信队列、告警与人工补偿 |

### 6.3 存储设计

存储设计要围绕数据类型展开，而不是一句“用MySQL”。

| 数据类型 | 推荐存储 | 说明 |
|----------|----------|------|
| 用户、设备、权限 | MySQL | 结构化强，事务要求高 |
| 设备最新状态 | Redis + MySQL | Redis读写快，MySQL持久化 |
| 设备历史状态 | TDengine/时序库 | 高写入、按时间查询 |
| 消息历史 | MySQL分表/ES | MySQL做状态，ES做检索 |
| 日志和链路数据 | ES/日志平台 | 便于排查问题 |
| 视频截图/录像 | 对象存储 | 大文件不进数据库 |

#### MySQL表设计要点

- 所有多租户表都带`tenant_id`
- 设备相关表带`device_id`并建立联合索引
- 历史消息按时间或设备维度分表
- 状态流转表保留操作日志，方便排查
- 唯一约束用于保证幂等，例如`order_no`、`message_id`、`task_id`

### 6.4 缓存设计

缓存不是越多越好，要说明缓存什么、失效策略是什么、如何保证一致性。

| 缓存内容 | 失效策略 | 风险 |
|----------|----------|------|
| 设备在线状态 | 心跳刷新TTL | 状态可能短暂不准 |
| 用户权限 | 短TTL + 变更主动删除 | 权限变更延迟 |
| 热门配置 | 本地缓存 + Redis | 多级缓存一致性 |
| 消息去重ID | TTL过期 | TTL太短可能重复处理 |

常见一致性方案：

- 读多写少：更新数据库后删除缓存
- 配置类数据：版本号 + 本地缓存定时刷新
- 强一致要求高的数据：不走缓存或只缓存只读副本

### 6.5 安全与权限设计

IoT系统安全很容易被追问。

| 场景 | 设计要点 |
|------|----------|
| 设备接入认证 | 设备唯一标识、密钥签名、证书、Token有效期 |
| 用户访问设备 | 用户-家庭-设备关系校验，设备分享权限 |
| 多租户隔离 | `tenant_id`强制过滤，网关和服务层双重校验 |
| API安全 | OAuth2/JWT、签名、防重放、限流 |
| 敏感数据 | 加密存储、脱敏展示、审计日志 |
| 海外合规 | 数据删除、数据导出、区域化存储 |

## 七、Step 5：高可用与异常处理

### 7.1 高可用设计清单

| 维度 | 方案 |
|------|------|
| 接入层 | 多节点部署，负载均衡，连接重连 |
| 服务层 | 无状态服务，水平扩展，熔断降级 |
| MQ | 集群部署，持久化，死信队列 |
| MySQL | 主从复制，读写分离，备份恢复 |
| Redis | Sentinel或Cluster，避免单点 |
| 跨区域 | 就近接入，区域容灾，数据同步 |
| 发布 | 灰度发布，快速回滚，兼容旧协议 |

### 7.2 限流与降级

面试官常问“流量突然暴涨怎么办”。

可以分层回答：

| 层级 | 限流对象 | 策略 |
|------|----------|------|
| 网关层 | 用户/IP/租户/API | 令牌桶、漏桶、黑白名单 |
| 设备层 | deviceId/productKey | 限制单设备上报频率 |
| 服务层 | 核心接口 | Sentinel、线程池隔离 |
| MQ层 | 消费速度 | 削峰填谷、批量消费 |
| 存储层 | 写入压力 | 批量写、异步写、分库分表 |

降级要分清优先级：

- 控制指令、告警消息优先保证
- 统计、日志、非实时分析可以延迟
- 图片/视频类大数据可以异步处理

### 7.3 监控告警

系统设计最后补监控，是很好的加分项。

| 模块 | 关键指标 |
|------|----------|
| 接入网关 | 在线连接数、连接成功率、鉴权失败率、心跳超时 |
| 消息系统 | 生产QPS、消费QPS、消息积压、投递延迟、失败率 |
| 业务接口 | RT、QPS、错误率、P95/P99 |
| 存储 | 慢SQL、连接池、主从延迟、磁盘空间 |
| 设备侧 | 上线率、离线率、指令ACK率 |
| 推送链路 | 到达率、点击率、通道失败率 |

回答模板：

> 我会对每条核心链路埋点，至少监控QPS、错误率、延迟和积压。对于消息推送，还要监控从业务产生消息到设备ACK的端到端耗时，这样线上出问题时能定位是网关、MQ、业务服务还是第三方通道的问题。

## 八、Step 6：总结取舍与演进

### 8.1 总结模板

系统设计题收尾时，用30秒总结：

```
总结一下，这个方案的核心是：
1. 接入层通过多协议网关统一设备连接和认证；
2. 业务层通过消息队列解耦上下行链路；
3. 存储层按数据类型拆分，热数据走Redis，结构化数据走MySQL，检索走ES；
4. 可靠性上通过ACK、幂等、重试、死信队列和监控告警保证消息可追踪。

这个方案适合百万级设备规模。如果未来扩展到千万级，我会进一步做接入层分区、消息分区、跨区域部署和冷热数据分层。
```

### 8.2 主动讲取舍

高级候选人一定要讲trade-off：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 强一致 | 数据准确，业务简单 | 性能低，复杂度高 |
| 最终一致 | 性能好，扩展性强 | 需要补偿和对账 |
| 单体服务 | 简单，开发快 | 扩展性和团队协作差 |
| 微服务 | 边界清晰，易扩展 | 分布式复杂度高 |
| Redis缓存 | 性能高 | 一致性和穿透风险 |
| MQ异步 | 削峰解耦 | 增加延迟和排障复杂度 |

## 九、常见系统设计题思路

### 9.1 设计一个视频云平台设备接入系统

#### 需求假设

- 支持IPC设备通过MQTT/HTTP/WebSocket接入
- 支持设备认证、设备绑定、状态上报、控制指令下发
- 百万级设备，要求秒级控制和高可用

#### 核心架构

```
设备
  → 接入网关（MQTT/WebSocket/HTTP）
  → 设备认证服务
  → 设备管理服务
  → 消息队列
  → 状态服务/控制服务/告警服务
  → MySQL + Redis + TDengine/ES
```

#### 关键设计点

| 模块 | 设计重点 |
|------|----------|
| 协议适配 | 统一内部消息模型，屏蔽MQTT/HTTP/WebSocket差异 |
| 设备认证 | 设备三元组、证书或Token，防止伪造设备 |
| 连接管理 | deviceId到gateway节点映射，支持控制指令路由 |
| 状态上报 | 最新状态进Redis，历史状态进时序库 |
| 指令下发 | 在线直推，离线存储，ACK确认，超时重试 |
| 安全 | TLS、签名、防重放、设备权限校验 |

#### 高频追问

- 设备重连后旧连接怎么处理？
- 如何防止设备伪造身份？
- 单台网关挂了，连接如何迁移？
- 指令下发如何保证不丢、不重复？
- 设备在线状态为什么不能只依赖Redis？

### 9.2 设计一个IoT消息推送系统

#### 设计思路

| 模块 | 方案 |
|------|------|
| 消息入口 | 业务系统调用推送API，生成messageId |

| 离线推送 | 保存离线消息，设备上线后拉取 |
| 可靠性 | ACK、重试、幂等、死信队列 |
| 监控 | 到达率、延迟、失败原因分类 |

#### 可讲的项目经验

可以结合你的推送系统经验：

- 日均6000万+消息
- 峰值QPS 1000+

- 历史消息做持久化和检索
- 消息到达率、延迟、失败原因需要链路监控

### 9.3 设计一个多租户IoT SaaS平台

#### 核心考点

租户隔离、权限模型、设备管理、配置化、扩展性。

#### 设计思路

| 模块 | 方案 |
|------|------|
| 租户模型 | tenant、user、role、permission、device、product |
| 数据隔离 | 共享库表 + tenant_id，核心大客户可独立库 |
| 权限控制 | RBAC + 数据权限，设备级/家庭级/组织级授权 |
| 设备管理 | 设备绑定、分享、转移、解绑 |
| 配置能力 | 租户级功能开关、套餐限制、资源配额 |
| 安全防护 | 所有查询强制带tenant_id，审计敏感操作 |

#### 技术取舍

| 隔离方案 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| 共享库共享表 | 成本低，运维简单 | 隔离弱，需要严格权限控制 | 中小租户 |
| 独立Schema | 隔离较好 | 运维复杂度增加 | 中大型租户 |
| 独立数据库 | 隔离最强 | 成本高，运维复杂 | 大客户/强合规 |

### 9.5 设计一个设备影子系统

#### 核心考点

设备状态同步、期望状态与上报状态、离线控制、版本冲突。

#### 设计思路

| 概念 | 含义 |
|------|------|
| reported | 设备实际上报的状态 |
| desired | 云端期望设备达到的状态 |
| delta | reported和desired之间的差异 |
| version | 状态版本号，解决并发覆盖 |

#### 数据流

```
App设置期望状态
  → 更新desired和version
  → 如果设备在线，推送delta
  → 设备执行后上报reported
  → 云端清理delta
```

关键点：

- 设备离线时仍可设置desired，设备上线后同步
- 用版本号避免旧状态覆盖新状态
- 影子数据适合Redis缓存最新状态，MySQL持久化关键版本

## 十、系统设计高频追问与回答方向

| 追问 | 回答方向 |
|------|----------|
| 为什么要用MQ？ | 解耦、削峰、异步、失败重试；代价是延迟和排障复杂度 |
| Redis和MySQL如何保证一致性？ | 先更新DB再删缓存；强一致场景不使用缓存；加版本号和重试 |
| 如何保证幂等？ | 唯一业务ID、状态机、唯一索引、幂等表、Redis去重 |
| 如何处理消息积压？ | 扩消费者、批量消费、定位慢消费者、临时降级、死信队列 |
| 如何做分库分表？ | 按租户、设备、时间或订单号；避免热点；考虑查询路径 |
| 如何做限流？ | 网关限流、用户/租户/设备维度、令牌桶、服务熔断 |
| 如何做灰度发布？ | 按租户、设备型号、区域、用户比例放量；监控指标回滚 |
| 如何排查线上问题？ | 看监控、日志、链路追踪、MQ积压、慢SQL、错误率变化 |

## 十一、常见扣分点

| 扣分点 | 正确做法 |
|--------|----------|
| 上来就堆技术名词 | 先澄清需求，再讲架构 |
| 没有容量估算 | 至少估QPS、存储、连接数 |
| 架构图没有数据流 | 按请求/消息流向讲 |
| 只讲正常流程 | 主动补异常、重试、降级 |
| 没有安全意识 | IoT场景必须讲设备认证和权限 |
| 不讲取舍 | 说明为什么选这个方案，代价是什么 |
| 所有题都套同一套架构 | 根据题目规模和约束调整 |
| 细节过深导致讲不完 | 只深入核心模块，其他模块点到为止 |
